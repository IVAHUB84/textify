argument-hint: [путь-к-шаблону]
---

Пользователь хочет перевести **уже существующий** проект на этот многоагентный workflow.

**Аргумент:** `$ARGUMENTS` — путь к каталогу шаблона-источника (например, `../template`). Может быть пустым.

Это **одноразовая, неразрушающая** операция: добавить каркас workflow и аккуратно слить конфликтные файлы, не теряя того, что уже есть в проекте. Документы — на **русском**, код/идентификаторы — на английском.

## Железные принципы миграции

1. **Неразрушающе.** Существующие `CLAUDE.md`, `README.md`, `.gitignore`, файлы в `.github/workflows/` — **никогда не перезаписывать**. Только дополнять/сливать.
2. **Не трогать продукт.** Исходный код, тесты, конфиги сборки, история git — вне зоны миграции.
3. **Аддитивно по умолчанию.** Файл из шаблона копируется, только если в проекте его ещё нет.
4. **Не угадывать.** Если стек, команда тестов, «поставляемый сервис или библиотека» неоднозначны — **спроси пользователя**.
5. **Не коммить и не пушь.** Миграция оставляет изменения в рабочей копии; как их закоммитить — решает пользователь.

## Шаг 0. Источник шаблона

1. Определи путь к шаблону:
   - если `$ARGUMENTS` задан — используй его;
   - иначе попробуй `../template` (соседний каталог);
   - если там нет признаков шаблона (`.claude/agents/`, `CLAUDE.md`, `deploy-guide.md`) — **спроси у пользователя путь**.
2. Если путь шаблона совпадает с текущим проектом — остановись: мигрировать сам шаблон нельзя.

## Шаг 1. Предполётная проверка

1. Убедись, что текущий каталог — git-репозиторий. Если нет — предупреди: миграция добавит файлы; пользователю стоит сначала `git init` (или работать осознанно).
2. `git status`. Если дерево грязное — предупреди, что миграция добавит/сольёт файлы поверх, и порекомендуй сначала закоммитить/застэшить или сделать ветку. Спроси разрешение продолжать на грязном дереве.
3. **Проверка на затёртый `CLAUDE.md`.** Сравни `CLAUDE.md` проекта с `CLAUDE.md` шаблона. Если они **идентичны** — почти наверняка собственный `CLAUDE.md` проекта был затёрт копией шаблона. Попробуй восстановить оригинал: `git show HEAD:CLAUDE.md`. Если оригинал есть в истории — используй его как базу для слияния (Шаг 4) и предупреди пользователя. Если истории нет — отметь это в отчёте как потенциальную потерю.

## Шаг 2. Анализ проекта

Собери факты (для отчёта и для заполнения проектной секции `CLAUDE.md`). Не угадывай молча — где неясно, спрашивай.

- **Стек:** по манифестам — `package.json` (Node/JS, фронтенд), `pyproject.toml`/`requirements.txt` (Python), `Cargo.toml` (Rust), `go.mod` (Go), `pom.xml`/`build.gradle` (Java), `composer.json` (PHP), `Gemfile` (Ruby) и т. п.
- **Команды сборки/тестов:** из скриптов манифеста (`package.json` → `scripts`), `Makefile`, `Taskfile`, CI-конфигов. Зафиксируй точные команды build / test / lint.
- **Frontend?** Признаки UI-фреймворка/сборщика (React, Vue, Svelte, Angular, Next.js, Vite, `index.html`, `web/`, `frontend/`, `client/`). Если да — применим e2e-гейт на Playwright (см. агент `tester`).
- **Поставляемый сервис или нет?** Сервис/web/API, который едет на стенд (наличие `Dockerfile`, HTTP-сервера, порта) → применим контракт поставки `deploy-guide.md`. Библиотека/CLI без поставки → контракт N/A. Если неочевидно — **спроси пользователя**.
- **Текущая версия (базовая линия):** из манифеста (`version`) и/или последнего тега `git tag --list 'v*' --sort=-v:refname` (первый). Это точка отсчёта semver для следующего `/adr`.
- **Существующий CI/CD:** что уже есть в `.github/workflows/` (особенно деплой). Существующий пайплайн **не трогаем**.

Выведи краткое резюме анализа и открытые вопросы, прежде чем менять файлы.

## Шаг 3. Каркас workflow (аддитивно — только то, чего нет)

Скопируй из шаблона-источника **только отсутствующие** элементы:

- `.claude/agents/*`, `.claude/commands/*` — допиши недостающие (часть уже есть, раз `/migrate` запустилась).
- Каталоги артефактов с их `README.md`-плейсхолдерами: `requirements/`, `ba-req/`, `adrs/`, `releases/`, `release-notes/`. **Не** копируй `requirements/req-example-001.md` (это плейсхолдер шаблона).
- `deploy-guide.md`, `deploy-diagram-automind.md` — если проект может деплоиться (или может в будущем); это справочник.
- `.github/workflows/deploy.yml` — **только если** проект поставляемый сервис **и** у него нет своего деплой-пайплайна. Копируй **дормантный** (как в шаблоне, без активного `push`-триггера). Если деплой-CI уже есть — не копируй, опиши существующий в отчёте и предложи, как привести к контракту (не выполняя слепых правок).

## Шаг 4. Слияние конфликтных файлов (сливать, не перезаписывать)

- **`CLAUDE.md`:**
  - Нет в проекте → создай из `CLAUDE.md` шаблона; затем допиши/заполни проектную секцию (стек, точные команды build/test/lint/e2e, признак поставляемости) по фактам из Шага 2.
  - Есть в проекте → **сохрани весь существующий контент**. Убедись, что присутствуют разделы workflow (Принцип, Структура каталогов, Соглашения, Production-готовность, Кто и когда вызывается, Принципы работы, Как запускать flow). Недостающие добавь под понятными заголовками — **без дублирования**. Заполни/допиши проектную секцию со стеком и командами. Существующие проектные инструкции не удаляй — на них опираются агенты.
- **`.gitignore`:** допиши недостающее — блок секретов окружения (`.env`, `deploy/.env`, при сохранении `*.env.example`) и `.claude/settings.local.json`. Не дублируй существующие правила.
- **`README.md`:** не перезаписывай. Максимум — предложи в отчёте добавить ссылку на раздел workflow в `CLAUDE.md`; правь только с согласия пользователя.

## Шаг 5. Базовая линия и снимок состояния

1. Зафиксируй **базовую версию** (из Шага 2) — от неё `/adr` отсчитает следующий semver. Отрази её в отчёте.
2. Напиши **снимок текущей архитектуры** как ADR: `adrs/adr-NNN.md` со следующим свободным `NNN` (обычно `001`), статус «принято», заголовок вида «Базовая линия: принятие workflow и снимок текущей архитектуры». Кратко: стек, структура модулей, текущая версия, поставляемость, существующий CI. Это даёт контекст будущим ADR. Ceremony с вариантами решения не требуется — это фиксация факта, а не выбор.

## Шаг 6. Отчёт о миграции

Выведи пользователю:
- **Добавлено:** какие файлы/каталоги созданы из шаблона.
- **Слито:** какие разделы дописаны в `CLAUDE.md`, какие строки — в `.gitignore`.
- **Обнаруженный стек и команды:** чтобы пользователь мог поправить, если что-то определено неверно.
- **Поставляемость:** решение (сервис/библиотека) и статус `deploy.yml` (добавлен дормантным / пропущен из-за существующего CI / N/A).
- **Базовая версия** и путь к baseline-ADR.
- **Конфликты и открытые вопросы**, требующие решения пользователя (в т. ч. возможная потеря затёртого `CLAUDE.md`).
- **Следующие шаги:** проверить слитый `CLAUDE.md`; затем обычный цикл для первой задачи — `/ba` → `/adr` → … Команды дальше **не запускай** автоматически.

## Когда остановиться и спросить
- Путь к шаблону не найден или неоднозначен.
- Непонятно, поставляемый ли это сервис, или неясны команды тестов/сборки.
- Существующий `CLAUDE.md`/CI содержит правила, конфликтующие с workflow, — опиши конфликт и спроси, как разрешать. Не разрешай молча в пользу шаблона.
