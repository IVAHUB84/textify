Пользователь запросил подготовку релиза.

**Аргумент:** `$ARGUMENTS` — версия релиза `vX.Y.Z`. Может быть пустым.

## Что делать

1. Определи целевую спецификацию (как в `/implement`).
2. Убедись, что репозиторий инициализирован. Если нет — сообщи пользователю; тестировщик не должен инициализировать репо в режиме `/release` без явной просьбы.

### Этап А. Тестирование и подготовка релиза

3. Запусти subagent через Agent:
   - `subagent_type`: `tester`
   - `description`: «Финальная проверка и подготовка релиза vX.Y.Z»
   - `prompt`: путь к спецификации, пути к связанным BA-требованиям (для критериев приёмки) и ADR, явное указание: «Выполни процесс из системного промпта: для frontend-проектов — наличие/создание e2e на Playwright, локальный прогон и обязательный e2e-гейт в `.github/workflows/`; прогон тестов, проверка критериев приёмки, осмысленные коммиты по группам изменений, тег `vX.Y.Z`, файл `release-notes/notes-vX.Y.Z.md`. Не выполняй `git push`. По окончании верни отчёт с хэшами коммитов, именем тега, статусом e2e/гейта и списком подтверждённых/неподтверждённых критериев приёмки.»

### Этап Б. Возможные исправления

4. Если тестировщик нашёл дефекты — запусти subagent `developer` с подробным отчётом и попроси починить. После исправлений — **снова `tester`** (он же делает коммиты и тег).
5. Лимит — 3 итерации.

### Финал

6. Проверь:
   - `git log --oneline -n 10` — есть релизные коммиты.
   - `git tag --list 'vX.Y.Z'` — тег присутствует.
   - `release-notes/notes-vX.Y.Z.md` существует.
   - Для frontend-проектов: e2e на Playwright прогнаны локально и в `.github/workflows/` присутствует обязательный e2e-гейт (без `continue-on-error`). Если frontend-части нет — тестировщик должен явно это отметить.
7. Выведи пользователю отчёт тестировщика и напомни о `/deploy` как следующем шаге.
