# CLAUDE.md

Этот документ описывает workflow разработки и соглашения проекта. Он автоматически загружается в контекст Claude Code.

## Проект: Textify

Telegram-бот, бесплатно превращающий голосовые/аудио и изображения в чистый структурированный текст. Подробности и схема — в [`README.md`](./README.md).

### Принятые решения

- **Стек:** Python + [aiogram](https://docs.aiogram.dev/) (Telegram), [faster-whisper](https://github.com/SYSTRAN/faster-whisper) (аудио→текст), [pytesseract](https://github.com/madmaze/pytesseract)/Tesseract (картинки→текст), [Groq](https://groq.com/) free API (структурирование текста).
- **Хостинг:** свой VPS (Linux). Whisper и Tesseract работают локально/офлайн; во внешний интернет ходит только Groq API.
- **Языки контента:** русский + английский. Whisper определяет язык автоматически; для OCR подключены пакеты Tesseract `rus` + `eng`.
- **Выход:** распознанный текст + краткая структура (заголовки, списки, ключевые пункты), формируется LLM (Groq).
- **Бот:** [@YourTextifyBot](https://t.me/YourTextifyBot).

### Системные зависимости (VPS)

```bash
sudo apt install -y ffmpeg tesseract-ocr tesseract-ocr-rus tesseract-ocr-eng
```

### Команды проекта

| Действие | Команда |
|---|---|
| Установка зависимостей | `pip install -r requirements.txt` |
| Запуск бота | `python bot.py` |
| Линт | `ruff check .` |

> Конкретные команды линта/тестов уточняются по мере появления конфигов; на них опираются агенты `developer`, `tester`, `maintainer`.

### Секреты

- `BOT_TOKEN` и `GROQ_API_KEY` хранятся **только** в `.env` (покрыт `.gitignore`). В коде — через переменные окружения, никогда не хардкодить.
- В репозиторий коммитится только `.env.example` с плейсхолдерами.
- Токен бота, если он был раскрыт, перевыпускать через @BotFather → `/revoke`.

## Принцип

Разработка построена вокруг **последовательной передачи артефактов** между специализированными агентами. Каждый агент имеет узкую зону ответственности и пишет свой документ. Следующий агент читает предыдущий артефакт и создаёт свой.

```
requirements/req-*.md   ── /ba ──►   ba-req/ba-requirements-*.md
                                        │
                                        ▼
                              /adr ──►  adrs/adr-*.md
                                        releases/release-spec-vX.Y.Z.md
                                        │
                                        ▼
                       /implement ──►  изменения кода + тесты
                                        ↑
                                        │ (повторно при отклонениях)
                                        ▼
                              architect (роль 2: контроль)
                                        │
                                        ▼
                          /review ──►  отчёт ревьюера
                                        ↑
                                        │ (повторно при замечаниях)
                                        ▼
                                    developer
                                        │
                                        ▼
                          /release ──► tests + commits + tag + release-notes/notes-*.md
                                        │
                                        ▼
                          /deploy  ──► чек-лист + git push (после подтверждения)
```

Альтернативный режим: **`/lead vX.Y.Z`** — после того, как BA-требование и ADR/спецификация готовы, Lead-агент автоматически прогоняет implement → arch-check → review → release → deploy.

## Структура каталогов

- `requirements/` — сырые требования от пользователя (`req-<name>-NNN.md`).
- `ba-req/` — оформленные бизнес-требования (`ba-requirements-NNN.md`).
- `adrs/` — Architecture Decision Records (`adr-NNN.md`).
- `releases/` — спецификации релизов (`release-spec-vX.Y.Z.md`).
- `release-notes/` — release notes (`notes-vX.Y.Z.md`).
- `roadmap.md` — необязательный, ведётся мейнтейнером.
- `.claude/agents/` — определения subagents.
- `.claude/commands/` — slash-команды.

## Соглашения

### Язык
- Документы (`ba-req`, `adrs`, `releases`, `release-notes`, `roadmap`, `CLAUDE.md`, `README.md`) — **на русском**.
- Код, идентификаторы, имена файлов кода, сообщения коммитов — **на английском**, если в проекте не принято иное.

### Версионирование (semver)
- **MAJOR (X)** — ломающие изменения публичного контракта.
- **MINOR (Y)** — обратимо-совместимое добавление функциональности.
- **PATCH (Z)** — багфиксы без изменения контракта.

### Нумерация документов
- `requirements/req-<name>-NNN.md` — номер сквозной, выдаётся при создании.
- `ba-req/ba-requirements-NNN.md` — номер совпадает с номером исходного требования (если возможно).
- `adrs/adr-NNN.md` — сквозная нумерация ADR, независимая от requirements.

### Git
- Коммиты — только в рамках `/release` (через тестировщика).
- Push — только в рамках `/deploy` (через мейнтейнера, после подтверждения пользователя).
- Никогда не использовать `--no-verify`, `--no-gpg-sign`, `git push --force` без явной просьбы пользователя.
- `git add -A` / `git add .` запрещены — добавлять файлы выборочно.
- Сообщения коммитов **не должны** содержать упоминаний AI-инструментов (`Generated with Claude` и т. п.), если пользователь не попросил иначе.

## Production-готовность (контракт поставки)

Каждый **поставляемый сервис** (web-приложение / API / сервис, который едет на production-стенд `app-01`) обязан соответствовать контракту из [`deploy-guide.md`](./deploy-guide.md) — это **источник истины** по поставке. Сама production-среда описана в `deploy-guide.md` и `deploy-diagram-automind.md`.

Контракт требует в репозитории четыре артефакта:

1. `Dockerfile` — приложение слушает **один** порт на `0.0.0.0`, multi-stage, по возможности non-root, с health-эндпоинтом.
2. `deploy/docker-compose.yml` — **production**-манифест (не dev): без `ports:`, без своих `postgres`/`redis`, `image:` из GHCR (не `build:`), сеть `edge` (`external: true`), traefik-лейблы с верным `loadbalancer.server.port`, `restart: unless-stopped`.
3. `deploy/.env.example` — контракт переменных окружения с плейсхолдерами. Реальный `.env` **не коммитится** (покрыт `.gitignore`).
4. `.github/workflows/deploy.yml` — build → push в GHCR → deploy по SSH → уведомление в Discord. Универсальный, имя продукта берётся из имени репозитория. **Входит в шаблон в дормантном виде** (только `workflow_dispatch`, чтобы шаблон сам себя не деплоил и не запускал Actions); на первом релизе продукта разработчик активирует push-триггер.

Эти артефакты — часть DoD релиза: архитектор включает их в спецификацию, разработчик создаёт/поддерживает их строго по `deploy-guide.md`, мейнтейнер проверяет их в чек-листе готовности перед `/deploy`. Полный список железных правил и антипаттернов **не дублируй** — он в `deploy-guide.md`.

**Проекты без поставки** (библиотека, CLI-утилита, которые не едут на стенд) от контракта освобождены — отметь это явно при планировании релиза.

## Кто и когда вызывается

| Команда | Subagent | Модель | Артефакт на выходе |
|---|---|---|---|
| `/ba` | `business-analyst` | opus | `ba-req/ba-requirements-NNN.md` |
| `/adr` | `architect` | opus | `adrs/adr-NNN.md`, `releases/release-spec-vX.Y.Z.md` |
| `/implement` | `developer` → `architect` | sonnet → opus | изменения кода + тесты; отчёт о соответствии ADR |
| `/review` | `code-reviewer` → `developer` | opus → sonnet | отчёт ревьюера; исправления |
| `/release` | `tester` (+ `developer` при дефектах) | sonnet | e2e на Playwright + release gate `.github/workflows/` (для frontend), коммиты, тег `vX.Y.Z`, `release-notes/notes-vX.Y.Z.md` |
| `/deploy` | `maintainer` | sonnet | чек-лист готовности (вкл. контракт поставки `deploy-guide.md` для сервисов), `git push` (после подтверждения) |
| `/lead vX.Y.Z` | `lead` (оркестратор) | opus | прогон всех шагов от implement до deploy |

## Принципы работы

1. **Артефакты — источник истины.** Не верь устным отчётам subagent'а — проверяй файл.
2. **Открытые вопросы блокируют переход.** Если у BA остались блокирующие вопросы — не запускать `/adr`. Если у архитектора — не запускать `/implement`.
3. **Лимит итераций.** Не более 3 циклов исправлений на каждом этапе. После третьего круга — эскалация пользователю.
4. **Push — всегда через подтверждение.** Никогда автоматически.
5. **Минимальные изменения.** Не добавлять ничего сверх того, что описано в спецификации.

## Как пользователю запускать flow

Полный цикл новой задачи:

```
1. Создать requirements/req-<name>-001.md   (вручную)
2. /ba 001                                  → ba-req/ba-requirements-001.md
3. /adr 001 v0.2.0                          → adrs/adr-001.md, releases/release-spec-v0.2.0.md
4. /implement v0.2.0                        → код + тесты (+ arch-check)
5. /review v0.2.0                           → ревью (+ возможные исправления)
6. /release v0.2.0                          → e2e/CI gate (frontend), коммиты, тег, release notes
7. /deploy v0.2.0                           → push (после подтверждения)
```

Ускоренный режим (когда BA + ADR уже готовы):

```
1. /lead v0.2.0
```

### Миграция существующего проекта на workflow

Если проект уже существует:

1. Скопировать в проект каталог `.claude/` из шаблона (агенты + команды — этого достаточно, чтобы стала доступна `/migrate`; копирование аддитивное).
2. Запустить `/migrate <путь-к-шаблону>` (например, `/migrate ../template`).

`/migrate` неразрушающа: аддитивные файлы копирует только если их нет, конфликтные (`CLAUDE.md`, `.gitignore`) **сливает**, а не перезаписывает (и восстанавливает затёртый `CLAUDE.md` из git history); применяет контракт поставки только если проект — поставляемый сервис без своего пайплайна; фиксирует текущую версию как базовую, пишет baseline-ADR и отчёт. Код, тесты и историю не трогает, не коммитит.
