# ADR-001: Каркас Telegram-бота на aiogram (long-polling) и адаптация деплой-контракта под бота без HTTP-порта

- **Дата:** 2026-06-22
- **Статус:** принято
- **Связанные BA-требования:** `ba-req/ba-requirements-001.md`
- **Связанные ADR:** —

## Контекст

Textify — Telegram-бот, который в будущих релизах будет распознавать аудио (faster-whisper),
изображения (Tesseract) и структурировать текст через Groq (см. `README.md`, `CLAUDE.md`).
Релиз v0.1.0 — первый релиз продукта. Его задача не пользовательская функция, а **проверенный
канал поставки** и **минимальный скелет**, на который безопасно наращивается функциональность:
команды `/start`, `/help`, ответ-заглушка на текст, конфиг `BOT_TOKEN` из `.env`, каркас каталогов
`handlers/`/`services/`, `requirements.txt`. Логики распознавания на этом этапе нет.

Действующие силы и ограничения:

- **Способ работы с Telegram.** Бот может получать обновления двумя способами: long-polling
  (бот сам опрашивает Telegram, входящий порт не нужен) или webhook (Telegram стучится на
  публичный HTTPS-эндпоинт бота). Выбор влияет на инфраструктуру поставки.
- **Целевой сервер — ivahub** (138.16.226.66): 1 vCPU, 1.9 GB RAM, Ubuntu 24.04, Docker 29.5.3.
  На сервере **нет Traefik и нет docker-сети `edge`**, нет общих PostgreSQL/Redis на `10.0.0.20`.
  Это не стенд `app-01` из `deploy-guide.md`.
- **Деплой-контракт `deploy-guide.md`** написан под HTTP-сервисы за Traefik на `app-01`. Его
  железные правила (нет `ports:`, нет своих stateful-сервисов, `image:` из GHCR, секреты только
  в `.env`, `restart: unless-stopped`) частично применимы, а часть (traefik-лейблы,
  `loadbalancer.server.port`, сеть `edge`, health-эндпоинт за прокси) — нет.
- **Ресурсное ограничение** (1 vCPU / 1.9 GB RAM) — для каркаса несущественно, но определяет
  будущие решения по выбору моделей распознавания.

## Рассматриваемые варианты

### Решение 1. Способ получения обновлений: long-polling vs webhook

#### Вариант A: long-polling
- Плюсы: не нужен входящий HTTP-порт, публичный домен, TLS-сертификат и обратный прокси;
  деплой сводится к одному контейнеру; проще локальный запуск (`python bot.py`); меньше движущихся
  частей на маломощном сервере.
- Минусы: бот сам держит соединение с Telegram (небольшой постоянный трафик опроса); на масштабе
  тысяч обновлений в секунду webhook эффективнее (для Textify не актуально).
- Последствия: у контейнера нет открытых портов; deploy-контракт в части Traefik/edge неприменим.

#### Вариант B: webhook
- Плюсы: меньше холостого трафика, лучше масштабируется на высокой нагрузке.
- Минусы: требует публичный HTTPS-эндпоинт → обратный прокси с валидным TLS на ivahub, где
  Traefik отсутствует; пришлось бы ставить и сопровождать прокси/сертификаты ради бота без иной
  нагрузки. Избыточно для текущих и ожидаемых объёмов.
- Последствия: усложняет инфраструктуру на сервере без Traefik; противоречит цели «минимальный
  каркас».

### Решение 2. Поставка бота без HTTP-порта: следовать `deploy-guide.md` буквально vs адаптировать

#### Вариант A: буквально по `deploy-guide.md`
- Плюсы: единообразие с другими продуктами Automind Studio.
- Минусы: контракт предполагает Traefik, сеть `edge`, `loadbalancer.server.port`, общие БД/Redis
  на `10.0.0.20` и сервер `app-01` — ничего из этого на ivahub нет; буквальное следование
  невыполнимо и приведёт к нерабочему манифесту.
- Последствия: манифест ссылается на несуществующую инфраструктуру.

#### Вариант B: адаптировать контракт — сохранить применимые правила, отбросить неприменимые
- Плюсы: рабочий прод-манифест под реальный сервер; сохраняются ценные инварианты контракта
  (GHCR-образ, отсутствие проброса портов, секреты через `.env`, `restart: unless-stopped`,
  деплой через CI/CD); отклонение задокументировано.
- Минусы: расхождение с шаблоном; требует явной фиксации, чтобы будущие агенты не «чинили»
  манифест под Traefik.
- Последствия: для бота вводится профиль поставки «без HTTP-порта».

### Решение 3. CI/CD-деплой на ivahub vs шаблонный `deploy.yml` под `app-01`

#### Вариант A: использовать шаблонный `deploy.yml` как есть
- Минусы: workflow жёстко завязан на org-секреты `semsys-ams` (`DEPLOY_HOST` = app-01) и образ
  `ghcr.io/semsys-ams/<repo>`; целевой сервер — ivahub, а не app-01.
- Последствия: деплой уйдёт не на тот сервер либо упадёт.

#### Вариант B: активировать `deploy.yml`, нацелив его на ivahub
- Плюсы: переиспользуется готовый пайплайн build → push GHCR → deploy по SSH → уведомление;
  меняется только адресат деплоя и реестр образа через секреты/переменные.
- Минусы: расхождение значений секретов с остальными продуктами (это нормально — сервер другой).
- Последствия: на первом релизе активируется push-триггер; `DEPLOY_HOST`/`DEPLOY_USER`/
  `DEPLOY_SSH_KEY` и путь образа должны указывать на ivahub.

## Решение

1. **Long-polling** (Решение 1, вариант A). Бот не имеет входящего HTTP-порта; запуск — long-polling
   из `bot.py`.
2. **Адаптированный профиль поставки «бот без HTTP-порта»** (Решение 2, вариант B). Сохраняются
   применимые железные правила `deploy-guide.md`; неприменимые — отбрасываются как осознанное
   отклонение (см. ниже).
3. **CI/CD на ivahub** (Решение 3, вариант B). Шаблонный `deploy.yml` активируется push-триггером,
   но нацеливается на сервер ivahub; путь образа и адресат SSH-деплоя должны соответствовать ivahub,
   а не `app-01`. Конкретные имена секретов и путь образа фиксируются в спецификации релиза.
4. **Архитектура каркаса.** Точка входа `bot.py`; хендлеры — в `handlers/` (роутеры aiogram);
   будущая бизнес-логика распознавания — в `services/`. Конфиг (`BOT_TOKEN`) загружается из
   окружения с явной валидацией; пустой/незаданный токен → завершение с понятной ошибкой.
   Никаких зависимостей распознавания (faster-whisper, pytesseract/Tesseract, Groq, ffmpeg) в этом
   релизе не вводится.

### Осознанное отклонение от `deploy-guide.md` (фиксируется этим ADR)

Поскольку бот работает по long-polling и **не слушает входящий порт**, а на ivahub нет Traefik
и сети `edge`, к Textify **НЕ применяются**:

- traefik-лейблы и `loadbalancer.server.port`;
- сеть `edge` (`external: true`);
- проброс портов (`ports:`);
- health-эндпоинт за обратным прокси;
- общие PostgreSQL/Redis на `10.0.0.20` (боту состояние не нужно).

При этом **СОХРАНЯЮТСЯ** применимые правила контракта:

- `Dockerfile` — multi-stage, по возможности non-root;
- `deploy/docker-compose.yml` — production-манифест с `image:` из GHCR (не `build:`),
  `restart: unless-stopped`, `env_file: .env`, без `ports:`;
- `deploy/.env.example` — плейсхолдеры, реальный `.env` не коммитится;
- `.github/workflows/deploy.yml` — деплой через CI/CD; на первом релизе активируется push-триггер,
  адресат — ivahub.

Это отклонение допустимо: `deploy-guide.md` описывает профиль HTTP-сервисов за Traefik на `app-01`;
Textify — иной профиль (бот без порта на другом сервере). Отклонение узкое, явно ограничено и не
ослабляет инварианты безопасности (секреты, отсутствие проброса портов).

## Последствия

- **Положительные:** простая и предсказуемая поставка одного контейнера; ранняя проверка
  деплой-канала до появления тяжёлой логики; сохранены ключевые инварианты безопасности;
  каркас `handlers/`/`services/` позволяет добавлять v0.2.0+ без реструктуризации.
- **Отрицательные / компромиссы:** профиль поставки Textify расходится с шаблоном Automind Studio —
  будущим агентам нельзя «чинить» манифест под Traefik; значения деплой-секретов отличаются от
  остальных продуктов (другой сервер).
- **Технический долг:** при будущем кратном росте нагрузки можно пересмотреть long-polling →
  webhook (потребует прокси/TLS на ivahub) — отдельным ADR. Выбор и ресурсный бюджет тяжёлых
  моделей распознавания (whisper) под 1 vCPU / 1.9 GB RAM — забота релизов v0.2.0–v0.4.0,
  отдельными ADR.

## Влияние на код

- **Новые модули/слои:** `bot.py` (точка входа, запуск polling); `handlers/` (роутеры команд и
  текста); `services/` (заготовка под будущую логику); загрузка конфигурации из окружения.
- **Новые зависимости:** `aiogram` и средство загрузки `.env` (`python-dotenv`). Зависимости
  распознавания не вводятся.
- **Артефакты поставки:** `Dockerfile`, `deploy/docker-compose.yml`, `deploy/.env.example`,
  активация push-триггера в `.github/workflows/deploy.yml` (нацелен на ivahub).
- **Что ломается:** ничего — это первый релиз; публичного контракта до него нет.
