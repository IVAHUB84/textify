# ADR-005: Структурирование распознанного текста через бесплатный LLM (Cloudflare Workers AI по умолчанию) с абстракцией провайдера, неблокирующим HTTP-вызовом и фолбэком на сырой текст

- **Дата:** 2026-06-22
- **Статус:** принято
- **Связанные BA-требования:** `ba-req/ba-requirements-005.md`
- **Связанные ADR:** ADR-001 (профиль поставки «бот без HTTP-порта», секреты через `.env`), ADR-002 (граница «хендлер ↔ сервис», неблокирующий сервисный слой), ADR-003 (неблокирующий сервис для аудио, паттерн `services/`), ADR-004 (улучшение OCR)

## Контекст

Релизы v0.1.0–v0.4.0 в проде на ivahub. Оба канала распознавания — OCR (`services/ocr.py`) и
транскрипция (`services/transcribe.py`) — отдают пользователю **сырой** текст. Структурирование
распознанного текста (заголовки/списки/ключевые пункты в Markdown) — заявленная ключевая ценность
продукта, до сих пор не реализованная. Релиз v0.5.0 доводит продукт до исходного ценностного
обещания.

Изначально структурирование планировалось через Groq free API, но **Groq гео-блокирует российские
IP** (CF 403): с ivahub эндпоинт недоступен (см. memory: гео-блок Groq по RU). Заказчик принял
решение использовать провайдера, работающего с RU-IP без прокси, — **Cloudflare Workers AI** — как
провайдер по умолчанию, сохранив Groq переключаемой опцией на будущее.

Действующие силы и ограничения:

- **Паттерн ADR-002/ADR-003.** Уже есть `bot.py`, `handlers/`, `services/` (рабочие `ocr.py`,
  `transcribe.py`, `preprocess.py`), `config.py`. Структурирование надо встроить тем же приёмом:
  чистая граница «хендлер ↔ сервис», хендлер не знает о провайдере LLM, сервис не знает об aiogram.
- **Новый класс задачи — I/O-bound (сетевой), а не CPU-bound.** В отличие от OCR/транскрипции
  (`asyncio.to_thread` поверх синхронного CPU-вызова), здесь основная работа — HTTP-запрос к
  внешнему API. Это первый исходящий сетевой сервис; неблокирование решается **нативно async**
  HTTP-клиентом, а не выносом в поток.
- **Ресурсы ivahub** — 1 vCPU / 1.9 GB RAM. Сетевая задача почти не добавляет нагрузки на CPU;
  главное требование — не блокировать event loop на время HTTP-запроса.
- **Внешняя зависимость вне нашего контроля.** Доступность провайдера, суточные лимиты бесплатного
  tier и максимальный размер запроса вне контроля проекта. Поэтому фолбэк на сырой текст обязателен,
  а длинный вход надо укладывать в лимит.
- **Секреты** (ADR-001, `CLAUDE.md`). Параметры провайдера (`CF_ACCOUNT_ID`, `CF_API_TOKEN`; для
  Groq — `GROQ_API_KEY`) — только из окружения/`.env`; в репозиторий — лишь плейсхолдеры в
  `.env.example` и `deploy/.env.example`. Это **первые пользовательские env-переменные** помимо
  `BOT_TOKEN`.
- **Профиль поставки ADR-001.** Long-polling, без `ports:`/Traefik/`edge`, образ из GHCR,
  `restart: unless-stopped`, `extra_hosts`-пиннинг Telegram. Релиз добавляет лишь **новое
  направление исходящего трафика** — к API Cloudflare, помимо `api.telegram.org`.

## Рассматриваемые варианты

### Решение 1. Абстракция выбора провайдера (Cloudflare по умолчанию, Groq — опция)

#### Вариант A: один монолитный модуль с `if LLM_PROVIDER == ...` внутри одной функции
- Плюсы: меньше файлов.
- Минусы: ветвление провайдеров размазано по коду структурирования; добавление третьего провайдера
  или Groq-ветки разрастается ветвлениями; тяжело тестировать выбор провайдера изолированно.
- Последствия: смешение «что делаем» (структурируем) и «через кого» (провайдер).

#### Вариант B: тонкая абстракция провайдера — общий контракт + реализации Cloudflare/Groq, выбор по `LLM_PROVIDER`
- Плюсы: `services/structure.py` отвечает за оркестрацию (промпт, фолбэк, усечение, выбор
  провайдера), провайдеры — за «как сходить в конкретный API». Переключение — чистый конфиг
  (`LLM_PROVIDER`), без правки кода (ФТ-3). Groq-ветка добавляется как реализация общего контракта,
  её код-путь существует и выбирается, боевая работа не требуется (ДОП-5). Провайдер тестируется
  мок-патчем HTTP-клиента без сети.
- Минусы: чуть больше структуры (внутренний интерфейс провайдера).
- Последствия: вводится внутренний контракт «провайдер структурирования»; дефолт — Cloudflare.

### Решение 2. Способ обращения к Cloudflare Workers AI: SDK vs прямой HTTP

#### Вариант A: официальный Python-SDK Cloudflare
- Плюсы: типизированные методы.
- Минусы: новая тяжёлая зависимость и её транзитивные пакеты ради одного POST-запроса; завязка на
  версии SDK; Groq-ветка потребовала бы **другой** SDK — два разных клиента вместо одного приёма.
- Последствия: лишний вес образа и связность с вендорским SDK.

#### Вариант B: прямой HTTP-вызов REST-эндпоинта через общий async-клиент
- Плюсы: Cloudflare Workers AI и Groq оба — простые HTTP/JSON API (Groq — OpenAI-совместимый
  chat/completions; Cloudflare — REST `…/ai/run/<model>`). Один async HTTP-клиент покрывает обе
  ветки; провайдер = URL + заголовки + форма тела + парсинг ответа. Минимальная зависимость,
  полный контроль над таймаутом. Эндпоинт Cloudflare:
  `POST https://api.cloudflare.com/client/v4/accounts/{CF_ACCOUNT_ID}/ai/run/{model}` с
  `Authorization: Bearer {CF_API_TOKEN}` и телом `{"messages": [...]}`.
- Минусы: парсинг JSON-ответа пишем сами (для Cloudflare результат в `result.response`).
- Последствия: добавляется один HTTP-клиент в `requirements.txt`; провайдеры — тонкие функции
  построения запроса/разбора ответа.

### Решение 3. Неблокирующий механизм для сетевого вызова (нативный async vs вынос в поток)

#### Вариант A: синхронный HTTP-клиент (`requests`) в `asyncio.to_thread`
- Плюсы: единообразие с OCR/транскрипцией (тоже `to_thread`).
- Минусы: для CPU-bound `to_thread` оправдан, но для сетевого I/O он держит поток занятым на всё
  время ожидания сети (десятки секунд при медленном/повисшем ответе); поток из пула блокируется
  впустую. Семантически неверный инструмент для сетевого ожидания.
- Последствия: расход потоков пула на сетевое ожидание.

#### Вариант B: нативный async HTTP-клиент (`httpx.AsyncClient`) с `await`
- Плюсы: `await client.post(...)` отдаёт управление event loop на время ожидания сети — бот
  обслуживает другие сообщения без занятого потока (ФТ-7/НФТ-2). `httpx` имеет первоклассный
  async-API и встроенные таймауты. Это правильный инструмент для I/O-bound сетевого вызова и
  отличает данный сервис от CPU-bound предшественников.
- Минусы: новая зависимость `httpx` (лёгкая, чистый Python + `httpcore`/`anyio`).
- Последствия: `services/structure.py` и провайдеры — нативно асинхронны (`async def`), без
  `to_thread`.

### Решение 4. Таймаут и политика ретраев

#### Вариант A: агрессивные ретраи с backoff
- Минусы: при недоступности/гео-блоке/исчерпании лимита ретраи лишь умножают задержку перед
  неизбежным фолбэком; пользователь дольше ждёт сырой текст; повышается расход суточного лимита.
- Последствия: медленный фолбэк, бессмысленная нагрузка на лимит.

#### Вариант B: единичная попытка с явным таймаутом, без ретраев, затем фолбэк
- Плюсы: соответствует НФТ-1/ОВ-3 (по умолчанию — единичная попытка с явным таймаутом). Любой сбой
  (сеть, 4xx/5xx, таймаут, превышение лимита, некорректный ответ) → быстрый детерминированный
  фолбэк на сырой текст. Явный общий таймаут запроса **30 секунд** — баланс между «дать LLM
  ответить» и «не заставлять пользователя ждать вечно»; на нём же таймаут защищает event loop от
  повисшего соединения.
- Минусы: на разовом транзиентном сбое сети пользователь получит сырой текст вместо
  структурированного (приемлемо — структурирование не критичный путь).
- Последствия: один POST с `timeout=30`, без повторов; все исключения ловятся и ведут в фолбэк.

### Решение 5. Стратегия для очень длинного входного текста (лимит размера запроса / суточный лимит)

#### Вариант A: разбивка на части, отдельные запросы, склейка результатов
- Минусы: множит число запросов (быстрее исчерпывает суточный лимит бесплатного tier), усложняет
  склейку Markdown (стыки заголовков/списков), добавляет латентность и точки отказа. BA допускает
  усечение по умолчанию; разбивку — только если оправдана. Для одного-двух пользователей и редких
  очень длинных входов разбивка не оправдана.
- Последствия: преждевременная сложность, риск по лимитам.

#### Вариант B: усечение входа по безопасному порогу символов перед отправкой
- Плюсы: один запрос на сообщение (бережёт суточный лимит); гарантированно укладывается в размер
  запроса провайдера; прост и предсказуем. Порог — **по числу символов**, безопасно ниже лимита
  модели (ориентир — **8000 символов**: с запасом укладывается в контекст лёгкой instruct-модели
  для входа+промпта+вывода на смешанном rus/eng). Усечение — по границе (обрезка хвоста сырого
  текста перед формированием промпта); пользователь всё равно получает структурированную бóльшую
  часть. Если запрос всё же не проходит (любой сбой) — фолбэк на **полный** сырой текст (НФТ-1).
- Минусы: на сверхдлинном входе хвост не структурируется (структурируется усечённая часть);
  приемлемо для текущих объёмов, жёсткие пользовательские лимиты — v0.6.0 (ОВ-6/ДОП-6).
- Последствия: перед вызовом провайдера сырой текст усекается до порога; фолбэк всегда отдаёт
  исходный (неусечённый) сырой текст.

### Решение 6. Выбор конкретной модели Cloudflare Workers AI (ОВ-2)

#### Вариант A: крупная модель ради качества
- Минусы: на бесплатном tier крупные модели дороже по нейронам/лимиту и медленнее; для задачи
  «оформить, не сочиняя» избыточны.
- Последствия: быстрее исчерпывается лимит, выше латентность.

#### Вариант B: лёгкая мультиязычная instruct-модель — `@cf/meta/llama-3.1-8b-instruct`
- Плюсы: доступна в бесплатном каталоге Workers AI, мультиязычна (rus+eng), instruct-формат
  (`messages`) подходит для задачи форматирования по системному промпту; разумный баланс
  качество/латентность/лимит на 8B. Имя модели — **конфигурируемое** через env
  (`CF_MODEL`, дефолт `@cf/meta/llama-3.1-8b-instruct`), чтобы сменить модель без правки кода, если
  каталог/доступность изменятся.
- Минусы: 8B-модель оформляет хорошо, но не идеально на шумном входе — это свойство входа, не
  дефект (ДОП-3/НФТ-7).
- Последствия: дефолтная модель зафиксирована, переопределяема env-переменной.

## Решение

1. **Тонкая абстракция провайдера** (Решение 1B). `services/structure.py` — оркестратор: строит
   промпт, усекает длинный вход, выбирает провайдера по `LLM_PROVIDER`, вызывает его, при любом
   сбое возвращает фолбэк на сырой текст. Реализации провайдеров — Cloudflare (дефолт) и Groq
   (опция «на будущее») — следуют общему внутреннему контракту «принять системный промпт + сырой
   текст → вернуть структурированный Markdown». Выбор провайдера — по значению `LLM_PROVIDER`
   (дефолт/незадано → Cloudflare), без правки кода (ФТ-2/ФТ-3).

2. **Прямой HTTP к REST-эндпоинту** (Решение 2B). Без вендорских SDK. Cloudflare:
   `POST https://api.cloudflare.com/client/v4/accounts/{CF_ACCOUNT_ID}/ai/run/{CF_MODEL}`,
   заголовок `Authorization: Bearer {CF_API_TOKEN}`, тело `{"messages":[{role:"system",...},
   {role:"user", content: <сырой текст>}]}`; результат структурирования читается из
   `result.response`. Groq-ветка — OpenAI-совместимый `chat/completions` с `GROQ_API_KEY`
   (код-путь существует, боевая работа не требуется).

3. **Неблокирующий нативный async** (Решение 3B). HTTP-вызов выполняется через `httpx.AsyncClient`
   с `await` — event loop не блокируется на время сети, поток пула не занимается ожиданием.
   `asyncio.to_thread` здесь **не** используется (это сетевой, не CPU-bound сервис). Это осознанное
   отличие от ADR-002/ADR-003 и предмет данного ADR (а не противоречие им).

4. **Единичная попытка, таймаут 30 с, без ретраев, затем фолбэк** (Решение 4B). Один POST с явным
   общим таймаутом 30 секунд. Любой сбой (сеть, DNS, 4xx/5xx, таймаут, превышение лимита,
   некорректная/пустая структура ответа) перехватывается и приводит к возврату **сырого** текста;
   бот не падает (ФТ-6/НФТ-1). Сбой логируется.

5. **Усечение длинного входа** (Решение 5B). Перед формированием промпта сырой текст усекается до
   безопасного порога ~8000 символов (конкретное число — на усмотрение разработчика в этом порядке
   величины; задаётся константой модуля). Один запрос на сообщение. Фолбэк всегда отдаёт **полный**
   исходный сырой текст, а не усечённый. Разбивка/склейка не реализуется (v0.6.0+, если потребуется).

6. **Модель Cloudflare — `@cf/meta/llama-3.1-8b-instruct`, переопределяемая env `CF_MODEL`**
   (Решение 6B). Лёгкая мультиязычная instruct-модель из бесплатного каталога Workers AI.

7. **Промпт структурирования** (ФТ-5). Системный промпт инструктирует модель: привести текст к
   читаемой структуре Markdown (заголовки, списки, ключевые пункты); **сохранять смысл, не
   выдумывать факты и не добавлять содержание, отсутствующее в исходнике**; не переводить и не
   менять язык контента (rus/eng — по исходнику); вернуть только оформленный текст без преамбул.
   Задача — оформление, а не генерация.

8. **Подключение к обоим каналам** (ФТ-4). Хендлеры `handlers/image.py` и `handlers/audio.py` после
   получения непустого сырого результата OCR/транскрипции пропускают его через
   `services.structure.structure_text(...)` перед отправкой пользователю. На **пустом** результате
   распознавания структурирование **не вызывается** — сохраняется прежнее сообщение о
   нераспознанном тексте (ФТ-9). Граница чистая: хендлер не знает о провайдере, сервис не знает об
   aiogram (ADR-002).

9. **Секреты только из окружения** (ФТ-8/НФТ-5). `config.py` читает `LLM_PROVIDER` (дефолт
   `cloudflare`), `CF_ACCOUNT_ID`, `CF_API_TOKEN`, `CF_MODEL` (дефолт модели), `GROQ_API_KEY`,
   опционально `GROQ_MODEL`, из окружения/`.env`. Плейсхолдеры добавляются в `.env.example` и
   `deploy/.env.example`. **Отсутствие** ключей провайдера при старте бота **не валит** процесс
   (в отличие от `BOT_TOKEN`) — структурирование тогда деградирует в фолбэк на сырой текст
   (распознавание остаётся рабочим). Секреты не хардкодятся и не коммитятся.

10. **Профиль поставки ADR-001 сохраняется** (НФТ-6). Long-polling, без `ports:`/Traefik/`edge`,
    образ из GHCR, `restart: unless-stopped`, `extra_hosts`-пиннинг Telegram — без изменений.
    Добавляется лишь исходящий доступ к `api.cloudflare.com`. Новых системных пакетов в `Dockerfile`
    нет (`httpx` — чистый Python-wheel). В `deploy/docker-compose.yml` структурно ничего не меняется,
    кроме того что реальный `.env` теперь несёт переменные провайдера (плейсхолдеры — в
    `deploy/.env.example`).

11. **Roadmap** (ФТ-10). Слот v0.5.0 переопределяется на «Структурирование (Cloudflare Workers AI;
    Groq — опция)» с фиксацией причины смены провайдера (гео-блок Groq по RU IP).

## Последствия

- **Положительные:** реализуется ключевая ценность продукта; вводится первый I/O-bound сервис с
  правильным нативным async-неблокированием; абстракция провайдера позволяет переключиться на Groq
  конфигом, когда доступность решится; фолбэк делает зависимость от внешнего LLM нехрупкой —
  распознавание остаётся рабочим при любом сбое LLM; поставка не меняется (тот же пайплайн +
  новые env-переменные).
- **Отрицательные / компромиссы:** появляется зависимость от внешнего бесплатного API (доступность,
  суточные лимиты, размер запроса вне контроля) — смягчена фолбэком и усечением; качество
  оформления зависит от модели и промпта (НФТ-7/ДОП-3); сверхдлинный хвост входа не структурируется
  (усечение). Достижимость Cloudflare с ivahub подтверждается приёмкой деплоем (ОВ-1/ДОП-1) — если
  не подтвердится, выбор дефолтного провайдера пересматривается отдельно.
- **Технический долг:** разбивка/склейка длинного текста, пользовательские лимиты размера входа,
  индикация прогресса и понятные сообщения о таймаутах/ошибках — v0.6.0 (ОВ-6). Боевое включение
  Groq и/или relay-прокси под Groq — отдельная будущая история (вне рамок). Кэширование/дедуп
  запросов под суточный лимит — не делается.

## Влияние на код

- **Новые модули:** `services/structure.py` (оркестратор: промпт, усечение, выбор провайдера по
  `LLM_PROVIDER`, async-вызов через `httpx`, фолбэк на сырой текст; реализации Cloudflare-дефолт и
  Groq-опция в этом же модуле или подмодуле провайдеров — на усмотрение разработчика, но с явной
  абстракцией провайдера).
- **Изменяемые модули:** `handlers/image.py` и `handlers/audio.py` — на непустом сыром результате
  вызвать `structure_text(...)` перед `message.answer(...)`; на пустом — прежнее поведение.
  `config.py` — чтение `LLM_PROVIDER`/`CF_ACCOUNT_ID`/`CF_API_TOKEN`/`CF_MODEL`/`GROQ_API_KEY`
  (без обязательной валидации, в отличие от `BOT_TOKEN`).
- **Зависимости:** `requirements.txt` += `httpx`. Вендорские SDK (Cloudflare/Groq) **не**
  добавляются.
- **Артефакты поставки:** `.env.example` и `deploy/.env.example` += плейсхолдеры `LLM_PROVIDER`,
  `CF_ACCOUNT_ID`, `CF_API_TOKEN`, `CF_MODEL`, `GROQ_API_KEY`. `Dockerfile` — без новых системных
  пакетов (httpx — pure-python; ставится в builder-стейдж как обычная Python-зависимость).
  `deploy/docker-compose.yml` (профиль/`extra_hosts`/отсутствие `ports:`) и
  `.github/workflows/deploy.yml` — без структурных изменений.
- **Что ломается:** публичный контракт каналов **не ломается** — на вход те же фото/аудио, на
  выходе тот же распознанный текст, но оформленный (при доступности LLM) или сырой (фолбэк).
  Изменение — обратимо-совместимое (minor). Внутренние сигнатуры `recognize_text`/`transcribe`
  не меняются.
