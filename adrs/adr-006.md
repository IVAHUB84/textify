# ADR-006: Учёт пользователей через outer-middleware aiogram и первое персистентное состояние (SQLite в volume), команда /stats для администратора

- **Дата:** 2026-06-22
- **Статус:** принято
- **Связанные BA-требования:** `ba-req/ba-requirements-006.md`
- **Связанные ADR:** ADR-001 (профиль поставки «бот без HTTP-порта»; фиксировал «боту состояние не нужно» — расширяется этим ADR), ADR-002 / ADR-003 (граница «хендлер ↔ сервис», вынос блокирующих операций из event loop через `asyncio.to_thread`), ADR-005 (структурирование)

## Контекст

Релизы v0.1.0–v0.5.0 в проде на ivahub: бот распознаёт фото (OCR), аудио (whisper) и
структурирует текст (Cloudflare Workers AI). До сих пор бот **stateless** — ADR-001 прямо
зафиксировал «боту состояние не нужно» и отбросил общие stateful-сервисы (PostgreSQL/Redis).

Релиз v0.6.0 требует видимости аудитории: сколько уникальных пользователей и как они используют
бота. Telegram **не отдаёт** число пользователей бота через API — считать обращения нужно самим.
Из этого вытекают силы и ограничения:

- **Учёт должен быть сквозным и не трогать хендлеры.** Факт обращения нужно фиксировать один раз
  на сообщение, независимо от того, какой из четырёх каналов (фото, аудио, текст, команды) его
  обработает. Правка каждого хендлера дублировала бы логику и нарушала бы границы ADR-002/003.
- **Появляется состояние.** Счётчики и времена обращений нужно хранить **между передеплоями** —
  контейнер пересоздаётся на каждом релизе. Это первое персистентное состояние продукта; формально
  противоречит формулировке ADR-001 и требует явной фиксации (BA ОВ-5).
- **Доступ к хранилищу блокирующий.** SQLite через стандартный `sqlite3` — синхронный и блокирует
  event loop. По принципу ADR-002/003 блокирующие вызовы выносятся из loop.
- **Конкурентный доступ.** Записи из middleware идут на каждое сообщение (частые), чтение — из
  `/stats` (редкое). Нужна корректность при параллельных обращениях из разных потоков `to_thread`.
- **Контейнер запускается не от root** (`appuser`, ADR-001 «по возможности non-root»; в Dockerfile
  уже есть `useradd appuser`). Хостовый каталог volume создаётся root → без явной настройки прав
  `appuser` не сможет писать в БД, и персистентность не заработает.
- **Профиль поставки ADR-001 сохраняется** (long-polling, без `ports:`/Traefik/`edge`, образ из
  GHCR, `restart: unless-stopped`, `extra_hosts`-пиннинг Telegram). Расширяется ровно одним
  элементом — локальным файловым volume; внешних БД/Redis/портов не добавляется.

## Рассматриваемые варианты

### Решение 1. Где фиксировать факт обращения

#### Вариант A: outer-middleware aiogram на observer messages
- Плюсы: один перехват на сообщение **до** роутинга по хендлерам; существующие хендлеры (OCR,
  audio, text, commands) не меняются; тип сообщения классифицируется централизованно по объекту
  `Message`; учёт полностью изолирован от бизнес-логики (соблюдает границы ADR-002/003).
- Минусы: middleware видит «сырое» сообщение и должна сама классифицировать тип, не зная, какой
  хендлер сработает (граничные случаи — ниже, Решение 2).
- Последствия: новый слой `middlewares/` (или модуль), регистрируемый на `dp.message.outer_middleware`.

#### Вариант B: править каждый хендлер (вызывать учёт внутри)
- Минусы: дублирование вызова учёта в 4+ хендлерах; правка стабильных модулей; легко забыть новый
  хендлер; смешение бизнес-логики и учёта. Нарушает «не трогать существующие хендлеры» (ФТ-1) и
  принцип минимальных изменений.
- Последствия: размазанная ответственность, хрупкость.

#### Вариант C: inner-middleware / флаги на роутерах
- Минусы: inner-middleware срабатывает только если сообщение дошло до конкретного хендлера; учёт
  пропустит сообщения, не подхваченные ни одним хендлером, и потребует регистрации на каждом
  роутере. Сложнее и менее полно, чем outer на observer.
- Последствия: неполный и более сложный учёт.

### Решение 2. Классификация типа сообщения в middleware

#### Вариант A: классифицировать по полям `Message` + сопоставить с реальными хендлерами
- Логика (приоритетный порядок, совпадает с фильтрами хендлеров):
  1. `message.text` начинается с `/` (это команда) → тип **command**;
  2. `message.photo` → **photo**;
  3. `message.document` с `mime_type`, начинающимся на `image/` → **photo** (image-документ
     обрабатывается OCR-каналом, ADR-002);
  4. `message.voice` или `message.audio`, либо `message.document` с `mime_type` на `audio/` →
     **audio** (ADR-003);
  5. прочий `message.text` (не команда) → **text**;
  6. всё остальное (стикеры, видео, прочие документы, локации и т. п.) → **other**.
- Плюсы: один счётчик соответствует одному реальному каналу обработки; классификация детерминирована
  и проверяема юнит-тестами; покрывает граничные случаи (image/audio-документы) корректно.
- Минусы: правила нужно держать согласованными с фильтрами хендлеров при будущих изменениях.

#### Вариант B: четыре счётчика без «other», неклассифицированное игнорировать
- Минусы: теряется видимость нетипового трафика; «прочее» молча пропадает. BA допускает уточнение
  (ДОП-2/ОВ-3), но явный счётчик «other» информативнее и не стоит почти ничего.

Выбран **A**: четыре содержательных счётчика (photo, audio, text, command) + **other** для
нетипового. Image-документ относим к **photo**, audio-документ — к **audio** (согласовано с
фактическими хендлерами).

### Решение 3. Расширение профиля поставки: где и как хранить состояние

#### Вариант A: SQLite-файл в примонтированном volume (выбран)
- Плюсы: один файл, ноль внешних зависимостей и портов; данные переживают передеплой через
  bind-mount `/opt/apps/textify/data:/app/data`; адекватно объёмам (один-два активных пользователя);
  минимальное и узкое расширение профиля ADR-001 — без внешней БД/Redis, без сети.
- Минусы: появляется состояние и эксплуатационное предусловие (права на каталог); SQLite требует
  аккуратности при конкурентном доступе (Решение 4).
- Последствия: ADR-001 «боту состояние не нужно» **осознанно расширяется** (см. ниже).

#### Вариант B: внешний PostgreSQL/Redis
- Минусы: на ivahub нет общих БД/Redis (ADR-001); поднимать БД ради двух пользователей — грубая
  избыточность; противоречит «минимальным изменениям» и духу профиля поставки.
- Последствия: преждевременная инфраструктура.

#### Вариант C: файл JSON в volume
- Минусы: нет транзакционности и атомарности инкрементов при конкурентных записях; ручная
  сериализация и блокировки; SQLite даёт это «из коробки».
- Последствия: самодельная конкурентность, риск гонок/повреждения файла.

### Решение 4. Модель доступа к SQLite при конкуренции (запись из middleware + чтение из /stats)

#### Вариант A: соединение-на-операцию + WAL (выбран)
- Каждая операция (`record_message`, `get_stats`) открывает короткоживущее соединение `sqlite3`
  внутри `asyncio.to_thread`, выполняет запрос, коммитит и закрывает. БД в режиме **WAL**
  (`PRAGMA journal_mode=WAL`) с `busy_timeout` (например 5000 мс).
- Плюсы: WAL допускает одновременное чтение во время записи; короткие соединения не удерживают
  блокировку; `busy_timeout` сглаживает редкие конфликты записи; запись атомарна за счёт UPSERT в
  одной транзакции; код прост, без разделяемого изменяемого состояния между потоками.
- Минусы: накладные расходы на открытие соединения на операцию (пренебрежимы при текущих объёмах);
  WAL создаёт вспомогательные файлы `-wal`/`-shm` в каталоге данных (нормально для bind-mount).
- Последствия: каждая функция storage самодостаточна; потокобезопасность обеспечивает сам SQLite.

#### Вариант B: одно долгоживущее соединение с сериализацией через `asyncio.Lock`/мьютекс
- Минусы: единый замок сериализует чтение и запись (даже редкое `/stats` блокирует записи и
  наоборот); общее изменяемое соединение между потоками `to_thread` требует `check_same_thread=False`
  и аккуратной синхронизации; сложнее и более хрупко, чем короткие соединения + WAL.
- Последствия: лишняя сериализация и общее состояние.

Выбран **A**: соединение-на-операцию + WAL — проще, корректнее при текущей нагрузке и не вводит
разделяемого состояния.

### Решение 5. Права на каталог данных для непривилегированного `appuser`

#### Вариант A: фиксированный UID/GID у `appuser` + chown каталога на хосте (выбран)
- В `Dockerfile` `appuser` создаётся с **фиксированным UID/GID** (например `10001:10001`). На сервере
  каталог `/opt/apps/textify/data` создаётся и переводится во владение этим UID:GID
  (`chown 10001:10001`). Контейнерный процесс пишет в БД как владелец.
- Плюсы: детерминированно и переносимо; не требует root внутри контейнера; явный, проверяемый шаг
  деплоя; устойчиво к пересозданию контейнера.
- Минусы: эксплуатационное предусловие (шаг в чек-листе мейнтейнера); при смене UID нужно повторить
  chown.
- Последствия: Dockerfile фиксирует UID/GID; чек-лист деплоя получает обязательный шаг chown.

#### Вариант B: запускать контейнер от root
- Минусы: противоречит «non-root по возможности» (ADR-001, контракт поставки); ослабляет
  безопасность ради удобства. Отклонён.

#### Вариант C: `chmod 0777` на каталог
- Минусы: мировой доступ на запись к данным; небезопасно. Отклонён в пользу chown на конкретный UID.

### Решение 6. Авторизация /stats и поведение для не-админа

- Админ определяется ENV `ADMIN_USER_ID` (целое, Telegram id), читается в `config.py`, **не
  хардкодится**; добавляется в `.env.example` и `deploy/.env.example` как контракт. Текущий админ —
  `33030141`.
- Поведение для не-админа: **короткий ответ «Команда недоступна.»** (BA ОВ-2, дефолт). Статистика
  не раскрывается. Выбор в пользу явного ответа (а не молчания) — предсказуемее для UX и проще
  тестируется; не несёт утечки данных.
- Если `ADMIN_USER_ID` не задан/пуст — `/stats` недоступна никому (нет валидного админа), чтобы
  ошибка конфигурации не открывала статистику.

### Решение 7. Политика сбоя записи учёта

- Запись учёта — **вспомогательный, не критичный** путь. Вся работа middleware с хранилищем
  оборачивается в `try/except`: при любой ошибке (БД недоступна, нет прав, блокировка) исключение
  логируется и **проглатывается**, после чего управление передаётся дальше по цепочке
  (`await handler(...)`). Основная обработка (OCR/аудио/текст/ответ пользователю) выполняется
  штатно. Тихая деградация: пользователь ничего не замечает, теряется только инкремент статистики.

## Решение

1. **Учёт через outer-middleware** на `dp.message.outer_middleware` (Решение 1A). Новый слой
   фиксирует факт обращения один раз на сообщение и передаёт управление дальше; существующие
   хендлеры не меняются.
2. **Классификация типа** по полям `Message` в порядке command → photo (фото + image-документ) →
   audio (voice/audio + audio-документ) → text → other (Решение 2A). Пять счётчиков.
3. **Первое персистентное состояние — SQLite в volume** (Решение 3A). Осознанное расширение профиля
   поставки ADR-001 (см. ниже).
4. **Соединение-на-операцию + WAL + `busy_timeout`** (Решение 4A); все вызовы — через
   `asyncio.to_thread` (преемственность ADR-002/003).
5. **Фиксированный UID/GID `appuser` (10001:10001) + chown каталога на хосте** (Решение 5A) —
   эксплуатационное предусловие деплоя.
6. **Авторизация `/stats` по `ADMIN_USER_ID`** из окружения; не-админу — «Команда недоступна.»;
   пустой `ADMIN_USER_ID` → недоступна никому (Решение 6).
7. **Тихая деградация при сбое учёта** через `try/except` + лог в middleware (Решение 7).
8. **Границы модулей.** Классификация и перехват — в `middlewares/stats.py`; вся работа с SQLite —
   в `services/stats.py`; команда вывода — в `handlers/commands.py` (`/stats`). Middleware не знает
   про SQLite-детали (вызывает сервис); сервис не знает про aiogram. Чтение `ADMIN_USER_ID` и пути
   к БД — в `config.py`.

### Осознанное расширение профиля поставки ADR-001 (фиксируется этим ADR)

ADR-001 зафиксировал «боту состояние не нужно» (в контексте отказа от внешних PostgreSQL/Redis на
`10.0.0.20`). Релиз v0.6.0 **вводит первое персистентное состояние** — локальный SQLite-файл в
bind-mount volume. Это **не отмена** ADR-001, а узкое осознанное расширение профиля «бот без
HTTP-порта»:

- **СОХРАНЯЕТСЯ:** long-polling; нет `ports:`/Traefik/`edge`; нет внешних БД/Redis/сети за
  состоянием; образ из GHCR; `restart: unless-stopped`; `extra_hosts`-пиннинг Telegram; non-root.
- **ДОБАВЛЯЕТСЯ ровно одно:** локальный файловый volume `/opt/apps/textify/data:/app/data` для
  SQLite и одна env-переменная `ADMIN_USER_ID`.

Будущим агентам: volume — не нарушение ADR-001, а зафиксированное здесь расширение. Внешних
stateful-сервисов по-прежнему нет.

### Схема хранилища

Одна таблица (агрегаты по пользователю; суммарные метрики `/stats` вычисляются SQL-агрегацией):

```sql
CREATE TABLE IF NOT EXISTS user_stats (
    user_id     INTEGER PRIMARY KEY,           -- Telegram id, без иных персональных данных
    first_seen  TEXT NOT NULL,                 -- ISO-8601 UTC, ставится один раз при создании
    last_seen   TEXT NOT NULL,                 -- ISO-8601 UTC, обновляется при каждом обращении
    photo_count    INTEGER NOT NULL DEFAULT 0,
    audio_count    INTEGER NOT NULL DEFAULT 0,
    text_count     INTEGER NOT NULL DEFAULT 0,
    command_count  INTEGER NOT NULL DEFAULT 0,
    other_count    INTEGER NOT NULL DEFAULT 0
);
```

- Время — **UTC, ISO-8601** (`datetime.now(timezone.utc)`); вывод `/stats` — также UTC (с пометкой).
- `record_message(user_id, msg_type)` — атомарный UPSERT: при отсутствии строки создаёт её с
  `first_seen=last_seen=now` и `<тип>_count=1`; при наличии — `last_seen=now`, инкремент нужного
  счётчика. `first_seen` **не перезаписывается** (идемпотентность первого обращения).
- `get_stats()` агрегирует: `COUNT(*)` уникальных, `SUM` по каждому счётчику и общая сумма,
  `MIN(first_seen)` и `MAX(last_seen)` по всей аудитории. **Содержимое сообщений не хранится**
  (НФТ-1/ФТ-9).

## Последствия

- **Положительные:** владелец получает видимость аудитории через `/stats`; учёт сквозной и не
  трогает бизнес-логику; первое состояние введено узко и переносимо (один файл, один volume);
  доступ к БД неблокирующий и корректный при конкуренции (WAL); сбой учёта не влияет на основную
  функцию; приватность сохранена (только `user_id` и агрегаты).
- **Отрицательные / компромиссы:** появляется эксплуатационное предусловие (каталог данных + chown
  на UID 10001) — без него запись в БД невозможна (риск Р-1); профиль поставки усложняется одним
  volume; правила классификации типа нужно держать согласованными с фильтрами хендлеров.
- **Технический долг:** один администратор (не список); агрегаты по всей аудитории (без срезов по
  периодам/персонам); нет веб-дашборда/экспорта — всё это вне рамок v0.6.0.

## Влияние на код

- **Новые модули:** `middlewares/__init__.py`, `middlewares/stats.py` (outer-middleware: классификация
  типа, вызов `record_message` в `try/except`, передача управления дальше); `services/stats.py`
  (инициализация БД/WAL, `record_message`, `get_stats` — всё через `asyncio.to_thread`).
- **Изменяемые модули:** `bot.py` (регистрация outer-middleware на `dp.message`, инициализация
  схемы БД при старте); `handlers/commands.py` (хендлер `/stats` с проверкой `ADMIN_USER_ID` и
  выводом сводки); `config.py` (чтение `ADMIN_USER_ID` и пути к файлу БД, опционально через ENV
  `STATS_DB_PATH` с дефолтом внутри `/app/data`).
- **Зависимости:** новых Python-пакетов нет (`sqlite3` — стандартная библиотека).
- **Артефакты поставки:** `Dockerfile` — фиксированный UID/GID у `appuser` (10001:10001);
  `deploy/docker-compose.yml` += `volumes: /opt/apps/textify/data:/app/data` (профиль в остальном
  без изменений: без `ports:`/Traefik/`edge`); `deploy/.env.example` и `.env.example` += плейсхолдер
  `ADMIN_USER_ID`. `.github/workflows/deploy.yml` — без изменений.
- **Что ломается:** публичный контракт каналов (фото/аудио/текст) не ломается — это minor.
  Добавляется команда `/stats` (для не-админа — отказ). Новое эксплуатационное предусловие (volume +
  права) — обязательно для персистентности.
