# ADR-007: Единая надёжная отдача длинного текстового результата (сплиттер + переход на файл), plain-text без parse_mode

- **Дата:** 2026-06-23
- **Статус:** принято
- **Связанные BA-требования:** `ba-req/ba-requirements-007.md`
- **Связанные ADR:** ADR-001 (профиль поставки «бот без HTTP-порта» — сохраняется без изменений), ADR-002 / ADR-003 (граница «хендлер ↔ сервис», неблокирующие операции в event loop), ADR-005 (структурирование текста в Markdown — формат отдаваемого результата), ADR-006 (образец стиля, преемственность профиля поставки и неблокирующих операций)

## Контекст

Релизы v0.1.0–v0.6.0 в проде на ivahub: бот распознаёт фото (OCR), аудио (whisper), структурирует
текст (Cloudflare Workers AI, Groq — опция) и ведёт учёт пользователей. Все пути вывода результата
отдают ответ **одним** вызовом `message.answer(...)`:

- `handlers/image.py` — `await message.answer(await structure_text(text))` (фото и image-документ);
- `handlers/audio.py` — `_handle_audio_bytes` → `await message.answer(await structure_text(text))`
  (voice, audio, audio-документ).

Telegram **жёстко ограничивает текстовое сообщение 4096 символами** (UTF-16 code units). На длинных
входах (развёрнутая речь, плотный скан, объёмный структурированный результат) Bot API возвращает
ошибку отправки `message is too long`, и результат **теряется** — именно на самых ценных длинных
входах. Это латентный дефект надёжности.

Действующие силы и ограничения:

- **Лимит 4096 — твёрдый** (ОГР-1). Нужен запас от границы: фактический подсчёт у Telegram — в
  UTF-16 code units, а Python `len(str)` считает code points; для строк с эмодзи/символами вне BMP
  это расходится. Запас закрывает это расхождение без введения точного UTF-16-подсчёта.
- **Серия быстрых сообщений упирается в rate limit** Telegram (ОГР-2) → нужна пауза между частями.
- **Блокирующие операции выносятся из event loop** (ADR-002/003, ОГР-4). Здесь блокирующих операций
  нет: разбивка строки и формирование `BytesIO` — лёгкий CPU, паузы — это `await asyncio.sleep`,
  отправка — `await`. Event loop не блокируется.
- **Нет новых внешних зависимостей** (ОГР-5): файл `.txt` формируется штатным `BufferedInputFile`
  из памяти.
- **Профиль поставки ADR-001 сохраняется** (ОГР-3): long-polling, без `ports:`/Traefik/`edge`,
  образ из GHCR, `restart: unless-stopped`. Изменения релиза — только код хендлеров и новый сервис;
  артефакты поставки не меняются.
- **Текущее поведение по разметке:** ответ уходит **без `parse_mode`**. ADR-005 заявляет Markdown
  как формат результата, но фактически Telegram его не парсит — пользователь видит сырой Markdown
  как читаемую структуру (заголовки `#`, списки `-`). Любая разбивка должна не разрушать эту
  визуальную структуру.
- **Дублирование логики.** Один и тот же `message.answer(...)` повторяется в трёх местах. Логику
  отдачи нужно собрать в одну переиспользуемую точку (НФТ-7), чтобы будущие пути вывода получали
  корректное поведение по длине автоматически.

## Рассматриваемые варианты

### Решение 1. Режим разметки при отдаче: plain-text vs включить parse_mode

#### Вариант A: оставить plain-text (без `parse_mode`) — как сейчас
- Плюсы: разбивка **безопасна** — не нужно отслеживать незакрытые markdown-сущности на границе
  части (иначе при `parse_mode` обрыв на `**bold` или внутри списка/код-блока даёт ошибку парсинга
  «can't parse entities» или порчу форматирования); поведение результата для пользователя не
  меняется (нулевой регресс визуала); сплиттер становится чистой строковой функцией без знания о
  разметке. Соответствует ДОП-1: «не ломать разметку» = «не рвать визуальную структуру».
- Минусы: Markdown остаётся сырым (заголовки/списки видны как символы `#`/`-`), а не «красивым»
  форматированием Telegram. Это **status quo**, релиз его не ухудшает.
- Последствия: сплиттер режет по строковым границам; маркеры разметки трактуются как обычный текст.

#### Вариант B: включить `parse_mode=MarkdownV2`/`HTML` при отдаче
- Плюсы: «красивое» форматирование в Telegram.
- Минусы: LLM-вывод не гарантирует валидный MarkdownV2 (требует экранирования спецсимволов
  `_ * [ ] ( ) ~ ` > # + - = | { } . !`); при сбое экранирования Bot API падает с ошибкой парсинга
  — **новый класс отказов прямо в релизе про надёжность**; разбивка должна гарантировать, что
  каждая часть — самостоятельно валидный документ (закрытые сущности), что резко усложняет сплиттер
  и противоречит цели «дешёвый фикс». Выходит за scope BA (релиз про **доставку**, не про парсинг).
- Последствия: новая точка отказа, сложный сплиттер; противоречит цели релиза.

### Решение 2. Стратегия разбивки длинного текста

#### Вариант A: иерархический жадный сплиттер по убыванию приоритета границ
Резать по приоритету: двойной перевод строки (абзац) → одиночный перевод строки → конец предложения
→ жёстко по лимиту (крайний случай). Жадно набирать максимально длинные части ≤ лимита, не разрывая
слова.
- Плюсы: естественные границы (НФТ-3, ФТ-5); абзацная структура и списки не рвутся посередине;
  гарантия `len(part) ≤ LIMIT` за счёт финального жёсткого среза для патологических входов
  (одно «слово»/строка длиннее лимита); чистая функция без сети — легко тестируется.
- Минусы: чуть сложнее наивного `text[i:i+LIMIT]`.
- Последствия: одна функция-сплиттер, тестируемая изолированно.

#### Вариант B: наивная нарезка `text[i:i+LIMIT]`
- Плюсы: тривиально.
- Минусы: рвёт слова и markdown-элементы посередине (нарушает ФТ-5, НФТ-3); нечитаемо.
- Последствия: не проходит критерии приёмки BA.

### Решение 3. Куда поместить логику отдачи

#### Вариант A: новый модуль `services/reply.py` с `async def send_result(message, text)`
- Плюсы: единая точка (НФТ-7, ФТ-1); хендлеры импортируют одну функцию вместо дублирования;
  чистый сплиттер выделяется отдельной функцией и тестируется без Telegram; согласуется с границей
  «хендлер ↔ сервис» (ADR-002/003) — отдача результата это инфраструктурная операция, не бизнес-логика.
- Минусы: новый модуль.
- Последствия: `handlers/image.py` и `handlers/audio.py` заменяют `message.answer(result)` на
  `send_result(message, result)`.

#### Вариант B: утилита внутри `handlers/` или хелпер в каждом хендлере
- Минусы: либо дублирование, либо размывание ответственности хендлеров; сплиттер тяжелее
  тестировать в отрыве от Telegram-типов.
- Последствия: нарушение НФТ-7.

### Решение 4. Граница перехода «несколько сообщений → один файл .txt»

#### Вариант A: переход на файл по числу частей (например, > 5 частей)
- Плюсы: прямой контроль анти-спама и анти-rate-limit — ограничивает максимум сообщений в серии;
  число пауз ограничено сверху.
- Минусы: порог в «частях» зависит от лимита.
- Последствия: естественная связка со сплиттером (сначала режем, считаем части).

#### Вариант B: переход на файл по абсолютной длине символов (например, > 20000)
- Плюсы: простой одиночный порог.
- Минусы: не контролирует напрямую число сообщений/пауз.
- Последствия: эквивалентно варианту A при согласованных значениях.

## Решение

1. **Plain-text без `parse_mode`** (Решение 1, вариант A). Результат отдаётся как обычный текст, как
   сейчас. Сплиттер **не обязан** учитывать markdown-сущности — маркеры разметки трактуются как
   обычный текст; «не ломать разметку» = «не рвать визуальную структуру (абзацы/строки/слова)».
   Включение `parse_mode` явно отложено (ОВ-1 закрыт в пользу status quo).
2. **Иерархический жадный сплиттер** (Решение 2, вариант A): приоритет границ
   абзац (`\n\n`) → строка (`\n`) → предложение → жёсткий срез по лимиту. Каждая часть ≤ `MAX_MESSAGE_LEN`.
   Полнота: конкатенация частей восстанавливает исходный текст по содержимому без потерь и дублей
   (допустима только нормализация ведущих/замыкающих пробелов на стыках).
3. **Новый модуль `services/reply.py`** (Решение 3, вариант A): публичная корутина
   `async def send_result(message: Message, text: str) -> None` + чистая функция-сплиттер
   `split_text(text: str, limit: int = MAX_MESSAGE_LEN) -> list[str]`. Хендлеры вызывают
   `send_result` вместо прямого `message.answer(result)`.
4. **Три режима по длине** в `send_result`:
   - `len(text) <= MAX_MESSAGE_LEN` → одно `message.answer(text)` (поведение не меняется);
   - иначе `split_text(...)`; если число частей `<= MAX_PARTS` → отправлять части последовательно,
     между соседними частями `await asyncio.sleep(PART_DELAY_SEC)` (анти-rate-limit);
   - если частей `> MAX_PARTS` → один файл `.txt` через `BufferedInputFile(text.encode("utf-8"), filename=...)`
     с короткой русской подписью; множество сообщений в этом случае не шлётся.
5. **Контракт входа `send_result`.** Принимает **уже непустой** результат. Решение о
   «не распознано»/«речь не распознана» остаётся в хендлерах (короткий `message.answer(...)`), как
   сейчас — служебные/короткие сообщения **через `send_result` не проходят** и не разбиваются
   (ФТ-10). Для устойчивости `send_result` на пустой/пробельной строке деградирует тихо
   (ничего не шлёт или один пустой-эквивалент — реализация на усмотрение, но без падения);
   штатно такой вызов не делается.
6. **Именованные константы** (ФТ-8) в `services/reply.py`:
   - `TELEGRAM_LIMIT = 4096` — жёсткий лимит Telegram (для справки/защиты);
   - `MAX_MESSAGE_LEN = 4000` — безопасный лимит части (запас от 4096 под UTF-16-расхождение и
     неточности);
   - `MAX_PARTS = 5` — порог перехода на файл по числу частей;
   - `PART_DELAY_SEC = 0.4` — пауза между частями (диапазон 0.3–0.5).
7. **Неблокирующесть** (НФТ-4, ADR-002/003): пауза — `await asyncio.sleep`, отдаёт управление
   event loop; число пауз ограничено сверху `MAX_PARTS - 1` (далее — файл, без серии пауз);
   формирование `BytesIO`/нарезка — лёгкий CPU. Обработка других апдейтов не блокируется.
8. **Устойчивость серии** (ОВ-4): отправка частей выполняется так, чтобы исключение на одной части
   не «роняло» бота (общий принцип проекта — без падения хендлера); конкретика обработки ошибок
   отправки — на усмотрение реализации, по умолчанию ошибка логируется и не прерывает обработку
   других апдейтов. Ретраи в scope не входят.
9. **Профиль поставки ADR-001 — без изменений.** Новых env-переменных, портов, зависимостей,
   изменений Docker/compose нет.

## Последствия

- **Положительные:** длинный результат гарантированно доходит целиком (НФТ-1, КП-10);
  единая переиспользуемая точка отдачи (НФТ-7, Сценарий E); чистый сплиттер покрыт юнит-тестами без
  сети; анти-rate-limit и анти-спам через паузу и переход на файл; нулевой регресс короткого
  ответа и служебных сообщений; новых зависимостей и изменений поставки нет.
- **Отрицательные / компромиссы:** Markdown остаётся сырым (не «красивое» форматирование Telegram) —
  это сознательный отказ от `parse_mode` ради надёжности; жёсткий срез как крайний случай может
  один раз разорвать аномально длинное «слово»/строку без естественной границы (неизбежно при
  отсутствии границ внутри лимита).
- **Технический долг:** «красивое» форматирование через `parse_mode` с экранированием и
  border-safe-разбивкой сущностей — отдельный будущий ADR, если потребуется. Точный UTF-16-подсчёт
  длины не вводится — закрыт запасом `MAX_MESSAGE_LEN`; при будущих проблемах с экзотическими
  символами можно ужесточить.

## Влияние на код

- **Новый модуль:** `services/reply.py` — `MAX_MESSAGE_LEN`/`MAX_PARTS`/`PART_DELAY_SEC`/
  `TELEGRAM_LIMIT`, `split_text(...)` (чистая функция), `send_result(message, text)` (корутина).
- **Изменяются:** `handlers/image.py` и `handlers/audio.py` — замена
  `await message.answer(await structure_text(...))` на `await send_result(message, await structure_text(...))`;
  служебные `NO_TEXT_MESSAGE`/`NO_SPEECH_MESSAGE` остаются на прямом `message.answer(...)`.
  Двойной отправки быть не должно: `structure_text` вызывается в хендлере, его результат
  передаётся в `send_result` ровно один раз.
- **Не изменяются:** `bot.py` (роутеры/middleware без изменений), `services/structure.py`,
  `services/ocr.py`, `services/transcribe.py`, `handlers/text.py`, `handlers/commands.py`,
  артефакты поставки (`Dockerfile`, `deploy/*`), `requirements.txt`.
- **Новые зависимости:** нет (`BufferedInputFile`, `BytesIO`, `asyncio` — штатные).
- **Что ломается:** ничего; короткий путь и служебные сообщения сохраняют поведение.
