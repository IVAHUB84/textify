# ADR-008: Облачная транскрипция аудио через Cloudflare Workers AI (whisper-large-v3-turbo) с абстракцией провайдера, переключателем ASR_PROVIDER и фолбэком на локальный faster-whisper

- **Дата:** 2026-06-23
- **Статус:** принято
- **Связанные BA-требования:** `ba-req/ba-requirements-008.md`
- **Связанные ADR:** ADR-001 (профиль поставки «бот без HTTP-порта», секреты через `.env` — сохраняется без изменений), ADR-003 (локальная транскрипция faster-whisper, неблокирующий CPU-bound сервис через `asyncio.to_thread`, веса в образе — **дополняется**: локальная модель становится фолбэком/режимом), ADR-005 (Cloudflare Workers AI для структурирования: аккаунт/токен, прямой HTTP через `httpx`, единичная попытка + таймаут + фолбэк — **переиспользуется как образец** для облачного провайдера ASR), ADR-007 (единая отдача результата через `send_result` — путь вывода не меняется)

## Контекст

Релизы v0.1.0–v0.7.0 в проде на ivahub. Транскрипция аудио сейчас выполняется **локальной** моделью
faster-whisper `base` (`int8`, CPU; ADR-003) на сервере с 1 vCPU / ~1.9 GB RAM. Это главное узкое
место продукта: распознавание медленное (десятки секунд на 1 vCPU), качество `base` ограничено, а
CPU-bound вызов нагружает единственное ядро и память, конкурируя с остальной работой бота.

В продукте уже работает инфраструктура **Cloudflare Workers AI** для структурирования (ADR-005):
тот же аккаунт/токен (`CF_ACCOUNT_ID` / `CF_API_TOKEN`), достижимость с RU-IP ivahub без прокси,
прямой async HTTP через `httpx` (уже зависимость). Эндпоинт транскрипции Cloudflare Whisper
**подтверждён вживую** с ivahub (HTTP 200).

Цель v0.8.0 — перенести **основную** транскрипцию аудио на Cloudflare Workers AI
`whisper-large-v3-turbo` (точнее, мультиязычнее, быстрее, снимает CPU/RAM-нагрузку с сервера),
оставив локальный faster-whisper **фолбэком** на любой сбой облака и **самостоятельным режимом**
при `ASR_PROVIDER=local`. Подтверждённый контракт вызова:

```
POST https://api.cloudflare.com/client/v4/accounts/{CF_ACCOUNT_ID}/ai/run/{модель}
Authorization: Bearer {CF_API_TOKEN}
тело JSON: {"audio": "<base64 аудио>"}
ответ: result.text (распознанный текст), result.transcription_info.language (язык)
```

Действующие силы и ограничения:

- **Образец абстракции — ADR-005.** Структурирование уже разделено на оркестратор + провайдеры с
  выбором по env. Транскрипцию надо привести к тому же приёму: оркестратор «аудио → текст» +
  реализации Cloudflare/локальная + выбор по `ASR_PROVIDER`. Граница «хендлер ↔ сервис» (ADR-002/003)
  сохраняется: `handlers/audio.py` не знает о провайдере, `services/transcribe.py` не знает об aiogram.
- **Два разных класса вызова в одном сервисе.** Облачный путь — I/O-bound, нативный async через
  `httpx` (как ADR-005). Локальный путь — CPU-bound, `asyncio.to_thread` поверх синхронного
  faster-whisper (как ADR-003). Сервис должен совмещать оба, не блокируя event loop ни в одном.
- **Зависимость от внешнего бесплатного API.** Доступность, суточные лимиты бесплатного tier и
  **максимальный размер запроса** Cloudflare вне контроля проекта. Отсюда обязательность фолбэка на
  локальную модель (ОГР-1) и обработка крупных файлов (ОГР-2).
- **base64 раздувает объём.** Cloudflare AI run принимает аудио как base64 в JSON-теле (~+33 % к
  размеру файла); крупные аудио могут не уложиться в лимит размера запроса (ОГР-2). Нужен
  детерминированный порог и стратегия.
- **Локальная модель остаётся в образе** (ADR-003 без изменений): веса `base` вшиты на этапе сборки,
  рантайм offline (`HF_HOME`, `HF_HUB_OFFLINE=1`). Образ не уменьшается — это сознательная цена
  страховки (фолбэк).
- **Раздельность переменных Cloudflare.** `CF_MODEL` (ADR-005) — модель **структурирования**;
  модель **ASR** должна жить в отдельной переменной, чтобы общий аккаунт не привёл к конфликту.
- **Секреты** (ADR-001, `CLAUDE.md`). `CF_ACCOUNT_ID` / `CF_API_TOKEN` уже введены ADR-005, есть в
  `.env` на сервере и в `deploy/.env.example`; новых **обязательных** секретов нет.
- **Профиль поставки ADR-001.** Long-polling, без `ports:`/Traefik/`edge`, образ из GHCR,
  `restart: unless-stopped`, `extra_hosts`-пиннинг Telegram. Исходящее направление к
  `api.cloudflare.com` уже используется (ADR-005) — нового направления трафика нет. `httpx` уже есть
  — новых зависимостей нет.
- **Путь вывода ADR-007 сохраняется.** Транскрипт уходит через `send_result`; интеграция провайдера
  не меняет вывод. Пустой результат → прежнее сообщение «Речь в аудио не распознана.».

## Рассматриваемые варианты

### Решение 1. Структура абстракции провайдера транскрипции

#### Вариант A: `if ASR_PROVIDER == ...` внутри одной функции `transcribe`
- Плюсы: меньше структуры.
- Минусы: ветвление провайдеров и оркестрация фолбэка размазываются по одной функции; тяжело
  тестировать выбор провайдера и фолбэк изолированно; расходится с уже принятым образцом ADR-005.
- Последствия: смешение «что делаем» и «через кого».

#### Вариант B: тонкая абстракция — оркестратор + реализации Cloudflare/локальная, выбор по `ASR_PROVIDER` (как ADR-005)
- Плюсы: `services/transcribe.py` отвечает за оркестрацию (выбор провайдера, проверку размера,
  фолбэк, логирование), провайдеры — за «как получить текст». Переключение — чистый конфиг без
  правки кода (ФТ-4). Облачный провайдер тестируется мок-патчем `httpx` без сети; локальный — мок
  `WhisperModel` без реальной модели; фолбэк тестируется как переход между ними.
- Минусы: чуть больше структуры (внутренний контракт провайдера).
- Последствия: вводится внутренний контракт «провайдер транскрипции»; дефолт — Cloudflare.

### Решение 2. Способ обращения к Cloudflare Whisper

#### Вариант A: вендорский SDK Cloudflare
- Минусы: тяжёлая зависимость ради одного POST; завязка на версии SDK; расходится с уже выбранным в
  ADR-005 прямым HTTP.
- Последствия: лишний вес образа, связность с SDK.

#### Вариант B: прямой HTTP к REST-эндпоинту через `httpx.AsyncClient` (как ADR-005)
- Плюсы: тот же приём, что и для структурирования; `httpx` уже зависимость; полный контроль над
  таймаутом и парсингом. Эндпоинт и форма тела подтверждены вживую (HTTP 200): тело
  `{"audio": "<base64>"}`, ответ `result.text` + `result.transcription_info.language`.
- Минусы: base64-кодирование и парсинг ответа пишем сами (тривиально).
- Последствия: облачный провайдер — тонкая функция «байты → base64 → POST → `result.text`».

### Решение 3. Неблокирующий механизм для двух классов вызова

#### Вариант A: оба пути через `asyncio.to_thread`
- Минусы: для сетевого облачного вызова `to_thread` держит поток пула занятым на всё время ожидания
  сети — семантически неверный инструмент (как разобрано в ADR-005, Решение 3).
- Последствия: расход потоков пула на сетевое ожидание.

#### Вариант B: облако — нативный async `httpx`; локальная модель — `asyncio.to_thread` (правильный инструмент под каждый класс)
- Плюсы: облачный путь `await client.post(...)` отдаёт управление event loop на время сети (ADR-005);
  локальный CPU-bound путь остаётся в `asyncio.to_thread` (ADR-003). Ни один путь не блокирует event
  loop. Каждый инструмент применён по назначению.
- Минусы: в одном сервисе сосуществуют две модели неблокирования (это осознанно и соответствует
  природе вызовов).
- Последствия: облачный провайдер — `async def` с `httpx`; локальный — `async def`-обёртка над
  существующим `_transcribe_sync` через `to_thread` (текущий код сохраняется).

### Решение 4. Таймаут и политика ретраев для облачного вызова

#### Вариант A: агрессивные ретраи с backoff
- Минусы: при недоступности/лимите ретраи лишь умножают задержку перед неизбежным фолбэком на
  локальную модель и расходуют лимит free-tier (как разобрано в ADR-005, Решение 4).
- Последствия: медленный фолбэк, лишняя нагрузка на лимит.

#### Вариант B: единичная попытка с явным таймаутом, без ретраев, затем фолбэк (как ADR-005)
- Плюсы: соответствует НФТ-1 (единичная попытка с явным таймаутом и фолбэком). Любой сбой → быстрый
  детерминированный фолбэк на локальную модель. Таймаут крупнее, чем у структурирования (30 с), так
  как транскрипция крупного аудио на стороне Cloudflare дольше: **60 секунд** — баланс «дать
  распознать крупный файл» и «не ждать вечно повисшее соединение».
- Минусы: на разовом транзиентном сбое сети аудио распознаётся локально (медленнее/качество `base`) —
  приемлемо, пользователь результат получает.
- Последствия: один POST с `timeout=60`, без повторов; все исключения и не-2xx ведут в фолбэк.

### Решение 5. Поведение для крупных аудио (лимит размера запроса Cloudflare, ОГР-2)

#### Вариант A: чанкинг аудио (декодирование через `av`/PyAV, несколько запросов, склейка)
- Минусы: множит число запросов (быстрее исчерпывает free-tier), усложняет склейку текста и стыки,
  добавляет латентность и точки отказа; для одного-двух пользователей не оправдан. BA допускает
  фолбэк по умолчанию (ОВ-1).
- Последствия: преждевременная сложность.

#### Вариант B: порог размера → крупные файлы сразу идут локально (без чанкинга)
- Плюсы: предсказуемо и просто; крупный файл не тратит запрос Cloudflare впустую (не упрётся в лимит
  размера) и не ждёт таймаут — сразу распознаётся локально. Один детерминированный порог-константа.
  Чанкинг отложен в технический долг.
- Минусы: крупный файл не выигрывает от облака (распознаётся локальной `base`) — приемлемо как
  страховка; редкий случай для голосовых.
- Последствия: оркестратор перед облачным путём проверяет размер исходных байтов аудио против порога;
  при превышении — сразу локальный путь (это не пользовательский лимит, файл не отвергается).

**Порог (вариант B, конкретно).** Cloudflare AI run ограничивает размер запроса; base64 раздувает
объём на ~33 %. Порог задаётся **по размеру исходных байтов аудио** (до кодирования) именованной
константой `_CF_MAX_AUDIO_BYTES = 8 * 1024 * 1024` (**8 МБ**). Обоснование: 8 МБ аудио → ~10.7 МБ
base64 в JSON-теле — с запасом под безопасный потолок размера запроса, при этом 8 МБ покрывает
подавляющее большинство голосовых/коротких аудио. Файлы крупнее порога идут локально (Решение 5B).

### Решение 6. Модель Cloudflare Whisper и отдельная переменная окружения

#### Вариант A: переиспользовать `CF_MODEL`
- Минусы: `CF_MODEL` — модель структурирования (ADR-005); общий аккаунт + одна переменная привели
  бы к конфликту назначений.
- Последствия: невозможно независимо менять ASR- и LLM-модели.

#### Вариант B: отдельная переменная `CF_WHISPER_MODEL`, дефолт `@cf/openai/whisper-large-v3-turbo`
- Плюсы: ASR-модель отделена от LLM-модели; смена модели без правки кода; дефолт — подтверждённая
  вживую `whisper-large-v3-turbo`.
- Минусы: ещё одна env-переменная (не секретная, с рабочим дефолтом).
- Последствия: `config.py` и `.env.example`/`deploy/.env.example` получают `CF_WHISPER_MODEL`.

## Решение

1. **Тонкая абстракция провайдера транскрипции** (Решение 1B, образец ADR-005). `services/transcribe.py`
   становится оркестратором: выбирает провайдера по `ASR_PROVIDER` (дефолт `cloudflare`), проверяет
   размер аудио, при облачном сбое выполняет фолбэк на локальную модель, логирует факт фолбэка.
   Реализации: облачный провайдер Cloudflare и локальный faster-whisper. Публичная корутина
   `transcribe(audio_bytes: bytes) -> str` сохраняет существующую сигнатуру (контракт для
   `handlers/audio.py` не меняется, ADR-007 путь вывода не затрагивается).

2. **Прямой HTTP к Cloudflare через `httpx`** (Решение 2B). Облачный провайдер:
   `POST https://api.cloudflare.com/client/v4/accounts/{CF_ACCOUNT_ID}/ai/run/{CF_WHISPER_MODEL}`,
   заголовок `Authorization: Bearer {CF_API_TOKEN}`, тело
   `{"audio": base64.b64encode(audio_bytes).decode()}`; результат читается из `result.text`
   (`.strip()`); язык доступен в `result.transcription_info.language` (для определения языка
   автоматически; насильно язык не переключается, НФТ-8).

3. **Два класса неблокирования** (Решение 3B). Облачный путь — нативный async `httpx.AsyncClient`
   с `await` (event loop не блокируется на сети). Локальный путь — существующий синхронный
   `_transcribe_sync` через `asyncio.to_thread` (ADR-003 без изменений: ленивый singleton
   `WhisperModel("base", ...)`, веса в образе, offline). `to_thread` для облачного пути **не**
   используется.

4. **Единичная попытка, таймаут 60 с, без ретраев, затем фолбэк** (Решение 4B). Один POST с явным
   `timeout=60`. Любой сбой облака (сеть, DNS, 4xx/5xx, таймаут, превышение лимита, некорректный/
   пустой по причине ошибки ответ, отсутствие CF-кредов) перехватывается оркестратором → фолбэк на
   локальную модель. Сбой облака **не ломает ответ** и не валит процесс; факт фолбэка логируется
   (см. ниже).

5. **Порог размера, крупные файлы — локально** (Решение 5B). Константа модуля
   `_CF_MAX_AUDIO_BYTES = 8 * 1024 * 1024`. Если `len(audio_bytes) > _CF_MAX_AUDIO_BYTES` —
   оркестратор при `ASR_PROVIDER=cloudflare` сразу идёт локальным путём, не обращаясь к Cloudflare
   (логируется причина). Чанкинг не реализуется (технический долг). Файл не отвергается — это не
   пользовательский лимит.

6. **Модель ASR — `@cf/openai/whisper-large-v3-turbo`, переменная `CF_WHISPER_MODEL`** (Решение 6B),
   **отдельная** от `CF_MODEL` структурирования.

7. **Переключатель `ASR_PROVIDER`** со значениями `cloudflare` | `local`, дефолт `cloudflare`
   (незадано/неизвестное значение → `cloudflare`). При `ASR_PROVIDER=local` оркестратор использует
   только локальную модель и **не обращается** к Cloudflare (Сценарий C). При облачном режиме без
   CF-кредов — поведение как при сбое: фолбэк на локальную модель (детерминированно, без падения).

8. **Логирование фолбэка** (НФТ-1, риск маскировки деградации). Оркестратор логирует:
   - на старте облачного пути — выбор провайдера/модель (debug/info, без секретов);
   - при любом облачном сбое — **предупреждение** (`logger.warning`/`logger.exception`) с причиной
     (тип исключения / HTTP-статус / «request too large by size threshold») и пометкой, что
     выполняется фолбэк на локальную модель. Это делает деградацию качества (turbo → `base`)
     **видимой в логах**, а не молчаливой.
   - при крупном файле (порог) — info/warning «audio exceeds CF size threshold, using local».

9. **Граница модулей и путь вывода без изменений.** `handlers/audio.py` по-прежнему вызывает
   `transcribe(audio_bytes)` и отдаёт результат через `send_result` (ADR-007); на пустом
   `result.text` — прежнее `NO_SPEECH_MESSAGE` (ФТ-11). Хендлер не знает о провайдере; сервис не
   знает об aiogram (ADR-002). `services/structure.py` (ADR-005) и `services/reply.py` (ADR-007) не
   меняются.

10. **Конфиг и секреты** (ФТ-8/ФТ-9, НФТ-5). `config.py` читает `ASR_PROVIDER` (дефолт `cloudflare`)
    и `CF_WHISPER_MODEL` (дефолт `@cf/openai/whisper-large-v3-turbo`); `CF_ACCOUNT_ID`/`CF_API_TOKEN`
    уже читаются окружением в `services/structure.py` и переиспользуются (сервис транскрипции читает
    их так же из окружения, как `structure.py`). Плейсхолдеры `ASR_PROVIDER` и `CF_WHISPER_MODEL`
    добавляются в `.env.example` и `deploy/.env.example`. Отсутствие новых переменных в `.env` на
    сервере **не валит** старт (есть рабочие дефолты; дефолт `cloudflare` включается сам). Секреты не
    хардкодятся и не коммитятся.

11. **Профиль поставки ADR-001 — без изменений** (НФТ-6). Long-polling, без `ports:`/Traefik/`edge`,
    образ из GHCR, `restart: unless-stopped`, `extra_hosts`-пиннинг. Исходящее направление к
    `api.cloudflare.com` уже есть (ADR-005). Локальная модель и её доставка в образ (ADR-003)
    сохраняются — **образ не уменьшаем**. Новых зависимостей нет: `httpx` и `faster-whisper` уже в
    `requirements.txt`; `base64` — стандартная библиотека.

12. **Roadmap** (ФТ-7 BA). Слот v0.8.0 = «Транскрипция аудио через Cloudflare Whisper
    (large-v3-turbo); локальная — фолбэк». v1.0.0 остаётся.

## Последствия

- **Положительные:** основная транскрипция — точнее (turbo), быстрее и снимает CPU/RAM-нагрузку с
  ivahub; абстракция провайдера единообразна со структурированием (ADR-005); фолбэк делает
  зависимость от внешнего API нехрупкой — пользователь получает результат при любом сбое облака;
  путь вывода (ADR-007) и контракт канала аудио не меняются; поставка не меняется (тот же пайплайн,
  новые переменные с дефолтами, без новых секретов/зависимостей/портов).
- **Отрицательные / компромиссы:** зависимость от внешнего бесплатного API (доступность, суточные
  лимиты, размер запроса вне контроля) — смягчена фолбэком и порогом размера; фолбэк **маскирует
  деградацию качества** (turbo → локальная `base`) — смягчено обязательным логированием факта
  фолбэка (Решение 8); образ не уменьшается (локальная модель остаётся как страховка); крупные
  файлы (>8 МБ) не выигрывают от облака (распознаются локально); в одном сервисе сосуществуют две
  модели неблокирования (осознанно — по природе вызовов).
- **Технический долг:** чанкинг крупного аудио для облачного пути (несколько запросов + склейка) —
  отложен (Решение 5A отклонён); ретраи/закэширование под лимиты free-tier — не делается; возможное
  будущее уменьшение образа (отказ от локальной модели) — только если откажемся от фолбэка
  (не в этом релизе).

## Влияние на код

- **Изменяемые модули:** `services/transcribe.py` — рефактор в оркестратор + провайдеры
  (Cloudflare-дефолт через `httpx`, локальный faster-whisper через `to_thread`; существующие
  `_get_model`/`_transcribe_sync` сохраняются как локальный путь), выбор по `ASR_PROVIDER`, порог
  размера `_CF_MAX_AUDIO_BYTES`, таймаут `_CF_TIMEOUT = 60.0`, фолбэк с логированием; публичная
  `transcribe(audio_bytes) -> str` сохраняет сигнатуру. `config.py` — чтение `ASR_PROVIDER`
  (дефолт `cloudflare`) и `CF_WHISPER_MODEL` (дефолт `@cf/openai/whisper-large-v3-turbo`), без
  обязательной валидации.
- **Не изменяемые модули:** `handlers/audio.py` (продолжает вызывать `transcribe` и `send_result`),
  `services/structure.py`, `services/reply.py`, `services/ocr.py`, `bot.py`, прочие хендлеры.
- **Зависимости:** `requirements.txt` — **без изменений** (`httpx` и `faster-whisper` уже есть;
  `base64` — stdlib).
- **Артефакты поставки:** `.env.example` и `deploy/.env.example` += плейсхолдеры `ASR_PROVIDER`,
  `CF_WHISPER_MODEL`. `Dockerfile` (веса whisper в образе, offline-env), `deploy/docker-compose.yml`,
  `.github/workflows/deploy.yml` — **без изменений**. Профиль поставки ADR-001 не меняется.
- **Что ломается:** публичный контракт канала аудио **не ломается** — на вход то же voice/audio/
  audio-документ, на выходе тот же транскрипт (через тот же `send_result`); меняется только
  **источник** транскрипта (облако по умолчанию, локально — фолбэк/режим). Изменение
  обратимо-совместимое (minor). Сигнатура `transcribe` сохраняется.
