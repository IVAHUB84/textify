# ADR-009: Доп-действия над результатом («Кратко» и «Перевести») через inline-кнопки и callback-хендлеры; LLM-действия над текстом видимого сообщения

- **Дата:** 2026-06-23
- **Статус:** принято
- **Связанные BA-требования:** `ba-req/ba-requirements-009.md`
- **Связанные ADR:** ADR-001 (профиль поставки «бот без HTTP-порта», секреты через `.env` — сохраняется без изменений), ADR-002 / ADR-003 (граница «хендлер ↔ сервис», неблокирующие операции в event loop), ADR-005 (CF Workers AI: `services/structure.py`, прямой async HTTP через `httpx`, таймаут/фолбэк, промпт, env `CF_*`/`LLM_PROVIDER`/`CF_MODEL` — переиспользуется паттерн и провайдер), ADR-006 (учёт через outer-middleware на `dp.message` — определяет, считаются ли callback'и в статистике), ADR-007 (`send_result` — единая отдача результата, переиспользуется и расширяется опциональным `reply_markup`)

## Контекст

Релизы v0.1.0–v0.8.0 в проде на ivahub: бот распознаёт фото (OCR), аудио (Cloudflare Whisper +
локальный фолбэк), структурирует текст через CF Workers AI (ADR-005) и надёжно отдаёт длинный
результат через `send_result` (ADR-007). Сейчас на отдаче результата взаимодействие заканчивается.

Релиз v0.9.0 добавляет **быстрые действия над уже полученным результатом**: под текстовым
результатом OCR и аудио показываются inline-кнопки **«Кратко»** (саммари в 3–5 пунктов) и
**«Перевести»** (перевод); по нажатию бот через тот же CF LLM формирует ответ над исходным текстом и
присылает его отдельным сообщением через `send_result`. LLM-слой уже работает, доступ к CF
подтверждён в проде (v0.5.0/v0.8.0) — добавление дёшево и опирается на проверенную инфраструктуру.

Действующие силы и ограничения:

- **Источник текста без хранилища (ОГР-1, ФТ-5).** BA запрещает отдельное хранилище исходных
  текстов. Единственный доступный во времени источник — `callback.message.text`, текст видимого
  сообщения, к которому прикреплены кнопки. Отсюда фундаментальное ограничение: действие возможно
  только когда **весь** исходный текст помещается в одно текстовое сообщение.
- **`send_result` умеет три режима** (ADR-007): одно сообщение / серия сообщений / файл `.txt`. В
  режимах «серия» и «файл» текст **не доступен целиком** через `callback.message.text` (несколько
  сообщений с кнопкой под последним → текст одной части; файл → у сообщения вообще нет `.text`).
  Прикреплять кнопки в этих режимах означало бы действовать над неполным/отсутствующим текстом →
  кнопки прикрепляются **только** в одно-сообщенческом режиме (ФТ-9, ОВ-6).
- **Лимит `callback_data` — 64 байта** (жёсткий лимит Bot API). Данные кнопки должны быть
  компактными; нельзя класть в них исходный текст.
- **`callback.message.text` может быть `None`/пустым** для file-сообщений и в редких случаях →
  callback-хендлер обязан это детерминированно обработать (ФТ-10, ОВ-4).
- **Переиспользование паттерна LLM (ADR-005).** Вызов CF — тем же приёмом: прямой async HTTP через
  `httpx`, секреты из окружения, явный таймаут, единичная попытка. Новый провайдер не вводится.
- **Неблокирование (ADR-002/003/005, НФТ-2).** Сетевой вызов CF и отдача — нативно async, без
  `to_thread`. Бот остаётся отзывчивым.
- **Учёт (ADR-006).** Middleware висит на `dp.message.outer_middleware` — на `Message`, не на
  `CallbackQuery`. Нажатия кнопок текущим учётом **не фиксируются**; нужно решить, расширять ли учёт.
- **Профиль поставки ADR-001, requirements.txt, env — без изменений** (`httpx` есть, CF-креды есть,
  inline-кнопки — штатный aiogram).

## Рассматриваемые варианты

### Решение 1. Прикрепление кнопок к результату: расширить `send_result` vs отдельная обёртка

#### Вариант A: расширить `send_result(message, text, reply_markup=None)` опциональным параметром
- Плюсы: единая точка отдачи (ADR-007) сохраняется; markup прикрепляется штатно к первому/единственному
  `message.answer`; хендлеры вызывают одну функцию; нет дублирования логики разбивки/файла.
- Минусы: `send_result` должна знать правило «markup только при одиночном сообщении» — но это
  естественно ложится на её же ветвление по длине.
- Последствия: `send_result` получает третий опциональный параметр; правило применяется внутри неё.

#### Вариант B: отдельная обёртка `send_result_with_actions(...)` поверх `send_result`
- Минусы: дублирует знание о режимах длины (чтобы понять, будет ли одно сообщение) либо повторяет
  ветвление; две точки отдачи; рассинхрон при будущих правках. Нарушает НФТ-9 (без дублирования).
- Последствия: размывание единой точки отдачи.

**Правило прикрепления markup (закрывает ОВ-6).** `reply_markup` передаётся в Telegram **только**
когда результат уходит **одним** сообщением (`len(text) <= MAX_MESSAGE_LEN`). При серии сообщений и
при переходе на файл — markup **игнорируется** (кнопки не показываются). Так гарантируется: если
кнопки под сообщением есть, то весь исходный текст доступен через `callback.message.text` — источник
для действия однозначен.

### Решение 2. Где разместить логику LLM-действий: расширить `services/structure.py` vs новый `services/llm.py`

#### Вариант A: новый модуль `services/llm.py` с `summarize(text)` и `translate(text)`
- Плюсы: `structure.py` остаётся узким (одна задача — структурирование); общий низкоуровневый код
  вызова провайдера (выбор провайдера, async POST, парсинг) переиспользуется; новый модуль группирует
  «действия над текстом через LLM» под будущее расширение (Сценарий G); тестируется изолированно.
- Минусы: нужно аккуратно переиспользовать `_build_provider`/`LLMProvider` из `structure.py`, не
  дублируя их.
- Последствия: `services/llm.py` импортирует провайдерный слой из `services/structure.py`; в
  `structure.py` провайдерный слой и константы (`_MAX_INPUT_CHARS`, `_TIMEOUT`) делаются доступными
  для переиспользования (по необходимости — повышение видимости отдельных имён).

#### Вариант B: добавить `summarize`/`translate` в `services/structure.py`
- Плюсы: всё в одном файле; провайдерный слой рядом.
- Минусы: модуль с именем «structure» обрастает несвязанными по семантике функциями (саммари,
  перевод) — размывание ответственности; хуже читается и растёт.
- Последствия: монолитный сервис.

Выбран **A**: `services/llm.py` для действий, переиспользующий провайдерный слой из `structure.py`.
Это соблюдает границу «одна задача — один сервис» и даёт точку расширения для будущих действий.

### Решение 3. Перевод: авто-направление по эвристике vs две кнопки EN/RU (закрывает ОВ-1)

#### Вариант A: одна кнопка «Перевести» + эвристика направления по символам
- Правило: если в тексте есть кириллица (доля/наличие) → переводим **на английский**; иначе →
  **на русский**. Эвристика — простой подсчёт кириллических символов, без сторонних библиотек.
- Плюсы: минимальный UI (две кнопки на сообщении: «Кратко», «Перевести»); покрывает основной кейс
  ru↔en одним тапом; `callback_data` компактен; нет сторонних зависимостей (ОГР-5).
- Минусы: на смешанном тексте направление может быть не тем, что хотел пользователь.
- Последствия: одна функция-эвристика `detect_lang`/направление, тестируемая изолированно.

#### Вариант B: две кнопки «На английский» / «На русский»
- Плюсы: пользователь явно выбирает направление.
- Минусы: три-четыре кнопки под сообщением (с «Кратко»); больше callback-вариантов; избыточно для
  ru/en (вне рамок прочих языков). BA дефолт — авто-направление.
- Последствия: больше UI и callback-ветвлений ради малой выгоды.

Выбран **A** (минимальный вариант, дефолт из BA): одна кнопка «Перевести» + эвристика
кириллица→en / иначе→ru. Обоснование: ru/en — единственные языки контента (ОГР), эвристика по
символам тривиальна и не требует зависимостей, UI остаётся из двух кнопок.

### Решение 4. Учёт callback'ов в статистике (ADR-006, закрывает вопрос учёта)

#### Вариант A: не считать callback'и — middleware остаётся только на `dp.message`
- Плюсы: минимальные изменения; схема `user_stats` (ADR-006) не меняется (нет колонки под действия);
  middleware и сервис статистики не трогаются; нажатия кнопок — производное действие над уже
  учтённым сообщением.
- Минусы: `/stats` не показывает, сколько раз нажимали кнопки.
- Последствия: callback-роутер не проходит через `StatsMiddleware`; статистика действий не ведётся.

#### Вариант B: расширить учёт на `dp.callback_query`
- Минусы: требует колонку/тип в схеме `user_stats` (миграция), новой классификации, правок
  middleware и `/stats`. Выходит за scope BA (нет требования считать действия).
- Последствия: лишняя сложность и миграция БД ради метрики вне scope.

Выбран **A**: callback'и **не считаются** в статистике v0.9.0. Middleware остаётся на `dp.message`.
Решение зафиксировано здесь; при будущей потребности учёт действий — отдельный релиз.

### Решение 5. Поведение при сбое CF и при недоступном исходном тексте (закрывает ОВ-3, ОВ-4)

- **Сбой CF** (сеть, 4xx/5xx, таймаут, лимит, пустой/некорректный ответ): единичная попытка с явным
  таймаутом (образец ADR-005, Решение 4), без ретраев. При сбое функция действия сигнализирует
  неудачу; callback-хендлер показывает пользователю **понятное русское сообщение** (через
  `callback.answer(...)` всплывающим алертом и/или коротким `message.answer`), сбой логируется, бот
  **не падает**. Действие над текстом — не критичный путь.
- **Пустой/`None` `callback.message.text`** (file-сообщение, устаревшее/отредактированное
  сообщение): callback-хендлер детерминированно отвечает `callback.answer("Текст недоступен")`
  (русское служебное сообщение), LLM **не вызывается**, бот не падает.
- **Повторное нажатие / устаревшее сообщение** (ОВ-4): каждое нажатие формирует ответ **заново**
  (без дедупликации); при наличии текста — действие выполняется, при отсутствии — служебный ответ.
  Простая и предсказуемая семантика.
- **Снятие «часиков».** `callback.answer()` вызывается всегда, чтобы убрать индикатор нажатия. На
  время работы LLM допустимо предварительно вызвать `callback.answer("Готовлю…")` (необязательный
  промежуточный ответ); финальный результат уходит отдельным сообщением через `send_result`. Без
  блокирования event loop.

## Решение

1. **Расширить `send_result`** (Решение 1A): сигнатура становится
   `async def send_result(message: Message, text: str, reply_markup: InlineKeyboardMarkup | None = None) -> None`.
   `reply_markup` прикрепляется к Telegram **только в одно-сообщенческом режиме**
   (`len(text) <= MAX_MESSAGE_LEN`). В режимах «серия» и «файл» markup **игнорируется**. Это и есть
   гарантия однозначности источника текста для callback.
2. **Новый сервис `services/llm.py`** (Решение 2A): публичные корутины
   `async def summarize(text: str) -> str | None` и `async def translate(text: str) -> str | None`.
   Возврат `None` (или пустая строка) сигнализирует неудачу/недоступность → callback-хендлер
   показывает служебное сообщение. Обе функции:
   - переиспользуют провайдерный слой из `services/structure.py` (выбор провайдера по `LLM_PROVIDER`,
     `_CloudflareProvider`/`_GroqProvider`, async POST через `httpx`), **не дублируя** его;
   - усекают вход до того же порога, что структурирование — переиспользовать `_MAX_INPUT_CHARS`
     (8000) из `structure.py` (Решение/ОВ-5);
   - используют явный таймаут (тот же `_TIMEOUT` = 30 с) и единичную попытку без ретраев;
   - ловят все исключения, логируют, возвращают `None` при любом сбое (НФТ-1).
3. **Промпты без выдумок** (НФТ-5, ДОП-5, образец ADR-005, Решение 7):
   - **summarize:** системный промпт инструктирует вернуть саммари исходного текста в **3–5 пунктов
     на языке исходника** (не менять язык), сохранять смысл, **не выдумывать** факты, вернуть только
     результат без преамбул; текст подаётся между маркерами, не как обращение к модели.
   - **translate:** перевести текст на **целевой язык** (см. п.4), переводить, **не добавляя**
     содержание, вернуть только перевод без преамбул; маркеры в ответ не включать.
4. **Перевод — авто-направление по эвристике** (Решение 3A): функция определения направления по
   наличию кириллицы — кириллица в тексте → целевой язык **английский**, иначе → **русский**.
   Эвристика — подсчёт кириллических символов (диапазон Unicode), без сторонних библиотек. Целевой
   язык передаётся в промпт `translate`.
5. **Новый модуль `handlers/actions.py`** с `Router` и двумя `CallbackQuery`-хендлерами:
   - `callback_data` компактный, в пределах 64 байт: **`"act:sum"`** (саммари), **`"act:tr"`**
     (перевод);
   - хендлер: вызывает `callback.answer()` (снять «часики», опционально `"Готовлю…"`); берёт
     `callback.message.text`; если он пуст/`None` → `callback.answer("Текст недоступен")` и выход;
     иначе вызывает `summarize`/`translate`; при `None` → служебное русское сообщение пользователю
     (через `callback.answer`/`message.answer`), без падения; при успехе → отдаёт результат через
     `send_result(callback.message, result)` (**без** markup — у ответа действия кнопок нет, чтобы не
     плодить цепочку действий над производным текстом).
6. **Фабрика клавиатуры.** Inline-клавиатура с кнопками «Кратко» (`act:sum`) и «Перевести»
   (`act:tr`) формируется штатными средствами aiogram (`InlineKeyboardMarkup`/`InlineKeyboardButton`
   или `InlineKeyboardBuilder`) в одном месте (например, функция в `handlers/actions.py` или
   `handlers/keyboards.py` — на усмотрение разработчика; держать подписи и `callback_data` в одной
   точке, НФТ-9).
7. **Подключение к каналам.** В `handlers/image.py` и `handlers/audio.py` на **непустом** результате
   распознавания результат отдаётся через `send_result(message, await structure_text(text), reply_markup=actions_keyboard())`.
   На **пустом** результате (NO_TEXT/NO_SPEECH) кнопки не показываются (ФТ-13) — прежнее поведение
   сохраняется. Правило «markup только при одиночном сообщении» применяется внутри `send_result`
   (п.1) — хендлеры всегда передают markup, `send_result` решает, прикреплять ли его.
8. **Регистрация роутера в `bot.py`.** `include_router(actions_router)` добавляется в
   `bot.py`. Порядок: `actions_router` обрабатывает `callback_query` — он не конфликтует с
   message-роутерами по типу апдейта, порядок относительно message-роутеров не критичен; разместить
   после message-роутеров (или рядом) — главное, что роутер зарегистрирован. `handlers/__init__.py`
   экспортирует `actions_router`.
9. **Callback'и не учитываются в статистике** (Решение 4A). `StatsMiddleware` остаётся на
   `dp.message.outer_middleware`; на `dp.callback_query` middleware **не вешается**; схема
   `user_stats` (ADR-006) **не меняется**.
10. **Сбой и граничные случаи** — по Решению 5: единичная попытка + таймаут; служебные русские
    сообщения при сбое и при недоступном тексте; каждое нажатие формирует ответ заново; `callback.answer()`
    всегда вызывается; бот не падает.
11. **Профиль поставки ADR-001 — без изменений.** Новых env, портов, зависимостей, изменений
    Docker/compose нет. `requirements.txt` — без изменений (`httpx` есть). Используется уже
    существующее исходящее направление к `api.cloudflare.com` (ADR-005).
12. **Roadmap.** Слот v0.9.0 фиксируется как «Доп-действия над результатом: Кратко и Перевести
    (inline-кнопки)». v1.0.0 (Продакшн-готовность) остаётся.

## Последствия

- **Положительные:** пользователь получает саммари/перевод в один тап над уже полученным текстом,
  бесплатно, не выходя из чата; переиспользуется проверенный LLM-слой (ADR-005) и единая отдача
  (ADR-007) без дублирования; правило «markup только при одиночном сообщении» делает источник текста
  однозначным и снимает необходимость в хранилище; `services/llm.py` — точка расширения под будущие
  действия (Сценарий G); сбой CF не делает бота хрупким; профиль поставки и зависимости не меняются.
- **Отрицательные / компромиссы:** для длинного результата (серия/файл) кнопки не показываются —
  сознательное упрощение из-за отсутствия хранилища (ОГР-1); эвристика направления перевода может
  ошибаться на смешанном тексте (приемлемо для ru/en); каждое нажатие — отдельный запрос к CF
  (учитывать лимиты free-tier, ДОП-2/НФТ-3, смягчено усечением входа и тем, что объёмы малы);
  callback'и не видны в `/stats`.
- **Технический долг:** учёт действий в статистике; Q&A-чат, TTS, перевод на произвольные языки,
  хранение текстов под действия над длинным результатом, две кнопки выбора направления — всё вне
  рамок v0.9.0 и при необходимости — отдельными релизами.

## Влияние на код

    направления перевода; переиспользует провайдерный слой и константы из `services/structure.py`;
    промпты саммари/перевода без выдумок.
  - `handlers/actions.py` — `Router`, два `CallbackQuery`-хендлера (`act:sum`, `act:tr`), фабрика
    inline-клавиатуры «Кратко»/«Перевести».
- **Изменяемые модули:**
  - `services/reply.py` — `send_result` получает опциональный `reply_markup`, прикрепляемый **только**
    в одно-сообщенческом режиме; экспорт провайдерного слоя/констант из `structure.py` (повышение
    видимости имён для переиспользования в `llm.py`) — по необходимости.
  - `handlers/image.py`, `handlers/audio.py` — на непустом результате передавать `reply_markup`
    клавиатуры действий в `send_result`; на пустом — без изменений.
  - `bot.py` — `include_router(actions_router)`.
  - `handlers/__init__.py` — экспорт `actions_router`.
  - `roadmap.md` — фиксация слота v0.9.0.
- **Не изменяются:** `middlewares/stats.py`, `services/stats.py`, схема `user_stats` (callback'и не
  учитываются); `services/ocr.py`, `services/transcribe.py`; `config.py` (новых env нет);
  `requirements.txt`; артефакты поставки (`Dockerfile`, `deploy/*`, `.github/workflows/deploy.yml`).
- **Новые зависимости:** нет (`httpx` есть; inline-кнопки/`CallbackQuery` — штатный aiogram).
- **Что ломается:** публичный контракт каналов не ломается — обратимо-совместимое добавление (minor);
  сигнатура `send_result` расширяется опциональным параметром с дефолтом (обратимо-совместима).
