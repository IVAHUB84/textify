# ADR-010: Продакшн-готовность v1.0.0 — наблюдаемость (logging), индикатор прогресса (ChatActionSender), защита слабого сервера глобальным семафором на тяжёлые локальные операции и фиксация публичного контракта 1.0.0

- **Дата:** 2026-06-23
- **Статус:** принято
- **Связанные BA-требования:** `ba-req/ba-requirements-010.md`
- **Связанные ADR:** ADR-001 (профиль поставки «бот без HTTP-порта», `Dockerfile` multi-stage/non-root, секреты через `.env` — сохраняется; единственное изменение `Dockerfile` — `PYTHONUNBUFFERED=1`), ADR-002 / ADR-003 (граница «хендлер ↔ сервис», неблокирование event loop, OOM-риск тяжёлых локальных операций на 1 vCPU / 1.9 GB — прямое основание для семафора), ADR-005 (CF Workers AI: структурирование `services/structure.py`, async `httpx`, env `CF_*`/`LLM_PROVIDER`/`CF_MODEL` — документируется, под семафор **не** попадает), ADR-006 (учёт/`/stats`, SQLite в volume, `ADMIN_USER_ID`, `STATS_DB_PATH` — документируется), ADR-007 (`send_result` — формат отдачи результата, часть фиксируемого контракта; обвязка ChatActionSender его не меняет), ADR-008 (CF Whisper + локальный фолбэк `services/transcribe.py` — локальный путь попадает под семафор, CF — нет), ADR-009 (доп-действия «Кратко»/«Перевести», `act:sum`/`act:tr` — часть фиксируемого контракта, индикатор прогресса добавляется в `handlers/actions.py`)

## Контекст

Релизы v0.1.0–v0.9.0 в проде на ivahub (1 vCPU / 1.9 GB RAM, long-polling, Docker, образ из GHCR;
профиль поставки «бот без HTTP-порта» — ADR-001). Функциональность продукта закрыта. v1.0.0 —
**стабилизация без новых пользовательских фич**: нужно закрыть эксплуатационные пробелы и
зафиксировать продукт как стабильный 1.0. BA-требование 010 описывает пять блоков: наблюдаемость,
индикатор прогресса, защита от перегрузки, документация, фиксация контракта.

Действующие силы и ограничения:

- **Слепые логи.** В `bot.py` нет настройки логирования; вывод процесса в контейнере буферизуется.
  Существующие `logger.*` (фолбэки CF в `structure.py`/`transcribe.py`/`llm.py`, исключения в
  `ocr.py`) при штатной работе не видны в `docker logs` — диагностика инцидентов на проде затруднена.
- **Отсутствие обратной связи.** OCR и локальный whisper-фолбэк занимают секунды; пользователь не
  видит признаков работы и может счесть бота «зависшим». Доп-действия (v0.9.0) и структурирование —
  тоже секунды.
- **Риск перегрузки слабого сервера.** Тяжёлые **локальные** операции — локальный OCR
  (`services/ocr._run_ocr` через `asyncio.to_thread`) и локальный whisper-фолбэк
  (`services/transcribe._transcribe_local` → `_transcribe_sync` через `asyncio.to_thread`) — на
  1 vCPU / 1.9 GB при параллельных запросах создают риск перегрузки CPU/OOM (отмечалось в
  ADR-002/003). Число одновременно выполняемых тяжёлых операций сейчас ничем не ограничено.
- **CF-вызовы — не локальная нагрузка.** ASR через CF Whisper (`transcribe._transcribe_cloudflare`)
  и LLM (`structure.structure_text`, `llm.summarize`/`translate`) уходят на внешнее API Cloudflare
  через async `httpx` — они **не** грузят CPU/RAM ivahub и держать их под локальным семафором нельзя
  (это сериализовало бы облачную работу без причины и провоцировало бы дедлок: CF-путь
  `transcribe()` при сбое падает в локальный фолбэк, который сам должен захватить семафор).
- **Устаревший `README.md`** описывает прежний стек (Groq основной, локальный whisper основной), не
  отражает фактическое состояние (CF Workers AI для ASR/LLM, локальный фолбэк, доп-действия,
  `/stats`), не содержит полного списка env и эксплуатационного раздела.
- **Незафиксированный публичный контракт.** До 1.0.0 контракт считался нестабильным (`roadmap.md`).
  С 1.0.0 его нужно явно зафиксировать, чтобы ломающие изменения трактовались как MAJOR.
- **Ограничения обвязки (ОГР-5).** Без новых внешних зависимостей: индикатор — штатный aiogram
  (`ChatActionSender`), семафор — `asyncio.Semaphore`, логирование — стандартный `logging`.

## Рассматриваемые варианты

### Решение 1. Настройка логирования: `logging.basicConfig` в `bot.py` vs отдельный logging-модуль/dictConfig

#### Вариант A: `logging.basicConfig(...)` в начале `main()` (до `start_polling`), уровень из `LOG_LEVEL`
- Плюсы: одна точка настройки в точке входа; ровно то, что просит BA (ФТ-1); `basicConfig`
  идемпотентен (ставит **один** root-handler, повторный вызов без `force` ничего не делает) — нет
  дубль-хендлеров; существующие `logger = logging.getLogger(__name__)` по всему коду немедленно
  начинают писать в stdout; формат с временем/уровнем/именем логгера задаётся `format=`.
- Минусы: глобальная конфигурация root-логгера — но для одного процесса-бота это и нужно.
- Последствия: `bot.py` импортирует `logging`/`os`, настраивает root до старта polling.

#### Вараинт B: отдельный модуль логирования с `dictConfig`/несколькими хендлерами
- Минусы: избыточно для одного процесса в stdout; риск нескольких хендлеров → дублирование строк;
  больше кода и поверхности ошибок; BA явно просит `basicConfig` (ФТ-1) и «не плодить дубль-хендлеры».
- Последствия: усложнение без выгоды.

Выбран **A**. Уровень читается из `LOG_LEVEL` (дефолт `INFO`); некорректное/пустое значение
**не роняет старт** — нормализация к `INFO` (закрывает ОВ-3). Формат включает время, уровень и имя
логгера (`%(asctime)s %(levelname)s %(name)s %(message)s` или эквивалент). `PYTHONUNBUFFERED=1` в
`Dockerfile` (ФТ-2) делает вывод небуферизованным в `docker logs`.

**Где разместить чтение `LOG_LEVEL` (закрывает часть ОВ-2/ОВ-6).** `LOG_LEVEL` читается в `bot.py`
напрямую из окружения (не через `config.py`), потому что `config.py` импортируется многими модулями
(в т.ч. в тестах) и уже валидирует `BOT_TOKEN`/`ADMIN_USER_ID`/`STATS_DB_PATH`; добавление туда
log-уровня смешало бы конфиг приложения с настройкой инфраструктуры логирования, которая нужна
строго в точке входа. Это деталь реализации, не контракт.

**Новые точки логирования (закрывает ОВ-2).** По умолчанию — только настройка + видимость уже
существующих `logger.*`. Допустимо добавить **одну** информационную строку о старте бота на уровне
`INFO` (чтобы `docker logs` был непуст при штатной работе — ФТ-3), и (опционально) `INFO`-строку о
срабатывании семафора (ожидание/освобождение). Секреты (`BOT_TOKEN`, `CF_API_TOKEN`) в логи **не
пишутся** (НФТ-1) — существующий код их не логирует, новых логов с секретами не вводить.

### Решение 2. Индикатор прогресса: `ChatActionSender` (контекстный менеджер) vs ручной периодический `send_chat_action`

#### Вариант A: `aiogram.utils.chat_action.ChatActionSender` как async-контекст вокруг тяжёлой обработки
- Плюсы: штатное средство aiogram (без новых зависимостей, ОГР-5); сам по себе **неблокирующий** —
  внутри крутит фоновую задачу, периодически шлющую `send_chat_action`, пока активен `async with`;
  Telegram-индикатор живёт максимум ~5 с и требует периодического обновления — `ChatActionSender`
  делает это автоматически, переживая многосекундные операции; чистое снятие индикатора по выходу из
  контекста (в т.ч. при исключении); не трогает существующий путь `send_result`.
- Минусы: оборачивает блок обработки — нужно аккуратно охватить именно тяжёлую часть, не саму
  отправку результата.
- Последствия: в каждом хендлере тяжёлая обработка заворачивается в `async with ChatActionSender(...)`.

#### Вариант B: ручной `bot.send_chat_action` + собственная фоновая задача обновления
- Минусы: дублирует то, что уже делает `ChatActionSender`; ручное управление жизненным циклом
  фоновой задачи и её отменой при ошибке — источник багов и подвисших задач.
- Последствия: лишний код, риск утечки задач.

Выбран **A**. Какой action где (закрывает «зафиксируй, какой action где»):

- `handlers/image.py` (OCR, `handle_photo` и `handle_image_document`): **`upload_document`** —
  семантически ближе всего к «готовлю и пришлю результат по файлу/изображению» и согласуется с
  возможной отдачей `send_result` файлом; допустимо `typing`, но фиксируем `upload_document`.
- `handlers/audio.py` (`handle_voice`, `handle_audio`, `handle_audio_document` — через общий
  `_handle_audio_bytes`): **`typing`**.
- `handlers/actions.py` (`act:sum`/`act:tr` — общий `_handle_action`): **`typing`** на время вызова
  LLM.

Индикатор охватывает тяжёлую часть (распознавание + структурирование / LLM-действие); сама отдача
`send_result` может остаться внутри контекста — это не нарушает её поведение (ADR-007). Сбой/таймаут
индикатора не должен ронять обработку (НФТ-4): `ChatActionSender` штатно гасит свои ошибки, но
оборачивание не должно перехватывать исключения бизнес-логики иначе, чем сейчас.

### Решение 3. Защита от перегрузки: глобальный `asyncio.Semaphore` на тяжёлые локальные операции — значение лимита и место размещения

#### Вариант A: один общий `asyncio.Semaphore` в модуле `services` (`services/__init__.py`), захват внутри сервисов вокруг `to_thread`
- Плюсы: один экземпляр на процесс (модуль импортируется один раз) — действительно **глобальное**
  ограничение; локализован в одном месте и переиспользуем под будущие тяжёлые операции (НФТ-8);
  захват внутри сервиса (`ocr.recognize_text`, `transcribe._transcribe_local`) гарантирует, что
  **любой** путь к локальной операции (в т.ч. фолбэк CF→local внутри `transcribe()`) проходит через
  семафор — хендлерам про семафор знать не нужно; `services/__init__.py` сейчас пуст — естественное
  место.
- Минусы: нужно аккуратно избежать **двойного захвата** в `transcribe` (CF-путь не держит семафор;
  локальный путь — и прямой режим `ASR_PROVIDER=local`, и фолбэк — захватывает семафор ровно один раз).
- Последствия: `services/ocr.py` и `services/transcribe.py` импортируют общий семафор и оборачивают
  им вызов `to_thread`.

#### Вариант B: семафор в хендлерах (вокруг вызовов сервисов)
- Минусы: фолбэк CF→local происходит **внутри** `transcribe()` — хендлер не знает, пошёл ли путь в
  облако или в локальную модель, и обернул бы семафором даже чистый CF-вызов (ненужная сериализация
  облака, риск дедлока при ожидании). Дублирование обёртки в нескольких хендлерах. Нарушает «семафор
  только на локальные операции».
- Последствия: неверная гранулярность, дедлок-риск.

Выбран **A**. **Лимит = 1** (`asyncio.Semaphore(1)`). Обоснование: 1 vCPU физически выполняет одну
CPU-bound задачу одновременно; локальный whisper и OCR — оба CPU/RAM-тяжёлые, а RAM 1.9 GB не
оставляет запаса под две одновременные тяжёлые операции (риск OOM — главная угроза НФТ-3). Лимит 1
сериализует тяжёлую локальную работу, давая максимальную защиту от OOM; пропускная способность
страдает минимально, т.к. дефолтный путь ASR — облачный CF (локальный whisper — лишь фолбэк), а OCR
быстр. Значение фиксируется как именованная константа (не «магическое число»), чтобы при будущем
апгрейде сервера менять в одном месте.

**Размещение.** Общий семафор объявляется в `services/__init__.py` (например,
`HEAVY_LOCAL_SEMAPHORE = asyncio.Semaphore(1)` с поясняющим именем константы лимита). `services/ocr.py`
и `services/transcribe.py` импортируют его и захватывают вокруг своего `asyncio.to_thread(...)`.

**Правило отсутствия дедлока/двойного захвата (критично):**
- `ocr.recognize_text`: `async with HEAVY_LOCAL_SEMAPHORE: await asyncio.to_thread(_run_ocr, ...)`.
- `transcribe._transcribe_local`: захватывает семафор вокруг `asyncio.to_thread(_transcribe_sync, ...)`
  — **единственное** место захвата для локального пути. И прямой режим (`ASR_PROVIDER=local`), и
  фолбэк после сбоя CF идут через `_transcribe_local`, поэтому захват в нём покрывает оба случая
  ровно один раз.
- `transcribe._transcribe_cloudflare` и весь CF-ветвящийся код в `transcribe()` **до** вызова
  `_transcribe_local` семафор **не держат**. Так фолбэк CF→local не пытается захватить семафор дважды
  и не ждёт его, пока сам же его держит. CF-вызовы (ASR/LLM, `structure`/`llm`) семафором **не**
  оборачиваются вовсе (ОВ-5).

#### Per-user мягкий лимит (закрывает ОВ-4) — НЕ делаем в 1.0

Опциональный мягкий лимит «один тяжёлый запрос на пользователя» в 1.0.0 **не вводится**. Обоснование
(минимализм, принцип «минимальные изменения» CLAUDE.md): глобальный семафор уже снимает главный риск
(OOM/перегрузка CPU) — это и есть НФТ-3. Per-user лимит требует учёта активных запросов по
пользователю (структура состояния, очистка, гонки) и нового пользовательского сообщения — это
сложность ради UX-нюанса, не ради устойчивости. Поведение при достижении лимита — **ожидание под
индикатором прогресса** (`ChatActionSender` активен, пока корутина ждёт семафор): пользователь видит
«печатает…», бот остаётся отзывчивым для других (event loop не блокируется — ожидание семафора
асинхронно, НФТ-2). При появлении потребности per-user лимит вводится отдельным релизом.

### Решение 4. Где фиксировать публичный контракт 1.0.0 (закрывает ОВ-1)

#### Вариант A: зафиксировать контракт в этом ADR (раздел «Публичный контракт 1.0.0») + отразить в README
- Плюсы: ADR — источник истины по архитектурным решениям и semver-baseline (Сценарий G); README —
  пользовательский источник истины. Контракт виден разработчику (ADR) и пользователю (README) без
  дублирования сути: ADR фиксирует контракт как baseline и правило MAJOR, README описывает то же с
  точки зрения пользователя.
- Последствия: ADR-010 содержит явный раздел контракта; README — раздел возможностей/команд.

#### Вариант B: только README
- Минусы: README — не место для semver-baseline; разработчик при планировании сверяется с ADR.

Выбран **A**: канонический контракт 1.0.0 фиксируется ниже в этом ADR; README его отражает для
пользователя.

## Решение

1. **Логирование (Решение 1A).** В `bot.py`, в начале `main()` до `start_polling`,
   `logging.basicConfig(level=<из LOG_LEVEL>, format=<время/уровень/имя логгера>)`. `LOG_LEVEL`
   читается из окружения (дефолт `INFO`); некорректное/пустое значение → `INFO`, старт не падает.
   Ровно один root-handler (`basicConfig` идемпотентен) — без дубль-хендлеров. Допустима одна
   `INFO`-строка о старте бота. Секреты не логируются.
2. **`PYTHONUNBUFFERED=1` (ФТ-2).** Добавляется как `ENV PYTHONUNBUFFERED=1` в `Dockerfile` —
   **единственное** изменение `Dockerfile`; профиль поставки ADR-001 сохраняется (multi-stage,
   non-root, без портов, GHCR-образ).
3. **Индикатор прогресса (Решение 2A).** `ChatActionSender` оборачивает тяжёлую обработку:
   `upload_document` в `handlers/image.py`, `typing` в `handlers/audio.py`, `typing` в
   `handlers/actions.py`. Неблокирующе; путь `send_result` (ADR-007) не меняется.
4. **Глобальный семафор (Решение 3A).** `asyncio.Semaphore(1)` в `services/__init__.py`, захват в
   `services/ocr.recognize_text` и `services/transcribe._transcribe_local` вокруг `asyncio.to_thread`.
   CF-вызовы (ASR/LLM) семафором не оборачиваются. Локальный путь захватывает семафор ровно один раз
   (через `_transcribe_local`), фолбэк CF→local дедлока не вызывает. Лимит — именованная константа.
5. **Per-user лимит — не вводится в 1.0** (Решение 3): только глобальный семафор; при достижении
   лимита — ожидание под индикатором прогресса.
6. **Документация.** `README.md` полностью переписывается под фактическое состояние: возможности
   (OCR с предобработкой + quality gate; аудио через CF Whisper large-v3-turbo + локальный фолбэк;
   структурирование CF; действия «Кратко»/«Перевести»; `/stats`); команды (`/start`, `/help`,
   `/stats`); полный список из 10 env-переменных с назначением/обязательностью/дефолтами; деплой
   (ivahub, GHCR, volume статистики); особенности (пиннинг Telegram-DC, офлайн-модель whisper в
   образе, гео-ограничения Groq по RU-IP); эксплуатация (логи `docker logs`/`LOG_LEVEL`, путь данных
   статистики `STATS_DB_PATH`/volume, переключение провайдеров `ASR_PROVIDER`/`LLM_PROVIDER`).
   Приводятся **плейсхолдеры**, не реальные секреты (НФТ-5).
7. **Фиксация публичного контракта 1.0.0 (Решение 4A)** — см. раздел ниже.
8. **Версия — MAJOR 1.0.0.** Все изменения обратимо-совместимы (обвязка + документация); MAJOR
   обусловлен фиксацией контракта (semver-baseline), а не ломающим изменением. `roadmap.md`: v1.0.0
   «Продакшн-готовность» — выпущен.

### Публичный контракт Textify 1.0.0 (semver-baseline)

С версии 1.0.0 следующий контракт считается **стабильным**; его ломающее изменение = **MAJOR**,
обратимо-совместимое добавление = MINOR, багфикс без изменения контракта = PATCH (CLAUDE.md).

**Команды:**
- `/start` — приветствие/описание.
- `/help` — справка по возможностям и командам.
- `/stats` — сводка статистики; доступна только администратору (`ADMIN_USER_ID`); прочим —
  короткий отказ.

**Входные каналы (распознаются):**
- фото (`message.photo`);
- изображение-документ (`document` с `mime_type`, начинающимся на `image/`);
- голосовое (`message.voice`);
- аудио (`message.audio`);
- аудио-документ (`document` с `mime_type`, начинающимся на `audio/`).

**Формат ответа:**
- plain text (без `parse_mode`);
- отдача через `send_result` (ADR-007): одно сообщение → серия сообщений по естественным границам
  (до `MAX_PARTS`) → файл `result.txt` при превышении (контракт ADR-007);
- при нераспознанном тексте/речи — служебное сообщение («текст не распознан» / «речь не
  распознана»);
- при недоступности LLM — тихий фолбэк на сырой текст (ADR-005); при сбое CF ASR — локальный
  фолбэк (ADR-008).

**Inline-действия** (под одно-сообщенческим результатом OCR/аудио — ADR-009):
- «Кратко» — `callback_data = act:sum`;
- «Перевести» — `callback_data = act:tr`.

**Профиль поставки** (ADR-001): long-polling, без входящих портов/Traefik/`edge`, образ из GHCR,
`restart: unless-stopped`, пиннинг Telegram-DC, volume статистики.

## Последствия

- **Положительные:** прод становится наблюдаемым (`docker logs` информативен и небуферизован) при
  нулевых новых зависимостях; пользователь видит, что бот работает (индикатор); слабый сервер
  защищён от OOM/перегрузки глобальным семафором без блокировки event loop; документация приведена к
  истине и пригодна для нового сопровождающего; контракт зафиксирован как semver-baseline.
- **Отрицательные / компромиссы:** лимит семафора = 1 сериализует тяжёлые локальные операции
  (приемлемо: дефолтный ASR — облачный, OCR быстр); per-user UX-лимит отложен (приемлемо: главный
  риск — OOM — закрыт глобальным семафором); `LOG_LEVEL=DEBUG` в проде может быть шумным —
  управляется эксплуатантом.
- **Технический долг:** per-user мягкий лимит; чанкинг крупного аудио для CF (наследие v0.8.0);
  mypy-stubs для `pytesseract`/`faster_whisper` (опционально, релиз не блокирует); возможный пересмотр
  лимита семафора при апгрейде сервера.
