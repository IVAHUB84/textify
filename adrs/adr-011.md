# ADR-011: Гарантия бесплатности при росте — дневной бюджет Cloudflare с авто-деградацией на локальные движки (персистентный счётчик в SQLite, единый гейт перед CF) + усиление дискаверабилити

- **Дата:** 2026-06-23
- **Статус:** принято
- **Связанные BA-требования:** `ba-req/ba-requirements-011.md`
- **Связанные ADR:** ADR-001 (профиль поставки «бот без HTTP-порта» — сохраняется без изменений), ADR-002 / ADR-003 (граница «хендлер ↔ сервис», неблокирование event loop через `asyncio.to_thread`), ADR-005 (CF-структурирование + тихий фолбэк на сырой текст — путь деградации структурирования), ADR-006 (SQLite в volume, WAL/UPSERT, соединение-на-операцию, `STATS_DB_PATH` — образец и хранилище для дневного счётчика), ADR-007 (`send_result`), ADR-008 (CF Whisper + локальный whisper-фолбэк — путь деградации ASR), ADR-009 (действия «Кратко»/«Перевести», обработка сбоев в `handlers/actions.py`), ADR-010 (логирование, глобальный семафор, публичный контракт 1.0.0 — релиз MINOR его не ломает)

## Контекст

Релизы v0.1.0–v1.0.0 в проде на ivahub (1 vCPU / 1.9 GB RAM, long-polling, Docker, образ из GHCR;
профиль поставки «бот без HTTP-порта» — ADR-001). Продукт стабилизирован и зафиксирован как 1.0
(ADR-010): контракт стабилен, функциональность закрыта.

Сверхцель продукта — **бесплатно для пользователей и для владельца**. Две ключевые внешние операции —
транскрипция (CF Whisper, ADR-008) и LLM (структурирование ADR-005, действия «Кратко»/«Перевести»
ADR-009) — идут через бесплатный tier **Cloudflare Workers AI = 10 000 нейронов/сутки** со сбросом в
00:00 UTC. Пока пользователей мало, лимит не достигается. Но рост (в планах — группы v1.2.0,
виральность v1.3.0) может вывести суточное потребление за free-tier, и Cloudflare начнёт
**тарифицировать** вызовы — прямое нарушение сверхцели «бесплатно для владельца».

BA-требование 011 (релиз v1.1.0, MINOR) закрывает два связанных пробела:

- **Гарантия бесплатности при росте (главное).** Персистентный **дневной счётчик** обращений к CF за
  текущие UTC-сутки и **дневной бюджет** (`CF_DAILY_BUDGET`); при достижении бюджета —
  **авто-деградация на локальные бесплатные движки без отказа пользователю** (ASR → локальный
  faster-whisper; структурирование → сырой текст; действия → понятное сообщение о лимите).
- **Дискаверабилити (дешёвый органический приток).** Усиление ключевых слов в уже заданных
  метаданных бота (`setup_bot_profile`, ADR-010 контракт) под поиск Telegram и готовый текст листинга
  для бесплатных каталогов (`docs/listing.md`); подача — ручной шаг владельца в чек-листе деплоя.

Действующие силы и ограничения:

- **Учёт должен быть персистентным и concurrency-safe.** Счётчик переживает передеплой/перезапуск
  контейнера и корректен при параллельных обращениях из разных потоков `asyncio.to_thread`.
  Инфраструктура для этого уже есть — SQLite в volume с WAL/UPSERT (ADR-006); вводить новое хранилище
  избыточно.
- **CF не возвращает «нейроны» удобным образом.** Точный учёт стоимости вызова в нейронах
  нетривиален; требование (ОВ-2) допускает консервативный прокси по числу вызовов.
- **Гейт должен стоять ровно на фактических обращениях к CF.** Локальные пути (Tesseract-OCR,
  локальный whisper-фолбэк) бесплатны и бюджет не тратят; ручной режим `ASR_PROVIDER=local` /
  `LLM_PROVIDER=groq` не должен инкрементировать счётчик CF.
- **Учёт — вспомогательный путь.** Его сбой (БД недоступна, нет прав, блокировка) не должен ронять
  обработку или блокировать пользователя — преемственность Решения 7 ADR-006 (тихая деградация учёта
  `/stats`).
- **Деградация без отказа.** ASR и структурирование имеют готовые локальные/фолбэк-пути (ADR-008,
  ADR-005). У LLM-действий локальной замены нет → пользователю нужно понятное сообщение, а не ошибка.
- **Профиль поставки ADR-001 сохраняется.** Новых портов, внешних зависимостей, направлений
  исходящего трафика нет; переиспользуется существующий volume данных (ADR-006) и направление к
  `api.cloudflare.com`. Добавляется лишь опциональная env `CF_DAILY_BUDGET`.
- **Наблюдаемость (ADR-010).** Переход в деградацию логируется (видно в `docker logs`); секреты в
  логи не пишутся.

## Рассматриваемые варианты

### Решение 1. Единица учёта бюджета

#### Вариант A: прокси по числу вызовов (1 вызов CF = 1 единица) (выбран)
- Плюсы: CF не отдаёт нейроны на вызов удобным/гарантированным образом; число вызовов считается
  тривиально и детерминированно; concurrency-safe UPSERT (ADR-006) ложится напрямую; достаточно для
  цели (страховка от выхода за free-tier), т.к. бюджет берётся консервативным с запасом под
  worst-case стоимость нейронов/вызов.
- Минусы: грубее точного учёта нейронов — Whisper-вызов и LLM-вызов считаются одинаково, хотя стоят
  разное число нейронов. Компенсируется консервативностью дефолта.
- Последствия: одна целочисленная единица на любой CF-вызов; схема хранилища простая
  (`date, count`).

#### Вариант B: точный учёт нейронов из ответа CF
- Минусы: CF не возвращает нейроны стабильно в ответе `ai/run`; парсинг хрупкий и провайдер-зависимый;
  усложняет код ради точности, не нужной для консервативной страховки. Отклонён (BA ОВ-2 допускает
  прокси).

#### Вариант C: прокси с весами (Whisper ≠ LLM по стоимости)
- Минусы: веса — это оценка нейронов «вручную», такая же неточная, как и сам прокси, но сложнее;
  при консервативном бюджете не даёт практической выгоды. Отклонён; при необходимости вводится позже
  без смены контракта (внутренняя деталь).

**Выбран A.** Единица бюджета — **число вызовов CF за UTC-сутки**.

### Решение 2. Значение дефолта `CF_DAILY_BUDGET`

Free-tier — 10 000 нейронов/сутки. Стоимость одного вызова в нейронах зависит от модели и длины
ввода; точные числа CF меняет и не гарантирует. Чтобы прокси «1 вызов = 1 единица» **гарантированно**
не вышел за 10 000 нейронов даже в worst-case, дефолт берётся консервативным: предполагаем
пессимистично, что один вызов (длинный Whisper-аудио ≤ 8 МБ или LLM на ~8000 символов) может стоить
порядка десятков нейронов. При worst-case ~20–30 нейронов/вузов и запасе безопасности дефолт
**`CF_DAILY_BUDGET = 300`** (≈ 300 вызовов/сутки) держит суммарное потребление заведомо ниже 10 000
нейронов даже при пиковом распределении и одновременно с запасом покрывает текущий реальный трафик
(единицы пользователей). Значение — **именованная константа** в коде (не «магическое число»),
переопределяемая env; владелец может поднять/опустить по факту наблюдения за реальной стоимостью
нейронов на проде (ОВ-7).

- Некорректное/пустое/нечисловое значение `CF_DAILY_BUDGET` **не роняет старт** — нормализация к
  дефолту 300 с `logger.warning` (преемственность нормализации `ADMIN_USER_ID` в `config.py` и
  `LOG_LEVEL` в ADR-010). Неположительное значение (≤ 0) трактуется как «бюджет недопустим» →
  нормализация к дефолту (бюджет 0 означал бы вечную деградацию — это не цель; намеренно отключить CF
  оператор может существующими переключателями `ASR_PROVIDER=local` / `LLM_PROVIDER=groq`).

### Решение 3. Где разместить механизм учёта/бюджета

#### Вариант A: новый модуль `services/budget.py` рядом со `services/stats.py` (выбран)
- Плюсы: одна узкая ответственность (дневной учёт CF), не смешивается со статистикой пользователей;
  переиспользует тот же паттерн (соединение-на-операцию, WAL, `busy_timeout`, UPSERT, доступ через
  `asyncio.to_thread`) и **тот же файл БД** (`STATS_DB_PATH`, тот же volume — ADR-006), но **отдельную
  таблицу**; переиспользуем под будущий рост (НФТ-9, группы/виральность); сервис не знает про aiogram
  (граница ADR-002/003).
- Минусы: ещё один модуль — но узкий и самодостаточный.
- Последствия: новый `services/budget.py`; инициализация таблицы при старте рядом с `init_db()`
  (ADR-006).

#### Вариант B: расширить `services/stats.py`
- Минусы: смешивает две разные ответственности (статистика аудитории vs страховка бюджета CF) в одном
  модуле; усложняет тестирование и будущее переиспользование. Отклонён в пользу разделения.

**Выбран A.** Отдельный модуль, общий файл БД, отдельная таблица.

### Решение 4. Структура хранилища

Отдельная таблица в существующем файле БД:

```sql
CREATE TABLE IF NOT EXISTS cf_usage (
    date  TEXT PRIMARY KEY,            -- UTC-дата 'YYYY-MM-DD'
    count INTEGER NOT NULL DEFAULT 0   -- число вызовов CF за эти UTC-сутки (прокси по вызовам)
);
```

- Ключ — **текущая UTC-дата** (`datetime.now(timezone.utc).date().isoformat()`), согласованно с
  ADR-006 (время — UTC).
- **Сброс по смене UTC-даты** реализуется естественно: при новых сутках ключ `date` другой → создаётся
  новая строка со счётчиком 0; чтение бюджета всегда идёт по сегодняшней дате. Старые строки
  **не чистятся** (объём ничтожен; история не вредит).
- Инкремент — атомарный UPSERT в одной транзакции (как `record_message` в ADR-006).

### Решение 5. Контракт гейта и точка его применения

Единая точка — две функции в `services/budget.py`:

- **`async def cf_budget_allow() -> bool`** — читает счётчик за текущую UTC-дату и возвращает
  `count < CF_DAILY_BUDGET`. **Политика fail-open** при любом сбое чтения (см. Решение 7): при
  исключении возвращает `True` (разрешить CF) с `logger.warning`.
- **`async def cf_budget_consume() -> None`** — атомарный UPSERT-инкремент счётчика за текущую
  UTC-дату; вызывается **только при фактическом обращении к CF** (после прохождения гейта, перед/при
  реальном CF-вызове). При сбое — `logger.warning`, без проброса исключения (учёт вспомогательный).

**Порядок «проверка → вызов → инкремент»** (Решение 6 про гонки). Применяется в **трёх** местах —
ровно там, где код фактически обращается к CF:

1. **`services/transcribe.py` (CF ASR).** Гейт ставится на **CF-ветке** `transcribe()` — после
   проверок `ASR_PROVIDER`, размера файла и наличия `CF_ACCOUNT_ID`/`CF_API_TOKEN`, **перед**
   `_transcribe_cloudflare(...)`. Если `cf_budget_allow()` вернул `False` → `logger.warning` о
   деградации и `return await _transcribe_local(audio_bytes)` (существующий путь ADR-008; семафор
   ADR-010 захватывается ровно один раз в `_transcribe_local`, дедлока нет). Если разрешено →
   `cf_budget_consume()` непосредственно вокруг вызова CF. Ручной режим `ASR_PROVIDER=local` и фолбэк
   по размеру/кредам идут в локальный путь **до** гейта → счётчик не растёт (ФТ-2, ФТ-12).
2. **`services/structure.py` (CF структурирование).** Гейт — в `structure_text()`, **после**
   `_build_provider()` и **только если провайдер — Cloudflare** (для `LLM_PROVIDER=groq` гейт CF не
   применяется). Если бюджет исчерпан → `logger.warning` и `return raw_text` (существующий тихий
   фолбэк ADR-005). Иначе → `cf_budget_consume()` вокруг `provider.complete(...)`.
3. **`services/llm.py` (CF действия «Кратко»/«Перевести»).** Гейт — в `summarize()` и `translate()`,
   после `build_provider()` и только для Cloudflare-провайдера. Если бюджет исчерпан → `logger.warning`
   и `return None` (существующий контракт «нет результата»; локальной замены нет). Иначе →
   `cf_budget_consume()` вокруг `provider.complete(...)`. **Сообщение пользователю формирует не
   сервис, а хендлер** (`handlers/actions.py`, см. Решение 8) — граница ADR-002/003 сохраняется.

**Определение «провайдер — Cloudflare».** Чтобы гейт применялся ровно к CF и не трогал Groq, в
`services/structure.py` вводится способ узнать тип активного провайдера без вызова сети (например,
проверка `isinstance(provider, _CloudflareProvider)` или явный флаг на провайдере). Это внутренняя
деталь модуля LLM, не контракт.

### Решение 6. Конкурентность и допустимый overshoot

Порядок «`cf_budget_allow()` → CF-вызов → `cf_budget_consume()`» не сериализуется глобальной
блокировкой. При параллельных запросах несколько корутин могут пройти `allow()` до того, как любая из
них успела `consume()` → возможен небольшой **overshoot** (счётчик чуть превысит бюджет). Это
**осознанный компромисс**:

- Альтернатива (атомарная «проверка-и-инкремент» под глобальным замком/в одной SQL-транзакции с
  блокировкой) сериализовала бы все CF-обращения и противоречила бы духу ADR-010 (CF-вызовы намеренно
  **не** сериализуются — это внешнее API, не локальная нагрузка). Накладные расходы и риск
  бутылочного горлышка не оправданы.
- Overshoot ограничен числом одновременно летящих CF-запросов (на ivahub — единицы), а бюджет
  консервативен (Решение 2) с запасом в десятки раз до free-tier. Малый overshoot гарантированно не
  выводит за 10 000 нейронов.

Сам инкремент (`consume`) атомарен (UPSERT в одной транзакции, WAL, `busy_timeout` — ADR-006), поэтому
учёт не теряется и не задваивается, процесс не падает (НФТ-6). Overshoot фиксируется как принятый риск,
а не дефект.

### Решение 7. Политика при сбое механизма учёта — fail-open (выбран)

#### Вариант A: fail-open — при сбое учёта разрешить CF-вызов (выбран)
- Плюсы: учёт/бюджет — **вспомогательный** механизм (страховка), приоритет — работа бота
  (преемственность Решения 7 ADR-006: сбой учёта `/stats` проглатывается, основная функция не
  страдает); сбой БД не должен ни ронять обработку, ни деградировать качество пользователю; CF-вызов
  при недоступности счётчика — безопаснее для UX.
- Минусы: при длительном сбое учёта страховка бюджета временно не действует (теоретически возможен
  выход за free-tier, если БД недоступна долго **и** трафик высок одновременно). Риск низкий: сбой
  персистентного SQLite в volume — редкая аномалия (фиксируется логом для оператора), а не штатный
  режим.
- Последствия: `cf_budget_allow()` при исключении → `True` + `logger.warning`; `cf_budget_consume()`
  при исключении → проглатывание + `logger.warning`.

#### Вариант B: fail-safe — при сбое учёта деградировать
- Минусы: сбой вспомогательного учёта ухудшал бы качество всем пользователям (локальный whisper / сырой
  текст / отказ действий) — отказ в пользу бюджета противоречит НФТ-2 «отсутствие отказа пользователю»
  и приоритету «работа бота важнее точности учёта». Отклонён.

**Выбран A (fail-open)** — согласуется с BA-рекомендацией (ОВ-4) и Решением 7 ADR-006.

### Решение 8. Сообщение пользователю при исчерпании бюджета для действий

Для «Кратко»/«Перевести» локальной замены нет. Сервис (`services/llm.py`) возвращает `None` при
исчерпании бюджета — как и при любой недоступности провайдера (существующий контракт ADR-009). Но
текущий хендлер на `None` отвечает общим «Не удалось выполнить действие. Попробуйте позже.», что для
исчерпанного бюджета неточно («завтра», а не «позже»).

Чтобы хендлер мог отличить «временный сбой» от «исчерпан дневной бюджет» **без раскрытия деталей CF**,
вводится способ различения исхода. Выбран минимальный: в `services/llm.py` определяется
**sentinel-значение** (например, модульная константа `BUDGET_EXCEEDED`, отдельный синглтон-объект),
которое `summarize`/`translate` возвращают **именно** при исчерпании бюджета CF (вместо `None`).
`handlers/actions.py` проверяет результат:

- `result is BUDGET_EXCEEDED` → показать фиксированное сообщение:
  **«Дневной бесплатный лимит исчерпан, попробуйте завтра.»**
- `result is None` → прежнее «Не удалось выполнить действие. Попробуйте позже.» (ADR-009).
- иначе → `send_result` (ADR-007).

Сообщение о лимите — **на русском**, в стиле существующих служебных сообщений (НФТ-8); «часики»
callback снимаются (`callback.answer` уже вызывается до обработки — ADR-009/010). Сообщение
фиксируется как часть контракта релиза (точная формулировка выше).

> Альтернатива «сервис сам возвращает текст сообщения» отклонена: формирование пользовательского
> текста — ответственность хендлера, не сервиса (граница ADR-002/003). Sentinel сохраняет границу и
> расширяемость (другие хендлеры могут реагировать иначе).

### Решение 9. Дискаверабилити: метаданные и листинг

- **Метаданные (`handlers/commands.py`, `setup_bot_profile`).** Усилить ключевые слова в
  `BOT_DESCRIPTION` (≤ 512) и `BOT_SHORT_DESCRIPTION` (≤ 120) под поиск Telegram: распознавание речи,
  транскрипция, OCR, перевод, голос в текст, RU/EN. Команды (`/start`, `/help`) — часть контракта
  1.0.0 (ADR-010), их набор **не меняется**; допустимо уточнить тексты-описания команд при сохранении
  смысла. Изменение текстов description/short — **не ломающее** (это не часть semver-контракта команд/
  каналов/формата ответа ADR-010) → MINOR.
- **Листинг (`docs/listing.md`).** Готовый текст для бесплатных каталогов (BotoStore, TeleHunt и т. п.):
  название, короткое и полное описание, категории, теги/ключевые слова (RU, с EN-вариантом по
  применимости). Пригоден для ручного копирования без доработки.
- **Подача — ручной шаг** владельца, фиксируется в чек-листе деплоя (мейнтейнер, `/deploy`).

## Решение

1. **Единица бюджета — число вызовов CF за UTC-сутки** (прокси, Решение 1A).
2. **Дефолт `CF_DAILY_BUDGET = 300`** — консервативный, с запасом до free-tier; именованная
   константа, переопределяемая env; некорректное/неположительное значение → дефолт без падения старта
   (Решение 2).
3. **Новый модуль `services/budget.py`** рядом со `services/stats.py`: тот же паттерн (соединение-на-
   операцию, WAL, `busy_timeout`, UPSERT, `asyncio.to_thread`), **тот же файл БД** `STATS_DB_PATH`,
   **отдельная таблица `cf_usage(date TEXT PRIMARY KEY, count INTEGER)`** (Решения 3A, 4).
4. **Единый гейт**: `cf_budget_allow() -> bool` и `cf_budget_consume() -> None`, порядок «проверка →
   CF-вызов → инкремент»; применён в трёх местах — `transcribe.py` (CF ASR), `structure.py`
   (CF структурирование), `llm.py` (CF действия). Инкремент — только при фактическом обращении к CF;
   локальные пути и ручные режимы бюджет не тратят (Решение 5).
5. **Overshoot при конкуренции — принятый компромисс** (без глобальной сериализации CF); сам
   инкремент атомарен; бюджет консервативен (Решение 6).
6. **Политика при сбое учёта — fail-open** (разрешить CF + `logger.warning`), приоритет — работа бота;
   преемственность ADR-006 (Решение 7).
7. **Деградация без отказа**, логируется (`logger.warning`, ADR-010): ASR → локальный whisper;
   структурирование → сырой текст; действия → `BUDGET_EXCEEDED` → хендлер показывает «Дневной
   бесплатный лимит исчерпан, попробуйте завтра.» (Решения 5, 8).
8. **Дискаверабилити**: усиление ключевых слов в `setup_bot_profile` в пределах лимитов Telegram;
   `docs/listing.md`; ручная подача — в чек-листе деплоя (Решение 9).
9. **Версия — MINOR 1.1.0.** Публичный контракт 1.0.0 (ADR-010) не ломается: при неисчерпанном
   бюджете поведение идентично 1.0.0; при исчерпанном — мягкая деградация без отказа в распознавании.
   `roadmap.md`: v1.1.0 — выпущен; добавлены v1.2.0 (группы) и v1.3.0 (виральность) как план.

### Композиция «переключатель провайдера + бюджет» (закрывает ОВ-3)

Гейт по бюджету применяется **строго к фактическим обращениям к CF**, поэтому ставится **внутри
CF-ветки** каждого оркестратора, после разрешения провайдера/режима:

- `ASR_PROVIDER=local` → код уходит в `_transcribe_local` **до** гейта; CF не вызывается, счётчик не
  растёт.
- `LLM_PROVIDER=groq` → активен `_GroqProvider`; гейт CF не применяется (Groq — отдельный free-tier,
  вне рамок этого механизма), счётчик CF не растёт.
- Фолбэки CF→local (по размеру файла, по отсутствию кредов, по сбою CF) идут в локальный путь и
  счётчик не трогают.

Таким образом бюджет и переключатели не конфликтуют: бюджет — дополнительный гейт **поверх** выбора
провайдера, видимый только на пути к CF.

## Последствия

- **Положительные:** рост трафика **никогда** не приводит к оплате Cloudflare (автоматическая
  страховка, НФТ-1); пользователь не получает отказа (ASR/структурирование деградируют тихо, действия
  — понятным сообщением, НФТ-2); механизм переиспользует существующее хранилище и паттерн (ADR-006),
  не вводя новых зависимостей/портов/направлений трафика (НФТ-7); наблюдаемость деградации в
  `docker logs` (НФТ-3); готовый материал и зафиксированный шаг для бесплатного продвижения.
- **Отрицательные / компромиссы:** прокси по вызовам грубее точного учёта нейронов (компенсируется
  консервативным дефолтом); возможен малый overshoot при конкуренции (принят, Решение 6); при
  длительном сбое учёта страховка временно не действует (fail-open, Решение 7 — осознанный приоритет
  работы бота); при исчерпании бюджета качество деградирует (whisper `base` медленнее/слабее,
  структурирование пропадает) — это и есть цель (бесплатность важнее качества при пике), факт
  логируется.
- **Технический долг:** точный учёт нейронов и веса по типам вызовов (если консервативный прокси
  окажется слишком грубым на практике, ОВ-7); опциональная индикация остатка бюджета в `/stats`
  (ОВ-5 — не делаем); автоматизация подачи листинга (вне рамок); старые строки `cf_usage` не чистятся
  (объём ничтожен — при желании добавить периодическую очистку позже).

## Влияние на код

- **Новые модули:** `services/budget.py` (инициализация таблицы `cf_usage`; `cf_budget_allow`,
  `cf_budget_consume` — через `asyncio.to_thread`, паттерн ADR-006; чтение `CF_DAILY_BUDGET` с
  дефолтом). `docs/listing.md` (текст листинга).

  - `services/transcribe.py` — гейт + `consume` на CF-ветке `transcribe()` (перед
    `_transcribe_cloudflare`), деградация в `_transcribe_local`; `logger.warning` о деградации.
  - `services/structure.py` — гейт + `consume` в `structure_text()` для CF-провайдера; деградация в
    сырой текст; способ определить «провайдер — Cloudflare».

    `BUDGET_EXCEEDED` при исчерпании; `logger.warning`.

  - `.env.example`, `deploy/.env.example` — плейсхолдер/дефолт `CF_DAILY_BUDGET`.
  - `roadmap.md` — v1.1.0 выпущен; v1.2.0 (группы), v1.3.0 (виральность) как план.
