# ADR-014: Проактивные группы — оффер-кнопка «Распознать» на медиа при отключённом Group Privacy, распознавание по тапу

- **Дата:** 2026-06-23
- **Статус:** принято
- **Связанные BA-требования:** `ba-req/ba-requirements-014.md`
- **Связанные ADR:** ADR-012 (режим групп — **надстраивается**: смена предпосылки privacy ON → privacy OFF, судьба прежнего триггера), ADR-009 (inline-кнопки «Кратко»/«Перевести», callback, правило `callback.answer()`, callback не считаются в статистике), ADR-007 (`send_result` — отдача результата reply на медиа), ADR-010 (семафор на тяжёлые локальные операции, `ChatActionSender`, публичный контракт 1.0.0 лички — не ломается), ADR-011 (дневной бюджет CF — переиспользуется), ADR-008 (ASR-пайплайн, `transcribe(force_local=...)`), ADR-006 (учёт `StatsMiddleware`, схема `user_stats` — не меняется), ADR-004/005 (OCR/структурирование — переиспользуются), ADR-001/002/003 (профиль поставки, границы «хендлер ↔ сервис», неблокирование — сохраняются)

## Контекст

Релизы v0.1.0–v1.3.0 в проде на ivahub (1 vCPU / 1.9 GB RAM, long-polling, Docker, образ из GHCR;
профиль поставки «бот без HTTP-порта» — ADR-001). v1.2.0 (ADR-012) ввёл режим групп **по явному
запросу**: пользователь делает reply на медиа с `/textify`/упоминанием бота, бот берёт медиа из
`reply_to_message` и прогоняет через общие функции обработки (`process_audio`, `process_photo`,
`process_image_document`), отдавая результат через `send_result` (ADR-007) с кнопками
«Кратко»/«Перевести» (ADR-009). ASR в группах форсируется локальным по `GROUP_ASR_LOCAL` (дефолт on),
не расходуя дневной CF-бюджет лички (ADR-011).

Сверхцель продукта — максимум пользователей бесплатно. Текущий групповой режим требует **знания
команды** `/textify` или умения упомянуть бота — это трение, а сам бот в группе остаётся невидимым,
пока его явно не позовут. BA-требование 014 (MINOR, v1.4.0) делает группы **проактивными**: бот сам
замечает появившееся медиа и предлагает распознать его компактной inline-кнопкой **«Распознать»**
reply на это медиа; расшифровка происходит **только по тапу**. Это убирает трение и делает бота
видимым всем участникам при каждом голосовом/фото — сильный органический рост без спама и без
перегрузки 1 vCPU (распознавание — по тапу, под семафором, локальный ASR).

Действующие силы и ограничения:

- **Смена предпосылки ADR-012 — Group Privacy (ОВ-7, ОГР-1).** ADR-012 строился на **включённом**
  privacy mode: бот видит в группе только адресованные ему команды/упоминания/ответы, и явно
  фиксировал «отключать privacy не требуется и нежелательно». Чтобы бот видел **любое** медиа группы
  и мог предложить кнопку, privacy mode должен быть **отключён** (`/setprivacy` → Disable + заново
  добавить бота). Это не противоречие требований, а **сознательная смена режима работы в группе**.
  Новый ADR должен явно надстроить ADR-012 под privacy OFF, сохранив «no spam» (реакция только на
  медиа, один оффер на медиа) и неизменность лички (ADR-010).
- **«No spam» при privacy OFF (НФТ-1).** При отключённом privacy Telegram доставляет боту **все**
  сообщения группы. Бот обязан реагировать **только** на поддерживаемое медиа (оффером) и на
  нажатие собственной кнопки; на обычный текст и нерелевантный трафик — молчать. Один оффер на одно
  медиа-сообщение.
- **Источник медиа при тапе (ОГР-6).** Оффер отправляется **reply на медиа**, поэтому при тапе
  исходное медиа доступно как `callback.message.reply_to_message`. Если оно недоступно
  (`None`/`InaccessibleMessage`/удалено/нет поддерживаемого медиа) — распознавание невозможно,
  обрабатывается служебно, без падения (ФТ-8).
- **Лимит `callback_data` 64 байта (ОГР-2).** Данные кнопки компактны и не несут содержимого медиа;
  исходное медиа берётся из `reply_to_message`, а не из `callback_data`.
- **Судьба прежнего триггера v1.2.0 (ОВ-1, ФТ-10).** Прежний путь «reply на медиа +
  `/textify`/упоминание» нужно либо сохранить как дополнительный, либо заменить кнопкой; приоритет —
  кнопка. При сохранении обоих путей — исключить дублирующее распознавание одного медиа.
- **Защита слабого сервера и бесплатности (НФТ-2, НФТ-3).** Распознавание — только по тапу, под
  глобальным семафором (ADR-010, лимит 1); группы → локальный ASR по `GROUP_ASR_LOCAL`; OCR локален;
  оффер-кнопка — лёгкая операция (reply с разметкой), семафор не держит.
- **Границы (ADR-002/003, НФТ-10).** Проактивный оффер и обработка callback живут в слое хендлеров;
  сервисы (OCR/ASR/LLM) про проактивный режим не знают; переиспользуются общие функции обработки из
  `handlers/audio.py`/`handlers/image.py` без дублирования пайплайна.
- **Профиль поставки (ADR-001, НФТ-5).** Без новых портов/внешних зависимостей/направлений трафика;
  новая обязательная env не нужна.

## Рассматриваемые варианты

### Решение 1. Group Privacy: ON (ADR-012) vs OFF (проактивный режим) — надстройка ADR-012

#### Вариант A: оставить privacy ON, дополнить v1.2.0-триггер
- Плюсы: ничего не менять в эксплуатации; бот видит только адресованное.
- Минусы: **технически невозможен** проактивный оффер — при privacy ON Telegram не доставляет боту
  произвольные медиа группы, бот не узнаёт о появлении голосового/фото, кнопку предложить не может.
  Не выполняет цель требования.

#### Вариант B: privacy OFF + проактивный оффер только на медиа
- Плюсы: бот видит всё медиа группы и предлагает кнопку — выполняет цель (убрать трение, проактивная
  экспозиция); «no spam» обеспечивается **фильтром группового роутера**: оффер только на
  поддерживаемое медиа, на текст бот молчит; распознавание — только по тапу.
- Минусы: бот видит весь трафик группы (приватность участников — отмечено в рисках); требуется ручной
  шаг @BotFather (`/setprivacy` → Disable + заново добавить бота); при наличии бота в нескольких
  группах он видит весь их медиатрафик (нагрузка — смягчена семафором/локальным ASR, расшифровка по
  тапу).

Выбран **B**. Это **сознательная смена режима относительно ADR-012**, а не противоречие: ADR-012
описывал режим под privacy ON (реакция только на reply+триггер), ADR-014 переводит группу в
проактивный режим под privacy OFF (реакция только на медиа оффером). Прежняя предпосылка ADR-012
(«privacy ON, отключать нежелательно») **надстраивается**: для v1.4.0 privacy OFF — обязательное
условие режима. Отключение privacy — обязательный **ручной** шаг деплоя (чек-лист `/deploy`, README),
без него офферы не появляются и бот не падает (поведение при privacy ON — отсутствие офферов).

### Решение 2. Реакция на медиа в группе: оффер-кнопка vs авто-распознавание

#### Вариант A: проактивный оффер-кнопкой, распознавание по тапу
- Плюсы: текст не вываливается без спроса (не спам в активной группе); распознавание контролируемо
  (только по тапу) — не перегружает 1 vCPU; кнопка видна всем участникам — органическая экспозиция;
  оффер — лёгкая операция (reply с разметкой), семафор не держит.
- Минусы: оффер на каждое медиа в активной группе = умеренный шум (митигировано «только медиа, один
  оффер на сообщение»; настройки на уровне группы — вне scope).

#### Вариант B: авто-распознавание всех медиа без тапа
- Минусы: спам (бот вываливает текст на каждое голосовое/фото); перегрузка 1 vCPU при всплеске
  голосовых; явно вне scope требования.

Выбран **A** (исходное требование, КП-1/ФТ-3). Авто-распознавание без тапа — вне рамок.

### Решение 3. Источник медиа при тапе и формат `callback_data` (ОВ-3, ФТ-7)

#### Вариант A: единый `callback_data` (`grec`), тип медиа определяется по `reply_to_message`
- Плюсы: один callback-хендлер, минимум кода; тип медиа (аудио/изображение) и так нужно определять по
  содержимому `reply_to_message` (там лежит реальное медиа); `callback_data` минимален (4 байта),
  заведомо в лимите 64; различение голос/текст в подписи кнопки — на этапе оффера, по типу медиа.
- Минусы: нет — определение типа по `reply_to_message` всё равно необходимо при тапе.

#### Вариант B: типизированный `callback_data` (`grec:a` / `grec:i`)
- Плюсы: тип известен из callback без чтения reply.
- Минусы: тип всё равно нужно перепроверить по `reply_to_message` (медиа могло быть удалено/изменено);
  два префикса = чуть больше кода; реальной выгоды нет.

Выбран **A** — единый `callback_data = "grec"`. Источник медиа — **только** `callback.message.reply_to_message`
(`callback_data` содержимого медиа не несёт — НФТ-8, ОГР-2). При тапе тип медиа определяется тем же
диспетчером по содержимому reply, что используется при триггере (переиспользование `_dispatch_media`).

### Решение 4. Судьба прежнего триггера v1.2.0 (ОВ-1, ФТ-10)

#### Вариант A: сохранить прежний триггер как дополнительный путь; приоритет UX — кнопка
- Плюсы: обратная совместимость для пользователей, привыкших к `/textify`/упоминанию (v1.2.0 в
  проде); дёшево — триггер-хендлеры и `_dispatch_media` уже существуют в `handlers/group.py`; оба
  пути (триггер и оффер-тап) ведут в **общий код** обработки reply-медиа (`_dispatch_media`) —
  единый пайплайн, нет дублирования логики.
- Минусы: формально два пути запуска; но дублирующего распознавания одного медиа не возникает —
  триггер запускается на reply+команду/упоминание, оффер-тап на callback кнопки; это разные
  пользовательские действия, каждое осознанное.

#### Вариант B: заменить триггер кнопкой (удалить триггер-хендлеры)
- Плюсы: один путь запуска.
- Минусы: ломает привычку пользователей v1.2.0; удаляет рабочий код без пользы; кнопка и триггер не
  конфликтуют. Регресс UX для тех, кто уже знает команду.

Выбран **A**. Прежний триггер v1.2.0 **сохраняется как дополнительный путь**, приоритетный UX —
проактивная кнопка. Оба пути ведут в общий `_dispatch_media(reply, bot)`. Дублирующего распознавания
одного медиа нет (триггер и тап — разные действия пользователя; «один оффер на медиа» гарантирует, что
сам оффер не дублируется).

### Решение 5. Граничные случаи при тапе (ФТ-8)

`callback.message.reply_to_message` может быть `None`, `InaccessibleMessage` (сообщение
удалено/устарело) или содержать неподдерживаемое медиа. Решение: callback-хендлер проверяет, что reply
— это `Message` (не `None`, не `InaccessibleMessage`) и содержит поддерживаемое медиа
(`_has_supported_media`); иначе показывает короткое русское служебное сообщение через
`callback.answer(..., show_alert=...)`, **не** вызывает пайплайн и **не** падает. `callback.answer()`
вызывается **всегда** (снять «часики» — преемственность ADR-009, Решение 5). При долгой обработке —
`callback.answer("Распознаю…")` и индикатор `ChatActionSender` (typing для аудио / upload_document для
изображений, ADR-010) — это уже встроено в общие функции обработки (`process_audio`/`process_photo`/
`process_image_document`).

### Решение 6. Учёт статистики (ОВ-5, ADR-006)

`StatsMiddleware` — outer-middleware на `dp.message`; групповые **медиа-сообщения** уже проходят через
него и учитываются существующим механизмом (`classify_message` по содержимому) — это приемлемо
(ФТ-12). Callback «grec» (нажатие кнопки «Распознать») **в статистике не считается** —
преемственность ADR-009 (Решение 4: callback'и в `/stats` не считаются; middleware висит на
`dp.message`, не на `dp.callback_query`). Схема `user_stats` и контракт `/stats` **не меняются**.
Метрика конверсии оффер→тап — отдельным релизом при необходимости (вне scope).

### Решение 7. Env для проактивного режима (ОВ-6)

Новая env **не вводится**. Режим включается фактом отключённого privacy + наличием бота в группе;
отдельный конфиг не обязателен (минимализм CLAUDE.md). Бесплатность группового ASR уже управляется
существующим `GROUP_ASR_LOCAL` (ADR-012). Профиль поставки ADR-001 без изменений; новых обязательных
секретов нет.

## Решение

1. **Group Privacy OFF — смена режима, надстройка ADR-012 (Решение 1B).** v1.4.0 работает при
   **отключённом** Group Privacy (`/setprivacy` → Disable + заново добавить бота — обязательный ручной
   шаг деплоя). Это сознательная смена предпосылки ADR-012 (которая строилась на privacy ON). «No
   spam» сохраняется фильтром группового роутера: реакция только на медиа. Личка (ADR-010) неизменна.
2. **Проактивный оффер-кнопка (Решение 2A).** В `group`/`supergroup` на сообщение с поддерживаемым
   медиа (`voice`/`audio`/`photo`/`document image/*`/`document audio/*`) бот отвечает **reply на это
   медиа** одним сообщением с компактной inline-кнопкой «Распознать»: для аудио/голоса (включая
   документ-аудио) — **«Распознать голос»**; для изображений (фото и документ-изображение) —
   **«Распознать текст»**. Распознанный текст в момент оффера **не отправляется** (только по тапу).
   На обычный текст и прочий трафик в группе бот **не реагирует**.
3. **Один оффер на одно медиа.** Бот по построению получает каждое входящее медиа-сообщение один раз
   и отвечает на него один раз (одно входящее = один ответ-оффер). Персистентный дедуп по `message_id`
   **не вводится** (нет хранилища состояния, не нужен): хендлер срабатывает на апдейт `message`
   (новое сообщение), а не на `edited_message`, поэтому редактирование/повторная доставка не плодят
   офферы. Это достаточная гарантия ФТ-4 без дополнительного кода.
4. **Распознавание по тапу (Решение 3A, 5).** Единый `callback_data = "grec"`. Callback-хендлер:
   (а) `callback.answer()` (снять часики) — всегда; (б) проверяет `callback.message.reply_to_message`:
   это должен быть `Message` (не `None`/`InaccessibleMessage`) с поддерживаемым медиа
   (`_has_supported_media`) — иначе короткое русское служебное сообщение через `callback.answer(...,
   show_alert=True)`, пайплайн не вызывается, бот не падает; (в) при валидном медиа —
   `callback.answer("Распознаю…")` и вызов общего диспетчера `_dispatch_media(reply, bot)`, который
   определяет тип медиа, скачивает его, прогоняет через `process_audio` (с `force_local` по
   `GROUP_ASR_LOCAL`) / `process_photo` / `process_image_document` (OCR локален), отдаёт результат
   через `send_result` (ADR-007) **reply на исходное медиа** с кнопками «Кратко»/«Перевести» (ADR-009);
   индикатор прогресса (`ChatActionSender`) — внутри общих функций (ADR-010).
5. **Прежний триггер v1.2.0 — сохранён как дополнительный путь (Решение 4A).** Хендлеры
   `Command("textify")` и упоминание бота остаются; приоритетный UX — кнопка. Оба пути (триггер и
   оффер-тап) ведут в общий `_dispatch_media`. Дублирующего распознавания одного медиа нет.
6. **Защита ресурсов и бесплатность (Решение 1B/2A).** Распознавание — только по тапу, под семафором
   (ADR-010); группы → локальный ASR (`GROUP_ASR_LOCAL`, дефолт on) — без расхода CF-бюджета; OCR
   локален. Оффер-кнопка — лёгкая операция (reply с разметкой), семафор не держит.
7. **Учёт (Решение 6).** Групповые медиа-сообщения учитываются `StatsMiddleware` существующим
   механизмом; callback «grec» **не считается** (преемственность ADR-009); схема `user_stats`
   неизменна.
8. **Env и поставка (Решение 7).** Новая env не вводится. Профиль поставки ADR-001 без изменений;
   новых внешних зависимостей и направлений трафика нет. Единственное эксплуатационное изменение —
   ручной шаг отключения Group Privacy (чек-лист `/deploy`, README).
9. **Личка не затрагивается.** Приватные хендлеры `handlers/image.py`/`handlers/audio.py` остаются
   `F.chat.type == "private"` (авто-распознавание без кнопок); `text_router` — private-only (в группе
   на текст не отвечать); контракт 1.0.0 (ADR-010) сохранён.
10. **Версия — MINOR 1.4.0.** Обратимо-совместимое добавление проактивного режима; публичный контракт
    1.0.0 лички не ломается. `roadmap.md`: v1.4.0 «Проактивные группы (кнопка Распознать)» — выпущен.

### Что НЕ входит (границы)

- Авто-распознавание медиа без тапа — нет (только оффер + тап).
- Настройки на уровне группы (вкл/выкл оффера админом, белые/чёрные списки чатов) — нет.
- Антиспам по частоте офферов сверх «один оффер на одно медиа» (rate-limit, троттлинг) — нет.
- Персистентный дедуп оффера по `message_id` — нет (достаточно «одно входящее = один ответ»).
- Изменение поведения в личке (ADR-010/012) — нет (авто-распознавание без офферов сохраняется).
- Учёт callback «grec» в статистике — нет (преемственность ADR-009).

## Последствия

- **Положительные:** группы становятся проактивными — кнопка «Распознать» видна всем участникам при
  каждом медиа, убирая трение (не нужно знать команду) и усиливая органический рост; распознавание
  по тапу не спамит и не перегружает 1 vCPU; личка неизменна (контракт 1.0.0); групповой ASR
  по-прежнему локален (CF-бюджет лички не расходуется); прежний триггер v1.2.0 сохранён (обратная
  совместимость) при общем коде обработки; ноль новых зависимостей, портов, env и направлений трафика
  (профиль ADR-001 сохранён).
- **Отрицательные / компромиссы:** privacy OFF означает, что бот **видит весь трафик группы**
  (приватность участников — осознанный компромисс ради проактивности; бот реагирует только на медиа и
  не сохраняет содержимое — ADR-006/НФТ-8); оффер на каждое медиа в активной группе — умеренный шум
  (митигировано «только медиа, один оффер на сообщение»); при наличии бота в нескольких группах он
  видит весь их медиатрафик (нагрузка смягчена семафором/локальным ASR, расшифровка — по тапу);
  обязательный ручной шаг @BotFather (privacy OFF) — без него режим не активен.
- **Технический долг:** настройки на уровне группы и rate-limit офферов — при появлении спроса;
  метрика конверсии оффер→тап — при необходимости; возможный персистентный дедуп — если обнаружатся
  дубли офферов на проде.
