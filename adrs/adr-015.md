# ADR-015: Прогрессивное раскрытие результата (суть → кратко → полностью), in-memory кэш исходного текста под кнопки, удаление «Перевести»

- **Дата:** 2026-06-23
- **Статус:** принято; **частично заменено ADR-016** — Решение 6 (удаление функции/кнопки «Перевести»)
  реверсировано: перевод возвращён в продукт (см. ADR-016, Решение 1). Прочие решения ADR-015
  (прогрессивное раскрытие, in-memory кэш, ленивый расчёт уровней) остаются в силе и надстраиваются.
- **Связанные BA-требования:** `ba-req/ba-requirements-015.md`
- **Связанные ADR:** ADR-009 (надстраивается: меняется источник текста для callback'ов, набор кнопок и поведение «Перевести» удаляется; перевод как функция выводится из контракта), ADR-010 (публичный контракт 1.0.0 лички — раздел inline-действий и формат немедленной выдачи меняются, см. «Влияние на контракт»), ADR-007 (`send_result` — отдача уровня «Полностью» и «Кратко», переиспользуется без изменений), ADR-014 (группы — охват прогрессивного раскрытия, дефолт «оставить прежнюю выдачу» с удалением «Перевести»), ADR-011 (дневной бюджет CF — дополнительный LLM-вызов на «суть», стратегия деградации), ADR-005 (структурирование `structure_text` — уровень «Полностью», переиспользуется), ADR-002/003 (граница «хендлер ↔ сервис», неблокирование), ADR-001 (профиль поставки «бот без HTTP-порта» — без изменений)

## Контекст

Релизы v0.1.0–v1.4.0 в проде на ivahub (1 vCPU / 2 GB RAM, long-polling, Docker, образ из GHCR; профиль
поставки «бот без HTTP-порта» — ADR-001). Сейчас на любое распознанное медиа в личке бот **немедленно**
отдаёт полный структурированный текст (`structure_text`, ADR-005) через `send_result` (ADR-007) с двумя
inline-кнопками — «Кратко» (`act:sum`) и «Перевести» (`act:tr`) (ADR-009). На коротком голосовом или
картинке это «стена текста» в ответ на простой запрос.

BA-требование 015 (MINOR, v1.5.0) вводит **прогрессивное раскрытие** в три уровня детализации:

- **Суть (превью)** — 1–2 предложения, показывается сразу, без действий пользователя;
- **Кратко** — саммари по ключевым пунктам (текущее поведение `summarize`), по кнопке;
- **Полностью** — полный структурированный текст (`structure_text`), по кнопке.

Под сутью — ровно две кнопки: «Показать полностью» и «Кратко». Кнопка «Перевести» **удаляется**.

Действующие силы и ограничения (из BA, п. 6 + открытые вопросы):

- **ОВ-1 — источник текста для кнопок.** ADR-009 строился на том, что текст видимого сообщения
  (`callback.message.text`) и есть полный результат — источник для «Кратко»/«Перевести». Теперь текст
  сообщения с кнопками — это **суть** (1–2 предложения), а не полный/исходный текст. Восстановить из сути
  ни «полностью», ни «кратко» нельзя (содержание потеряно). Нужен источник исходного распознанного текста
  при тапе, **без тяжёлого внешнего стейта** (ОГР-1: 1 vCPU / 2 GB, long-polling, без очередей/внешних
  БД-стора под это).
- **ОВ-2 — бюджет CF.** «Суть» — дополнительный LLM-вызов на **каждое** распознанное медиа сверх
  структурирования. Это ~удвоение расхода `CF_DAILY_BUDGET` (ADR-011) на путь распознавания лички. Нужна
  осознанная деградация при исчерпании бюджета (ФТ-10, КП-11), без падения.
- **ОВ-3 — идемпотентность кнопок.** Повторные тапы «Показать полностью»/«Кратко» — слать заново или нет.
  С продуктовой стороны допустимы оба варианта при отсутствии падения (КП-12).
- **ОВ-4 — охват групп.** Применять ли прогрессивное раскрытие в группах (`handlers/group.py`, ADR-014).
  Продуктовый дефолт BA — оставить группам прежнюю выдачу, но убрать «Перевести».
- **Лимит `callback_data` 64 байта** — содержимое медиа в кнопку не положить; нужен компактный ключ.
- **`send_result` (ADR-007)** уже умеет отдавать длинный текст (части/файл) и прикрепляет `reply_markup`
  только в одно-сообщенческом режиме (ADR-009, Решение 1). Уровень «Полностью» переиспользует это
  без изменений.
- **Неблокирование (ADR-002/003)** — доп-вызов LLM на «суть» нативно async (httpx), event loop не
  блокирует.
- **Профиль поставки ADR-001** — без новых портов/направлений трафика. CF-направление уже используется.

## Рассматриваемые варианты

### Решение 1. Источник исходного текста для кнопок при тапе (закрывает ОВ-1)

#### Вариант A: in-memory LRU-кэш `message_id сообщения-с-сутью → исходный распознанный текст`
- При успешном распознавании хендлер: получает `raw_text` (OCR/ASR), генерирует «суть», отправляет
  сообщение с сутью и кнопками, **по `message_id` отправленного сообщения** кладёт `raw_text` в общий
  кэш (`dict` с ограничением размера + TTL). Callback по тапу берёт `raw_text` из кэша по
  `callback.message.message_id` и прогоняет через `structure_text` («Полностью») или `summarize`
  («Кратко»).
- Плюсы: исходный текст доступен **целиком** независимо от длины (снимает ограничение ADR-009 «кнопки
  только под одно-сообщенческим результатом» — теперь сообщение-с-сутью всегда короткое, а полный текст
  лежит в кэше); один источник истины (исходный распознанный текст), оба уровня пересобираются из него
  свежим LLM-вызовом по требованию; кэш — чистый процесс-локальный `dict`, не внешний стор (ОГР-1
  соблюдён); естественная связка с `message_id`.
- Минусы: кэш **не переживает рестарт** контейнера и вытесняется по размеру/TTL — после рестарта или по
  истечении TTL тап по старой кнопке даёт «текст недоступен» (детерминированно, без падения — КП-12);
  память на кэш ограничивается LRU + TTL (на 1 vCPU / 2 GB при умеренном трафике лички — пренебрежимо).
- Последствия: новый лёгкий модуль кэша; хендлеры распознавания регистрируют `raw_text` после отправки
  сообщения с сутью; callback-хендлеры читают из кэша вместо `callback.message.text`.

#### Вариант B: предрасчёт всех уровней при распознавании, текст «полностью» прячется в сообщении
- Скрыть полный текст в самом сообщении (zero-width/служебный блок) или предрассчитать все три уровня
  заранее.
- Минусы: zero-width-приём хрупок, ломает атрибуцию и упирается в лимит 4096 (длинный текст не прячется);
  предрасчёт всех уровней = 3 LLM-вызова на каждое медиа (втрое бьёт по бюджету ОВ-2) ради уровней,
  которые пользователь может не открыть. Противоречит цели экономии.
- Последствия: либо хрупко, либо дорого по бюджету.

#### Вариант C: остаться на `callback.message.text` + ничего не прятать
- Минусы: текст сообщения теперь — суть; пересборка «кратко»/«полностью» из сути теряет содержание
  (ФТ-9: уровни обязаны быть монотонно полнее, из 1–2 предложений полный текст не получить). Нарушает
  КП-4/КП-5/КП-9. Неприемлемо.

Выбран **A**. In-memory LRU+TTL кэш `message_id → raw_text` — минимальный стейт, укладывающийся в ОГР-1,
дающий **целостный** исходный текст для обоих уровней без удвоения/утроения LLM-вызовов и без хрупких
приёмов. Утрата кэша при рестарте/TTL обрабатывается служебным сообщением (КП-12). Это **надстройка
ADR-009**: источник текста для callback'ов меняется с «текст видимого сообщения» на «исходный текст по
ключу `message_id`».

### Решение 2. Генерация «сути»: где и как (закрывает часть ОВ-2)

#### Вариант A: новая операция `summarize_gist(text) -> str | None | BUDGET_EXCEEDED` в `services/llm.py`
- Плюсы: «суть» — новое действие над текстом через LLM, его место — `services/llm.py` рядом с `summarize`
  (граница «одна задача — один сервис», ADR-009 Решение 2); переиспользует тот же провайдерный слой и
  бюджет-гейт (`cf_budget_allow`/`cf_budget_consume`), что `structure_text`/`summarize`; промпт —
  «изложи суть в 1–2 предложения на языке оригинала, ничего не добавляй» (преемственность анти-инъекции
  промптов структуры/саммари).
- Минусы: ещё одна функция в `llm.py` — приемлемо.
- Последствия: один новый публичный async-метод; контракт возврата как у `summarize`
  (`str | None | _BudgetExceededType`).

#### Вариант B: вычислять суть внутри `structure_text` одним вызовом (структура + суть)
- Минусы: смешивает две задачи в одном промпте/ответе (парсинг двух частей из одного ответа — хрупко);
  размывает `structure.py`; усложняет деградацию (нельзя независимо отдать структуру при провале сути).
- Последствия: хрупкий парсинг, размывание ответственности.

Выбран **A**: отдельная операция `summarize_gist` в `services/llm.py`, та же модель провайдера и тот же
бюджет-гейт.

### Решение 3. Влияние на бюджет и деградация «сути» при исчерпании (закрывает ОВ-2, ФТ-10/КП-11)

Путь распознавания лички теперь делает **два** CF-вызова на медиа: `structure_text` (как раньше — но
он расходуется лениво только при тапе «Полностью», см. ниже) и `summarize_gist` (новый, на каждое медиа).
Чтобы не удваивать расход вслепую, фиксируется **ленивая модель**:

- **«Суть» считается сразу** (один CF-вызов на медиа) — это и есть немедленный ответ.
- **«Полностью» и «Кратко» считаются по тапу** (`structure_text`/`summarize`) — пользователь, которому
  хватило сути, не тратит бюджет на структурирование. Это **снижает** средний расход по сравнению с
  текущим поведением (где `structure_text` считался всегда), частично компенсируя добавленный вызов сути.

Деградация «сути» при `BUDGET_EXCEEDED` / сбое LLM / пустом ответе (выбор зафиксирован):

#### Вариант A: при недоступности сути — отдать сырой распознанный текст превью + кнопки
- Минусы: сырой OCR/ASR может быть длинным — не «суть», ломает идею превью; для длинного сырого текста
  кнопки под одно-сообщенческим режимом не показались бы.

#### Вариант B (выбран): при недоступности сути — короткое служебное превью-сообщение + те же две кнопки
- При `summarize_gist` → `BUDGET_EXCEEDED`: отправить короткое служебное сообщение
  («Суть недоступна (дневной лимит). Нажмите «Показать полностью» или «Кратко».») **с теми же двумя
  кнопками**; `raw_text` всё равно кладётся в кэш по `message_id` → кнопки рабочие. Бот не падает (КП-11).
- При `summarize_gist` → `None` (сбой/пустой ответ): аналогичное короткое служебное превью
  («Не удалось сформировать суть. Доступны «Показать полностью» и «Кратко».») + те же кнопки + `raw_text`
  в кэш.
- Плюсы: предсказуемая деградация без падения; пользователь сохраняет доступ к обоим уровням; превью
  остаётся коротким; расход бюджета на тап «Полностью»/«Кратко» происходит только по явному действию.

Выбран **B**. Деградация сути → короткое служебное превью с рабочими кнопками (исходный текст уже в кэше).

### Решение 4. Идемпотентность повторных тапов (закрывает ОВ-3)

#### Вариант A (выбран): каждый тап формирует ответ заново, без дедупликации
- Преемственность ADR-009 (Решение 5: «каждое нажатие формирует ответ заново»). Простая предсказуемая
  семантика; повторный тап «Полностью»/«Кратко» — ещё один LLM-вызов и ещё одно сообщение. Падения нет
  (КП-12). Каждый тап под бюджет-гейтом (`structure_text`/`summarize` уже содержат гейт) — повторные тапы
  при исчерпанном бюджете деградируют штатно (структура → сырой текст для «Полностью»; «Кратко» →
  `BUDGET_EXCEEDED` → служебное сообщение).
- Минусы: повторные тапы тратят бюджет — приемлемо, объёмы малы; антиспам по тапам — вне scope.

#### Вариант B: дедуп — помнить, что уровень уже отправлен, второй тап игнорировать/всплывашка
- Минусы: требует доп-состояния «какие уровни уже отданы для message_id» (рост стейта, гонки); выгода —
  только экономия редких повторных тапов. Избыточно для v1.5.0.

Выбран **A** — без дедупликации, преемственно ADR-009.

### Решение 5. Охват групп (закрывает ОВ-4)

#### Вариант A (выбран, продуктовый дефолт BA): группы оставляют прежнюю выдачу, «Перевести» убирается
- В группах (`handlers/group.py`, путь `_dispatch_media` → `process_audio`/`process_photo`/
  `process_image_document`) сохраняется немедленная выдача полного структурированного текста; меняется
  только клавиатура — она теперь содержит **одну** кнопку «Кратко» (через ту же обновлённую
  `actions_keyboard`), «Перевести» отсутствует (ФТ-6, КП-10). Превью-суть в группы **не вносится**.
- Плюсы: единообразие по удалению «Перевести»; экономия бюджета (в группах ASR локален, но «суть» —
  это CF-вызов; не вводя суть в группы, не нагружаем CF групповым трафиком); меньший объём релиза;
  личка и группы используют общий хендлер распознавания — нужно развести поведение (см. ниже).
- Минусы: UX лички и групп различается (в личке — суть+кнопки, в группе — полный текст+«Кратко»).
  Полный перенос превью в группы — отдельная итерация.

#### Вариант B: применить прогрессивное раскрытие и в группах
- Минусы: «суть» в группе = CF-вызов на каждое распознавание группой (бюджет, ОГР-3); больше объём;
  выходит за продуктовый дефолт BA (требует подтверждения владельца).

Выбран **A**. **Важное следствие для архитектуры:** `process_audio`/`process_photo`/
`process_image_document` сейчас **общие** для лички и групп (личка зовёт напрямую, группа — через
`_dispatch_media`). Прогрессивное раскрытие должно применяться **только в личке**. Чтобы не дублировать
пайплайн, в общие функции обработки добавляется флаг режима выдачи (например, `progressive: bool`,
по умолчанию `False` = прежняя выдача полного текста + клавиатура «Кратко»). Личные хендлеры
(`handlers/audio.py`/`handlers/image.py`) вызывают с `progressive=True`; групповой `_dispatch_media`
оставляет дефолт `False`. Это сохраняет единый пайплайн (ADR-014, без дублирования) и разводит поведение
одним параметром.

### Решение 6. Удаление функции `translate` (ОВ исходного требования)

> **Реверсировано ADR-016 (2026-06-24).** Функция `translate` и кнопка «Перевести» возвращены в
> продукт в v1.6.0 по BA-016 (паритет по возможностям). Текст ниже сохранён как историческая запись
> решения v1.5.0; актуальное состояние — ADR-016, Решение 1.


Кнопка «Перевести» удаляется из клавиатуры; callback `act:tr` и хендлер `handle_translate` удаляются.
Функция `translate` в `services/llm.py` после этого **больше нигде не используется** → удаляется вместе
с промптом `_TRANSLATE_SYSTEM` и хелперами направления перевода (`_has_cyrillic`/`_target_language`),
**если** они не нужны для других операций. (Проверка: `_has_cyrillic`/`_target_language` использовались
только для `translate` — удаляются; язык «сути» определяет сама LLM по тексту, как в `structure`/
`summarize`.) Это убирает мёртвый код и упрощает контракт. Удаление `translate` — внутренняя функция
сервиса, не публичный пользовательский контракт; пользовательский контракт меняется в части «нет кнопки
Перевести» (см. ниже).

## Решение

1. **Прогрессивное раскрытие в личке (Решение 1A, 2A, 3B).** На успешно распознанном медиа в личке
   хендлер: (а) получает `raw_text` (OCR/ASR); (б) вызывает `summarize_gist(raw_text)` — «суть»;
   (в) отправляет **первым** сообщением суть (или служебное превью при недоступности сути — Решение 3B)
   с inline-клавиатурой из двух кнопок «Показать полностью» (`act:full`) и «Кратко» (`act:sum`);
   (г) кладёт `raw_text` в in-memory кэш по `message_id` отправленного сообщения. Полный
   структурированный текст и саммари **не считаются** на этом этапе.
2. **Кэш исходного текста (Решение 1A).** Новый процесс-локальный кэш `message_id → raw_text` с
   ограничением размера (LRU) и TTL (именованные константы). Публичные функции: положить
   (`put(message_id, text)`), взять (`get(message_id) -> str | None`). Чистый модуль без сети/БД,
   укладывается в ОГР-1. Утрата записи (рестарт/TTL/вытеснение) → callback отдаёт «Текст недоступен»
   (КП-12), бот не падает.
3. **Новая операция «суть» (Решение 2A).** В `services/llm.py` —
   `async def summarize_gist(text: str) -> str | None | _BudgetExceededType`: тот же провайдерный слой и
   бюджет-гейт, что у `summarize`/`structure_text`; промпт «изложи суть в 1–2 предложения на языке
   оригинала, ничего не добавляя/не отвечая на текст»; усечение входа `MAX_INPUT_CHARS`; единичная
   попытка + таймаут; все исключения ловятся, при сбое/пустом ответе → `None`.
4. **Callback «Показать полностью» (`act:full`).** Хендлер: `callback.answer("Готовлю…")`; проверяет, что
   `callback.message` — `Message`; берёт `raw_text = cache.get(callback.message.message_id)`; если `None`
   → `callback.answer("Текст недоступен", show_alert=True)` и выход (LLM не вызывается); иначе под
   `ChatActionSender(typing)` вызывает `structure_text(raw_text)` (ADR-005) и отдаёт результат через
   `send_result(callback.message, result)` **без** `reply_markup` (преемственность ADR-009 — нет цепочки
   действий над производным текстом). `structure_text` уже деградирует на сырой текст при исчерпании
   бюджета/сбое (ADR-005/011) — отдельной обработки не требуется.
5. **Callback «Кратко» (`act:sum`).** Хендлер: `callback.answer("Готовлю…")`; проверка `Message`; берёт
   `raw_text` из кэша (как в п. 4); при `None` → «Текст недоступен»; иначе под `ChatActionSender(typing)`
   вызывает `summarize(raw_text)`; при `BUDGET_EXCEEDED` → служебное «Дневной бесплатный лимит исчерпан,
   попробуйте завтра.»; при `None` → «Не удалось выполнить действие. Попробуйте позже.»; при успехе →
   `send_result(callback.message, result)` без `reply_markup`. Источник текста — **кэш по `message_id`**,
   а не `callback.message.text` (изменение относительно ADR-009).
6. **Удаление «Перевести» (Решение 6).** Из `handlers/actions.py` удаляются callback `act:tr` и хендлер
   `handle_translate`; из `services/llm.py` удаляются `translate`, `_TRANSLATE_SYSTEM`, `_has_cyrillic`,
   `_target_language` (не используются больше нигде); `__all__` в `llm.py` обновляется
   (`summarize`, `summarize_gist`, `BUDGET_EXCEEDED`).
7. **Клавиатура (Решение 5).** `actions_keyboard()` становится параметризуемой режимом раскрытия:
   - для лички (превью-суть): две кнопки — «Показать полностью» (`act:full`) и «Кратко» (`act:sum`);
   - для прежней немедленной выдачи (группы): одна кнопка — «Кратко» (`act:sum`).
   Реализация — одна фабрика с параметром (например, `actions_keyboard(progressive: bool)`); подписи и
   `callback_data` держатся в одной точке. `callback_data` компактны (`act:full`, `act:sum`) — в лимите
   64 байт.
8. **Охват групп (Решение 5A).** Группы сохраняют немедленную выдачу полного текста; клавиатура — одна
   кнопка «Кратко», без «Перевести». Общие функции `process_audio`/`process_photo`/
   `process_image_document` получают параметр `progressive: bool = False`: личные хендлеры зовут с
   `True` (превью-суть + две кнопки + кэш), групповой `_dispatch_media` — с дефолтом `False` (прежняя
   выдача + одна кнопка «Кратко»). Источник текста для группового «Кратко» — кэш по `message_id`
   сообщения-результата (тот же механизм; `raw_text` кладётся в кэш и в режиме `progressive=False` под
   ключом `message_id` отправленного результата).
9. **Идемпотентность (Решение 4A).** Каждый тап формирует ответ заново, без дедупликации; падения нет.
10. **Нераспознанное медиа (ФТ-7/КП-7).** Поведение прежнее: при пустом `raw_text` — служебное сообщение
    (`NO_SPEECH_MESSAGE`/`NO_TEXT_MESSAGE`), **без** превью, кнопок и записи в кэш. Не меняется.
11. **Атрибуция (ФТ-8/КП-8).** `ATTRIBUTION_FOOTER` через `send_result` сохраняется как сейчас (подпись
    добавляется в одно-сообщенческом режиме). На сообщении-сути подпись не обязательна (короткое превью);
    на уровнях «Полностью»/«Кратко» — как раньше через `send_result`. (Поведение подписи — деталь
    реализации `send_result`, ADR-007; не меняется.)
12. **Неблокирование/семафор/индикатор.** `summarize_gist` — async httpx, без `to_thread`, под бюджет-
    гейтом, без семафора (CF-вызов, ADR-010). Тяжёлые локальные операции (OCR/локальный ASR) остаются
    под семафором в своих сервисах (ADR-010, не меняется). `ChatActionSender` оборачивает тяжёлую часть
    в хендлерах распознавания и в callback-хендлерах (ADR-010, преемственно).
13. **Версия — MINOR 1.5.0.** Обратимо-совместимое изменение UX личного канала; ломающего изменения
    публичного контракта 1.0.0 в смысле MAJOR нет (входные каналы, команды, формат ответа `send_result`
    неизменны). Изменяется набор inline-действий и форма немедленной выдачи — фиксируется как изменение
    контракта 1.0.0 в части inline-действий (см. ниже), трактуется как MINOR-добавление/замена UX, не
    как слом входного/выходного контракта. `roadmap.md`: v1.5.0 «Прогрессивное раскрытие результата
    (суть → кратко → полностью)».

### Влияние на публичный контракт 1.0.0 (ADR-010)

Раздел «Inline-действия» контракта 1.0.0 обновляется (надстройка ADR-009/ADR-010):

- **Было:** под одно-сообщенческим результатом — «Кратко» (`act:sum`) и «Перевести» (`act:tr`).
- **Стало (личка):** под сообщением-сутью — «Показать полностью» (`act:full`) и «Кратко» (`act:sum`);
  «Перевести» удалена; перевод как функция недоступен.
- **Стало (группы):** под результатом — «Кратко» (`act:sum`); «Перевести» удалена.
- **Источник текста для действий:** изменён с `callback.message.text` на in-memory кэш
  `message_id → исходный распознанный текст`. Доступность действий ограничена временем жизни записи
  (TTL/LRU/рестарт); при недоступности — служебное «Текст недоступен» (без падения).
- **Немедленная выдача лички:** теперь не полный текст, а «суть» (1–2 предложения); полный текст — по
  кнопке «Показать полностью».

Это сознательная **MINOR-эволюция UX** при сохранении входных каналов, команд и формата отдачи
(`send_result`, ADR-007). Удаление перевода — осознанное сужение функциональности (BA scope, не баг).

### Что НЕ входит (границы)

- Перевод в любом виде (кнопка/функция) — удаляется, не заменяется.
- Прогрессивное раскрытие в группах — нет (дефолт: прежняя выдача + «Кратко»).
- Дедуп повторных тапов / антиспам по тапам — нет (каждый тап заново).
- Персистентный (переживающий рестарт) или внешний стор исходного текста — нет (in-memory LRU+TTL).
- Кэширование результатов между разными медиа, пользовательские настройки уровня по умолчанию — нет.
- «Красивое» форматирование через `parse_mode` — нет (ADR-007).
- Новые env / секреты / порты / зависимости — нет.

## Последствия

- **Положительные:** меньше «стены текста» — короткая суть сразу, детали по запросу (цель BA); три
  явно различимых уровня (суть < кратко < полностью); исходный текст для кнопок берётся из лёгкого
  in-memory кэша по `message_id` — снимается ограничение ADR-009 «кнопки только под одно-сообщенческим
  результатом» (теперь полный текст любой длины доступен по кнопке через `send_result`); ленивый расчёт
  «полностью»/«кратко» по тапу частично компенсирует добавленный вызов «сути» по бюджету; единый пайплайн
  распознавания (личка/группы) сохранён одним параметром `progressive`; удалён мёртвый код перевода;
  профиль поставки, env, зависимости неизменны.
- **Отрицательные / компромиссы:** in-memory кэш не переживает рестарт и ограничен TTL/LRU — старые
  кнопки могут «протухать» («Текст недоступен», без падения, КП-12); «суть» — дополнительный CF-вызов на
  каждое распознавание лички (бюджет ОГР-3, смягчено ленивым расчётом остальных уровней и деградацией
  Решение 3B); UX лички и групп различается (суть+2 кнопки vs полный текст+«Кратко») — осознанный дефолт
  BA; удаление перевода — потеря функции (по требованию BA, не регресс).
- **Технический долг:** прогрессивное раскрытие в группах; персистентный/переживающий рестарт источник
  текста под кнопки (если протухание окажется заметным на проде); дедуп/антиспам тапов; метрики
  использования уровней — всё при появлении спроса, отдельными релизами.
