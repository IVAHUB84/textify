# ADR-017: Дневной лимит распознаваний на пользователя и мягкий subscription-gate на Telegram-канал

- **Дата:** 2026-06-24
- **Статус:** принято
- **Связанные BA-требования:** `ba-req/ba-requirements-017.md`
- **Связанные ADR:** ADR-001 (профиль поставки «бот без HTTP-порта», long-polling, секреты через `.env` — сохраняется, расширяется env-переменными), ADR-006 (SQLite-стор `stats.db` в volume, соединение-на-операцию + WAL, атомарный UPSERT, тихая деградация — переиспользуется и расширяется новой таблицей), ADR-002/ADR-003 (граница «хендлер ↔ сервис», вынос блокирующих SQLite-операций из event loop через `asyncio.to_thread` — соблюдается), ADR-016 (меню inline-действий над результатом — действия лимит не расходуют)

## Контекст

Бот Textify работает на VPS ivahub (1 vCPU / 2 GB, long-polling). Транскрипция whisper —
основной потребитель CPU; до сих пор частота распознаваний ничем не ограничена, и один
хеви-пользователь способен монополизировать CPU. Параллельно у владельца есть бизнес-цель —
рост Telegram-канала ради монетизации рекламой (веха 1000+ подписчиков). Требование req-017
решает обе задачи одним механизмом: **дневной лимит распознаваний** с **мягким gate на
подписку канала** (исчерпал базовый лимит — предложить подписку ради повышенного лимита).

Силы и ограничения, формирующие решение:

- **Лимит расходует только распознавание входящего медиа** (voice/audio/photo/image-документ),
  но **не** кнопки-действия над уже распознанным результатом (ADR-016: Кратко/Перевести/Тайм-коды/
  Субтитры/Задачи/PDF). Значит, точка списания лимита — не сквозная middleware (она считает все
  сообщения, ADR-006), а **общие функции обработки медиа**, через которые проходят оба входа —
  и личка, и группы.
- **Появляется новое персистентное состояние** — дневной счётчик и пометка подписчика. Оно
  должно переживать рестарт/редеплой (как `stats.db`, ADR-006), но при этом ежедневно
  сбрасываться. Новый стор не нужен — `stats.db` уже существует и смонтирован в volume.
- **Гонки.** Параллельные сообщения одного пользователя не должны приводить к недо-/переучёту —
  инкремент обязан быть атомарным (паттерн `record_message`, ADR-006; НФТ-4).
- **`getChatMember` — сетевой вызов** к Telegram API, требующий, чтобы **бот был администратором
  канала** (ОГР-2). На каждое сообщение его дёргать нельзя (нагрузка, лимиты) — нужен короткий
  кэш статуса (ОВ-2). Канал недоступен / бот не админ / API падает — бот **не должен падать**,
  а деградировать к базовому лимиту (ФТ-12).
- **Выключенный режим.** При незаданном `REQUIRED_CHANNEL` gate выключен полностью — dev/CI и
  текущий прод (канал ещё не создан) работают как прежде. Это сохраняет MINOR-характер релиза.
- **Соседний слой req-018.** В коде уже есть `services/referrals.py` (виральный слой, задел под
  req-018). В req-017 он **не используется**; учёт лимита не должен конфликтовать с его схемой
  (отдельная таблица, без пересечения ключей).
- **Профиль поставки ADR-001 сохраняется**: long-polling, без `ports:`/Traefik/`edge`, образ из
  GHCR, `restart: unless-stopped`. Релиз вводит только новые env-переменные и переиспользует volume.

## Рассматриваемые варианты

### Решение 1. Где списывать лимит (точка учёта)

#### Вариант A: явный вызов в общих функциях обработки медиа (выбран)
Списание лимита делается в `process_audio` / `process_photo` / `process_image_document`
(`handlers/audio.py`, `handlers/image.py`) — через них проходят **все** распознавания: и личные
хендлеры (`handle_voice`/`handle_audio`/…), и групповой `_dispatch_media`.
- Плюсы: списывается ровно факт распознавания медиа (ФТ-1), не команды и не кнопки-действия
  (ФТ-2 — `handlers/actions.py` эти функции не зовёт); один проход охватывает личку и группы по
  инициатору (ОВ-4); проверка лимита и gate-сообщение естественно встают перед скачиванием медиа.
- Минусы: правка трёх стабильных функций; нужно аккуратно определить «инициатора» (личка — автор
  сообщения, группа — `from_user` исходного медиа-сообщения/коллбэка).
- Последствия: gate-логика — отдельный модуль `handlers/gate.py`; учёт — сервис `services/limits.py`.

#### Вариант B: outer-middleware (как ADR-006)
- Минусы: middleware видит **все** сообщения, включая команды, текст и **callback-кнопки** действий
  (ADR-016). Чтобы не списывать лимит на действия, пришлось бы воспроизводить классификацию и
  исключения внутри middleware и отделять «распознавание» от «кнопки» — дублирование и хрупкость.
  Gate-сообщение из middleware до роутинга нарушает границу слоёв.
- Последствия: смешение учёта статистики (ADR-006) и бизнес-логики лимита; легко ошибиться в ФТ-2.

#### Вариант C: декоратор на хендлеры
- Минусы: личные и групповой входы устроены по-разному (хендлеры vs callback `_dispatch_media`);
  декоратор на разнородные сигнатуры неудобен и не покрывает группу единообразно.

Выбран **A**: единая точка в общих функциях обработки медиа — минимально, охватывает оба входа,
естественно исключает кнопки-действия.

### Решение 2. Граница суток и схема счётчика (ОВ-1, ОВ-3)

#### Вариант A: календарный день UTC, ключ суток `YYYY-MM-DD`, атомарный UPSERT (выбран)
Таблица `daily_usage(user_id, day TEXT, count INTEGER, PRIMARY KEY(user_id, day))`. Ключ суток —
строка `datetime.now(timezone.utc).strftime("%Y-%m-%d")`. Инкремент — атомарный UPSERT в одной
транзакции (паттерн `record_message`/`record_referral`, ADR-006):
`INSERT INTO daily_usage(user_id, day, count) VALUES(?,?,1) ON CONFLICT(user_id, day) DO UPDATE SET count = count + 1`.
Чтение остатка — `SELECT count … WHERE user_id=? AND day=?` (нет строки → 0 за сегодня).
- Плюсы: UTC уже сквозной во всём проекте (`stats.db`, `referrals`, бюджет CF — все UTC); ключ-строка
  суток даёт сброс «бесплатно» — на новые сутки просто нет строки, счёт начинается с 0; UPSERT
  атомарен под WAL + busy_timeout, гонки исключены (НФТ-4, КП-15); персистентность переживает
  редеплой (ФТ-4, КП-4); по строке суток в ключе старые дни самоустаревают и легко чистятся.
- Минусы: накапливаются строки прошлых суток (по строке на (user, day)) — фоновая/ленивая чистка
  записей старше N суток желательна; объёмы малы (один-два активных пользователя, ОГР-1).
- Последствия: новая таблица в `stats.db`; функции `record_recognition`/`usage_today` в
  `services/limits.py`.

#### Вариант B: скользящее окно 24 ч (timestamp каждого распознавания)
- Минусы: хранить отметку каждого распознавания и считать «за последние 24 ч» — больше строк,
  сложнее запрос, нет дешёвого «сброса»; пользы над календарным днём для данной задачи нет.

#### Вариант C: счётчик в существующей таблице `user_stats` + поле «дата сброса»
- Минусы: смешивает накопительную статистику (ADR-006, не сбрасывается) с дневным счётчиком
  (сбрасывается); требует ручной логики сброса при чтении (race-prone). Отдельная таблица чище и
  не трогает контракт ADR-006.

Выбран **A**: календарный день **UTC**, отдельная таблица `daily_usage`, ключ `(user_id, day)`,
атомарный UPSERT.

### Решение 3. Кэш статуса подписки (ОВ-2)

#### Вариант A: in-memory TTL-кэш подтверждённой подписки, TTL 600 с (выбран)
Подтверждённая подписка кэшируется в памяти (`dict[user_id] -> expiry_monotonic`) на
`SUBSCRIPTION_TTL_SECONDS = 600` (10 мин). На горячем пути (списание лимита) сначала смотрим кэш:
есть живая запись → пользователь подписчик, `getChatMember` **не вызывается**. Нет/истёк → лимит
считается базовым (gate сработает при исчерпании). Реальный `getChatMember` дёргается **только** по
кнопке «Проверить» (редко) и при успехе кладёт пользователя в кэш на TTL.
- Плюсы: на каждое сообщение к Telegram API не ходим (КП-10); отписавшийся через ≤ 10 мин снова
  упирается в базовый лимит — кэш сам инвалидируется по TTL (ОВ-2, ФТ-11, Сценарий F); состояние
  эфемерно — потеря при рестарте безопасна (пользователь просто перепроверится по «Проверить»);
  не вводит ещё одной персистентной сущности.
- Минусы: после рестарта подтверждённые статусы теряются — но это лишь означает повторный
  `getChatMember` по «Проверить» (дёшево, редко); короткое окно (≤ TTL), в котором отписавшийся
  ещё считается подписчиком (приемлемо, ДОП-3).
- Последствия: модуль кэша внутри `services/subscription.py` (память) рядом с обёрткой `getChatMember`.

#### Вариант B: персистить статус подписчика в `stats.db`
- Минусы: статус меняется (отписка) — персистентная пометка живёт дольше реального членства, нужна
  отдельная инвалидация; усложняет схему и приватность ради сомнительной выгоды. По TTL-кэшу
  актуальность восстанавливается естественно. Отклонён.

#### Вариант C: проверять `getChatMember` на каждое распознавание (без кэша)
- Минусы: лишний сетевой вызов на каждое медиа, риск rate-limit Telegram, лишняя задержка перед
  распознаванием. Противоречит НФТ-1. Отклонён.

Выбран **A**: in-memory TTL-кэш (600 с) подтверждённой подписки; `getChatMember` — только по
кнопке «Проверить».

### Решение 4. Деградация при недоступности проверки подписки (ФТ-12, ОГР-2)

Любой сбой `getChatMember` (бот не админ канала, канал недоступен, ошибка API, таймаут) ловится в
обёртке `services/subscription.py` в `try/except`: исключение логируется (`logger.warning`), функция
возвращает «подписка не подтверждена» (а не падает). Эффект — пользователь трактуется как
неподтверждённый, действует **базовый** лимит без повышения. Бот продолжает работать (КП-11, НФТ-2).
То же при незаданном `REQUIRED_CHANNEL`: обёртка сразу возвращает «gate выключен», `getChatMember`
не вызывается вовсе (ФТ-10, КП-9).

### Решение 5. Поведение после подтверждения подписки (ОВ-5)

#### Вариант A: попросить прислать медиа заново (выбран)
По успешной кнопке «Проверить» бот помечает пользователя подписчиком (кладёт в кэш на TTL),
отвечает дружелюбным сообщением «Подписка подтверждена, лимит повышен — пришлите медиа ещё раз» и
**не** обрабатывает прежнее медиа автоматически.
- Плюсы: проще и надёжнее — не нужно кэшировать байты исходного медиа между gate-сообщением и
  подтверждением (на 1 vCPU / 2 GB лишний стейт нежелателен, ОГР-1); меньше пограничных случаев
  (медиа устарело/недоступно/большое); естественный для пользователя шаг.
- Минусы: пользователь отправляет медиа повторно (одно лишнее действие). Приемлемо для редкого
  события «только что подписался».
- Последствия: gate-сообщение и подтверждение не несут состояния «отложенного медиа».

#### Вариант B: сразу обработать текущее медиа после «Проверить»
- Минусы: требует держать байты/ссылку на исходное медиа в кэше до клика (память, TTL, инвалидация,
  обработка «медиа устарело»). Существенно больше сложности ради экономии одного действия
  пользователя. Отклонён на 1 vCPU / 2 GB.

Выбран **A**: после подтверждения — попросить прислать заново.

### Решение 6. Охват групп (ОВ-4)

В группах (v1.4.0/v1.5.0) распознавание по кнопке «Распознать» **учитывает** дневной лимит — по
инициатору (`from_user` исходного медиа-сообщения / коллбэка), так как Решение 1A списывает в общих
`process_*`. Однако **gate (предложение подписки) в группах не показывается**: при исчерпании
лимита в группе бот отвечает коротким нейтральным сообщением о лимите, без inline-кнопок подписки.
- Обоснование: защита CPU нужна везде (учёт по инициатору справедлив и в группах, ДОП-4); gate же —
  диалоговый UX лички, в групповом чате он шумен и неуместен. Gate-сообщение с кнопками
  показывается только в `private` (ФТ-15, КП-14). Существующее поведение групп не ломается.

### Решение 7. Место показа остатка лимита (ОВ-6)

Остаток показывается в **`/start`** (видно каждому пользователю в личке: строка «Сегодня доступно
N из M распознаваний») и дублируется агрегатом в **`/stats`** для администратора (например, число
подтверждённых подписчиков в кэше и/или суммарный учёт за сегодня — по месту в существующем
выводе). Основная пользовательская точка — `/start` (КП-13). Формат — дружелюбная русская строка,
без давления.

### Решение 8. Конфигурация (ФТ-3, ФТ-9, ФТ-16, НФТ-6)

Три новые env-переменные читаются в `config.py` тем же паттерном, что `ADMIN_USER_ID`/`CF_DAILY_BUDGET`
(strip, валидация, дефолт при пустом/некорректном, `logger.warning`):
- `REQUIRED_CHANNEL` — строка `@username` или числовой `chat_id`; пусто/не задано → gate выключен.
- `DAILY_LIMIT_FREE` — int, дефолт **3**; при ≤ 0 или нечисловом → дефолт + warning.
- `DAILY_LIMIT_SUBSCRIBED` — int, дефолт **30**; та же валидация.
Реальные значения — только в `.env` (не коммитится); плейсхолдеры — в `.env.example` и
`deploy/.env.example` (НФТ-6).

## Решение

1. **Точка учёта** — явный вызов в `process_audio`/`process_photo`/`process_image_document`
   (Решение 1A): покрывает личку и группы, исключает кнопки-действия.
2. **Счётчик** — новая таблица `daily_usage(user_id, day, count, PK(user_id, day))` в `stats.db`,
   ключ суток — календарный день **UTC** `YYYY-MM-DD`, атомарный UPSERT (Решение 2A); доступ —
   соединение-на-операцию + WAL + busy_timeout через `asyncio.to_thread` (ADR-006).
3. **Два уровня лимита** из env `DAILY_LIMIT_FREE` (3) / `DAILY_LIMIT_SUBSCRIBED` (30) (Решение 8).
4. **Кэш подписки** — in-memory TTL 600 с подтверждённой подписки; `getChatMember` только по кнопке
   «Проверить» (Решение 3A).
5. **Деградация** — сбой `getChatMember` / незаданный `REQUIRED_CHANNEL` → «не подтверждено» /
   «gate выключен», бот не падает, лог (Решение 4).
6. **После подтверждения** — попросить прислать медиа заново (Решение 5A).
7. **Группы** — лимит учитывается по инициатору, gate-сообщение с кнопками только в `private`
   (Решение 6).
8. **Остаток лимита** — в `/start` (пользователю) и `/stats` (агрегат админу) (Решение 7).
9. **Границы модулей.** `services/limits.py` — учёт дневного лимита (схема, `record_recognition`,
   `usage_today`, всё через `to_thread`); `services/subscription.py` — обёртка над
   `bot.get_chat_member` + in-memory кэш статуса + чтение `REQUIRED_CHANNEL`; `handlers/gate.py` —
   gate-логика (проверка лимита, формирование gate-сообщения и клавиатуры, callback «Проверить»);
   `handlers/audio.py`/`handlers/image.py` — вызов точки учёта/гейта в общих функциях; `config.py` —
   чтение env. Сервисы не знают про aiogram-хендлеры; хендлеры не знают про SQLite-детали.

### Расширение, а не отмена ADR-006/ADR-001

ADR-006 ввёл первое персистентное состояние (`stats.db` в volume). Релиз v1.7.0 **добавляет одну
таблицу** в тот же файл и две функции доступа тем же паттерном (соединение-на-операцию, WAL,
атомарный UPSERT, `to_thread`, тихая деградация) — без новой инфраструктуры, volume и портов.
Профиль поставки ADR-001 сохраняется; добавляются ровно три env-переменные. Соседний
`services/referrals.py` (req-018) не затрагивается — его таблица `referrals` независима от
`daily_usage`.

### Схема нового хранилища

```sql
CREATE TABLE IF NOT EXISTS daily_usage (
    user_id INTEGER NOT NULL,           -- Telegram id (без иных персональных данных)
    day     TEXT    NOT NULL,           -- календарные сутки UTC, 'YYYY-MM-DD'
    count   INTEGER NOT NULL DEFAULT 0, -- число распознаваний за эти сутки
    PRIMARY KEY (user_id, day)
);
```

- Списание: атомарный UPSERT `count = count + 1` за `(user_id, today_utc)`.
- Чтение остатка: `count` за сегодня (нет строки → 0).
- Подписчик/нет определяет уровень лимита (`DAILY_LIMIT_SUBSCRIBED` vs `DAILY_LIMIT_FREE`); сама
  пометка подписчика — эфемерная, в in-memory кэше (Решение 3), не в БД.
- Хранятся только `user_id`, день и счётчик; содержимое сообщений — нет (НФТ-5, преемственно ADR-006).

### Порядок на горячем пути (псевдокод, в общих `process_*`)

```
limit = DAILY_LIMIT_SUBSCRIBED if subscription.is_subscriber_cached(user_id) else DAILY_LIMIT_FREE
used  = await limits.usage_today(user_id)
if used >= limit:
    if private and gate_enabled and not subscriber:
        отправить gate-сообщение (кнопки «Открыть канал» / «Проверить»)   # Сценарий B
    else:
        отправить нейтральное «дневной лимит исчерпан»                     # группа / подписчик
    return  # медиа не скачивается и не распознаётся
await limits.record_recognition(user_id)   # атомарный инкремент ДО скачивания/распознавания
... существующая обработка ...
```

Списание — **до** скачивания и распознавания (защита CPU/трафика). Сбой записи учёта ловится и не
прерывает доставку результата (НФТ-2, преемственно ADR-006 Решение 7).

## Последствия

- **Положительные:** защита CPU от хеви-юзеров (базовый лимит); канал-конверсия мотивированной
  аудитории через мягкий gate; первое впечатление новых пользователей не страдает (gate невидим до
  исчерпания базового лимита, НФТ-8); персистентный учёт переживает редеплой и сбрасывается посуточно;
  гонки исключены атомарным UPSERT; нагрузка на Telegram API минимизирована кэшем; полная деградация
  при недоступности канала; выключенный режим сохраняет dev/CI/текущий прод; профиль поставки и
  ADR-001/006 не нарушены; слой req-018 не затронут.
- **Отрицательные / компромиссы:** правка трёх общих функций обработки медиа; после «Проверить»
  пользователь шлёт медиа повторно (Решение 5A); накопление строк прошлых суток в `daily_usage`
  (нужна ленивая/периодическая чистка); эфемерный кэш подписки теряется при рестарте (повторный
  `getChatMember` по «Проверить» — дёшево); короткое окно (≤ TTL) после отписки, где статус ещё
  «подписчик».
- **Технический долг:** нет реферального повышения лимита (req-018); нет жёсткого gate; нет антифрода
  по мультиаккаунтам; чистка старых строк `daily_usage` — простая (ленивая при записи или на старте),
  без отдельного планировщика; кэш подписки — без проактивной инвалидации по событию отписки (только TTL).

## Влияние на код

- **Новые модули:**
  - `services/limits.py` — `init_limits_db()` (создаёт таблицу `daily_usage`, WAL, busy_timeout),
    `record_recognition(user_id)` (атомарный UPSERT за сегодня, `to_thread`),
    `usage_today(user_id) -> int` (`to_thread`); опционально `purge_old_days()`.
  - `services/subscription.py` — `is_gate_enabled()`, `is_subscriber_cached(user_id) -> bool`,
    `async check_subscription(bot, user_id) -> bool` (обёртка `get_chat_member` в `try/except` +
    запись в кэш при успехе), `channel_url()`; in-memory TTL-кэш (600 с); чтение `REQUIRED_CHANNEL`.
  - `handlers/gate.py` — `async enforce_limit(...) -> bool` (проверка лимита/гейта; True = разрешить
    обработку, False = лимит исчерпан, сообщение уже отправлено), фабрика gate-клавиатуры
    («Открыть канал» url, «Проверить» callback), callback-хендлер «Проверить» (`check_subscription`,
    повышение/отказ), `gate_router`. Все сообщения — на русском (ФТ-13).
- **Изменяемые модули:**
  - `handlers/audio.py`, `handlers/image.py` — в начале `process_audio`/`process_photo`/
    `process_image_document` вызвать `enforce_limit` (личка vs группа определяется по типу чата);
    при отказе — `return` до скачивания; при разрешении — `record_recognition`.
  - `handlers/commands.py` — `/start`: строка остатка лимита для пользователя; `/stats`: агрегат
    подписчиков/учёта (по месту).
  - `bot.py` — `init_limits_db()` на старте; регистрация `gate_router`.
  - `config.py` — чтение `REQUIRED_CHANNEL`, `DAILY_LIMIT_FREE`, `DAILY_LIMIT_SUBSCRIBED`.
- **Зависимости:** новых Python-пакетов нет (`sqlite3` стандартный; `get_chat_member` — aiogram).
- **Артефакты поставки:** Dockerfile / `deploy/docker-compose.yml` / `.github/workflows/deploy.yml` —
  **без изменений** (профиль ivahub/long-polling/GHCR не меняется, новых портов/volume/зависимостей
  нет; таблица создаётся в существующем `stats.db`). `.env.example` и `deploy/.env.example` += три
  плейсхолдера. **Деплой-предусловие:** бот должен быть администратором канала `REQUIRED_CHANNEL`
  (ОГР-2, ОВ-7) — иначе деградация по ФТ-12.
- **Что ломается:** публичный контракт входных каналов не ломается. Новое поведение — лимит и gate —
  при заданном `REQUIRED_CHANNEL`; при незаданном поведение прежнее. Релиз — **MINOR**.
