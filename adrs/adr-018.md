# ADR-018: Реферальная награда — повышение дневного лимита распознаваний за приглашённых друзей

- **Дата:** 2026-06-24
- **Статус:** принято
- **Связанные BA-требования:** `ba-req/ba-requirements-018.md`
- **Связанные ADR:** ADR-017 (дневной лимит и subscription-gate v1.7.0; данный ADR — **надстройка над `enforce_limit`**: бинарный уровень `subscribed`/`free` становится базой, к которой прибавляется реферальный бонус — без слома логики v1.7.0), ADR-006 (SQLite-стор `stats.db`: соединение-на-операцию + WAL + `busy_timeout`, доступ через `asyncio.to_thread`, тихая деградация — переиспользуется), ADR-001 (профиль поставки «бот без HTTP-порта», long-polling, секреты через `.env` — сохраняется, расширяется двумя env-переменными), реферальная механика v1.3.0 (`services/referrals.py`: таблица `referrals` с `referred_id` PRIMARY KEY, индекс `idx_referrals_referrer`, `count_referrals` — переиспользуется **как есть**, новый стор не заводится)

## Контекст

В v1.7.0 (ADR-017) введён дневной лимит распознаваний с двумя уровнями (`DAILY_LIMIT_FREE`,
дефолт 3 / `DAILY_LIMIT_SUBSCRIBED`, дефолт 30) и мягкий subscription-gate. Точка применения —
`enforce_limit` в `handlers/gate.py`, где лимит выбирается **бинарно**: подписчик получает
`DAILY_LIMIT_SUBSCRIBED`, остальные — `DAILY_LIMIT_FREE`.

Требование req-018 вводит **второй рычаг роста** поверх subscription-gate — реферальную награду:
число приглашённых пользователем друзей повышает его собственный дневной лимит. Это даёт прямой
мотив звать друзей (виральный рост), **независимый** от подписки на канал и складывающийся с ней.

Силы и ограничения, формирующие решение:

- **Вся инфраструктура рефералов уже существует с v1.3.0** и не создаётся заново: реф-ссылка в
  `/start` (`handlers/commands.py`), фиксация реферала (`record_referral`, таблица `referrals`,
  `referred_id` PRIMARY KEY с `INSERT OR IGNORE`), `count_referrals(user_id)` (`COUNT(*)` по
  индексу `idx_referrals_referrer`). Требование — **только слой награды**: связать
  `count_referrals` с эффективным лимитом в `enforce_limit`.
- **Горячий путь на 1 vCPU.** `enforce_limit` вызывается на **каждое** распознавание медиа.
  Добавление реферального бонуса означает чтение `count_referrals` — ещё один `SELECT COUNT(*)`
  по индексу. Сейчас на горячем пути уже есть один SQLite-`SELECT` (`limits.usage_today`); второй
  по индексу дёшев, но трафик лички низкий, а число рефералов меняется редко — это создаёт
  возможность для дешёвого TTL-кэша (ОВ-1).
- **Обратная совместимость с v1.7.0 — жёсткое требование.** При **нуле рефералов** эффективный
  лимит обязан равняться текущему поведению v1.7.0 (база/подписчик). Формула
  `база + min(r·PER, CAP)` при `r=0` даёт `база + 0` = поведение v1.7.0 (ФТ-6, КП-6).
- **Защита ресурса whisper.** Потолок `REFERRAL_BONUS_CAP` гарантирует, что никто не получит
  безлимит за счёт рефералов (НФТ-3).
- **Деградация.** Сбой чтения числа рефералов не должен ломать распознавание: при ошибке —
  откат к лимиту v1.7.0 без бонуса, лог (НФТ-2, преемственно ADR-006/ADR-017).
- **Анти-фрод — вне рамок.** Тривиальная накрутка уже снижена существующей схемой v1.3.0
  (`referred_id` PRIMARY KEY исключает повторный учёт одного приглашённого; `referrer_id != user_id`
  исключает самоприглашение; реферал фиксируется только для `is_new_user`). Серьёзный анти-фрод по
  мультиаккаунтам отложен (ОВ-6, ДОП-2).
- **Профиль поставки ADR-001 сохраняется**: long-polling, без `ports:`/Traefik/`edge`, образ из
  GHCR, `restart: unless-stopped`. Релиз вводит только две env-переменные и переиспользует volume.

## Рассматриваемые варианты

### Решение 1. Где живёт расчёт эффективного лимита (граница модулей)

#### Вариант A: новая функция в `services/limits.py` + обёртка бонуса, `enforce_limit` её зовёт (выбран)
Расчёт эффективного лимита выносится в чистую функцию `services/limits.py`
(`effective_daily_limit(base_limit, referral_count) -> int`), а получение числа рефералов с кэшем —
в обёртку (Решение 3). `enforce_limit` остаётся тонким: получает `subscriber`, базу, число
рефералов, остаток и сравнивает.
- Плюсы: формула — чистая, легко тестируется изолированно (КП-1/КП-3); `enforce_limit` и `/start`
  переиспользуют **одну** формулу (нет расхождения «лимит в gate ≠ лимит в /start»); граница
  «хендлер ↔ сервис» соблюдена (ADR-002/003) — хендлер не считает арифметику бонуса сам.
- Минусы: ещё одна функция в `limits.py`. Незначительно.
- Последствия: `handlers/gate.py` и `handlers/commands.py` зовут общую формулу.

#### Вариант B: вся арифметика бонуса внутри `enforce_limit`
- Минусы: `/start` (Задача про вывод) вынужден дублировать ту же арифметику — два источника правды
  об эффективном лимите, риск расхождения формулировок. Хендлер начинает считать бизнес-арифметику.
  Отклонён.

Выбран **A**: чистая формула в `services/limits.py`, переиспользуемая `enforce_limit` и `/start`.

### Решение 2. Кэшировать `count_referrals` на горячем пути или читать напрямую (ОВ-1)

#### Вариант A: in-memory TTL-кэш числа рефералов, TTL 600 с (выбран)
По аналогии с кэшем подписки v1.7.0 (`services/subscription.py`, `SUBSCRIPTION_TTL_SECONDS = 600`)
вводится in-memory кэш `dict[user_id] -> (count, expiry_monotonic)` в `services/referrals.py` с
`REFERRAL_COUNT_TTL_SECONDS = 600`. На горячем пути (`enforce_limit`) сначала смотрим кэш: живая
запись → используем её, `COUNT` **не выполняется**; нет/истёк → один `SELECT COUNT(*)` по индексу,
кладём в кэш на TTL. **Инвалидация:** при фиксации нового реферала (`record_referral`) запись
пригласившего из кэша **удаляется** — чтобы рост числа рефералов отражался без ожидания TTL.
- Плюсы: повторные распознавания в окне TTL не порождают новый SQL `COUNT` (КП-13); согласованность
  с уже принятым паттерном кэша подписки (один стиль, легко сопровождать); число рефералов меняется
  редко (только при `/start` нового пользователя по ссылке) — точечная инвалидация даёт почти
  мгновенное отражение нового реферала без TTL-задержки; кэш эфемерен, потеря при рестарте
  безопасна (восстановится первым же `COUNT`); не вводит новой персистентной сущности.
- Минусы: ещё один in-memory кэш (память ничтожна — int+float на пользователя); короткое окно
  рассинхронизации, если реферал зафиксирован в другом процессе (на ivahub процесс один — неактуально).
- Последствия: модуль кэша внутри `services/referrals.py` рядом с `count_referrals`; `record_referral`
  инвалидирует запись пригласившего.

#### Вариант B: читать `COUNT` напрямую на каждое распознавание (без кэша)
- Плюсы: проще, всегда актуально; `COUNT(*)` по индексу `idx_referrals_referrer` дёшев, трафик лички
  низкий (один-два активных пользователя, ОГР-1) — нагрузка реально пренебрежима.
- Минусы: второй SQLite-`SELECT` на горячем пути на каждое медиа (первый — `usage_today`); на 1 vCPU
  при будущем росте трафика лишний `to_thread` + открытие соединения на каждое распознавание; нет
  согласованности с уже выбранным для подписки паттерном кэша. КП-13 прямо предполагает кэш.
- Последствия: меньше кода, но горячий путь дороже и стиль расходится с v1.7.0.

**Обоснование выбора A.** На сегодняшней нагрузке оба варианта корректны и B был бы достаточен.
Решающие аргументы за A: (1) **симметрия с кэшем подписки v1.7.0** — тот же TTL 600 с, тот же стиль
in-memory словаря, тот же горизонт инвалидации; разработчик и ревьювер видят один паттерн в обоих
сервисах; (2) **точечная инвалидация** при `record_referral` даёт лучшее из обоих миров —
актуальность как у прямого чтения, дешевизна как у кэша; (3) КП-13 явно просит, чтобы повторные
распознавания в окне TTL не порождали новый `COUNT`. Цена — несколько строк кэша, идентичных уже
существующему паттерну. Выбран **A**.

### Решение 3. Формула эффективного лимита и обёртка получения бонуса

Формула (чистая, `services/limits.py`):
```
effective_daily_limit(base_limit, referral_count) =
    base_limit + min(referral_count * REFERRAL_BONUS_PER, REFERRAL_BONUS_CAP)
```
где `base_limit` = `DAILY_LIMIT_SUBSCRIBED` (подписчик) или `DAILY_LIMIT_FREE` (остальные) — как в
v1.7.0. При `referral_count = 0` → `base_limit` (обратная совместимость, ФТ-6).

Получение числа рефералов с кэшем — `async cached_referral_count(user_id) -> int` в
`services/referrals.py` (Решение 2A). Бонус = `min(count·PER, CAP)`.

### Решение 4. Дефолты `REFERRAL_BONUS_PER` / `REFERRAL_BONUS_CAP` (ОВ-2, ОВ-3)

BA даёт ориентир «+3 за реферала, потолок +30». Принимаем его как дефолты:
- **`REFERRAL_BONUS_PER = 3`** — каждый приглашённый друг добавляет +3 распознавания/сутки. Симметрично
  базовому `DAILY_LIMIT_FREE = 3`: один друг фактически удваивает базовый лимит — мотив ощутим.
- **`REFERRAL_BONUS_CAP = 30`** — потолок бонуса. 10 друзей выводят бонус на потолок. Для
  неподписчика это даёт `3 + 30 = 33` (≈ уровень подписчика + чуть выше), для подписчика —
  `30 + 30 = 60`. Бонус ощутим, но whisper-ресурс защищён (НФТ-3): ни один пользователь не получает
  безлимит, потолок фиксирован и независим от подписки.

Обоснование величин: PER=3 делает первый же реферал заметным (удвоение базы); CAP=30 ограничивает
суммарную дополнительную нагрузку и совпадает по порядку с `DAILY_LIMIT_SUBSCRIBED`, оставаясь
управляемым через env. Оба читаются из окружения, дефолт при пустом/некорректном (Решение 8).

### Решение 5. Формат вывода эффективного лимита в `/start` (ОВ-4)

`/start` уже показывает реф-ссылку, «Приглашено: N» и строку остатка лимита v1.7.0
(`handlers/commands.py`). Строка остатка **расширяется** так, чтобы учитывать бонус и показывать
слагаемые — без дублирования уже выводимого блока рефералов. Формат (русский, дружелюбный):

```
Сегодня доступно <remaining> из <effective> распознаваний.
```
и, **только при наличии бонуса** (`referral_count > 0` и бонус > 0), дополнительная строка-расшифровка:
```
Лимит: <base> базовых + <bonus> за <N> друзей.
```
где `effective = effective_daily_limit(base, count)`, `remaining = max(0, effective - usage_today)`,
`bonus = effective - base`. При нуле рефералов расшифровка не показывается — вывод визуально
совпадает с v1.7.0 (минимальное изменение, ФТ-9). Существующая строка «Приглашено: N» сохраняется
как есть; реф-ссылка с призывом — уже есть.

- Плюсы: компактно, расшифровка появляется только когда есть что объяснять; переиспользует
  `effective_daily_limit` (одна формула с gate, Решение 1A); не ломает существующий каркас вывода.
- Альтернатива (всегда показывать «база + бонус» даже при нуле) отклонена: шум для большинства
  пользователей без рефералов и визуальное отличие от v1.7.0 без пользы.

### Решение 6. Формат gate-сообщения с двумя путями (ОВ-3)

При исчерпании эффективного лимита неподтверждённым пользователем **в личке** gate-сообщение
расширяется вторым путём — «или пригласи друзей». Решение: **не добавлять новую callback-кнопку**
(ОГР-3, экономия и простота), а дополнить **текст** gate-сообщения второй строкой и переиспользовать
**уже существующую** в `/start` кнопку «Поделиться ботом» — но в gate-клавиатуру добавить **url-кнопку
«Пригласить друзей»** на тот же `https://t.me/share/url`-линк, что уже строится в `commands.py`.

#### Вариант A: текст + url-кнопка «Пригласить друзей» рядом с «Открыть канал» (выбран)
Gate-клавиатура: ряд 1 — «Открыть канал» (url) и «Пригласить друзей» (url на share-ссылку);
ряд 2 — «Я подписался / Проверить» (callback `gate:chk`, как в v1.7.0). Текст:
```
Вы исчерпали дневной лимит распознаваний.

Чтобы поднять лимит:
• подпишитесь на наш канал — лимит вырастет;
• или пригласите друзей по своей ссылке — за каждого +<PER> распознаваний в день.
```
- Плюсы: оба пути видны (КП-10); url-кнопки **не тратят** `callback_data` (ОГР-3 неактуален для url);
  share-ссылка переиспользует логику `_build_ref_link`/`_build_share_keyboard` из `commands.py`
  (минимальные изменения, DRY); кнопка «Проверить» v1.7.0 сохранена без изменений.
- Минусы: общий доступ к построению share-url из `gate.py` — выносится в общий хелпер (или вызов
  существующего). Незначительно.
- Последствия: gate-клавиатура расширяется одной url-кнопкой; текст — два пути.

#### Вариант B: добавить callback-кнопку «Как пригласить» с инструкцией
- Минусы: лишний callback-хендлер и состояние ради текста, который можно показать сразу; перегружает
  клавиатуру. Отклонён.

#### Вариант C: только текст, без второй кнопки
- Минусы: пользователю негде сразу взять свою ссылку — путь «пригласить» становится умозрительным.
  Отклонён в пользу A (одна url-кнопка дешева и действенна).

Выбран **A**: текст с двумя путями + url-кнопка «Пригласить друзей» (share-ссылка) + сохранённая
кнопка «Проверить». При `is_private=False` (группа) или выключенном gate — поведение v1.7.0
(нейтральное сообщение), второй путь не показывается.

### Решение 7. Уведомление пригласившему при новом реферале (ОВ-5, ФТ-12) — отложить

#### Вариант A: не включать в v1.8.0 (выбран)
Уведомление «по твоей ссылке пришёл друг, лимит вырос» **выносится за рамки v1.8.0**.
- Обоснование: BA помечает ФТ-12 как **опциональное**, а принцип релиза — минимальные изменения.
  Корректное уведомление требует: (1) отправки сообщения пригласившему из хендлера `/start` нового
  пользователя (новый путь доставки в чужой чат), (2) управляющего флага вкл/откл (новая env), (3)
  обработки сбоя доставки (пригласивший заблокировал бота / не начинал диалог → `bot.send_message`
  упадёт) так, чтобы не сломать фиксацию реферала и `/start` нового пользователя. Это заметный объём
  и новые краевые случаи ради вторичного UX. Точечная инвалидация кэша (Решение 2A) и так
  гарантирует, что лимит пригласившего вырастет к следующему распознаванию; пригласивший видит рост
  в своём `/start`.
- Последствия: ФТ-12/КП-14 переносятся в технический долг / отдельное требование; env для флага не
  вводится в этом релизе.

#### Вариант B: включить с флагом `REFERRAL_NOTIFY`
- Минусы: дополнительная env, новый путь доставки, обработка сбоя, тесты — против принципа
  минимального релиза. Отклонён в пользу A.

Выбран **A**: отложить уведомление. Релиз остаётся минимальным.

### Решение 8. Анти-фрод (ОВ-6) — подтверждение достаточности текущей схемы

Подтверждаем: для слоя награды **отдельный анти-фрод не вводится**. Действующая схема v1.3.0 уже
исключает тривиальную накрутку:
- `referred_id` — PRIMARY KEY + `INSERT OR IGNORE`: один и тот же приглашённый учитывается **один
  раз**, повторные `/start` по ссылке не накручивают `count_referrals` (ДОП-1).
- `cmd_start` фиксирует реферал только при `referrer_id != user_id` — **самоприглашение исключено**
  (ДОП-2).
- Реферал фиксируется только для `is_new_user` — повторный заход существующего пользователя по чужой
  ссылке реферала не создаёт.

Остаётся накрутка через создание новых Telegram-аккаунтов — она ограничена стоимостью создания
аккаунта (номер телефона) и потолком `REFERRAL_BONUS_CAP` (даже массовая накрутка не даёт безлимит).
Серьёзный анти-фрод по мультиаккаунтам отложен до появления злоупотреблений (вне рамок, ДОП-2).
Решение слоя награды: **полагаться на существующие инварианты + `CAP`**, новый анти-фрод не строить.

### Решение 9. Конфигурация (ФТ-5, ФТ-11)

Две новые env читаются в `config.py` тем же паттерном, что `DAILY_LIMIT_FREE`/`DAILY_LIMIT_SUBSCRIBED`
(`os.environ.get(...).strip()`, `int(...)`, дефолт при пустом/некорректном/≤ 0, `logger.warning`).
Существующего хелпера `_read_positive_int` в `config.py` **нет** — паттерн инлайнится повторно, как
для лимитов v1.7.0 (для согласованности; вынос общего хелпера — опциональный рефакторинг вне рамок):
- `REFERRAL_BONUS_PER` — int, дефолт **3**; при ≤ 0 или нечисловом → дефолт + warning.
- `REFERRAL_BONUS_CAP` — int, дефолт **30**; та же валидация. (Допускается `CAP = 0` как осознанное
  «выключить бонус»? — нет: трактуем ≤ 0 как некорректное → дефолт, чтобы не путать с выключением;
  отключение бонуса достигается `REFERRAL_BONUS_PER`, но и он ≤ 0 → дефолт. Полное выключение бонуса
  как сценарий не требуется — при нуле рефералов поведение и так равно v1.7.0.)

Реальные значения — только в `.env` (не коммитится); плейсхолдеры — в `.env.example` и
`deploy/.env.example` (НФТ-5).

## Решение

1. **Формула** — чистая `effective_daily_limit(base, referral_count) = base + min(count·PER, CAP)`
   в `services/limits.py` (Решение 1A, 3); переиспользуется `enforce_limit` и `/start`. При
   `count = 0` → `base` (обратная совместимость v1.7.0, ФТ-6).
2. **Число рефералов с кэшем** — `cached_referral_count(user_id)` в `services/referrals.py`:
   in-memory TTL-кэш 600 с поверх существующего `count_referrals`, **точечная инвалидация** в
   `record_referral` (Решение 2A).
3. **Дефолты** — `REFERRAL_BONUS_PER = 3`, `REFERRAL_BONUS_CAP = 30` (Решение 4).
4. **Точка применения** — `enforce_limit` (`handlers/gate.py`): база v1.7.0 + реферальный бонус;
   допуск/отказ и остаток — относительно **эффективного** лимита (ФТ-7).
5. **`/start`** — строка остатка использует эффективный лимит; расшифровка «база + бонус за N
   друзей» только при наличии бонуса (Решение 5).
6. **Gate-сообщение** — текст с двумя путями + url-кнопка «Пригласить друзей» (share-ссылка) +
   сохранённая кнопка «Проверить»; только в личке при включённом gate (Решение 6).
7. **Уведомление пригласившему** — **отложено** (Решение 7A); в v1.8.0 не входит.
8. **Анти-фрод** — отдельный не вводится; опора на инварианты v1.3.0 + `CAP` (Решение 8).
9. **Конфигурация** — `REFERRAL_BONUS_PER`/`REFERRAL_BONUS_CAP` в `config.py` паттерном лимитов
   v1.7.0 (Решение 9); плейсхолдеры в обоих `.env.example`.
10. **Деградация** — сбой чтения числа рефералов/расчёта бонуса → откат к лимиту v1.7.0 без бонуса,
    лог; распознавание не прерывается (НФТ-2, КП-12). `cached_referral_count` при ошибке `COUNT`
    возвращает `0` (бонус 0 = поведение v1.7.0).
11. **Границы модулей.** `services/limits.py` — чистая формула эффективного лимита (без SQLite, без
    рефералов внутри); `services/referrals.py` — `count_referrals` + кэш + инвалидация (доступ к
    `referrals` через `to_thread`); `handlers/gate.py` — `enforce_limit` собирает базу (подписка) +
    бонус (рефералы) + остаток, формирует gate-сообщение; `handlers/commands.py` — вывод `/start`;
    `config.py` — env. Сервисы не знают про aiogram; хендлеры не считают арифметику бонуса сами.

### Надстройка над ADR-017, а не его отмена

ADR-017 ввёл бинарный выбор уровня в `enforce_limit`. ADR-018 **оборачивает** этот уровень в
формулу `база + бонус`: бинарный уровень становится `base_limit`, бонус прибавляется поверх. При
нуле рефералов и/или дефолтах поведение **идентично** v1.7.0 (ФТ-6, КП-6) — поэтому релиз MINOR.
Логика подписки, кэш статуса, gate-callback «Проверить», точка списания в `process_*`, схема
`daily_usage` — **не меняются**. Таблица `referrals` (v1.3.0) переиспользуется как есть, новой
схемы БД нет. Профиль поставки ADR-001 сохраняется; добавляются ровно две env-переменные.

### Порядок на горячем пути (псевдокод, `enforce_limit`)

```
subscriber = subscription.is_subscriber_cached(user_id)
base  = DAILY_LIMIT_SUBSCRIBED if subscriber else DAILY_LIMIT_FREE
count = await referrals.cached_referral_count(user_id)   # кэш TTL 600 с; при ошибке → 0
limit = limits.effective_daily_limit(base, count)        # base + min(count*PER, CAP)
used  = await limits.usage_today(user_id)
if used >= limit:
    if private and gate_enabled and not subscriber:
        отправить gate-сообщение (два пути: подписка + пригласить друзей)
    else:
        отправить нейтральное «дневной лимит исчерпан»   # группа / подписчик / gate выключен
    return False
await limits.record_recognition(user_id)
return True
```

Реферальный бонус считается **между** определением базы и сравнением остатка — минимальная вставка
в существующий поток v1.7.0. Сбой `cached_referral_count` → `0` → база без бонуса (деградация).

## Последствия

- **Положительные:** второй независимый рычаг роста (виральность) поверх subscription-gate; мотив
  звать друзей даже без подписки на канал; при нуле рефералов — полная совместимость с v1.7.0
  (нет регрессии); потолок `CAP` защищает CPU whisper от безлимита; кэш числа рефералов с точечной
  инвалидацией — дёшев на горячем пути и согласован с кэшем подписки v1.7.0; формула эффективного
  лимита — одна на gate и `/start` (нет расхождения); инфраструктура рефералов v1.3.0 и слой лимитов
  v1.7.0 переиспользованы без изменения механики; профиль поставки ADR-001/006 не нарушен; новой
  схемы БД нет.
- **Отрицательные / компромиссы:** второй in-memory кэш (память ничтожна, но ещё одна эфемерная
  сущность); на горячем пути при холодном кэше — второй SQLite-`SELECT` (`COUNT`, дёшев по индексу);
  уведомление пригласившему отсутствует — рост лимита виден только в его `/start` (Решение 7A);
  анти-фрод по мультиаккаунтам не закрыт (опора на стоимость аккаунта + `CAP`).
- **Технический долг:** уведомление пригласившему о новом реферале (ФТ-12, отложено — Решение 7);
  анти-фрод по мультиаккаунтам (вне рамок); общий хелпер `_read_positive_int` в `config.py`
  (паттерн чтения int-env инлайнится третий раз — кандидат на вынос, опционально); кэш числа
  рефералов — без проактивной инвалидации по событиям, кроме `record_referral` (для одного процесса
  ivahub достаточно).

## Влияние на код

- **Изменяемые модули:**
  - `services/referrals.py` — добавить in-memory TTL-кэш числа рефералов (`REFERRAL_COUNT_TTL_SECONDS = 600`,
    `dict[int, tuple[int, float]]`), `async cached_referral_count(user_id) -> int` (живая запись →
    из кэша; иначе `count_referrals` + кэш; при исключении → `0` + лог), инвалидацию записи
    пригласившего в `record_referral` (удалить из кэша). Существующие `count_referrals`/
    `record_referral`/`total_referrals`/`top_referrers` сохраняются.
  - `services/limits.py` — добавить чистую `effective_daily_limit(base_limit: int, referral_count: int) -> int`
    = `base_limit + min(referral_count * config["REFERRAL_BONUS_PER"], config["REFERRAL_BONUS_CAP"])`.
    Существующие функции учёта не меняются.
  - `handlers/gate.py` — `enforce_limit`: между базой и сравнением вставить
    `count = await referrals.cached_referral_count(user_id)` и
    `limit = limits.effective_daily_limit(base, count)`; расширить gate-сообщение текстом двух путей
    и url-кнопкой «Пригласить друзей» (share-ссылка); callback «Проверить» — без изменений.
  - `handlers/commands.py` — `/start`: строку остатка считать от эффективного лимита; при бонусе > 0
    добавить расшифровку «база + бонус за N друзей»; блок реф-ссылки/«Приглашено» — без изменений;
    переиспользовать построение share-url для gate-кнопки (общий хелпер или вызов существующего).
  - `config.py` — чтение `REFERRAL_BONUS_PER` (дефолт 3), `REFERRAL_BONUS_CAP` (дефолт 30) паттерном
    лимитов v1.7.0; ключи в возвращаемый `config`.
- **Зависимости:** новых Python-пакетов нет (`sqlite3` стандартный; кэш — стандартные `dict`/`time`).
- **Артефакты поставки:** `Dockerfile` / `deploy/docker-compose.yml` / `.github/workflows/deploy.yml` —
  **без изменений** (профиль ivahub/long-polling/GHCR не меняется; новых портов/volume/зависимостей
  нет; таблица `referrals` уже существует). `.env.example` и `deploy/.env.example` += два
  плейсхолдера (`REFERRAL_BONUS_PER`, `REFERRAL_BONUS_CAP`).
- **Что ломается:** ничего из публичного контракта. При нуле рефералов и/или дефолтах поведение
  идентично v1.7.0. Новое поведение — повышение лимита за рефералов и второй путь в gate-сообщении.
  Релиз — **MINOR**.
