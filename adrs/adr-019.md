# ADR-019: «Скан»-качество видимой страницы в PDF из фото (рендер из очищенного изображения, режимы цвет/чб, JPEG-сжатие, деградация к оригиналу)

- **Дата:** 2026-06-25
- **Статус:** принято
- **Связанные BA-требования:** `ba-req/ba-requirements-019.md`
- **Связанные ADR:** ADR-016 (надстраивается: Решение 6 «Searchable PDF из фото» —
  `recognize_pdf`/`_run_ocr_pdf`; видимый слой PDF теперь рендерится из очищенного изображения,
  текстовый слой и контракт кнопки «📄 PDF» сохраняются), ADR-002/003 (граница «хендлер ↔ сервис»,
  неблокирование: тяжёлая CV-обработка вне event loop под `HEAVY_LOCAL_SEMAPHORE`), ADR-001 (профиль
  поставки «бот без HTTP-порта», long-polling, GHCR, ivahub — без изменений), ADR-017 (дневной лимит:
  PDF лимит не расходует — без изменений)

## Контекст

С v1.6.0 (BA-016 / ADR-016, Решение 6) под результатом распознавания фото доступна кнопка
**«📄 PDF»**, отдающая **searchable PDF** = исходное изображение + невидимый текстовый слой
Tesseract. Сейчас `services/ocr.py::_run_ocr_pdf` передаёт в `pytesseract.image_to_pdf_or_hocr`
**сырой оригинал** — перекошенный, с тенями и цветным фоном стола. При этом очищенное изображение в
проекте **уже строится** (`services/preprocess.py::preprocess_image`: EXIF-поворот, ресайз,
бинаризация, deskew), но используется **только** для извлечения OCR-текста и в видимую страницу не
попадает. Визуальный результат кнопки почти не отличается от пересланного снимка — главная ценность
фичи («аккуратный документ, а не фото на столе») теряется.

**Цель.** Рендерить видимую страницу PDF из **очищенного** изображения (выравнивание перекоса,
подъём контраста / снятие теней, ресайз к разумному размеру), добавить два режима вывода
(цвет/серый и чб), вшивать страницу как JPEG (меньше итоговый размер) и **обязательно** деградировать
к оригиналу при любом сбое, чтобы результат не был хуже v1.6.0 ни на одном входе.

**Действующие силы и ограничения:**
- Прод ivahub: **1 vCPU / 2 GB**, CPU — узкое место (ОГР-1). Тяжёлая локальная обработка идёт под
  существующим `HEAVY_LOCAL_SEMAPHORE` и вне event loop через `to_thread` (как `recognize_pdf`).
- Текущий `preprocess_image` сразу переводит в grayscale и бинаризует — для цвет/серый-режима
  страницы нужна отдельная ветка обработки, не ломающая ветку для OCR-текста (ОГР-2).
- Лимит `callback_data` 64 байта (ОГР-3); русскоязычные подписи в стиле меню действий (НФТ-6).
- **Без новых внешних зависимостей** по умолчанию: используется уже установленный OpenCV / Pillow /
  Tesseract (НФТ-7).
- Searchable-контракт «выделение текста попадает в слова» требует, чтобы координаты невидимого слоя
  совпадали с видимой (повёрнутой/очищенной) страницей (ФТ-2).

**Ключевой технический момент.** Если deskew/очистка применяются к видимой странице **отдельно** от
текстового слоя, координаты слоя поедут относительно картинки и выделение будет промахиваться мимо
слов. Чтобы этого избежать, проще скормить Tesseract **уже очищенное/выровненное** изображение
**целиком** и дать ему самому положить текстовый слой по этой странице — тогда видимый слой и
текстовый слой строятся из **одного и того же** массива пикселей и согласованы по построению. Это
дешевле и надёжнее, чем синхронизировать два независимых преобразования.

Конфликтов с действующими ADR нет: релиз **улучшает существующую функцию** searchable PDF
(ADR-016, Решение 6), сохраняет её публичный контракт за счёт обязательной деградации к текущему
поведению и не меняет профиль поставки (ADR-001).

## Рассматриваемые варианты

### Решение 1. Согласование текстового слоя с видимой страницей (ОВ-4, ФТ-2)

#### Вариант A (выбран): один проход Tesseract по уже очищенному изображению — видимый слой и текстовый слой из одного массива
- `image_to_pdf_or_hocr` получает **очищенную/выровненную** страницу (RGB/серую или бинаризованную в
  зависимости от режима). Tesseract сам кладёт невидимый текстовый слой по координатам этой же
  картинки, которая и становится видимой страницей. Видимый и текстовый слой согласованы **по
  построению** — нечего синхронизировать.
- Плюсы: координаты слоя совпадают со страницей без ручной геометрии; минимальная поверхность —
  меняется только то, что подаётся в Tesseract; нет второго преобразования координат.
- Минусы: текстовый слой в PDF строится по **очищенному** изображению, а не по тому, что давало лучший
  OCR-текст в `recognize_text` (там выбирается лучший из двух кандидатов). Это **осознанно**: PDF —
  про выделяемость/поиск, а не про максимальное качество текста; `recognize_text` (немедленная выдача)
  **не трогается**, его двухкандидатная логика сохраняется.

#### Вариант B: deskew видимой страницы + ручной пересчёт bbox текстового слоя той же матрицей поворота
- Минусы: хрупкая ручная геометрия (поворот bbox вокруг центра, обрезка по границам), легко
  рассинхронизировать; больше кода и тестов; ради точности, которую Вариант A даёт бесплатно.

Выбран **A** (подтверждает ДОП-2 BA-требования). Это закрывает ОВ-4.

### Решение 2. Пайплайн очистки видимой страницы и рефакторинг `preprocess.py` (ОВ-2, ОВ-6, ОГР-2, ФТ-1/8)

#### Вариант A (выбран): новая функция рендера в `preprocess.py` с общими шагами, цвет-/чб-ветки расходятся в конце; `preprocess_image` (для OCR-текста) не трогаем
- Добавить `render_page(image_bytes: bytes, mode: str) -> PIL.Image.Image`, возвращающую **очищенную
  PIL-страницу, готовую к подаче в Tesseract** (а не numpy-массив для OCR). Общие шаги выделить в
  приватные хелперы, переиспользуемые обеими задачами:
  - `_load_oriented(image_bytes) -> PIL.Image` — открыть + `ImageOps.exif_transpose` (EXIF-поворот).
  - `_resize_to_target(...)` — ресайз к разумной стороне (см. Решение 4): downscale если больше,
    upscale если слишком мелкое (для читаемости).
  - `_deskew` — **переиспользовать существующий** (работает по бинарной маске, см. Решение 3).
  - Режим **`"doc"` (цвет/серый, дефолт):** EXIF → ресайз → подъём контраста/снятие теней (метод —
    Решение 5) → deskew по бинарной маске, поворот применяется к **серой/цветной** странице тем же
    углом → JPEG. Результат — выровненная контрастная страница, **не** бинаризованная.
  - Режим **`"scan"` (чб):** EXIF → ресайз → adaptive threshold (как `preprocess_image`) → deskew →
    бинарная страница. По сути — то, что уже умеет `preprocess_image`, обёрнутое в PIL.
- `preprocess_image` (массив для OCR-текста, используется `recognize_text`) **остаётся как есть** —
  ветка OCR не ломается (ОГР-2). Угол deskew вычисляется по бинарной маске, но **применяется** к
  странице соответствующего режима (серой/цветной в `"doc"`, бинарной в `"scan"`).
- Плюсы: ветка OCR-текста не затронута; общие шаги переиспользованы без дублирования; цвет-режим не
  обязан бинаризовать (ОГР-2 решён ветвлением, а не переписыванием); `_deskew` переиспользован (ОВ-6).
- Минусы: `_deskew` сейчас принимает и возвращает бинарный массив — для `"doc"`-режима нужно отделить
  **вычисление угла** (по бинарной маске) от **применения поворота** (к цветной/серой странице).
  Небольшой рефактор `_deskew` на две функции (`_skew_angle` + `_rotate`) — допустим, не ломает
  текущий вызов в `preprocess_image`.

#### Вариант B: единая функция, всегда возвращающая бинаризованную страницу (как `preprocess_image`), просто отдать её в PDF
- Минусы: цветной/серый режим невозможен (всегда чб) — не закрывает ФТ-3; бинаризация может «выжечь»
  бледный/мелкий текст на части входов (ФТ-8), а деградации к серому нет.

Выбран **A**. Закрывает ОВ-6 (рефактор: выделить рендер-ветку отдельно от OCR-ветки) и ОГР-2.

### Решение 3. Критерий «не трогать ровное» (ОВ-3, ФТ-7)

#### Вариант A (выбран): порог угла deskew + сохранение уже встроенного «потолка» угла; без агрессивной правки
- **Deskew пропускается**, если `abs(angle) < _MIN_DESKEW_ANGLE` (порог ~**0.5°**): мелкий угол —
  шум, поворот только размывает интерполяцией. Уже встроенный верхний потолок `abs(angle) > 15°`
  (см. `_deskew`) сохраняется — большой «угол» это, как правило, ложное срабатывание `minAreaRect`
  на пёстром фоне, его лучше не применять.
- **Подъём контраста — мягкий и идемпотентный** для уже контрастных входов (autocontrast с малым
  `cutoff`, CLAHE с умеренным `clipLimit`): на уже ровном/контрастном изображении он близок к
  no-op, поэтому отдельный «детектор контраста» не нужен — метод сам по себе не «выжигает» (Решение
  5). Это упрощает критерий ФТ-7 до одного порога угла + выбора неагрессивного метода контраста.
- Плюсы: один понятный порог; не плодим эвристик; согласуется с уже имеющимся ограничением угла.
- Минусы: возможны редкие случаи реального наклона < 0.5°, который останется — несущественно
  визуально.

Выбран **A**. Закрывает ОВ-3 (ФТ-7).

### Решение 4. Целевой DPI/размер и качество JPEG (ОВ-5, ФТ-4, НФТ-3)

#### Вариант A (выбран): ресайз по максимальной стороне ~2000 px (≈ A4 @ ~170–200 dpi), JPEG quality 75
- **Целевая максимальная сторона `_PAGE_MAX_SIDE = 2000`** px — близко к A4 @ ~200 dpi по короткой
  стороне, достаточно для читаемости мелкого шрифта, при этом заметно меньше типичного фото с
  телефона (3000–4000 px). Слишком мелкие входы (< ~1500 px) допускается слегка апскейлить для
  читаемости (как `preprocess_image` уже делает для OCR).
- **JPEG `_PAGE_JPEG_QUALITY = 75`** — баланс размер/читаемость; для чб-режима то же качество (или
  выше — бинарное изображение жмётся отлично).
- Tesseract `image_to_pdf_or_hocr` принимает PIL-изображение, и DPI страницы в PDF берётся из
  `image.info["dpi"]`; задаём `dpi=(200, 200)`, чтобы видимая страница имела корректный физический
  размер.
- Эти значения — **константы в коде** (`services/ocr.py`/`services/preprocess.py`), **без env**
  (ДОП-3, НФТ-5). Если архитектор/эксплуатация позже захотят настройку — вынести в env отдельным
  релизом.
- Плюсы: предсказуемый, меньший итоговый PDF (КП-4); читаемость сохранена; без новых env.
- Минусы: фиксированные константы — компромисс «один размер для всех»; приемлемо для бесплатного бота.

#### Вариант B: вынести DPI/качество/сторону в env-переменные
- Минусы: новые env без явной потребности (BA: «параметры — константы, если архитектор не решит
  иначе»); лишняя поверхность конфигурации и плейсхолдеры в обоих `.env.example`. Отклонён —
  не требуется в этом релизе.

Выбран **A**. Закрывает ОВ-5. **Новых env-переменных нет** (НФТ-5, КП-10).

### Решение 5. Метод подъёма контраста / снятия теней (ОВ-2, ФТ-8)

#### Вариант A (выбран): для `"doc"` — CLAHE по L-каналу (или grayscale) + лёгкий autocontrast; для `"scan"` — существующий adaptive threshold
- **`"doc"`-режим:** перевести в grayscale (или работать по L-каналу), применить **CLAHE**
  (`cv2.createCLAHE`, умеренный `clipLimit≈2.0`, `tileGridSize=(8,8)`) — локальное выравнивание
  освещённости снимает тени без «выжигания» бледного текста (в отличие от глобального
  autocontrast/бинаризации). Дефолтный результат — **серая** контрастная страница (цвет фона стола
  уходит, бледный текст сохраняется). Цвет можно сохранить, но grayscale надёжнее снимает цветной
  фон и компактнее — серый принят как практический дефолт «doc».
- **`"scan"`-режим:** существующий `cv2.adaptiveThreshold` (как в `preprocess_image`) — классический
  чб-скан.
- CLAHE на уже контрастном входе близок к no-op (clipLimit ограничивает усиление) → не ухудшает
  ровное (ФТ-7, Решение 3).
- Плюсы: локальный метод снимает тени и не «выжигает» мелкий/бледный текст (ФТ-8); CLAHE уже в
  OpenCV — без новых зависимостей (НФТ-7); один метод для дефолта.
- Минусы: дефолт `"doc"` отдаёт **серую**, а не цветную страницу — для документов это обычно лучше
  (убирает цветной фон), а название режима «Документ (цвет/серый)» это допускает (BA: «цветная **или
  серая**»). Полноцветный вывод — возможное будущее уточнение, не в scope.

#### Вариант B: глобальный `ImageOps.autocontrast` / гистограммная нормализация
- Минусы: глобальный контраст «выжигает» при неравномерном освещении (тени) — ровно то, что ФТ-8
  запрещает; хуже на фото со столом.

Выбран **A**. Закрывает ОВ-2 (ФТ-8).

### Решение 6. UX режимов и дефолт (ОВ-1, ФТ-3)

#### Вариант A (выбран): одна кнопка «📄 PDF» = дефолт «Документ» + вторая кнопка «📄 Скан (ч/б)»
- В клавиатуре изображений (`with_pdf`) — **две** кнопки в одном ряду:
  - «📄 PDF» → callback `act:pdf` (как сейчас) → режим **`"doc"`** (дефолт, выровненная контрастная
    серая страница).
  - «📄 Скан (ч/б)» → новый callback `act:pdfbw` → режим **`"scan"`** (бинаризованная страница).
- Дефолт — **«Документ»** (цвет/серый выровненный), как просит требование (ОВ-1, ФТ-3). Существующий
  callback `act:pdf` сохраняет смысл «дать PDF» → обратная совместимость старых сообщений с кнопкой.
- Плюсы: один тап до результата для обоих режимов (без промежуточного меню выбора); короткие коды
  `act:pdf`/`act:pdfbw` укладываются в 64 байта (ОГР-3); минимум нового кода — второй хендлер —
  тонкая обёртка над общим.
- Минусы: две кнопки в ряду PDF вместо одной — приемлемо, ряд PDF отдельный.

#### Вариант B: одна кнопка + выбор режима после нажатия (вложенное меню/edit-клавиатура)
- Минусы: лишний тап и состояние; больше callback-кодов и обработки; UX тяжелее ради двух режимов.

Выбран **A**. Закрывает ОВ-1. Подписи русские, в стиле меню (НФТ-6, КП-11).

### Решение 7. Обязательная деградация к оригиналу (ФТ-5/6, НФТ-2)

`_run_ocr_pdf(image_bytes, mode)`:
1. Попытаться построить очищенную страницу `render_page(image_bytes, mode)` и
   `image_to_pdf_or_hocr` по ней.
2. **Любое** исключение или пустой результат на шаге 1 → `logger.warning`/`exception` (факт
   деградации логируется, НФТ-2) → **fallback**: `image_to_pdf_or_hocr` по **оригиналу** (поведение
   v1.6.0).
3. Если и fallback падает → вернуть `None` (хендлер показывает «Не удалось сформировать PDF», как
   сейчас).

Так результат **никогда не хуже** v1.6.0 (ДОП-4, ФТ-6): в худшем случае получаем тот же
оригинал-PDF, что и сегодня. Закрывает ФТ-5/ФТ-6, КП-5/КП-6.

### Решение 8. Ресурсные ограничения (ФТ-9, НФТ-1, КП-8)

Вся CV-обработка рендера выполняется **внутри** `_run_ocr_pdf`, который уже вызывается из
`recognize_pdf` через `asyncio.to_thread` под `HEAVY_LOCAL_SEMAPHORE`. Дополнительной параллельной
тяжёлой нагрузки **не вводится** — один семафор сериализует и `recognize_text`, и `recognize_pdf`
(ADR-002/003/016). `recognize_pdf` получает аргумент `mode`; хендлеры передают `"doc"`/`"scan"`.
Закрывает ФТ-9, НФТ-1, КП-8.

### Решение 9. Версия — MINOR 1.9.0 (ОВ-7, НФТ-4)

Обратимо-совместимое **улучшение существующей функции**: публичный контракт кнопки «📄 PDF» не
ломается (по-прежнему приходит searchable PDF с выделяемым текстом), обязательная деградация
гарантирует, что поведение не хуже v1.6.0. Новый режим «Скан (ч/б)» и callback `act:pdfbw` — это
**добавление**. Поведение прочих кнопок, дневной лимит (PDF лимит не расходует, ADR-017), профиль
поставки — без изменений. → **MINOR, v1.9.0** (подтверждено, ориентир BA). `roadmap.md` (если
ведётся): строка v1.9.0.

**Метаданные/имя файла PDF** (ОВ-7, необязательная косметика): остаётся текущее `document.pdf`;
осмысленное имя/`title` — **вне scope** этого релиза (BA: не критерий приёмки). Минимизация scope —
только видимый слой PDF.

## Решение

Резюме принятых решений:

1. **Согласование слоя** (Реш. 1A) — Tesseract строит видимый и текстовый слой из **одного**
   очищенного изображения; ручной пересчёт bbox не делаем (ОВ-4).
2. **Рендер-ветка в `preprocess.py`** (Реш. 2A) — новая `render_page(image_bytes, mode)`, общие
   шаги (EXIF/ресайз/deskew-угол) переиспользованы; `preprocess_image` (OCR-текст) **не тронут**;
   `_deskew` разделён на вычисление угла и применение поворота (ОВ-6, ОГР-2).
3. **«Не трогать ровное»** (Реш. 3A) — порог угла `_MIN_DESKEW_ANGLE ≈ 0.5°` + сохранён потолок 15°;
   мягкий идемпотентный контраст вместо отдельного детектора (ОВ-3, ФТ-7).
4. **Размер/качество** (Реш. 4A) — максимальная сторона `≈ 2000` px, JPEG quality `75`, dpi 200,
   **константы без env** (ОВ-5, ФТ-4, НФТ-3/НФТ-5).
5. **Контраст/тени** (Реш. 5A) — CLAHE по grayscale для `"doc"`; adaptive threshold для `"scan"`;
   не «выжигает» бледный текст (ОВ-2, ФТ-8).
6. **UX** (Реш. 6A) — две кнопки: «📄 PDF» (`act:pdf`, дефолт «Документ») и «📄 Скан (ч/б)»
   (`act:pdfbw`, режим «Скан»); русские подписи (ОВ-1, ФТ-3, КП-11).
7. **Деградация** (Реш. 7) — сбой рендера → fallback на оригинал (v1.6.0) + лог; сбой и его → `None`
   (ФТ-5/6, НФТ-2).
8. **Ресурсы** (Реш. 8) — вся обработка под существующим `HEAVY_LOCAL_SEMAPHORE` вне event loop;
   новой параллельной CV-нагрузки нет (ФТ-9, НФТ-1).
9. **Версия — MINOR 1.9.0** (Реш. 9); метаданные/имя файла PDF — вне scope (ОВ-7).
10. **Без новых внешних зависимостей и без новых env** (НФТ-7, НФТ-5): CLAHE/adaptive threshold —
    OpenCV; JPEG/EXIF — Pillow; searchable PDF — Tesseract.

## Последствия

- **Положительные:** видимая страница PDF выглядит как из сканер-приложения (выровнена, контрастна,
  тени сняты), а не как фото на столе; два режима (документ/скан) одним тапом; итоговый PDF меньше
  (JPEG + ресайз, НФТ-3/КП-4); текстовый слой согласован с видимой страницей по построению (ФТ-2);
  результат **никогда не хуже** v1.6.0 за счёт деградации (ФТ-6); ветка OCR-текста и профиль поставки
  не затронуты; без новых зависимостей/env/портов.
- **Отрицательные / компромиссы:** дефолт «Документ» отдаёт **серую** (не полноцветную) страницу —
  обычно лучше для документов, но это сознательное сужение; текстовый слой PDF строится по очищенному
  изображению, а не по «лучшему из двух кандидатов» как `recognize_text` (PDF — про выделяемость, не
  про максимум качества текста); фиксированные константы размера/качества (один размер для всех);
  `_deskew` разделён на две функции — небольшой рефактор с риском регрессии угла (покрыть тестом).
- **Технический долг:** многостраничный PDF из альбома (вариант B BA) — следующее требование;
  авто-кроп/коррекция перспективы (вариант C) — позже; полноцветный «doc»-режим; вынос
  размера/качества в env при появлении спроса; осмысленное имя/`title` PDF.
