# ADR-021: Транскрипция речи из видео — извлечение аудиодорожки через PyAV и переиспользование аудио-пайплайна

- **Дата:** 2026-06-25
- **Статус:** принято
- **Связанные BA-требования:** `ba-req/ba-requirements-021.md`
- **Связанные ADR:** ADR-001 (профиль поставки «бот без HTTP-порта», long-polling, GHCR, секреты через `.env` — **сохраняется**, новых артефактов нет), ADR-003 / ADR-008 (пайплайн транскрипции `services/transcribe.py`: CF Whisper по умолчанию + локальный faster-whisper-фолбэк, декод через `av.open` — переиспользуется как есть), ADR-010 (глобальный `HEAVY_LOCAL_SEMAPHORE` на тяжёлые локальные операции на 1 vCPU / 2 GB; раздел «Публичный контракт 1.0.0», расширяемый этим релизом перечнем входных каналов), ADR-015 (прогрессивное раскрытие суть→кратко→полностью), ADR-016 (HTML-вывод, меню действий, персистентный `result_cache`), ADR-014 (проактивные группы, кнопка «Распознать», `_has_supported_media`), ADR-017 (дневной лимит и subscription-gate), ADR-018 (реферальная награда)

## Контекст

Textify распознаёт голос/аудио (ADR-003/008) и изображения/OCR (v0.2.0), отдаёт результат с
прогрессивным раскрытием (ADR-015) и меню действий (ADR-016), под дневным лимитом/gate (ADR-017) и
реферальными бонусами (ADR-018). Видео бот сейчас **игнорирует**, хотя видео с речью (лекции,
кружочки `video_note`, пересланные ролики) — частый формат, интуитивно совпадающий для пользователя
с голосовым.

BA-требование 021 вводит **новый входной канал — транскрипцию речи из видео** (не OCR кадров).
Все продуктовые решения приняты заказчиком: принимать `video` / `video_note` / документ `video/*`;
извлекать аудиодорожку через уже подключённый PyAV (`av`), **без системного ffmpeg**; подавать в
существующий пайплайн `services/transcribe.py`; результат — как у аудио (прогрессив + те же кнопки);
переиспользовать гард 20 МБ, лимит/gate, рефералы, `result_cache`; видео без аудио/без речи →
дружелюбное сообщение без падения. На этап ADR делегированы **технические** вопросы ОВ-1…ОВ-6.

Действующие силы и ограничения:

- **Слабый сервер.** ivahub 1 vCPU / 2 GB, long-polling; whisper — основной потребитель CPU
  (ADR-010). Извлечение аудио и число перекодировок должны быть экономны; тяжёлая локальная работа —
  под существующим `HEAVY_LOCAL_SEMAPHORE` (ADR-010 Решение 3).
- **Два провайдера ASR с разным контрактом входа.** CF Whisper (`_transcribe_cloudflare`) ждёт
  **аудио-payload** (base64 аудио-контейнера), видео-байты не принимает. Локальный faster-whisper
  (`_transcribe_sync` → `model.transcribe(BytesIO(...))` → `av.open`) технически умеет открыть и
  видео-контейнер, взяв из него аудиодорожку. `transcribe_with_timestamps(audio_bytes, ...)` —
  единая точка входа, принимающая один `bytes`-payload и сама выбирающая провайдера/фолбэк
  (ADR-008). Подать в неё **видео**-байты нельзя: CF-ветка их отвергнет.
- **Существующий аудио-хендлер уже обобщён.** `handlers/audio.process_audio(media_message,
  reply_target, bot, audio_bytes, force_local, progressive)` принимает готовые `audio_bytes` и
  делает всё остальное: транскрипция, порог тайм-кодов, прогрессив (ADR-015), клавиатура (ADR-016),
  `result_cache`, сообщение «речь не распознана». Три аудио-хендлера (`voice`/`audio`/документ
  `audio/*`) сводятся к «скачать → гард → лимит → `process_audio`».
- **Точка списания лимита фиксирована.** `enforce_limit` (ADR-017) списывает единицу
  (`record_recognition`) **при допуске, до распознавания** — то есть до того, как известно, есть ли в
  медиа речь. Так уже ведут себя аудио и фото (голос без речи / фото без текста тоже списывают
  единицу). Это часть контракта ADR-017.
- **`video_note` без mime.** Кружочки — отдельный тип `message.video_note` без mime (всегда mp4);
  `video` — `message.video` (есть `mime_type`); видео-файл — `message.document` с `mime_type`,
  начинающимся на `video/`.
- **Группы (ADR-014).** `_has_supported_media` / `_dispatch_media` покрывают аудио и фото; видео там
  не распознаётся.
- **Без новых зависимостей.** PyAV (`av`) уже в окружении (транзитивно через faster-whisper, который
  использует его для декода). Системный ffmpeg **не добавляется** (ОГР-3, НФТ-3).

Противоречий действующим ADR нет: релиз **переиспользует** пайплайн транскрипции, формат вывода,
гард, лимит/gate, рефералы, `result_cache` и сохраняет профиль поставки ADR-001. Это **MINOR** —
обратимо-совместимое добавление входного канала, без слома публичного контракта (ADR-010).

## Рассматриваемые варианты

### Решение 1 (ОВ-1). Способ извлечения аудио и подача в пайплайн транскрипции

#### Вариант A: единое извлечение аудиодорожки PyAV → один аудио-payload в существующий `transcribe_with_timestamps`
Новый хелпер `extract_audio(video_bytes) -> bytes | None` через PyAV: `av.open(BytesIO(video_bytes))`,
выбрать аудиопоток, ремультиплексировать/перекодировать его в стандартный аудио-контейнер (например
WAV PCM 16 kHz mono или MP3) и вернуть **аудио**-байты. Результат подаётся в **немодифицированный**
`transcribe_with_timestamps(audio_bytes, ...)`, который сам решает CF/local/фолбэк (ADR-008). Если
аудиопотока нет → вернуть `None`.
- Плюсы: **один** путь и **одна** перекодировка независимо от провайдера; CF получает корректный
  аудио-payload (закрывает ОГР-4); local получает чистое аудио (короче декод, меньше RAM, чем
  открывать видео-контейнер целиком); `services/transcribe.py` и его контракт **не меняются** (ФТ-3,
  ДОП-2); провайдерская логика/фолбэк/бюджет CF переиспользуются без дублирования; «нет аудиодорожки»
  определяется на этапе извлечения (`None`) — чисто и до похода в whisper.
- Минусы: одна обязательная перекодировка даже для local (которому теоретически можно было бы отдать
  видео напрямую) — но это **ровно одна** перекодировка, выполняемая под `HEAVY_LOCAL_SEMAPHORE`,
  и она же нужна для CF; «бесплатного» прямого пути для local нет, т.к. провайдер выбирается **внутри**
  `transcribe_with_timestamps`, и хендлер заранее не знает, пойдёт ли запрос в CF или в local.
- Последствия: новый хелпер на PyAV (в `services/transcribe.py` рядом с декодом или в `services/media.py`);
  декод/энкод аудио — CPU-работа → под `HEAVY_LOCAL_SEMAPHORE` (ADR-010).

#### Вариант B: два пути — для local подавать видео-байты прямо в `av.open`, для CF извлекать аудио
Ветвление по провайдеру: при local-пути отдать `transcribe` сырые видео-байты (faster-whisper откроет
видео через `av.open` и сам возьмёт аудиодорожку), при CF — извлечь аудио PyAV.
- Плюсы: для local-пути формально ноль «наших» перекодировок (всё внутри faster-whisper).
- Минусы: провайдер выбирается **внутри** `transcribe_with_timestamps` (включая фолбэк CF→local при
  сбое, исчерпании бюджета CF, превышении `_CF_MAX_AUDIO_BYTES`) — хендлер **не знает** заранее, какой
  путь сработает, поэтому «отдать видео только в local» требует **протащить видео-байты внутрь**
  `transcribe.py` и продублировать там провайдерскую развилку → ломает контракт ADR-008 и границу
  «хендлер ↔ сервис»; CF-ветка всё равно требует извлечения; «нет аудиодорожки» при прямой подаче
  видео в faster-whisper всплывает как невнятная ошибка декода, а не как чистый `None`; открытие
  видео-контейнера целиком тяжелее для RAM, чем чистое аудио. Сложнее и опаснее ради экономии на
  фолбэк-пути, который и так редкий.
- Последствия: модификация `services/transcribe.py` под видео, дублирование развилки, риск регрессий
  аудио-канала.

Выбран **A**. Извлекаем аудиодорожку **один раз** PyAV в стандартный аудио-контейнер и подаём в
**неизменный** `transcribe_with_timestamps`. Минимум перекодировок (одна), сохранение контракта
ADR-008/ADR-010, чистое определение «нет аудио» (`None`) до whisper, CF получает валидный
аудио-payload. Извлечение — под `HEAVY_LOCAL_SEMAPHORE`. Целевой формат — лёгкий для CF и whisper
(WAV PCM 16 kHz mono либо MP3; точный кодек/контейнер — деталь реализации, критерий: декодируется
faster-whisper и принимается CF Whisper, размер минимизирован под 1 vCPU). **Конкретный кодек выбора
кодеков PyAV без доп. системных кодеков** — на разработчике; если целевой энкодер недоступен в сборке
PyAV, допустим иной общедоступный аудио-кодек, лишь бы без системного ffmpeg.

### Решение 2 (ОВ-2). Диспетчеризация `video_note` (нет mime)

#### Вариант A: один обработчик аудио-из-видео, три фильтра-входа, общий путь обработки
Три aiogram-фильтра — `F.video`, `F.video_note`, `F.document & mime startswith "video/"` — ведут в
**общую** функцию `process_video(...)`, отличающуюся только тем, какой attachment скачивается. Дальше
единый путь: гард → лимит → `extract_audio` → `process_audio`.
- Плюсы: `video_note` без mime обрабатывается тем же кодом — отличается лишь источник attachment
  (`message.video_note` vs `message.video` vs `message.document`); нет дублирования логики;
  симметрично трём аудио-хендлерам (`voice`/`audio`/документ).
- Последствия: `handlers/video.py` с тремя тонкими хендлерами и одной общей `process_video`.

#### Вариант B: отдельная ветка для `video_note`
- Минусы: дублирование пути обработки ради единственного отличия (источник attachment); лишний код.

Выбран **A**. `video_note` диспетчеризуется по типу сообщения (`F.video_note`), а не по mime;
mime-фильтр (`video/*`) применяется только к документу. Все три формы сходятся в общий путь.

### Решение 3 (ОВ-3). Охват групп в первом релизе

#### Вариант A: первый релиз видео — только личный чат; группы отложены
Новый `video_router` фильтруется `F.chat.type == "private"` (как `audio_router`/`image_router`).
`handlers/group.py` (`_has_supported_media`, `_dispatch_media`) **не трогаем** — видео в группах в
этом релизе не предлагается.
- Плюсы: минимальный scope (принцип «минимальные изменения», CLAUDE.md); основной сценарий (личка)
  закрыт; не трогаем чувствительную групповую диспетчеризацию (ADR-014) и её контракт; меньше
  поверхности регрессий; экономия CPU в групповых чатах (видео тяжелее аудио).
- Минусы: видео в группах не распознаётся в v1.11.0 — переносится в следующий релиз.
- Последствия: `handlers/group.py` без изменений; КП-11 удовлетворён вариантом «отложено и явно
  отражено».

#### Вариант B: сразу включить видео в группах по механизму v1.4.0
Расширить `_has_supported_media` (+ `_is_audio_media`/`_dispatch_media`) на `video`/`video_note`/
документ `video/*`, добавить ветку извлечения аудио в `_dispatch_media`.
- Минусы: расширяет scope и затрагивает контракт групп (ADR-014); видео тяжелее аудио — кнопка
  «Распознать» на каждое групповое видео нагружает 1 vCPU сильнее; больше тестов и риск регрессий
  проактивных групп; BA допускает оба варианта (ОВ-3, ФТ-13).
- Последствия: правки `handlers/group.py`, расширение тестов групп.

Выбран **A** (только личка в v1.11.0). Обоснование: минимизация scope и риска под ограничение 1 vCPU;
группы — естественный отдельный инкремент. Это **явно** фиксируется в спецификации (КП-11, вариант
«отложено»). При появлении потребности видео в группах вводится следующим MINOR по механизму ADR-014.

### Решение 4 (ОВ-4). Кап размера/длительности

#### Вариант A: только существующий гард 20 МБ, без отдельного лимита длительности
- Плюсы: 20 МБ — практический потолок загрузки ботом через getFile (ДОП-4, ОГР-2); это и так сильно
  ограничивает длину видео; переиспользуется `is_oversized`/`OVERSIZED_MESSAGE` (ADR-010) без нового
  контракта и нового сообщения; меньше кода.
- Минусы: теоретически очень короткое, но «плотное» видео ≤ 20 МБ может дать длинную дорожку — но это
  тот же риск, что у аудио ≤ 20 МБ, и он уже принят продуктом.
- Последствия: гард применяется к `file_size` видео-attachment до скачивания, как у аудио.

#### Вариант B: дополнительный лимит длительности видео
- Минусы: новый контракт/сообщение/конфиг ради краевого случая, уже косвенно ограниченного 20 МБ;
  длительность точно известна только после демукса; усложнение без явной потребности.

Выбран **A**. Только гард 20 МБ (`handlers/limits.py`), как у прочего медиа (ФТ-5, КП-5). Отдельного
лимита длительности нет.

### Решение 5 (ОВ-5). Чанкинг длинных аудио из видео

Вне рамок релиза. Переиспользуется существующая логика `transcribe_with_timestamps` **как есть**:
сверхбольшое аудио (> `_CF_MAX_AUDIO_BYTES`) уже деградирует на локальный whisper (ADR-008/010);
чанкинг для CF — наследуемый техдолг (отмечен ещё в ADR-010), не вводится этим релизом (ФТ-3, НФТ-1).

### Решение 6 (ОВ-6). Поведение лимита при «нет речи / нет аудиодорожки»

#### Вариант A: списывать единицу, как у аудио/фото (точка списания ADR-017 не меняется)
`enforce_limit` (ADR-017) списывает единицу **при допуске, до распознавания** — для видео тот же
порядок: гард → `enforce_limit` (списание) → `extract_audio` → транскрипция. Видео без аудио/без речи
расходует единицу так же, как голос без речи или фото без текста.
- Плюсы: контракт лимита/gate **не меняется** (ФТ-6, ФТ-12, КП-6) — это прямое требование BA «контракт
  лимита не меняется»; симметрия со всеми каналами; нет нового кода учёта/возврата; нет гонок
  «списать/вернуть»; защита от абуза (нельзя слать «пустые» видео бесконечно).
- Минусы: пользователь, приславший видео без речи, теряет единицу — но ровно как с пустым голосовым;
  справедливость учёта, не основной путь (BA: «затрагивает только справедливость»).
- Последствия: ничего нового; единый порядок «лимит до обработки».

#### Вариант B: не списывать при «нет речи» (возврат/откат единицы)
- Минусы: требует **изменения** точки списания ADR-017 (списывать после успеха, либо возвращать
  единицу) → меняет контракт лимита для **всех** каналов или вводит видео-специфичный путь;
  рассинхрон с аудио/фото; гонки и сложность; противоречит BA «контракт лимита не меняется».
- Последствия: слом симметрии и контракта ради краевого случая.

Выбран **A**. Видео списывает единицу как любое медиа; точка списания ADR-017 не меняется. Это
сознательный компромисс: справедливость учёта приносится в жертву неизменности контракта и симметрии
(требование BA приоритетнее).

## Решение

1. **Извлечение аудио (Решение 1A).** Новый хелпер `extract_audio(video_bytes: bytes) -> bytes | None`
   на PyAV (`av`): открыть `BytesIO(video_bytes)`, выбрать аудиопоток, перекодировать его в
   стандартный аудио-контейнер (WAV PCM 16 kHz mono или MP3 — деталь реализации), вернуть
   аудио-байты; **нет аудиопотока → `None`**. Размещение — `services/transcribe.py` (рядом с декодом)
   или `services/media.py`. CPU-работа выполняется **под `HEAVY_LOCAL_SEMAPHORE`** (ADR-010) через
   `asyncio.to_thread`. Системный ffmpeg **не добавляется** (PyAV-кодеки).
2. **Пайплайн без изменений (Решение 1A).** Извлечённые аудио-байты подаются в **немодифицированный**
   `transcribe_with_timestamps(audio_bytes, force_local=...)` (ADR-008): CF Whisper по умолчанию,
   локальный фолбэк, бюджет CF, порог `_CF_MAX_AUDIO_BYTES` — всё переиспользуется.
3. **Хендлер видео (Решение 2A).** Новый `handlers/video.py`: три тонких хендлера-входа
   (`F.video`, `F.video_note`, `F.document & mime startswith "video/"`), фильтр
   `router.message.filter(F.chat.type == "private")`, общая `process_video(...)`. Путь:
   гард 20 МБ → `enforce_limit` → скачать → `extract_audio` → при `None` дружелюбное сообщение
   «в видео нет звуковой дорожки…» → иначе `process_audio(..., progressive=True)`.
4. **Результат как у аудио.** Переиспользуется `handlers/audio.process_audio` (прогрессив ADR-015,
   клавиатура ADR-016, `result_cache`, тайм-коды, сообщение «речь не распознана»). **Без дублирования**
   логики выдачи.
5. **Гард 20 МБ (Решение 4A).** `is_oversized(<attachment>.file_size)` / `OVERSIZED_MESSAGE`
   (`handlers/limits.py`) — без изменений.
6. **Лимит/gate/рефералы (Решение 6A).** `enforce_limit` (ADR-017) вызывается до обработки, списывает
   единицу; рефералы (ADR-018) применяются через него. Контракт **не меняется**; видео без речи
   списывает единицу как любое медиа.
7. **Группы (Решение 3A).** В v1.11.0 видео — **только личный чат**; `handlers/group.py` не трогаем
   (отложено, явно отражено — КП-11).
8. **Чанкинг (Решение 5).** Вне рамок; переиспользуется логика транскрипции как есть.
9. **Регистрация канала.** `handlers/__init__.py` экспортирует `video_router`; `bot.py` включает его
   `dp.include_router(video_router)`. Раздел «Входные каналы» публичного контракта (ADR-010)
   дополняется видео (см. ниже).
10. **Ошибки и сообщения.** Любая ошибка `extract_audio`/декода обрабатывается без падения бота
    (`try/except` с понятным русским сообщением); все тексты канала — на русском (ФТ-11).
11. **Версия — MINOR 1.11.0.** Новый входной канал без слома контрактов существующих каналов и
    поставки.

### Дополнение к публичному контракту (semver-baseline, ADR-010)

Раздел «Входные каналы (распознаются)» дополняется (обратимо-совместимо, MINOR):
- видео (`message.video`);
- видео-кружок (`message.video_note`);
- видео-документ (`document` с `mime_type`, начинающимся на `video/`).

Семантика: распознаётся **речь из аудиодорожки** видео (не OCR кадров); формат ответа, действия,
лимит/gate, рефералы, гард — как у аудио-канала. Профиль поставки (ADR-001) — без изменений.

## Последствия

- **Положительные:** новый востребованный входной канал при **нулевых** новых зависимостях и без
  системного ffmpeg; максимальное переиспользование (`transcribe` без правок, `process_audio` без
  дублирования, гард/лимит/gate/рефералы/кэш как есть); одна перекодировка на запрос под
  `HEAVY_LOCAL_SEMAPHORE` — предсказуемая нагрузка на 1 vCPU; чистое определение «нет аудиодорожки»
  (`None`) до whisper; контракты ADR-008/010/017/018 не тронуты.
- **Отрицательные / компромиссы:** обязательна одна перекодировка аудио даже для local-пути (Решение
  1A — приемлемо: она же нужна для CF, и фолбэк-путь редкий); видео в группах отложено (Решение 3A —
  следующий инкремент); видео без речи списывает единицу лимита (Решение 6A — в пользу неизменности
  контракта); чанкинг длинного аудио из видео не вводится (Решение 5 — наследуемый техдолг).
- **Технический долг:** видео в группах (ADR-014); чанкинг крупного аудио для CF (наследие ADR-010);
  возможный «лёгкий» прямой путь видео→local без перекодировки при будущем рефакторинге `transcribe`;
  опциональный лимит длительности при появлении потребности.
