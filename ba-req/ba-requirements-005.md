# BA-требования 005: Структурирование распознанного текста через бесплатный LLM (Cloudflare Workers AI) (v0.5.0)

- **Источник:** `requirements/req-textify-005.md`
- **Дата:** 2026-06-22
- **Статус:** черновик

## 1. Контекст и бизнес-цель

Textify — Telegram-бот, который бесплатно превращает голосовые/аудио и изображения в чистый
**структурированный** текст (см. `README.md`, `CLAUDE.md`). Функциональность развивалась поэтапно:
каркас (v0.1.0), OCR изображений (v0.2.0), аудио-транскрипция (v0.3.0), улучшение качества OCR
(v0.4.0). Все эти релизы в проде на ivahub.

Сейчас оба канала распознавания — OCR (`services/ocr.py`) и транскрипция (`services/transcribe.py`) —
отдают пользователю **сырой** текст без постобработки. Между тем структурирование распознанного
текста (заголовки, списки, ключевые пункты в Markdown) — это **заявленная ключевая ценность
продукта** (README/roadmap): обещание «не сырой текст, а аккуратно оформленный». Эта ценность пока
не реализована.

Изначально структурирование планировалось через Groq free API, но **Groq гео-блокирует российские
IP** (CF 403), и с сервера ivahub API недоступен (см. memory: гео-блок Groq по RU). Поэтому слот
несколько раз сдвигался по roadmap. Заказчик принял решение: реализовать структурирование на
**полностью бесплатном провайдере, работающем с RU-IP без прокси** — **Cloudflare Workers AI** —
сделав его провайдером по умолчанию, и сохранить Groq как опцию на будущее (когда доступность
решится), переключаемую без переписывания кода.

Цель релиза v0.5.0 — **довести продукт до его исходного ценностного обещания**: на вход голос/аудио
или фото, на выходе — распознанный текст, приведённый LLM к читаемой структуре в Markdown. При
сбое/недоступности LLM пользователь всё равно получает сырой распознанный текст (структурирование —
улучшение поверх, не критичный путь).

Стейкхолдеры:
- **Владелец продукта** (заказчик) — хочет реализовать заявленную ценность (структурированный
  вывод) на бесплатной инфраструктуре, работающей с ivahub; выбрал Cloudflare как дефолт, Groq —
  опцию; завёл бесплатный аккаунт Cloudflare и токен.
- **Конечный пользователь Telegram** — присылает аудио или фото и получает аккуратно оформленный
  текст вместо «простыни» сырого распознавания; при недоступности LLM получает сырой текст без
  ошибки.
- **Команда разработки** (последующие агенты) — получает новый сервисный слой `services/structure.py`
  с абстракцией провайдера, встроенный в оба существующих хендлера.

## 2. Область (scope)

### В рамках задачи

- **Новый сервисный слой структурирования** (`services/structure.py`): принимает на вход сырой
  распознанный текст, обращается к LLM-провайдеру и возвращает структурированный Markdown
  (заголовки, списки, ключевые пункты).
- **Провайдер по умолчанию — Cloudflare Workers AI** (бесплатный tier, не блокирует RU-IP, работает
  с ivahub напрямую без прокси). Конкретная модель — лёгкая мультиязычная (Llama 3.x / Qwen) —
  выбирается и обосновывается архитектором в ADR (см. ОВ-2).
- **Абстракция выбора провайдера через переменную окружения `LLM_PROVIDER`**: значение по умолчанию —
  Cloudflare; Groq — альтернативное значение «на будущее». Смена провайдера выполняется
  конфигурацией, без правки кода.
- **Подключение структурирования к обоим входным каналам** — изображения (`handlers/image.py`) и
  аудио (`handlers/audio.py`): сырой результат OCR/транскрипции перед отправкой пользователю
  проходит через структурирование. Это общий сквозной сценарий.
- **Промпт структурирования**, ориентированный на смешанный rus+eng контент: сохраняет смысл,
  **не выдумывает факты и не дописывает то, чего нет в исходнике**; задача — оформление, а не
  генерация нового содержания.
- **Фолбэк на сырой текст**: при недоступности/ошибке/таймауте LLM бот отдаёт сырой распознанный
  текст и не падает.
- **Неблокирующий вызов LLM**: обращение к внешнему API — сетевой I/O; интеграция в event loop
  aiogram должна быть корректной (не блокировать loop на время HTTP-запроса).
- **Секреты провайдера — только через `.env`**: `CF_ACCOUNT_ID`, `CF_API_TOKEN` (для Groq-ветки —
  `GROQ_API_KEY`); плейсхолдеры — в `.env.example` (и `deploy/.env.example`). Никогда не хардкодить.
- **Обновление `roadmap.md`**: слот v0.5.0 переопределяется со «Структурирование (Groq)» на
  «Структурирование (Cloudflare Workers AI; Groq — опция)».

### Вне рамок (explicitly out of scope)

- **Возврат к Groq как дефолту / настройка relay-прокси под Groq** — отдельная будущая история
  (Groq-ветка в этом релизе реализуется лишь как переключаемая опция, её боевое использование не
  требуется и не проверяется на проде).
- **Индикация прогресса обработки и пользовательские лимиты размера входного файла** — релиз
  UX-устойчивости (v0.6.0).
- **Тонкая настройка стиля структуры под разные типы контента**, выбор между несколькими моделями в
  рантайме, пользовательский выбор формата вывода.
- **Изменение логики самого распознавания** (OCR v0.2.0/v0.4.0, транскрипция v0.3.0): их контракт
  «сырой текст» сохраняется как вход для структурирования; способ приёма и скачивания файлов не
  меняется.
- **Изменение профиля поставки** (ADR-001): бот остаётся long-polling без HTTP-порта.

## 3. Действующие лица и сценарии использования

**Пользователь Telegram.** Цель — получить распознанный текст в аккуратном, читаемом виде, который
удобно скопировать и использовать.
- Сценарий A (аудио): пользователь отправляет голосовое/аудио → бот транскрибирует и возвращает
  **структурированный** Markdown (заголовки/списки/ключевые пункты) вместо сырого транскрипта.
- Сценарий B (изображение): пользователь отправляет фото/изображение-документ → бот распознаёт OCR
  и возвращает **структурированный** Markdown вместо сырого распознанного текста.
- Сценарий C (фолбэк): LLM-провайдер недоступен/вернул ошибку/превысил таймаут → бот возвращает
  **сырой** распознанный текст без сообщения об ошибке/падения; пользователь всё равно получает
  результат распознавания.
- Сценарий D (пустой результат распознавания): распознавание не нашло текста → поведение прежнее
  (сообщение «текст не распознан»); LLM не вызывается на пустом входе.

**Эксплуатирующий инженер / владелец.** Цель — развернуть структурирование на ivahub на бесплатной
инфраструктуре.
- Сценарий E (конфигурация и деплой): владелец заводит бесплатный аккаунт Cloudflare, получает
  `CF_ACCOUNT_ID` и `CF_API_TOKEN`, кладёт их в `.env`; деплоит образ → бот по умолчанию использует
  Cloudflare Workers AI, обращается к нему по сети из ivahub без прокси.
- Сценарий F (переключение провайдера в будущем): при решении проблемы доступности Groq владелец
  меняет `LLM_PROVIDER` на Groq и задаёт `GROQ_API_KEY` в `.env` — без изменения кода.

## 4. Функциональные требования

- **ФТ-1.** Бот предоставляет сервисный слой структурирования (`services/structure.py`), который
  принимает сырой распознанный текст и возвращает структурированный Markdown (заголовки, списки,
  ключевые пункты), полученный от LLM-провайдера.
- **ФТ-2.** Провайдер LLM по умолчанию — **Cloudflare Workers AI**. Выбор провайдера управляется
  переменной окружения `LLM_PROVIDER`; при незаданном/дефолтном значении используется Cloudflare.
- **ФТ-3.** Поддерживается **абстракция провайдера**: предусмотрена альтернативная ветка **Groq**,
  выбираемая значением `LLM_PROVIDER`, без изменения кода при переключении. Боевое использование
  Groq в этом релизе не требуется (опция «на будущее»).
- **ФТ-4.** Структурирование подключено к **обоим** входным каналам: результат OCR
  (`handlers/image.py`) и результат транскрипции (`handlers/audio.py`) перед отправкой пользователю
  проходят через `services/structure.py`.
- **ФТ-5.** Промпт структурирования ориентирован на rus+eng контент и инструктирует модель
  **сохранять смысл, не выдумывать факты и не добавлять содержание, отсутствующее в исходном
  тексте**; задача — оформление, а не генерация нового материала.
- **ФТ-6.** При недоступности, ошибке или таймауте LLM бот **возвращает сырой распознанный текст**
  и не завершается с ошибкой (фолбэк). Сбой структурирования не прерывает доставку результата
  распознавания пользователю.
- **ФТ-7.** Вызов LLM выполняется **неблокирующим** образом по отношению к event loop aiogram
  (асинхронный сетевой I/O либо вынос блокирующего вызова из loop) — бот остаётся отзывчивым для
  других сообщений во время HTTP-запроса к провайдеру.
- **ФТ-8.** Секреты провайдера читаются **только из окружения/`.env`**: `CF_ACCOUNT_ID`,
  `CF_API_TOKEN` (для Groq-ветки — `GROQ_API_KEY`). В `.env.example` и `deploy/.env.example`
  добавлены соответствующие плейсхолдеры. Секреты не хардкодятся и не коммитятся.
- **ФТ-9.** Если распознавание не дало текста (пустой результат), LLM **не вызывается**, и
  сохраняется прежнее поведение канала (сообщение о нераспознанном тексте).
- **ФТ-10.** `roadmap.md` обновлён: слот v0.5.0 переопределён на структурирование через Cloudflare
  Workers AI (дефолт) с Groq как опцией; отражена причина смены провайдера (гео-блок Groq по RU IP).

## 5. Нефункциональные требования

- **НФТ-1 (Надёжность / устойчивость к сбоям).** Зависимость от внешнего LLM-API не должна делать
  основную функцию (распознавание) хрупкой: любой сбой структурирования (сеть, 4xx/5xx, таймаут,
  превышение лимита, некорректный ответ) обрабатывается фолбэком на сырой текст (ФТ-6). Таймаут на
  обращение к провайдеру задаётся явно (конкретное значение — задача архитектора, см. ОВ-3).
- **НФТ-2 (Ресурсы / совместимость со средой).** Задача сетевая (I/O-bound), а не CPU-bound, и
  почти не добавляет нагрузки на CPU ivahub (1 vCPU, 1.9 GB RAM). Главное требование к ресурсам —
  **не блокировать event loop** на время HTTP-запроса (ФТ-7).
- **НФТ-3 (Стоимость / бесплатность).** Используется бесплатный tier Cloudflare Workers AI без
  привязки карты. Необходимо учитывать **суточные лимиты бесплатного tier** и **ограничение размера
  запроса**: очень длинные транскрипты при необходимости усекаются/разбиваются (стратегия и пороги —
  задача архитектора, см. ОВ-4). Превышение лимита трактуется как сбой → фолбэк (НФТ-1).
- **НФТ-4 (Сетевая доступность с ivahub).** Провайдер по умолчанию должен быть **достижим с ivahub
  напрямую, без прокси** (в отличие от Groq, который гео-блокирует RU IP). Это ключевая причина
  выбора Cloudflare. Реальная достижимость эндпоинта Cloudflare с ivahub проверяется при приёмке
  (см. ОВ-1).
- **НФТ-5 (Безопасность секретов).** `CF_ACCOUNT_ID`, `CF_API_TOKEN`, `GROQ_API_KEY` — только через
  `.env` (покрыт `.gitignore`); в репозиторий коммитятся лишь плейсхолдеры в `.env.example`/
  `deploy/.env.example`. Соответствует разделу «Секреты» `CLAUDE.md`.
- **НФТ-6 (Преемственность поставки).** Сохраняется профиль поставки «бот без HTTP-порта» (ADR-001):
  long-polling, без `ports:`/Traefik/`edge`, образ из GHCR, `restart: unless-stopped`, деплой на
  ivahub через CI/CD. Сохраняется чистая граница «хендлер ↔ сервис» (ADR-002/ADR-003). Релиз не
  ослабляет эти инварианты. Добавляется **исходящий** сетевой доступ к API Cloudflare (новое
  направление трафика наружу, помимо `api.telegram.org`).
- **НФТ-7 (Качество структурирования).** На типовом rus/eng распознанном тексте вывод читаемо
  структурирован (заголовки/списки/ключевые пункты, валидный Markdown) и **не искажает смысл и не
  добавляет вымышленных фактов** относительно исходного текста (ФТ-5). Способ проверки — см. ОВ-5.
- **НФТ-8 (Локализация интерфейса).** Системные сообщения бота — на русском. Контент структурируется
  на языке исходника (русский и/или английский), язык контента не переключается насильно.

## 6. Ограничения и допущения

Ограничения:
- **ОГР-1.** Структурирование зависит от **внешнего бесплатного LLM-API**; его доступность,
  суточные лимиты и размер запроса вне контроля проекта. Поэтому фолбэк на сырой текст обязателен
  (ФТ-6, НФТ-1), а длинный вход усекается/разбивается (НФТ-3).
- **ОГР-2.** Провайдер по умолчанию должен работать с **RU-IP ivahub без прокси**. Groq этому не
  удовлетворяет (гео-блок, CF 403) и потому остаётся лишь переключаемой опцией, не дефолтом.
- **ОГР-3.** Бот работает в Docker-контейнере; любые новые Python-зависимости (HTTP-клиент/SDK
  провайдера, если потребуются) должны быть в `requirements.txt` и образе. Конкретный способ
  обращения к Cloudflare (прямой HTTP vs SDK) — задача архитектора (ОВ-2).
- **ОГР-4.** Контракт распознавания (входы каналов, способ скачивания файлов через запиннённый
  `api.telegram.org`) и профиль поставки (ADR-001/ADR-002/ADR-003) не меняются; релиз не вводит
  входящих портов.
- **ОГР-5.** Секреты — только через `.env`; реальный `.env` не коммитится (НФТ-5).

Допущения:
- **ДОП-1.** Эндпоинт Cloudflare Workers AI достижим с ivahub напрямую (в отличие от Groq). Это
  допущение проверяется при приёмке деплоем на ivahub (ОВ-1); если оно не подтвердится, провайдер
  по умолчанию придётся пересмотреть.
- **ДОП-2.** Бесплатного tier Cloudflare (без карты) достаточно для ожидаемых объёмов обращений
  одного-двух пользователей; превышение лимита покрывается фолбэком (НФТ-1/НФТ-3).
- **ДОП-3.** Качество структурирования зависит от выбранной модели и промпта; абсолютная точность
  оформления не гарантируется — задаётся читаемая структура без искажения смысла (НФТ-7). На
  плохо распознанном (шумном) входе результат может быть ограниченным — это свойство входа, не
  дефект структурирования.
- **ДОП-4.** Образ обновляется и деплоится через тот же CI/CD-пайплайн на ivahub, что и
  v0.1.0–v0.4.0; добавляются лишь новые env-переменные провайдера.
- **ДОП-5.** Groq-ветка проверяется на уровне корректного переключения по `LLM_PROVIDER` (код-путь
  существует и выбирается конфигом); её боевая работоспособность с прода не требуется в этом релизе
  (Groq недоступен с ivahub).
- **ДОП-6.** Жёсткие пользовательские лимиты размера входа — вне рамок (v0.6.0); внутреннее
  усечение/разбивка длинного текста под лимит запроса провайдера (НФТ-3) не является пользовательским
  лимитом и не отвергает файл.

## 7. Критерии приёмки

- **КП-1.** Сервис `services/structure.py` принимает сырой распознанный текст и возвращает
  структурированный Markdown (заголовки/списки/ключевые пункты), полученный от LLM. (ФТ-1, ФТ-5)
- **КП-2.** Провайдер по умолчанию — Cloudflare Workers AI; при незаданном/дефолтном `LLM_PROVIDER`
  используется именно он. (ФТ-2)
- **КП-3.** Значение `LLM_PROVIDER` переключает реализацию между Cloudflare и Groq без изменения
  кода (абстракция провайдера присутствует и работает). (ФТ-3)
- **КП-4.** Структурирование применяется к результату **обоих** каналов: и аудио
  (`handlers/audio.py`), и изображений (`handlers/image.py`) — пользователь получает оформленный
  Markdown в обоих сценариях. (ФТ-4)
- **КП-5.** При недоступности/ошибке/таймауте LLM бот возвращает **сырой** распознанный текст и не
  падает; сбой структурирования не прерывает доставку результата распознавания. (ФТ-6, НФТ-1)
- **КП-6.** Вызов LLM не блокирует event loop: во время HTTP-запроса к провайдеру бот остаётся
  отзывчивым для других сообщений. (ФТ-7, НФТ-2)
- **КП-7.** Секреты провайдера читаются только из `.env`/окружения (`CF_ACCOUNT_ID`, `CF_API_TOKEN`,
  для Groq — `GROQ_API_KEY`); в `.env.example` и `deploy/.env.example` присутствуют плейсхолдеры;
  секреты не захардкожены и не закоммичены. (ФТ-8, НФТ-5)
- **КП-8.** На пустом результате распознавания LLM не вызывается; сохраняется прежнее сообщение о
  нераспознанном тексте. (ФТ-9)
- **КП-9.** Структурированный вывод на типовом rus/eng тексте читаемо оформлен (валидный Markdown)
  и не искажает смысл / не добавляет вымышленных фактов относительно исходника. (ФТ-5, НФТ-7)
- **КП-10.** Очень длинный распознанный текст обрабатывается без падения: укладывается в лимит
  запроса провайдера (усечение/разбивка) либо корректно уходит в фолбэк на сырой текст. (НФТ-3,
  ОГР-1)
- **КП-11.** `roadmap.md` обновлён: v0.5.0 — структурирование через Cloudflare Workers AI (дефолт),
  Groq — опция; причина смены провайдера отражена. (ФТ-10)
- **КП-12.** Профиль поставки сохранён: long-polling, без `ports:`/Traefik/`edge`, образ из GHCR,

## 8. Открытые вопросы

Блокирующих вопросов нет: требование детально специфицировано и непротиворечиво относительно
действующих ADR. Некритичные вопросы (уточняются при проектировании/реализации или подтверждаются
при приёмке, оформление документа не блокируют):

- **ОВ-1 (проверяется при приёмке, не блокирует BA).** Реальная достижимость эндпоинта Cloudflare
  Workers AI с ivahub без прокси (ДОП-1/НФТ-4/КП-13). По требованию заказчика Cloudflare выбран
  именно как работающий с RU-IP без прокси; подтверждается деплоем. Если не подтвердится — выбор
  дефолтного провайдера пересматривается отдельно.
- **ОВ-2 (архитектор, `/adr`).** Конкретная модель Cloudflare Workers AI (лёгкая мультиязычная —
  Llama 3.x / Qwen и её параметры), а также способ обращения к API (прямой HTTP-клиент vs SDK) и
  выбор асинхронного механизма неблокирования (async HTTP-клиент vs вынос вызова из loop) — задача
  архитектора, не заказчика.
- **ОВ-3 (архитектор, `/adr`).** Конкретное значение таймаута на обращение к LLM и политика
  ретраев (есть/нет, сколько) — задача архитектора (НФТ-1); по умолчанию — единичная попытка с
  явным таймаутом и фолбэком, без агрессивных ретраев.
- **ОВ-4 (архитектор, `/adr`).** Стратегия работы с очень длинным входом под лимит размера запроса
  провайдера (порог усечения vs разбивка на части и склейка результата) и учёт суточных лимитов
  бесплатного tier (НФТ-3/КП-10). По умолчанию допустимо усечение с фолбэком; разбивку выбирает
  архитектор, если она оправдана.
- **ОВ-5 (заказчик/архитектор, не блокирует).** Способ демонстрации качества структурирования
  (НФТ-7/КП-9): достаточно качественной проверки на репрезентативных rus/eng примерах или нужен
  фиксированный набор эталонов. По умолчанию — качественная проверка, формальные метрики не
  требуются.
- **ОВ-6 (наследуется, вне рамок).** Индикация прогресса, понятные сообщения о таймаутах/ошибках
  для пользователя и пользовательские лимиты размера входа остаются за v0.6.0; данный релиз даёт
  только «тихий» фолбэк на сырой текст.

Противоречий действующим ADR (ADR-001 — профиль поставки, ADR-002 — неблокирующий сервисный слой,
ADR-003/ADR-004 — паттерны распознавания) нет. Релиз вводит **новое направление исходящего сетевого
трафика** (к API Cloudflare, помимо `api.telegram.org`) и **первый I/O-bound (сетевой) сервис** —
в отличие от прежних CPU-bound сервисов на `asyncio.to_thread`; конкретный неблокирующий механизм
для сетевого вызова — предмет нового ADR (ОВ-2), а не противоречие существующим решениям.
