# BA-требования 006: Учёт пользователей и команда /stats для администратора (v0.6.0)

- **Источник:** `requirements/req-textify-006.md`
- **Дата:** 2026-06-22
- **Статус:** черновик

## 1. Контекст и бизнес-цель

Textify — Telegram-бот, бесплатно превращающий голос/аудио и изображения в чистый
структурированный текст (см. `README.md`, `CLAUDE.md`). Функциональность развивалась поэтапно:
каркас (v0.1.0), OCR (v0.2.0), аудио-транскрипция (v0.3.0), улучшение OCR (v0.4.0),
структурирование текста через Cloudflare Workers AI (v0.5.0). Все релизы в проде на ivahub
(Telegram-бот на long-polling, Docker, образ из GHCR; профиль поставки «бот без HTTP-порта» —
ADR-001).

До настоящего момента бот **stateless**: он ничего не хранит между обработками сообщений
(ADR-001 явно фиксировал «боту состояние не нужно»). Из-за этого нельзя ответить на простой
вопрос владельца: **сколько у бота уникальных пользователей и как они им пользуются**. Telegram
**не отдаёт** «число подписчиков» бота через API, поэтому единственный способ получить эту
метрику — **считать обращения самостоятельно**.

Цель релиза v0.6.0 — дать владельцу видимость аудитории и характера использования бота:
фиксировать факт обращения каждого пользователя и агрегаты по типам обработанного контента,
а по запросу администратора выводить сводку командой **`/stats`**. Учёт начинается **с момента
внедрения** (исторических данных нет — бот их не хранил). Сбор организуется так, чтобы **не
трогать логику существующих хендлеров** и **не нарушать приватность** (храним только `user_id`
и агрегаты, без содержимого сообщений).

Стейкхолдеры:
- **Владелец продукта / администратор** (заказчик, Telegram id `33030141`, @Alb0203) — хочет
  видеть размер аудитории и разбивку использования; единственный получатель `/stats`.
- **Конечный пользователь Telegram** — для него поведение бота не меняется; учёт прозрачен и не
  затрагивает приватность (содержимое сообщений не сохраняется).
- **Эксплуатирующий инженер / владелец** — обеспечивает персистентность данных через volume на
  ivahub, чтобы статистика переживала передеплой.
- **Команда разработки** (последующие агенты) — получает новый слой учёта (middleware aiogram +
  персистентный сервис на SQLite) и новую админ-команду.

## 2. Область (scope)

### В рамках задачи

- **Учёт обращений через middleware aiogram.** При каждом входящем сообщении один раз (независимо
  от того, какой хендлер его обработает) фиксируется факт обращения пользователя. Middleware не
  изменяет логику существующих хендлеров (OCR, аудио, текст, команды) и не вмешивается в их вывод.
- **Хранимые данные по пользователю:** `user_id` (уникальный идентификатор Telegram), время
  **первого** обращения, время **последнего** обращения, **счётчики по типам обработанного
  контента**: фото/изображение, аудио/голос, текст, команды.
- **Персистентное хранилище — SQLite-файл в примонтированном volume**, чтобы данные переживали
  передеплой (контейнер пересоздаётся при каждом релизе). Путь к БД — внутри каталога volume
  (`/app/data`), volume монтируется с хоста (`/opt/apps/textify/data:/app/data`).
- **Неблокирующая запись:** обращения к SQLite (блокирующие) выносятся из event loop aiogram
  (через `asyncio.to_thread` или аналог) — по тому же принципу, что уже принят для CPU-bound
  операций (ADR-002/ADR-003).
- **Команда `/stats` — только для администратора.** Выводит: число уникальных пользователей,
  суммарное число обработанных сообщений, разбивку по типам (фото/изображение, аудио/голос, текст,
  команды), дату первого и дату последнего обращения (по всей аудитории).
- **Авторизация администратора через переменную окружения `ADMIN_USER_ID`** (в `.env`, не
  хардкодить). Для не-администраторов `/stats` недоступна (молчит либо отвечает «команда
  недоступна» — формулировка уточняется, см. ОВ-2). Текущий админ — Telegram id `33030141`.
- **Правка `deploy/docker-compose.yml`:** добавление `volumes:` для каталога данных; добавление
  плейсхолдера `ADMIN_USER_ID` в `.env.example` и `deploy/.env.example`.
- **Эксплуатационное условие деплоя:** на сервере создаётся каталог данных
  (`/opt/apps/textify/data`) с правом записи для непривилегированного пользователя контейнера
  (`appuser`), так как образ запускается не от root.
- **Приватность:** хранятся только `user_id` и агрегаты; **содержимое сообщений и распознанный
  текст не сохраняются**.
- **Обновление `roadmap.md`:** слот v0.6.0 переопределяется на «Учёт пользователей и /stats»;
  релиз «UX и устойчивость» сдвигается на **v0.7.0**.

### Вне рамок (explicitly out of scope)

- **Веб-дашборд, экспорт данных, графики** — не делаем в этом релизе.
- **Хранение содержимого сообщений / распознанного текста / любых персональных данных** сверх
  `user_id` и агрегатов.
- **Ограничения скорости / квоты на пользователя** (rate limiting) — это релиз «UX и устойчивость»
  (v0.7.0).
- **Индикация прогресса, понятные сообщения об ошибках/таймаутах** — также v0.7.0.
- **Статистика в разрезе отдельного пользователя, временные срезы (за день/неделю), графики
  активности** — сводка `/stats` агрегатная по всей аудитории; детализация по периодам/персонам не
  входит в релиз.
- **Несколько администраторов / роли** — поддерживается один `ADMIN_USER_ID` (см. ОВ-1).
- **Изменение профиля поставки в части long-polling / отсутствия HTTP-порта** (ADR-001): бот
  остаётся без входящего порта. Профиль расширяется только добавлением персистентного volume.

## 3. Действующие лица и сценарии использования

**Администратор (владелец).** Цель — видеть аудиторию и характер использования бота.
- Сценарий A (просмотр статистики): администратор отправляет боту `/stats` → бот возвращает
  сводку: число уникальных пользователей, суммарно обработанных сообщений, разбивка по типам
  (фото/изображение, аудио/голос, текст, команды), дата первого и дата последнего обращения.

**Конечный пользователь Telegram (не администратор).** Цель — пользоваться ботом как обычно;
поведение для него не меняется.
- Сценарий B (обычное обращение): пользователь отправляет фото/аудио/текст/команду → бот
  обрабатывает как прежде; параллельно (прозрачно для пользователя) middleware фиксирует факт
  обращения и инкрементирует соответствующий счётчик типа.
- Сценарий C (первое обращение): пользователь обращается впервые → создаётся запись с его
  `user_id`, временем первого обращения; последующие обращения обновляют время последнего
  обращения и счётчики.
- Сценарий D (попытка вызвать `/stats` не-администратором): не-администратор отправляет `/stats`
  → команда недоступна (бот молчит или отвечает «команда недоступна»); статистика не раскрывается.

**Эксплуатирующий инженер / владелец.** Цель — обеспечить сохранность статистики между релизами.
- Сценарий E (деплой с персистентностью): инженер создаёт на ivahub каталог данных с правом записи
  для `appuser`, деплоит образ с volume → данные учёта переживают пересоздание контейнера при
  следующих релизах.
- Сценарий F (передеплой): выходит новый релиз, контейнер пересоздаётся → накопленная статистика
  сохраняется (читается из той же SQLite в volume), `/stats` показывает непрерывные данные.

## 4. Функциональные требования

- **ФТ-1.** При каждом входящем сообщении бот **один раз** (через middleware aiogram, независимо
  от хендлера) фиксирует факт обращения пользователя. Middleware **не изменяет** логику и вывод
  существующих хендлеров.
- **ФТ-2.** Для каждого уникального `user_id` хранятся: идентификатор пользователя, **время
  первого** обращения и **время последнего** обращения. Первое обращение фиксируется один раз
  (при создании записи), последнее — обновляется при каждом обращении.
- **ФТ-3.** Для каждого пользователя ведутся **счётчики по типам обработанного контента**:
  фото/изображение, аудио/голос, текст, команды. При обращении инкрементируется счётчик,
  соответствующий типу входящего сообщения.
- **ФТ-4.** Данные учёта сохраняются в **персистентном хранилище SQLite**, расположенном в
  каталоге примонтированного volume, и **переживают пересоздание контейнера** при передеплое.
- **ФТ-5.** Запись/чтение данных учёта выполняются **неблокирующим** образом по отношению к event
  loop aiogram (блокирующие вызовы SQLite выносятся из loop) — бот остаётся отзывчивым для других
  сообщений во время операций с БД.
- **ФТ-6.** Бот предоставляет команду **`/stats`**, которая выводит: число **уникальных
  пользователей**, **суммарное** число обработанных сообщений, **разбивку по типам**
  (фото/изображение, аудио/голос, текст, команды), **дату первого** и **дату последнего**
  обращения по всей аудитории.
- **ФТ-7.** `/stats` доступна **только администратору**, определяемому переменной окружения
  `ADMIN_USER_ID`. Для пользователя, чей `user_id` не равен `ADMIN_USER_ID`, команда недоступна
  (бот не раскрывает статистику; конкретное поведение — молчание или ответ «команда недоступна» —
  см. ОВ-2).
- **ФТ-8.** Идентификатор администратора читается **только из окружения/`.env`** (`ADMIN_USER_ID`)
  и **не хардкодится**. В `.env.example` и `deploy/.env.example` добавлен плейсхолдер
  `ADMIN_USER_ID`.
- **ФТ-9.** Сохраняются **только** `user_id` и агрегаты (время первого/последнего обращения,
  счётчики по типам). **Содержимое сообщений и распознанный текст не сохраняются** ни в каком виде.
- **ФТ-10.** `deploy/docker-compose.yml` содержит `volumes:` для каталога данных
  (`/opt/apps/textify/data:/app/data`), а путь к файлу БД указывает внутрь смонтированного каталога.
- **ФТ-11.** `roadmap.md` обновлён: слот v0.6.0 переопределён на «Учёт пользователей и команда
  /stats»; релиз «UX и устойчивость» перенесён на v0.7.0.

## 5. Нефункциональные требования

- **НФТ-1 (Приватность / минимизация данных).** Хранится минимально необходимый набор: `user_id`
  и агрегаты. Содержимое сообщений, распознанный текст, имена/контакты и иные персональные данные
  сверх `user_id` **не собираются и не хранятся** (ФТ-9). Это соответствует принципу приватности
  из требования и духу `CLAUDE.md`.
- **НФТ-2 (Производительность / отзывчивость).** Учёт не должен снижать отзывчивость бота:
  операции с SQLite выполняются вне event loop (ФТ-5). Накладные расходы на учёт пренебрежимы для
  ожидаемых объёмов (один-два активных пользователя) и не нагружают ivahub (1 vCPU, 1.9 GB RAM).
- **НФТ-3 (Надёжность / устойчивость учёта).** Сбой записи в хранилище **не должен ломать основную
  функцию** бота (распознавание/ответ пользователю): обработка сообщения завершается штатно даже
  при ошибке учёта (политика обработки сбоя записи — «тихая» деградация без падения; детали — см.
  ОВ-3). Учёт — вспомогательная функция, не критичный путь доставки результата пользователю.
- **НФТ-4 (Персистентность / сохранность данных).** Данные учёта обязаны сохраняться между
  передеплоями за счёт размещения SQLite в примонтированном volume (ФТ-4). Потеря накопленной
  статистики при штатном релизе недопустима.
- **НФТ-5 (Совместимость со средой исполнения).** Образ запускается **не от root**
  (непривилегированный `appuser`); каталог данных на хосте должен иметь права на запись для этого
  пользователя. Без корректных прав запись в БД невозможна — это эксплуатационное предусловие
  деплоя (см. КП-9, ОВ-4).
- **НФТ-6 (Безопасность доступа).** Статистика — чувствительная для владельца информация; доступ к
  `/stats` ограничен единственным администратором по `ADMIN_USER_ID` (ФТ-7). Идентификатор
  администратора — секрет конфигурации в `.env`, не в коде (ФТ-8, НФТ-7).
- **НФТ-7 (Управление конфигурацией секретов).** `ADMIN_USER_ID` задаётся только через
  `.env`/окружение; в репозиторий коммитятся лишь плейсхолдеры в `.env.example`/
  `deploy/.env.example`. Соответствует разделу «Секреты» `CLAUDE.md`.
- **НФТ-8 (Преемственность поставки).** Сохраняется профиль поставки «бот без HTTP-порта»
  (ADR-001): long-polling, без `ports:`/Traefik/`edge`, образ из GHCR, `restart: unless-stopped`,
  деплой на ivahub через CI/CD. Релиз **расширяет** профиль ровно одним элементом — **персистентным
  volume для данных** (ADR-001 ранее фиксировал «боту состояние не нужно»; теперь состояние
  появляется — это требует фиксации в новом ADR, см. ОВ-5).
- **НФТ-9 (Локализация интерфейса).** Сообщения бота (вывод `/stats`, при необходимости — ответ о
  недоступности команды) — на русском, в соответствии с принятыми соглашениями.

## 6. Ограничения и допущения

Ограничения:
- **ОГР-1.** Telegram **не предоставляет** число подписчиков/пользователей бота через API; метрика
  аудитории строится исключительно на собственном учёте обращений. Поэтому статистика отражает
  только пользователей, обратившихся **после** внедрения учёта.
- **ОГР-2.** Учёт ведётся **с момента внедрения**: исторических данных нет (бот был stateless),
  ретроспективно посчитать прошлые обращения невозможно.
- **ОГР-3.** Запись в SQLite — **блокирующая** операция; она обязана выполняться вне event loop
  (ФТ-5), как уже принято для других блокирующих операций (ADR-002/ADR-003).
- **ОГР-4.** Образ запускается **не от root** (`appuser`); персистентность работает только при
  корректных правах на каталог данных на хосте (НФТ-5).
- **ОГР-5.** Профиль поставки (ADR-001) сохраняется: бот остаётся long-polling без входящего
  HTTP-порта; релиз добавляет только volume для данных, не вводит портов/Traefik/edge.
- **ОГР-6.** `ADMIN_USER_ID` — только через `.env`; реальный `.env` не коммитится (НФТ-7).

Допущения:
- **ДОП-1.** Один администратор достаточно для текущих нужд; `ADMIN_USER_ID` хранит один
  идентификатор. Поддержка нескольких администраторов — вне рамок (см. ОВ-1).
- **ДОП-2.** Тип обращения для счётчиков можно однозначно определить в middleware по типу
  входящего сообщения (фото/изображение, аудио/голос, текст, команда). Граничная классификация
  (например, документ-изображение, пересланное сообщение, неподдерживаемый тип) уточняется при
  проектировании (см. ОВ-3) и не блокирует BA.
- **ДОП-3.** Объёмы обращений невелики (один-два активных пользователя), поэтому SQLite в одном
  файле на volume — достаточное и адекватное хранилище; выбор конкретной схемы БД — задача
  архитектора (ОВ-6).
- **ДОП-4.** Каталог данных на ivahub (`/opt/apps/textify/data`) создаётся и настраивается на
  права записи для `appuser` в рамках деплоя; это эксплуатационный шаг, фиксируемый в чек-листе
  деплоя.
- **ДОП-5.** Образ обновляется и деплоится через тот же CI/CD-пайплайн на ivahub, что и
  v0.1.0–v0.5.0; добавляются лишь новая env-переменная (`ADMIN_USER_ID`) и монтирование volume.
- **ДОП-6.** Учёт через middleware покрывает все каналы (фото, аудио, текст, команды) единообразно,
  не требуя правки существующих хендлеров (ФТ-1).

## 7. Критерии приёмки

- **КП-1.** При обращении любого пользователя (фото/аудио/текст/команда) факт обращения
  фиксируется ровно один раз, без изменения поведения и вывода существующих хендлеров. (ФТ-1)
- **КП-2.** Для нового пользователя создаётся запись с его `user_id` и временем первого обращения;
  при повторных обращениях время первого обращения не меняется, а время последнего обращения
  обновляется. (ФТ-2)
- **КП-3.** Счётчики по типам (фото/изображение, аудио/голос, текст, команды) корректно
  инкрементируются в соответствии с типом входящего сообщения. (ФТ-3)
- **КП-4.** `/stats`, вызванная администратором, выводит: число уникальных пользователей,
  суммарное число обработанных сообщений, разбивку по типам, дату первого и дату последнего
  обращения. (ФТ-6)
- **КП-5.** `/stats`, вызванная не-администратором, не раскрывает статистику (бот молчит либо
  отвечает «команда недоступна»). (ФТ-7)
- **КП-6.** Администратор определяется значением `ADMIN_USER_ID` из окружения; идентификатор не
  захардкожен; в `.env.example` и `deploy/.env.example` присутствует плейсхолдер `ADMIN_USER_ID`.
  (ФТ-7, ФТ-8, НФТ-7)
- **КП-7.** В хранилище присутствуют только `user_id` и агрегаты; содержимое сообщений и
  распознанный текст нигде не сохраняются. (ФТ-9, НФТ-1)
- **КП-8.** Операции с БД не блокируют event loop: во время записи/чтения учёта бот остаётся
  отзывчивым для других сообщений. (ФТ-5, НФТ-2)
- **КП-9.** После передеплоя (пересоздания контейнера) накопленная статистика сохраняется и
  `/stats` показывает непрерывные данные — за счёт SQLite в примонтированном volume и корректных
  прав записи для `appuser`. (ФТ-4, ФТ-10, НФТ-4, НФТ-5)
- **КП-10.** `deploy/docker-compose.yml` содержит `volumes:` для каталога данных, путь к БД —
  внутри смонтированного каталога; профиль поставки сохранён (long-polling, без `ports:`/Traefik/
  `edge`, образ из GHCR, `restart: unless-stopped`). (ФТ-10, НФТ-8)
- **КП-11.** Сбой записи учёта не приводит к падению бота и не прерывает обработку/ответ
  пользователю. (НФТ-3)
- **КП-12.** `roadmap.md` обновлён: v0.6.0 — «Учёт пользователей и /stats»; «UX и устойчивость»
  перенесён на v0.7.0. (ФТ-11)

## 8. Открытые вопросы

Блокирующих вопросов нет: требование детально специфицировано. Ниже — некритичные вопросы;
уточняются при проектировании (`/adr`) или приёмке и оформление документа **не блокируют**.

- **ОВ-1 (заказчик, не блокирует).** Достаточно ли **одного** администратора (`ADMIN_USER_ID`),
  или в обозримом будущем понадобится список администраторов? По умолчанию — один администратор
  (текущий — id `33030141`); поддержка нескольких — отдельная история.
- **ОВ-2 (заказчик/UX, не блокирует).** Поведение `/stats` для не-администратора: **молчание** или
  явный ответ «команда недоступна»? Требование допускает оба. По умолчанию предлагается явный
  короткий ответ на русском (без раскрытия деталей).
- **ОВ-3 (архитектор, `/adr`).** (а) Правила классификации типа обращения в middleware для
  граничных случаев (документ-изображение vs фото, пересланные сообщения, неподдерживаемые/прочие
  типы, нужен ли отдельный счётчик «прочее»). (б) Политика обработки сбоя записи учёта (НФТ-3):
  «тихая» деградация без падения предполагается по умолчанию.
- **ОВ-4 (эксплуатация / архитектор, проверяется при деплое).** Конкретные команды/шаги создания
  каталога `/opt/apps/textify/data` на ivahub и выставления прав записи для `appuser` (UID/GID
  контейнерного пользователя) — фиксируются в спецификации релиза и чек-листе деплоя. Подтверждается
  при приёмке деплоем (КП-9).
- **ОВ-5 (архитектор, `/adr`) — расширение профиля поставки.** ADR-001 ранее зафиксировал, что
  **«боту состояние не нужно»** и среди отброшенных правил — общие stateful-сервисы. Данный релиз
  **вводит первое персистентное состояние** (SQLite в volume). Это не противоречие, а осознанное
  **расширение** профиля поставки «бот без HTTP-порта»: добавляется локальный файловый volume (без
  внешней БД/Redis, без портов). Расширение должно быть **явно зафиксировано новым ADR**, чтобы
  будущие агенты не трактовали volume как нарушение ADR-001.
- **ОВ-6 (архитектор, `/adr`).** Конкретная схема SQLite (таблицы/поля), формат хранения времени
  (UTC/таймзона) и представления времён в выводе `/stats`, а также механизм неблокирующего доступа
  (`asyncio.to_thread` vs async-драйвер) — задача архитектора, не заказчика. По умолчанию —
  `asyncio.to_thread` (преемственность с ADR-002/ADR-003), время в UTC.

Противоречий действующим ADR нет. Единственная точка, требующая явной фиксации, — расширение
профиля поставки персистентным volume (ОВ-5): формально ADR-001 говорил об отсутствии состояния,
поэтому новый ADR должен задокументировать появление локального состояния как осознанное и узкое
расширение (без портов, без внешних stateful-сервисов).
