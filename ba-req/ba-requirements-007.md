# BA-требования 007: Надёжная отдача длинного текстового результата (v0.7.0)

- **Источник:** `requirements/req-textify-007.md`
- **Дата:** 2026-06-23
- **Статус:** черновик

## 1. Контекст и бизнес-цель

Textify — Telegram-бот, бесплатно превращающий голос/аудио и изображения в чистый
структурированный текст (см. `README.md`, `CLAUDE.md`). Функциональность развивалась поэтапно:
каркас (v0.1.0), OCR (v0.2.0), аудио-транскрипция (v0.3.0), улучшение OCR (v0.4.0),
структурирование текста через LLM (v0.5.0/v0.5.1), учёт пользователей и `/stats` (v0.6.0). Все
релизы в проде на ivahub (Telegram-бот на long-polling, Docker, образ из GHCR; профиль поставки
«бот без HTTP-порта» — ADR-001).

В текущей реализации все пути вывода результата отдают ответ одним вызовом `message.answer(...)`
(см. `handlers/image.py`, `handlers/audio.py`, оба после `structure_text(...)`). Telegram
**жёстко ограничивает текстовое сообщение 4096 символами**: на длинных входах (развёрнутая речь,
плотный скан, объёмный структурированный результат) сообщение **отклоняется с ошибкой отправки
либо результат теряется**. Это латентный дефект надёжности, проявляющийся именно на самых ценных
длинных входах — то есть там, где польза от бота максимальна.

Цель релиза v0.7.0 (в части надёжной отдачи) — **гарантировать, что длинный результат доходит до
пользователя целиком**, через единую точку отправки текстового результата, применяемую во всех
путях вывода. Поведение по длине: короткий результат — одним сообщением; длиннее безопасного
лимита — несколькими сообщениями с разбивкой по естественным границам и небольшой задержкой между
частями; очень длинный — одним вложением `.txt`, чтобы не спамить десятком сообщений. При разбивке
не должна разрушаться разметка ответа.

Это «дешёвый фикс с высокой отдачей по надёжности»: новых внешних зависимостей не требуется,
профиль поставки не меняется.

Стейкхолдеры:
- **Конечный пользователь Telegram** (основной выгодоприобретатель) — получает результат целиком,
  без потерь и без ошибок отправки, в читаемом виде.
- **Владелец продукта** (заказчик) — снимает латентный дефект надёжности на ключевом длинном
  кейсе.
- **Команда разработки** (последующие агенты) — получает единый модуль отправки результата,
  переиспользуемый всеми хендлерами.

## 2. Область (scope)

### В рамках задачи

- **Единая точка отправки текстового результата.** Вводится один общий способ отправки
  распознанного/структурированного результата пользователю (например, отдельный модуль/функция
  отправки результата). Все пути вывода результата используют его **вместо прямого**
  `message.answer(result)`.
- **Применение во всех путях вывода результата:**
  - OCR-результат — `handlers/image.py` (фото и изображение-документ);
  - транскрипт аудио — `handlers/audio.py` (голос, аудио, аудио-документ);
  - результат структурирования — это значение, возвращаемое `services/structure.py`
    (`structure_text(...)`) и отдаваемое пользователю в тех же хендлерах; единая отправка
    применяется к **финальному ответу**, формируемому в хендлерах (там, где сейчас стоит
    `message.answer(await structure_text(text))`).
- **Поведение по длине результата (три режима):**
  - **Короткий** (≤ безопасного текстового лимита, ориентир ~4000 символов с запасом к лимиту
    Telegram 4096) — отправляется **одним сообщением**.
  - **Длинный** (больше безопасного лимита, но не превышает порог перехода на файл) — **разбивается
    на несколько сообщений по естественным границам** (абзацы/строки/предложения — не посреди
    слова) и отправляется по частям; между частями выдерживается **небольшая задержка**, чтобы не
    спровоцировать ограничение скорости (rate limit) Telegram.
  - **Очень длинный** (превышает порог перехода на файл) — отправляется **одним вложением `.txt`**
    из памяти (без записи на диск), вместо десятка сообщений.
- **Пороги — в константы.** Безопасный лимит длины сообщения и порог перехода на файл выносятся в
  именованные константы (конкретные значения и место хранения — за архитектором).
- **Сохранение разметки при разбивке.** Если ответ отдаётся с разметкой (Markdown), части режутся
  так, чтобы **не разрывать элементы разметки** (заголовки/списки/блоки). Конкретная стратегия
  (как именно резать без поломки разметки и/или переход на plain-text/файл для проблемных случаев)
  — решение архитектора.
- **Сохранение существующих коротких ответов и сообщений-заглушек.** Поведение для коротких
  результатов и для служебных сообщений (`NO_TEXT_MESSAGE`, `NO_SPEECH_MESSAGE`, ответы команд)
  не должно регрессировать.

### Вне рамок (explicitly out of scope)

- **Индикатор прогресса** («обрабатываю…») — отдельное UX-улучшение (бэклог).
- **Logging / структурированное логирование** — бэклог.
- **Изменение подписей к медиа** (caption, лимит 1024): в этом релизе результат отдаётся
  текстовыми сообщениями и файлом `.txt`, **подписи к медиа не используются и не меняются**.
- **Изменение логики распознавания/структурирования** (OCR, транскрипция, LLM-форматирование):
  релиз касается **только доставки** уже сформированного результата.
- **Новые внешние зависимости** — отправка файла выполняется штатными средствами aiogram (например,
  `BufferedInputFile`) из памяти; новых пакетов/сервисов не вводится.
- **Перенос транскрипции на Cloudflare** (идея #1), **summary/перевод** (идея #3) — отдельные
  релизы.
- **Rate limiting / квоты на пользователя** — отдельная история темы «UX и устойчивость».
- **Изменение профиля поставки** (long-polling, без портов/Traefik/edge — ADR-001): остаётся без
  изменений.

## 3. Действующие лица и сценарии использования

**Конечный пользователь Telegram.** Цель — получить распознанный/структурированный результат
целиком и в читаемом виде, независимо от его объёма.
- Сценарий A (короткий результат): пользователь отправляет короткое голосовое/фото/аудио →
  результат укладывается в безопасный лимит → бот отвечает **одним сообщением** (как сейчас).
- Сценарий B (длинный результат): пользователь отправляет длинную речь/плотный скан → результат
  превышает безопасный лимит, но не достигает порога файла → бот отправляет **несколько
  сообщений**, разбитых по естественным границам, с небольшой задержкой между ними; разметка не
  разрушена; ни одна часть не теряется.
- Сценарий C (очень длинный результат): результат превышает порог перехода на файл → бот
  отправляет **один файл `.txt`** с полным результатом вместо множества сообщений.
- Сценарий D (граница лимита): результат близок к лимиту Telegram (например, 4000–4096) → за счёт
  запаса безопасного лимита бот не упирается в ошибку отправки; результат доходит целиком.

**Команда разработки (последующие агенты).** Цель — иметь единый, переиспользуемый способ отдачи
результата.
- Сценарий E (переиспользование): новый путь вывода результата (текущий или будущий) использует
  единую точку отправки и автоматически получает корректное поведение по длине, без дублирования
  логики разбивки.

## 4. Функциональные требования

- **ФТ-1.** Вводится **единая точка отправки текстового результата** пользователю; все пути вывода
  распознанного/структурированного результата используют её **вместо прямого**
  `message.answer(result)`.
- **ФТ-2.** Единая отправка применяется во **всех** путях вывода результата: OCR
  (`handlers/image.py` — фото и изображение-документ), аудио (`handlers/audio.py` — голос, аудио,
  аудио-документ), а также к финальному ответу со структурированным результатом
  (`structure_text(...)`), отдаваемому в этих хендлерах.
- **ФТ-3.** Если длина результата **≤ безопасного лимита** (ориентир ~4000 символов, с запасом к
  лимиту Telegram 4096) — результат отправляется **одним сообщением**.
- **ФТ-4.** Если длина результата **превышает безопасный лимит**, но **не превышает порог перехода
  на файл** — результат **разбивается на несколько сообщений** и отправляется по частям. Каждая
  часть не превышает безопасный лимит.
- **ФТ-5.** Разбивка выполняется **по естественным границам** (абзацы/строки/предложения) — слово
  **не разрывается** посередине.
- **ФТ-6.** Между отправкой соседних частей выдерживается **небольшая задержка** для снижения риска
  срабатывания ограничения скорости (rate limit) Telegram.
- **ФТ-7.** Если длина результата **превышает порог перехода на файл** — результат отправляется
  **одним вложением `.txt`** (из памяти, без записи на диск), а не множеством сообщений.
- **ФТ-8.** Безопасный лимит длины сообщения и порог перехода на файл задаются **именованными
  константами** (конкретные значения и расположение — задача архитектора).
- **ФТ-9.** При разбивке результата с разметкой (Markdown) **элементы разметки не разрываются**
  между частями (заголовок/список/блок не рвутся посередине); для случаев, когда сохранить
  целостность разметки в рамках лимита невозможно, применяется fallback (переход на plain-text
  и/или на файл) — конкретная стратегия определяется архитектором (см. ОВ-1, ОВ-2).
- **ФТ-10.** Поведение для **коротких результатов** и для **служебных сообщений** (отсутствие
  распознанного текста/речи, ответы команд) **не регрессирует**: они отправляются как прежде —
  одним сообщением.
- **ФТ-11.** Полнота доставки: при любом из трёх режимов (одно сообщение / несколько сообщений /
  файл) пользователь получает результат **целиком**, без потери частей.

## 5. Нефункциональные требования

- **НФТ-1 (Надёжность доставки).** На любом допустимом результате — включая результаты длиной
  у самой границы лимита Telegram (4096) и многократно превышающие её — отправка **не должна
  завершаться ошибкой «message is too long»** и **не должна терять часть ответа**. Это основная
  цель релиза.
- **НФТ-2 (Совместимость с ограничениями Telegram).** Решение учитывает лимит текстового сообщения
  4096 символов и риск ограничения скорости при серии сообщений (ФТ-6). Подписи к медиа (1024) в
  этом релизе не используются.
- **НФТ-3 (Читаемость результата).** Длинный результат остаётся читаемым: разбивка по
  естественным границам (ФТ-5) и сохранение разметки (ФТ-9) не превращают ответ в визуальный
  «мусор» с разорванными словами/элементами форматирования.
- **НФТ-4 (Отзывчивость / неблокирование).** Отправка результата (включая паузы между частями и
  формирование файла) **не блокирует** event loop aiogram — бот остаётся отзывчивым для других
  пользователей. Преемственность с принятым принципом неблокирующих операций (ADR-002/ADR-003).
- **НФТ-5 (Отсутствие новых зависимостей и сохранение профиля поставки).** Не вводятся новые
  внешние зависимости; файл `.txt` формируется штатными средствами aiogram из памяти. Профиль
  поставки «бот без HTTP-порта» (long-polling, без `ports:`/Traefik/`edge`, образ из GHCR,
  `restart: unless-stopped`) сохраняется без изменений (ADR-001).
- **НФТ-6 (Локализация).** Любые служебные подписи, сопровождающие отправку файла (имя файла или
  короткое пояснение, если оно вообще нужно), оформляются в соответствии с принятым языком
  интерфейса (русский); необходимость такой подписи уточняется (см. ОВ-3).
- **НФТ-7 (Сопровождаемость).** Логика отдачи результата сосредоточена в **одном** модуле/функции
  (ФТ-1), исключая дублирование по хендлерам и облегчая повторное использование (Сценарий E).

## 6. Ограничения и допущения

Ограничения:
- **ОГР-1.** Telegram ограничивает текстовое сообщение **4096 символами**; превышение приводит к
  ошибке отправки. Это первичная причина дефекта и ключевое ограничение релиза.
- **ОГР-2.** Серия быстрых сообщений может упереться в ограничение скорости (rate limit) Telegram —
  отсюда требование небольшой задержки между частями (ФТ-6).
- **ОГР-3.** Профиль поставки (ADR-001) сохраняется: бот остаётся long-polling без входящего
  HTTP-порта; релиз не вводит портов/Traefik/edge.
- **ОГР-4.** Блокирующие операции должны выноситься из event loop (ADR-002/ADR-003); паузы между
  частями выполняются асинхронно, не блокируя loop (НФТ-4).
- **ОГР-5.** Новых внешних зависимостей быть не должно; отправка файла — штатными средствами
  aiogram из памяти (НФТ-5).

Допущения:
- **ДОП-1.** В текущем коде ответ отдаётся вызовом `message.answer(...)` **без `parse_mode`**,
  то есть Markdown сейчас фактически отправляется как обычный текст (Telegram его не парсит).
  Релиз касается **доставки** и не обязан вводить серверный парсинг Markdown; требование «не ломать
  разметку» трактуется как «не разрывать визуальную структуру результата при разбивке». Вопрос о
  том, должен ли результат отдаваться с включённым `parse_mode`, относится к проектированию
  (см. ОВ-1) и BA не блокирует.
- **ДОП-2.** Конкретные значения безопасного лимита (~4000) и порога перехода на файл, а также
  единица разбивки (абзац/строка/предложение) и её приоритет — параметры реализации; задаются
  архитектором (см. ОВ-2). Требование фиксирует лишь, что пороги вынесены в константы (ФТ-8).
- **ДОП-3.** Ожидаемые объёмы невелики (один-два активных пользователя на ivahub, 1 vCPU /
  1.9 GB RAM), поэтому накладные расходы на разбивку/паузы/формирование файла пренебрежимы.
- **ДОП-4.** Имя файла-вложения и наличие сопроводительного пояснения — деталь UX (ОВ-3); по
  умолчанию предполагается осмысленное имя `.txt` без обязательного дополнительного сообщения.

## 7. Критерии приёмки

- **КП-1.** Все пути вывода результата (OCR в `handlers/image.py`; аудио в `handlers/audio.py`;
  структурированный ответ через `structure_text(...)`) отправляют результат через **единую точку
  отправки**, а не прямым `message.answer(result)`. (ФТ-1, ФТ-2)
- **КП-2.** Результат длиной ≤ безопасного лимита отправляется **одним сообщением** (поведение не
  изменилось). (ФТ-3)
- **КП-3.** Результат длиннее безопасного лимита, но в пределах порога файла, отправляется
  **несколькими сообщениями**, каждое из которых не превышает безопасный лимит; ни одна часть не
  потеряна. (ФТ-4, ФТ-11)
- **КП-4.** Разбивка не разрывает слова посередине — части режутся по естественным границам.
  (ФТ-5)
- **КП-5.** Между отправкой соседних частей присутствует небольшая задержка. (ФТ-6)
- **КП-6.** Результат длиннее порога перехода на файл отправляется **одним вложением `.txt`** с
  полным содержимым, без записи на диск; множество сообщений в этом случае не отправляется.
  (ФТ-7, ФТ-11)
- **КП-7.** Безопасный лимит и порог перехода на файл заданы именованными константами (не
  «магическими числами», разбросанными по коду). (ФТ-8)
- **КП-8.** Результат с разметкой при разбивке не имеет разорванных посередине элементов
  форматирования; для проблемных случаев применяется согласованный fallback (plain-text/файл).
  (ФТ-9, НФТ-3)
- **КП-9.** Служебные сообщения и короткие ответы ведут себя как прежде (отсутствие
  распознавания текста/речи, ответы команд — одним сообщением). (ФТ-10)
- **КП-10.** На результате длиной у границы 4096 и кратно больше отправка не падает с ошибкой
  «message is too long» и доставляет результат целиком. (НФТ-1, ФТ-11)
- **КП-11.** Отправка длинного результата (паузы между частями, формирование файла) не блокирует
  обработку сообщений других пользователей. (НФТ-4)
- **КП-12.** Новые внешние зависимости не добавлены; профиль поставки (long-polling, без
  `ports:`/Traefik/`edge`, образ из GHCR, `restart: unless-stopped`) сохранён. (НФТ-5)

## 8. Открытые вопросы

Блокирующих вопросов нет: требование детально специфицировано, а граничные решения (значения
порогов, единица и стратегия разбивки, использование `parse_mode`) **явно делегированы
архитектору** самим требованием («Решение — за архитектором»). Ниже — некритичные вопросы;
уточняются при проектировании (`/adr`) или приёмке и оформление документа **не блокируют**.

- **ОВ-1 (архитектор, `/adr`).** Использование `parse_mode` (Markdown) при отправке результата.
  Сейчас ответ уходит без `parse_mode` (Markdown виден как сырой текст). Нужно решить: включать ли
  серверный парсинг Markdown в этом релизе или сохранить текущее поведение, и как это согласуется с
  требованием «не ломать разметку при разбивке». По умолчанию релиз трактует разметку как
  визуальную структуру текста; включение `parse_mode` BA не требует и не запрещает.
- **ОВ-2 (архитектор, `/adr`).** Конкретные значения безопасного лимита (~4000) и порога перехода
  на файл (в скольких «частях» начинать слать файл), приоритет единиц разбивки
  (абзац → строка → предложение → жёсткий срез по лимиту) и величина задержки между частями.
- **ОВ-3 (UX/заказчик, не блокирует).** Нужно ли сопровождать файл `.txt` коротким пояснением
  («результат целиком во вложении») и каким должно быть имя файла. По умолчанию — осмысленное имя
  без обязательного дополнительного сообщения.
- **ОВ-4 (архитектор, `/adr`).** Поведение при частичном сбое отправки серии сообщений (например,
  ошибка на 3-й из 5 частей): прекращать, ретраить или продолжать. По умолчанию ожидается
  устойчивое поведение без падения бота; детализация — за архитектором.

Противоречий действующим ADR нет. Релиз сохраняет профиль поставки ADR-001 (без HTTP-порта) и
принцип неблокирующих операций ADR-002/ADR-003 (паузы/формирование файла не блокируют event loop).
