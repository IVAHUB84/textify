# BA-требования 008: Транскрипция аудио через Cloudflare Workers AI whisper-large-v3-turbo с переключателем провайдера и фолбэком на локальную модель (v0.8.0)

- **Источник:** `requirements/req-textify-008.md`
- **Дата:** 2026-06-23
- **Статус:** черновик

## 1. Контекст и бизнес-цель

Textify — Telegram-бот, который бесплатно превращает голосовые/аудио и изображения в чистый
структурированный текст (см. `README.md`, `CLAUDE.md`). Продукт развивался поэтапно: каркас
(v0.1.0), OCR изображений (v0.2.0), аудио-транскрипция через локальный faster-whisper (v0.3.0),
улучшение OCR (v0.4.0), структурирование через Cloudflare Workers AI (v0.5.0), учёт пользователей
(v0.6.0), надёжная отдача длинного результата (v0.7.0). Релизы v0.1.0–v0.7.0 в проде на ivahub.

Сейчас транскрипция аудио выполняется **локальной** моделью faster-whisper `base`
(`compute_type=int8`, CPU) на сервере ivahub с 1 vCPU и ~1.9 GB RAM (ADR-003). Это **главное узкое
место продукта**: распознавание медленное (десятки секунд на 1 vCPU), качество модели `base`
ограниченное, а тяжёлый CPU-bound вызов нагружает единственное ядро и память сервера, конкурируя с
остальной работой бота.

Параллельно в продукт уже интегрирована инфраструктура **Cloudflare Workers AI** — она используется
для структурирования текста (ADR-005), работает с RU-IP ivahub без прокси (в отличие от
гео-блокирующего Groq), использует тот же аккаунт и токен (`CF_ACCOUNT_ID` / `CF_API_TOKEN`) и
вызывается прямым async HTTP через `httpx` (уже зависимость проекта). Эндпоинт транскрипции
Cloudflare Whisper **подтверждён вживую с ivahub** (HTTP 200).

Цель релиза v0.8.0 — **перенести основную транскрипцию аудио с локальной модели на Cloudflare
Workers AI `whisper-large-v3-turbo`**: модель заметно точнее и мультиязычнее, ответ быстрее, а
нагрузка на CPU/RAM сервера снимается. При этом локальная модель **остаётся в образе как фолбэк** —
при любом сбое Cloudflare (сеть, лимит тарифа, ошибка ответа) бот незаметно для пользователя
распознаёт аудио локально, и сбой не ломает ответ. Провайдер транскрипции делается переключаемым
конфигурацией (`ASR_PROVIDER`), по умолчанию — Cloudflare.

Стейкхолдеры:
- **Владелец продукта** (заказчик) — хочет повысить качество и скорость распознавания речи и
  разгрузить CPU/RAM ivahub, переиспользуя уже оплаченную (бесплатную) и проверенную инфраструктуру
  Cloudflare без новых зависимостей.
- **Конечный пользователь Telegram** — присылает голос/аудио и получает более точную и быструю
  транскрипцию; при сбое облачного провайдера всё равно получает результат (через локальный фолбэк),
  без ошибки.
- **Команда разработки** (последующие агенты) — получает в `services/transcribe.py` абстракцию
  провайдера транскрипции (Cloudflare-дефолт + локальный faster-whisper) с фолбэком, по образцу
  абстракции провайдера структурирования (ADR-005).

## 2. Область (scope)

### В рамках задачи

- **Провайдер транскрипции Cloudflare Workers AI** в `services/transcribe.py`: распознавание аудио
  через Cloudflare Whisper по подтверждённому контракту вызова:
  - `POST https://api.cloudflare.com/client/v4/accounts/{CF_ACCOUNT_ID}/ai/run/{модель}`;
  - заголовок `Authorization: Bearer {CF_API_TOKEN}`;
  - тело JSON `{"audio": "<base64 аудио>"}`;
  - ответ: распознанный текст в `result.text`, определённый язык в
    `result.transcription_info.language`.
- **Модель по умолчанию** — `@cf/openai/whisper-large-v3-turbo`, вынесена в переменную окружения
  `CF_WHISPER_MODEL` (смена модели без правки кода).
- **Неблокирующий async-вызов** через `httpx` (уже зависимость, используется в
  `services/structure.py`) с разумным явным таймаутом — обращение к Cloudflare не блокирует event
  loop aiogram.
- **Переключатель провайдера `ASR_PROVIDER`** со значениями `cloudflare` | `local`, **по умолчанию
  `cloudflare`**. Смена провайдера — конфигурацией, без правки кода.
- **Фолбэк на локальный faster-whisper** при любом сбое Cloudflare-провайдера (сетевой сбой, лимит
  тарифа, ошибка/некорректный ответ, таймаут): локальная модель остаётся в образе как страховка,
  распознавание выполняется локально, ответ пользователю не ломается.
- **Сохранение локального провайдера** как самостоятельного режима (`ASR_PROVIDER=local`) и как
  фолбэка — локальная модель `base` и её доставка в образ (ADR-003) **не удаляются**.
- **Поведение для крупных аудио**: base64 раздувает объём запроса, а Cloudflare AI run имеет
  ограничение размера запроса. Релиз обязан предусмотреть корректное поведение для крупных файлов
  (фолбэк на локальную модель и/или понятное поведение/сообщение); сбой по размеру не должен ломать
  ответ. Конкретная стратегия (чанкинг vs фолбэк/сообщение) и **порог размера** — задача архитектора
  (см. ОВ-1); по умолчанию допустим фолбэк на локальную модель для слишком крупных файлов.
- **Переиспользование секретов Cloudflare**: `CF_ACCOUNT_ID` и `CF_API_TOKEN` уже есть в `.env` на
  сервере и в `deploy/.env.example` (введены в ADR-005). Новые переменные `ASR_PROVIDER` и
  `CF_WHISPER_MODEL` добавляются плейсхолдерами в `.env.example` (и, для единообразия,
  `deploy/.env.example`).
- **Сохранение существующего контракта канала аудио**: способ приёма аудио
  (`message.voice` / `message.audio` / audio-документы), скачивание штатным клиентом aiogram через
  запиннённый `api.telegram.org`, автоопределение языка (ru/en), последующее структурирование
  результата (ADR-005) и надёжная отдача длинного результата (ADR-007) — не меняются. Меняется
  только **способ получения** транскрипта (облако vs локально).

### Вне рамок (explicitly out of scope)

- **Summary / перевод транскрипта** (идея #3) — отдельный релиз v0.9.0.
- **Удаление локальной модели faster-whisper из образа** — модель остаётся как фолбэк/режим `local`.
- **Диаризация** (разделение по спикерам) и **таймкоды** в ответе пользователю.
- **Изменение логики структурирования** (ADR-005), **учёта пользователей** (v0.6.0) и **отдачи
  длинного результата** (ADR-007) — их контракты сохраняются; меняется только источник транскрипта.
- **Изменение профиля поставки** (ADR-001): бот остаётся long-polling без HTTP-порта; новых
  входящих портов нет.
- **Боевая работа провайдера структурирования** и его переменные (`LLM_PROVIDER`, `CF_MODEL`,
  `GROQ_API_KEY`) — данный релиз их не трогает; `CF_WHISPER_MODEL` — отдельная переменная для ASR,
  не пересекающаяся с `CF_MODEL` структурирования.

## 3. Действующие лица и сценарии использования

**Пользователь Telegram.** Цель — получить точную и быструю текстовую расшифровку сказанного.
- Сценарий A (облачная транскрипция, дефолт): пользователь отправляет голосовое/аудио/аудио-документ
  → бот скачивает файл, отправляет его в Cloudflare Whisper и возвращает распознанный текст (далее —
  по существующему пути: структурирование и отдача результата). Для пользователя поведение прежнее,
  но качество/скорость выше.
- Сценарий B (фолбэк на локальную модель): провайдер по умолчанию (Cloudflare) недоступен/вернул
  ошибку/превысил таймаут/файл слишком крупный → бот распознаёт аудио **локальной** faster-whisper и
  возвращает результат **без ошибки**. Пользователь незаметно для себя получает транскрипт.
- Сценарий C (явный локальный режим): при `ASR_PROVIDER=local` бот распознаёт аудио только локальной
  моделью (как до v0.8.0), не обращаясь к Cloudflare.
- Сценарий D (пустой результат распознавания): речь не распознана (тишина/шум, пустой результат) →
  сохраняется прежнее поведение канала (сообщение о нераспознанной речи); поведение не зависит от
  выбранного провайдера.

**Эксплуатирующий инженер / владелец.** Цель — развернуть облачную транскрипцию на ivahub,
переиспользуя имеющиеся секреты Cloudflare.
- Сценарий E (деплой и конфигурация): владелец деплоит обновлённый образ; `CF_ACCOUNT_ID` /
  `CF_API_TOKEN` уже в `.env`; `ASR_PROVIDER` не задан → используется дефолт `cloudflare`,
  `CF_WHISPER_MODEL` не задан → используется `@cf/openai/whisper-large-v3-turbo`. Бот транскрибирует
  через Cloudflare; при недоступности облака — локальный фолбэк.
- Сценарий F (переключение провайдера): владелец задаёт `ASR_PROVIDER=local` (или меняет
  `CF_WHISPER_MODEL`) в `.env` — поведение меняется конфигурацией, без правки кода.

## 4. Функциональные требования

- **ФТ-1.** В `services/transcribe.py` добавлен провайдер транскрипции **Cloudflare Workers AI**,
  выполняющий распознавание аудио по подтверждённому контракту:
  `POST .../accounts/{CF_ACCOUNT_ID}/ai/run/{модель}`, заголовок `Authorization: Bearer
  {CF_API_TOKEN}`, тело `{"audio": "<base64 аудио>"}`, ответ — текст из `result.text` (язык — из
  `result.transcription_info.language`).
- **ФТ-2.** Модель Cloudflare Whisper по умолчанию — `@cf/openai/whisper-large-v3-turbo`,
  переопределяется переменной окружения `CF_WHISPER_MODEL` без изменения кода.
- **ФТ-3.** Обращение к Cloudflare выполняется **неблокирующим** async-способом через `httpx`
  (существующая зависимость) с явным таймаутом; бот остаётся отзывчивым для других сообщений во
  время сетевого запроса.
- **ФТ-4.** Провайдер транскрипции выбирается переменной окружения **`ASR_PROVIDER`** со значениями
  `cloudflare` | `local`; при незаданном/дефолтном значении используется **`cloudflare`**.
  Переключение выполняется конфигурацией, без правки кода.
- **ФТ-5.** При **любом** сбое Cloudflare-провайдера (сетевой сбой, недоступность, 4xx/5xx, таймаут,
  превышение лимита тарифа, некорректный/пустой по причине ошибки ответ, превышение допустимого
  размера запроса) бот выполняет **фолбэк на локальный faster-whisper** и возвращает результат
  распознавания; сбой облака **не ломает ответ** пользователю и не валит процесс. Сбой логируется.
- **ФТ-6.** Локальный провайдер faster-whisper (`base`, `int8`, CPU; ADR-003) **сохраняется** — как
  фолбэк (ФТ-5) и как явный режим при `ASR_PROVIDER=local`. Локальная модель и её доставка в образ
  не удаляются.
- **ФТ-7.** Для крупных аудиофайлов (base64-объём которых превышает безопасный порог размера запроса
  Cloudflare) предусмотрено корректное поведение: распознавание не падает с необработанной ошибкой —
  выполняется фолбэк на локальную модель и/или иное определённое поведение (стратегия и порог — см.
  ОВ-1). Ответ пользователю не ломается.
- **ФТ-8.** Секреты Cloudflare **переиспользуются** из окружения/`.env` (`CF_ACCOUNT_ID`,
  `CF_API_TOKEN`); они уже введены ADR-005 и присутствуют на сервере и в `deploy/.env.example`.
  Секреты не хардкодятся и не коммитятся.
- **ФТ-9.** Новые переменные окружения `ASR_PROVIDER` и `CF_WHISPER_MODEL` добавлены плейсхолдерами
  в `.env.example` (и в `deploy/.env.example` для единообразия) с дефолтными значениями
  (`cloudflare` и `@cf/openai/whisper-large-v3-turbo` соответственно).
- **ФТ-10.** Существующий контракт канала аудио сохраняется: приём `message.voice` /
  `message.audio` / audio-документов, скачивание штатным клиентом aiogram через запиннённый
  `api.telegram.org`, автоопределение языка (ru/en), последующее структурирование (ADR-005) и отдача
  результата (ADR-007) работают как раньше независимо от провайдера транскрипции.
- **ФТ-11.** На пустом результате распознавания (речь не распознана) сохраняется прежнее поведение
  канала (сообщение о нераспознанной речи), независимо от выбранного провайдера.

## 5. Нефункциональные требования

- **НФТ-1 (Надёжность / устойчивость к сбоям).** Перенос транскрипции на внешний API не должен
  делать распознавание хрупким: любой сбой Cloudflare детерминированно покрывается фолбэком на
  локальную модель (ФТ-5/ФТ-7). Таймаут на обращение к Cloudflare задаётся **явно** (конкретное
  значение и политика ретраев — задача архитектора, см. ОВ-2); по умолчанию — единичная попытка с
  явным таймаутом и фолбэком, без агрессивных ретраев (по образцу ADR-005).
- **НФТ-2 (Ресурсы / разгрузка сервера).** Облачная транскрипция — сетевая (I/O-bound) задача;
  основная цель по ресурсам — **снять CPU/RAM-нагрузку транскрипции с ivahub** (1 vCPU / 1.9 GB RAM)
  в штатном (облачном) пути и **не блокировать event loop** на время сетевого запроса (ФТ-3). При
  фолбэке/режиме `local` сохраняются ограничения ADR-003 (CPU-bound вне event loop, модель `base`,
  `int8`).
- **НФТ-3 (Стоимость / бесплатность).** Используется бесплатный tier Cloudflare Workers AI (тот же
  аккаунт, что и для структурирования). Необходимо учитывать **лимиты бесплатного tier** и
  **ограничение размера запроса** (base64 раздувает объём): превышение трактуется как сбой → фолбэк
  на локальную модель (НФТ-1/ФТ-7). Локальный фолбэк смягчает риск исчерпания лимитов.
- **НФТ-4 (Сетевая доступность с ivahub).** Эндпоинт Cloudflare Whisper должен быть достижим с
  ivahub без прокси. Достижимость **подтверждена вживую** (HTTP 200) и согласуется с уже работающим
  структурированием на Cloudflare (ADR-005) — основной сетевой риск снят; финально подтверждается
  приёмкой деплоем (см. ОВ-3).
- **НФТ-5 (Безопасность секретов).** `CF_ACCOUNT_ID`, `CF_API_TOKEN` — только через `.env` (покрыт
  `.gitignore`); в репозиторий — лишь плейсхолдеры в `.env.example` / `deploy/.env.example`.
  Соответствует разделу «Секреты» `CLAUDE.md`. Новые переменные `ASR_PROVIDER` / `CF_WHISPER_MODEL`
  не секретны, но также задокументированы плейсхолдерами (ФТ-9).
- **НФТ-6 (Преемственность поставки).** Сохраняется профиль поставки «бот без HTTP-порта» (ADR-001):
  long-polling, без `ports:`/Traefik/`edge`, образ из GHCR, `restart: unless-stopped`, деплой на
  ivahub через CI/CD, `extra_hosts`-пиннинг Telegram. Сохраняется граница «хендлер ↔ сервис»
  (ADR-002/ADR-003). Релиз использует уже существующее исходящее направление к `api.cloudflare.com`
  (введено ADR-005) — нового направления трафика не добавляется. Новых системных пакетов в образ не
  вводится (`httpx` уже есть); локальная модель и её доставка в образ сохраняются.
- **НФТ-7 (Качество транскрипции).** Облачная `whisper-large-v3-turbo` мультиязычна (ru/en,
  автоопределение) и ожидаемо точнее локальной `base`. Абсолютная точность не гарантируется и
  зависит от качества входного аудио. Локальный фолбэк сохраняет прежний (минимально приемлемый)
  уровень распознавания при недоступности облака.
- **НФТ-8 (Локализация интерфейса).** Системные сообщения бота — на русском. Контент распознаётся на
  языке исходника (ru/en, автоопределение); язык контента не переключается насильно.

## 6. Ограничения и допущения

Ограничения:
- **ОГР-1.** Транскрипция по умолчанию зависит от **внешнего бесплатного API** Cloudflare Workers
  AI; его доступность, лимиты тарифа и **максимальный размер запроса** вне контроля проекта. Поэтому
  фолбэк на локальную модель обязателен (ФТ-5/ФТ-7, НФТ-1/НФТ-3).
- **ОГР-2.** Cloudflare AI run принимает аудио как **base64 в JSON-теле**, что **раздувает объём
  запроса** (~+33 % к размеру файла); крупные аудио могут не уложиться в лимит размера запроса —
  отсюда требование к поведению для крупных файлов (ФТ-7, ОВ-1).
- **ОГР-3.** Локальный фолбэк работает в ограничениях ivahub (1 vCPU / 1.9 GB RAM): модель `base`,
  `int8`, CPU-bound вне event loop (ADR-003). Локальная модель **остаётся в образе** (рост образа
  относительно «облачного-только» варианта сохраняется — это сознательная цена страховки).
- **ОГР-4.** Бот работает в Docker-контейнере; обращение к Cloudflare использует уже имеющуюся
  зависимость `httpx` — **новых зависимостей не добавляется**. Конкретные детали реализации
  (структура абстракции провайдера, кодирование base64, таймаут) — задача архитектора.
- **ОГР-5.** Контракт канала аудио и профиль поставки (ADR-001/ADR-002/ADR-003) и последующие пути
  (структурирование ADR-005, отдача результата ADR-007) не меняются; релиз не вводит входящих портов
  и не меняет способ работы с Telegram (long-polling).
- **ОГР-6.** Секреты — только через `.env`; реальный `.env` не коммитится (НФТ-5).

Допущения:
- **ДОП-1.** Эндпоинт Cloudflare Whisper достижим с ivahub без прокси — **подтверждено вживую**
  (HTTP 200) и согласуется с работающим структурированием на Cloudflare (ОВ-3).
- **ДОП-2.** Бесплатного tier Cloudflare достаточно для ожидаемых объёмов обращений одного-двух
  пользователей; превышение лимита покрывается фолбэком на локальную модель (НФТ-3).
- **ДОП-3.** Качество транскрипции зависит от модели Cloudflare и качества входного аудио; на
  шумном/тихом входе результат может быть ограниченным — это свойство входа, не дефект. При сбое
  облака качество деградирует до локальной `base` (фолбэк), но результат пользователь получает.
- **ДОП-4.** Образ обновляется и деплоится через тот же CI/CD-пайплайн на ivahub, что и
  v0.1.0–v0.7.0; на сервере уже заданы `CF_ACCOUNT_ID` / `CF_API_TOKEN`; новые переменные
  `ASR_PROVIDER` / `CF_WHISPER_MODEL` имеют рабочие дефолты и могут быть не заданы.
- **ДОП-5.** `CF_WHISPER_MODEL` (модель ASR) и `CF_MODEL` (модель структурирования, ADR-005) —
  **разные** переменные с разным назначением; их раздельность исключает конфликт при общем аккаунте
  Cloudflare.
- **ДОП-6.** Жёсткие пользовательские лимиты длительности/размера аудио и индикация прогресса
  остаются вне рамок (наследие границ v0.3.0/v0.5.0); внутреннее ограничение размера запроса под
  лимит Cloudflare (ФТ-7) не является пользовательским лимитом и не отвергает файл, а уводит его в
  локальный фолбэк.

## 7. Критерии приёмки

- **КП-1.** При `ASR_PROVIDER=cloudflare` (или незаданном) бот распознаёт присланное аудио через
  Cloudflare Whisper по подтверждённому контракту вызова и возвращает транскрипт. (ФТ-1, ФТ-3, ФТ-4)
- **КП-2.** Модель Cloudflare по умолчанию — `@cf/openai/whisper-large-v3-turbo`; задание
  `CF_WHISPER_MODEL` переопределяет модель без изменения кода. (ФТ-2)
- **КП-3.** Значение `ASR_PROVIDER` переключает источник транскрипции между Cloudflare и локальной
  моделью без изменения кода; дефолт — `cloudflare`. (ФТ-4, ФТ-6)
- **КП-4.** При сбое Cloudflare (недоступность/ошибка/таймаут/лимит) бот выполняет фолбэк на
  локальный faster-whisper и возвращает результат **без ошибки**; процесс не падает, сбой
  логируется. (ФТ-5, НФТ-1)
- **КП-5.** При `ASR_PROVIDER=local` бот распознаёт аудио только локальной моделью, не обращаясь к
  Cloudflare; локальная модель и её доставка в образ сохранены. (ФТ-6)
- **КП-6.** Для аудио, слишком крупного для лимита размера запроса Cloudflare, распознавание не
  падает с необработанной ошибкой: выполняется фолбэк на локальную модель и/или иное определённое
  поведение; ответ пользователю не ломается. (ФТ-7, НФТ-3, ОГР-2)
- **КП-7.** Обращение к Cloudflare не блокирует event loop: во время сетевого запроса бот остаётся
  отзывчивым для других сообщений. (ФТ-3, НФТ-2)
- **КП-8.** Секреты Cloudflare читаются только из `.env`/окружения и переиспользуются
  (`CF_ACCOUNT_ID`, `CF_API_TOKEN`); в `.env.example` и `deploy/.env.example` присутствуют
  плейсхолдеры новых переменных `ASR_PROVIDER` и `CF_WHISPER_MODEL`; секреты не захардкожены и не
  закоммичены. (ФТ-8, ФТ-9, НФТ-5)
- **КП-9.** Существующий контракт канала аудио сохранён: приём voice/audio/audio-документов,
  скачивание через запиннённый `api.telegram.org`, автоопределение языка (ru/en), последующее
  структурирование (ADR-005) и отдача результата (ADR-007) работают как раньше при любом провайдере.
  (ФТ-10)
- **КП-10.** На пустом результате распознавания сохраняется прежнее сообщение о нераспознанной речи,
  независимо от провайдера. (ФТ-11)
- **КП-11.** Профиль поставки сохранён: long-polling, без `ports:`/Traefik/`edge`, образ из GHCR,
  `restart: unless-stopped`; новых зависимостей и новых направлений исходящего трафика не добавлено
  (Cloudflare уже использовался), локальная модель в образе сохранена. (НФТ-6)
- **КП-12.** При деплое на ivahub облачная транскрипция (Cloudflare) реально работает с присланным
  аудио (подтверждение достижимости/работоспособности эндпоинта), а при искусственном сбое облака
  срабатывает локальный фолбэк. (НФТ-4, ОВ-3)

## 8. Открытые вопросы

Блокирующих вопросов нет: контракт вызова Cloudflare Whisper подтверждён вживую (HTTP 200),
переключатель/фолбэк/переменные окружения специфицированы, противоречий действующим ADR нет.
Некритичные вопросы (уточняются при проектировании/реализации или подтверждаются приёмкой,
оформление документа не блокируют):

- **ОВ-1 (архитектор, `/adr`).** Стратегия и **порог размера** для крупных аудио: фолбэк на
  локальную модель при превышении безопасного порога base64-объёма vs чанкинг аудио (декодирование
  через `av`/PyAV уже есть в образе) с несколькими запросами и склейкой. По умолчанию допустим
  фолбэк на локальную модель для слишком крупных файлов; чанкинг выбирает архитектор, если оправдан
  (с учётом лимитов бесплатного tier). Конкретный порог — на усмотрение архитектора (ФТ-7).
- **ОВ-2 (архитектор, `/adr`).** Конкретное значение таймаута на обращение к Cloudflare и политика
  ретраев (есть/нет, сколько) — задача архитектора (НФТ-1); по умолчанию — единичная попытка с явным
  таймаутом и фолбэком, без агрессивных ретраев (по образцу ADR-005, Решение 4).
- **ОВ-3 (проверяется при приёмке, не блокирует BA).** Финальное подтверждение работоспособности
  Cloudflare Whisper с прода ivahub на реальном аудио и корректного срабатывания локального фолбэка
  при искусственном сбое (КП-12). Достижимость уже подтверждена вживую (HTTP 200) — риск низкий.
- **ОВ-4 (архитектор, `/adr`).** Структура абстракции провайдера транскрипции в
  `services/transcribe.py` (общий контракт «аудио → текст» с реализациями Cloudflare/локальный и
  оркестрацией фолбэка) — по образцу абстракции провайдера структурирования (ADR-005, Решение 1);
  деталь реализации, не заказчика.
- **ОВ-5 (архитектор, не блокирует).** Поведение фолбэка при `ASR_PROVIDER=local` и одновременной
  недоступности/непригодности локальной модели (теоретический край) — детерминированное сообщение об
  ошибке распознавания; конкретика — за архитектором/реализацией.

Противоречий действующим ADR нет. ADR-003 (локальная транскрипция) **дополняется**, а не
отменяется: локальная модель сохраняется как фолбэк/режим. ADR-005 (Cloudflare Workers AI для
структурирования) **переиспользуется** по инфраструктуре (аккаунт, токен, паттерн `httpx`,
исходящее направление к `api.cloudflare.com`); переменная ASR-модели `CF_WHISPER_MODEL` отделена от
`CF_MODEL` структурирования во избежание конфликта. ADR-001 (профиль поставки) и ADR-007 (отдача
результата) сохраняются без изменений.
