# BA-требования 009: Доп-действия над результатом — «Кратко» и «Перевести» через inline-кнопки (v0.9.0)

- **Источник:** `requirements/req-textify-009.md`
- **Дата:** 2026-06-23
- **Статус:** черновик

## 1. Контекст и бизнес-цель

Textify — Telegram-бот, который бесплатно превращает голосовые/аудио и изображения в чистый
структурированный текст (см. `README.md`, `CLAUDE.md`). Продукт развивался поэтапно: каркас
(v0.1.0), OCR изображений (v0.2.0), аудио-транскрипция (v0.3.0), улучшение OCR (v0.4.0),
структурирование через Cloudflare Workers AI (v0.5.0), учёт пользователей и `/stats` (v0.6.0),
надёжная отдача длинного результата через `send_result` (v0.7.0, ADR-007), перенос транскрипции на
Cloudflare Whisper с фолбэком (v0.8.0). Релизы v0.1.0–v0.8.0 в проде на ivahub (long-polling,
Docker, образ из GHCR; профиль поставки «бот без HTTP-порта» — ADR-001).

Сейчас бот отдаёт пользователю распознанный и структурированный текст — и на этом взаимодействие
заканчивается. Платные боты-конкуренты (TAK! TEXT, Memo AI, Transcriber) удерживают аудиторию
именно **доп-ценностью** над уже полученным текстом: краткое саммари и перевод. У Textify
**LLM-слой уже работает** (Cloudflare Workers AI, ADR-005: `services/structure.py`, прямой async
HTTP через `httpx`, секреты `CF_ACCOUNT_ID`/`CF_API_TOKEN`/`CF_MODEL`/`LLM_PROVIDER`, доступ к CF
с ivahub подтверждён), поэтому добавление этих действий **дёшево** и опирается на проверенную
инфраструктуру.

Цель релиза v0.9.0 — дать пользователю **быстрые действия над уже полученным результатом** прямо
из чата: под текстовым результатом OCR и аудио показывать inline-кнопки **«Кратко»** (саммари в
3–5 пунктов) и **«Перевести»** (перевод), по нажатию которых бот через тот же CF LLM формирует
соответствующий ответ над исходным текстом и присылает его отдельным сообщением. Это рост
удержания при минимальной стоимости разработки; аудитория уже измеряется через `/stats` (v0.6.0).

Стейкхолдеры:
- **Конечный пользователь Telegram** (основной выгодоприобретатель) — получает дополнительную
  ценность (саммари, перевод) над распознанным текстом в один тап, бесплатно, не выходя из чата.
- **Владелец продукта** (заказчик) — повышает удержание и конкурентоспособность, переиспользуя уже
  оплаченную (бесплатную) и проверенную инфраструктуру Cloudflare без новых зависимостей и без
  изменения профиля поставки.
- **Команда разработки** (последующие агенты) — получает в `services/structure.py` (или соседнем
  сервисе) переиспользуемый паттерн «действие над текстом через LLM» и обработчики callback-кнопок
  в хендлерах.

## 2. Область (scope)

### В рамках задачи

- **Inline-кнопки под текстовым результатом** OCR и аудио:
  - **«Кратко»** — по нажатию формируется саммари исходного текста в **3–5 пунктов на языке
    текста** (ru/en — по исходнику, без насильственной смены языка).
  - **«Перевести»** — по нажатию формируется перевод исходного текста. Целевой язык (и его выбор —
    одна кнопка с авто-направлением vs две кнопки EN/RU) — **решение архитектора**; разумный дефолт
    из требования: текст русский → перевод на английский, иначе → на русский; либо две кнопки
    EN/RU.
- **Размещение кнопок** под итоговым текстовым результатом в `handlers/image.py` (OCR: фото и
  изображение-документ) и `handlers/audio.py` (аудио: голос, аудио, аудио-документ) — там, где
  сейчас результат отдаётся пользователю.
- **Обработка нажатия** (callback): по нажатию кнопки бот вызывает **CF Workers AI LLM** (тот же
  провайдер/паттерн, что `services/structure.py`: прямой async HTTP через `httpx`, секреты из
  окружения, явный таймаут) с **соответствующим промптом** (саммари / перевод) над исходным
  текстом.
- **Источник исходного текста** для действия — **текст сообщения, к которому прикреплены кнопки**
  (`callback.message.text`), **без отдельного хранилища**: между перезапусками/во времени тексты не
  хранятся.
- **Отдача ответа действия** — отдельным сообщением через **`send_result`** (v0.7.0, ADR-007),
  чтобы длинный ответ (саммари/перевод) не обрезался лимитом Telegram 4096 и доходил целиком.
- **Поведение при многосообщенческом/файловом результате (v0.7.0).** Если итоговый результат ушёл
  **несколькими сообщениями или файлом `.txt`** (т.е. исходный текст не помещается в одном
  сообщении и недоступен целиком через `callback.message.text`) — кнопки в таком случае **можно не
  показывать** (упрощение; точное правило и его реализация — решение архитектора).
- **Языки контента** — ru + en (как в проекте); системный интерфейс — русский.

### Вне рамок (explicitly out of scope)

- **Вопросы по содержимому (Q&A-чат)** над текстом — отдельная идея на будущее.
- **Хранение текстов** между перезапусками или отдельное хранилище исходных текстов под кнопки —
  действия работают строго по тексту видимого сообщения (`callback.message.text`).
- **Перевод на произвольные языки** сверх ru/en.
- **Озвучка результата (TTS).**
- **Изменение логики распознавания/структурирования** (OCR, транскрипция, базовое структурирование
  ADR-005) — релиз добавляет действия **над уже сформированным** результатом, не меняя его
  получение.
- **Изменение единой отдачи результата `send_result`** (ADR-007) — она **переиспользуется** для
  отправки ответа действия, её контракт не меняется.
- **Изменение профиля поставки** (ADR-001): бот остаётся long-polling без входящего HTTP-порта;
  новых портов/зависимостей/направлений исходящего трафика не вводится (CF уже используется).
- **Боевое включение Groq / смена провайдера** — действия используют тот же провайдер, что и
  структурирование (по умолчанию Cloudflare); переключение провайдера — наследие ADR-005, релиз его
  не трогает.

## 3. Действующие лица и сценарии использования

**Конечный пользователь Telegram.** Цель — быстро получить саммари или перевод уже распознанного
текста, не копируя его во внешние сервисы.

- **Сценарий A (саммари).** Пользователь прислал фото/аудио → получил текстовый результат с
  кнопками под ним → нажал **«Кратко»** → бот формирует саммари в 3–5 пунктов на языке текста и
  присылает его **отдельным сообщением**.
- **Сценарий B (перевод).** Пользователь под текстовым результатом нажимает **«Перевести»** → бот
  формирует перевод (целевой язык — по дефолту/выбору согласно решению архитектора) и присылает его
  **отдельным сообщением**.
- **Сценарий C (длинный ответ действия).** Саммари или перевод получается длиннее лимита Telegram →
  ответ доходит целиком за счёт `send_result` (несколько сообщений или файл `.txt`, ADR-007), без
  ошибки «message is too long» и без потери содержания.
- **Сценарий D (сбой LLM при действии).** CF LLM недоступен/вернул ошибку/таймаут → действие не
  ломает бота: пользователь получает понятное поведение (например, краткое сообщение, что действие
  сейчас недоступно); процесс не падает (точная формулировка/стратегия — за архитектором, см. ОВ-3).
- **Сценарий E (результат ушёл несколькими сообщениями/файлом).** Исходный результат был длинным и
  отдан несколькими сообщениями или файлом (v0.7.0) → кнопки под ним **могут не показываться**
  (упрощение); пользователь не получает действий над недоступным целиком текстом.
- **Сценарий F (повторное нажатие / устаревшее сообщение).** Пользователь нажимает кнопку повторно
  или на старом сообщении → бот корректно отрабатывает callback (формирует ответ заново либо
  детерминированно обрабатывает случай), без падения; конкретика — за архитектором/реализацией
  (см. ОВ-4).

**Команда разработки (последующие агенты).** Цель — переиспользуемый и предсказуемый механизм
«действие над текстом через LLM».

- **Сценарий G (переиспользование).** Новое действие над текстом (текущее или будущее) опирается на
  тот же паттерн вызова CF LLM (промпт + исходный текст → ответ) и на `send_result` для отдачи, без
  дублирования логики.

## 4. Функциональные требования

- **ФТ-1.** Под итоговым **текстовым** результатом OCR (`handlers/image.py`: фото и
  изображение-документ) и аудио (`handlers/audio.py`: голос, аудио, аудио-документ) отображаются
  inline-кнопки **«Кратко»** и **«Перевести»**.
- **ФТ-2.** По нажатию **«Кратко»** бот формирует **саммари исходного текста в 3–5 пунктов** на
  **языке текста** (ru/en по исходнику; язык не переключается насильно) и присылает его **отдельным
  сообщением**.
- **ФТ-3.** По нажатию **«Перевести»** бот формирует **перевод исходного текста** и присылает его
  **отдельным сообщением**. Целевой язык и способ его выбора (одна кнопка с авто-направлением
  рус→en / иначе→ru, либо две кнопки EN/RU) — **определяет архитектор** (см. ОВ-1); по умолчанию
  допустимо авто-направление рус→en, иначе→ru.
- **ФТ-4.** Ответ действия формируется вызовом **CF Workers AI LLM тем же паттерном, что
  `services/structure.py`** (прямой async-вызов через `httpx`, секреты из окружения, явный таймаут),
  с **соответствующим действию промптом** (саммари / перевод) над исходным текстом. Релиз не вводит
  новый провайдер и не меняет провайдера структурирования.
- **ФТ-5.** Источник исходного текста для действия — **текст сообщения, к которому прикреплены
  кнопки** (`callback.message.text`). **Отдельное хранилище текстов не вводится**; тексты не
  хранятся между перезапусками/во времени.
- **ФТ-6.** Ответ действия (саммари/перевод) отправляется через **`send_result`** (ADR-007), чтобы
  длинный ответ доходил **целиком** (одно сообщение / несколько сообщений / файл `.txt`) и не падал
  с ошибкой «message is too long».
- **ФТ-7.** Обращение к CF LLM при действии выполняется **неблокирующим** async-способом (через
  `httpx`, как ADR-005) с явным таймаутом; бот остаётся отзывчивым для других сообщений во время
  сетевого запроса по действию.
- **ФТ-8.** При **сбое** CF LLM (сетевой сбой, недоступность, 4xx/5xx, таймаут, превышение лимита
  тарифа, некорректный/пустой ответ) действие **не ломает бота**: процесс не падает, сбой
  логируется, пользователь получает понятное детерминированное поведение (конкретная
  формулировка/стратегия — за архитектором, см. ОВ-3).
- **ФТ-9.** Если итоговый результат был отдан **несколькими сообщениями или файлом** (v0.7.0) —
  кнопки под ним **могут не показываться** (упрощение). Точное правило (когда показывать/не
  показывать) определяет архитектор; кнопки гарантированно показываются как минимум для результата,
  целиком доступного через `callback.message.text` (одно текстовое сообщение).
- **ФТ-10.** Нажатие inline-кнопки корректно подтверждается на стороне Telegram (callback
  обрабатывается без «зависшего» индикатора нажатия) и не приводит к падению бота; обработка
  повторных нажатий/устаревших сообщений детерминирована (см. ОВ-4).
- **ФТ-11.** Секреты CF (`CF_ACCOUNT_ID`, `CF_API_TOKEN`) и параметры провайдера/модели
  (`LLM_PROVIDER`, `CF_MODEL`) **переиспользуются** из окружения/`.env` (введены ADR-005); новые
  секреты не вводятся, не хардкодятся и не коммитятся. Если для действий нужны дополнительные
  несекретные параметры (например, имя файла-вложения или промпты-константы) — оформляются как
  константы/несекретная конфигурация (деталь реализации, см. ОВ-2).
- **ФТ-12.** Существующее поведение каналов сохраняется: распознавание (OCR/аудио),
  структурирование (ADR-005), пустой результат (сообщение о нераспознанном тексте/речи) и отдача
  результата (`send_result`, ADR-007) работают как раньше; добавление кнопок и действий —
  обратимо-совместимое расширение, не ломающее текущие пути.
- **ФТ-13.** На **пустом** результате распознавания (текст/речь не распознаны) кнопки действий **не
  показываются** — действовать не над чем; сохраняется прежнее служебное сообщение канала.

## 5. Нефункциональные требования

- **НФТ-1 (Надёжность / устойчивость к сбоям).** Действия над текстом не должны делать бота
  хрупким: любой сбой CF LLM детерминированно обрабатывается без падения процесса (ФТ-8) — по
  образцу политики фолбэка ADR-005/ADR-008. Таймаут на обращение к CF задаётся явно; политика
  ретраев — за архитектором, по умолчанию единичная попытка с явным таймаутом (образец ADR-005,
  Решение 4).
- **НФТ-2 (Отзывчивость / неблокирование).** Сетевой вызов CF при действии и отдача ответа (включая
  возможные паузы между частями и формирование файла внутри `send_result`) **не блокируют** event
  loop aiogram — бот остаётся отзывчивым для других пользователей (преемственность ADR-002/ADR-003,
  ADR-005, ADR-007).
- **НФТ-3 (Стоимость / бесплатность).** Действия используют **бесплатный tier** Cloudflare Workers
  AI (тот же аккаунт, что структурирование и транскрипция). Каждое нажатие — отдельный запрос к CF;
  необходимо учитывать лимиты бесплатного tier и ограничение размера запроса (длинный исходный текст
  раздувает запрос). Превышение лимита/размера трактуется как сбой → детерминированное поведение
  (ФТ-8). Усечение слишком длинного входа перед отправкой — допустимо по образцу ADR-005 (Решение
  5); конкретика — за архитектором (см. ОВ-5).
- **НФТ-4 (Совместимость с ограничениями Telegram).** Ответ действия учитывает лимит текстового
  сообщения 4096 (закрывается `send_result`, ФТ-6) и риск rate limit при серии частей (также внутри
  `send_result`, ADR-007). Inline-клавиатура формируется штатными средствами aiogram.
- **НФТ-5 (Качество результата действия).** Качество саммари/перевода зависит от модели CF
  (`CF_MODEL`) и качества исходного текста; абсолютная точность не гарантируется. Промпты должны
  инструктировать модель **не выдумывать факты** (для саммари — оформить смысл исходника; для
  перевода — переводить, не добавляя содержание), по образцу промпта структурирования (ADR-005,
  Решение 7). На шумном/коротком входе результат может быть ограниченным — это свойство входа, не
  дефект.
- **НФТ-6 (Безопасность секретов).** `CF_ACCOUNT_ID`, `CF_API_TOKEN` — только через окружение/`.env`
  (покрыт `.gitignore`); в репозиторий — лишь плейсхолдеры (уже введены ADR-005). Соответствует
  разделу «Секреты» `CLAUDE.md`. Новых секретов релиз не вводит.
- **НФТ-7 (Преемственность поставки).** Сохраняется профиль поставки «бот без HTTP-порта» (ADR-001):
  long-polling, без `ports:`/Traefik/`edge`, образ из GHCR, `restart: unless-stopped`,
  `extra_hosts`-пиннинг Telegram. Сохраняется граница «хендлер ↔ сервис» (ADR-002/ADR-003). Релиз
  использует **уже существующее** исходящее направление к `api.cloudflare.com` (введено ADR-005) —
  нового направления трафика и **новых зависимостей** не добавляется (`httpx` уже есть; aiogram
  inline-клавиатуры — штатные).
- **НФТ-8 (Локализация интерфейса).** Подписи кнопок («Кратко», «Перевести», и при выборе — EN/RU),
  служебные сообщения о сбое/недоступности действия — на **русском**. Контент саммари формируется на
  языке исходника; перевод — на целевой язык по решению архитектора.
- **НФТ-9 (Сопровождаемость).** Логика действия (построение промпта, вызов CF LLM, отдача через
  `send_result`) переиспользует существующий паттерн `services/structure.py` и единую точку отдачи
  (ADR-007), без дублирования; промпты и подписи кнопок локализованы в одном месте (деталь
  реализации).

## 6. Ограничения и допущения

Ограничения:
- **ОГР-1.** Источник текста действия ограничен **`callback.message.text`** — текстом видимого
  сообщения с кнопками; отдельного хранилища нет (ФТ-5). Следствие: для результата, не помещающегося
  в одно текстовое сообщение (несколько сообщений/файл, v0.7.0), целостный исходный текст
  недоступен — отсюда допущение «кнопки можно не показывать» (ФТ-9).
- **ОГР-2.** Действия зависят от **внешнего бесплатного API** Cloudflare Workers AI; доступность,
  лимиты тарифа и максимальный размер запроса вне контроля проекта → требуется детерминированное
  поведение при сбое (ФТ-8, НФТ-1) и учёт длины входа (НФТ-3).
- **ОГР-3.** Лимит текстового сообщения Telegram 4096 и риск rate limit при серии частей —
  закрываются переиспользованием `send_result` (ADR-007); релиз не дублирует эту логику.
- **ОГР-4.** Профиль поставки (ADR-001) и границы «хендлер ↔ сервис» (ADR-002/ADR-003) сохраняются;
  релиз не вводит входящих портов и не меняет способ работы с Telegram (long-polling).
- **ОГР-5.** Новых внешних зависимостей быть не должно: `httpx` уже есть (ADR-005), inline-кнопки и
  обработка callback — штатные средства aiogram.
- **ОГР-6.** Секреты — только через `.env`; реальный `.env` не коммитится (НФТ-6).

Допущения:
- **ДОП-1.** Доступ к CF Workers AI с ivahub **подтверждён** и работает в проде для структурирования
  (ADR-005) и транскрипции (v0.8.0); тот же аккаунт/токен/паттерн `httpx` переиспользуются для
  действий — основной сетевой риск снят.
- **ДОП-2.** Ожидаемые объёмы невелики (один-два активных пользователя на ivahub, 1 vCPU /
  1.9 GB RAM); накладные расходы на дополнительный запрос по нажатию пренебрежимы, лимитов
  бесплатного tier ожидаемо достаточно (превышение — покрывается ФТ-8).
- **ДОП-3.** Целевой язык перевода и способ его выбора (одна кнопка с авто-направлением vs две
  кнопки EN/RU) **делегированы архитектору** самим требованием; дефолт — рус→en, иначе→ru. BA это не
  блокирует (см. ОВ-1).
- **ДОП-4.** Правило показа кнопок при многосообщенческом/файловом результате (показывать только для
  одно-сообщенческого текста) **делегировано архитектору** как упрощение (ФТ-9, ОВ-6); BA это не
  блокирует.
- **ДОП-5.** Промпты саммари/перевода строятся по образцу промпта структурирования (ADR-005,
  Решение 7): сохранять смысл, не выдумывать факты, вернуть только результат без преамбул; для
  саммари — 3–5 пунктов на языке исходника, для перевода — перевод на целевой язык.
- **ДОП-6.** Образ обновляется и деплоится тем же CI/CD на ivahub; CF-секреты уже заданы на сервере;
  новых обязательных env-переменных релиз не вводит (опциональные несекретные параметры, если
  понадобятся, имеют рабочие дефолты).

## 7. Критерии приёмки

- **КП-1.** Под итоговым текстовым результатом OCR (`handlers/image.py`) и аудио
  (`handlers/audio.py`) отображаются inline-кнопки «Кратко» и «Перевести». (ФТ-1)
- **КП-2.** Нажатие «Кратко» возвращает саммари исходного текста в 3–5 пунктов на языке текста,
  присланное отдельным сообщением. (ФТ-2)
- **КП-3.** Нажатие «Перевести» возвращает перевод исходного текста (целевой язык/способ выбора —
  согласно решению архитектора; по умолчанию рус→en, иначе→ru), присланный отдельным сообщением.
  (ФТ-3, ОВ-1)
- **КП-4.** Ответ действия формируется через CF Workers AI LLM тем же паттерном, что
  `services/structure.py` (async `httpx`, секреты из окружения, явный таймаут), с промптом действия
  над исходным текстом; новый провайдер не вводится. (ФТ-4, ФТ-7)
- **КП-5.** Исходный текст действия берётся из `callback.message.text`; отдельное хранилище текстов
  не добавлено. (ФТ-5, ОГР-1)
- **КП-6.** Длинный ответ действия (саммари/перевод) доходит до пользователя целиком через
  `send_result` (одно сообщение / несколько сообщений / файл `.txt`), без ошибки «message is too
  long». (ФТ-6, НФТ-4)
- **КП-7.** При сбое CF LLM по действию (недоступность/ошибка/таймаут/лимит) бот не падает, сбой
  логируется, пользователь получает понятное детерминированное поведение; обработка других
  сообщений не нарушается. (ФТ-8, НФТ-1, НФТ-2)
- **КП-8.** Обращение к CF по действию не блокирует event loop: во время сетевого запроса бот
  остаётся отзывчивым для других сообщений. (ФТ-7, НФТ-2)
- **КП-9.** Если итоговый результат был отдан несколькими сообщениями или файлом (v0.7.0), кнопки
  под ним могут не показываться согласно правилу архитектора; для одно-сообщенческого текстового
  результата кнопки показываются. (ФТ-9, ОВ-6)
- **КП-10.** На пустом результате распознавания кнопки действий не показываются; сохраняется прежнее
  служебное сообщение канала. (ФТ-13)
- **КП-11.** Нажатие кнопки корректно подтверждается на стороне Telegram (без зависшего индикатора),
  повторные нажатия/устаревшие сообщения обрабатываются детерминированно, бот не падает. (ФТ-10,
  ОВ-4)
- **КП-12.** Секреты CF и параметры провайдера/модели переиспользуются из окружения/`.env`
  (`CF_ACCOUNT_ID`, `CF_API_TOKEN`, `LLM_PROVIDER`, `CF_MODEL`); новых секретов нет, секреты не
  захардкожены и не закоммичены. (ФТ-11, НФТ-6)
- **КП-13.** Существующие пути сохранены: распознавание, структурирование (ADR-005), отдача
  результата (`send_result`, ADR-007), сообщения о пустом результате работают как раньше; добавление
  кнопок/действий обратимо-совместимо. (ФТ-12)
- **КП-14.** Профиль поставки сохранён: long-polling, без `ports:`/Traefik/`edge`, образ из GHCR,
  `restart: unless-stopped`; новых зависимостей и новых направлений исходящего трафика не добавлено
  (CF уже использовался). (НФТ-7)
- **КП-15.** Подписи кнопок и служебные сообщения о недоступности действия — на русском. (НФТ-8)

## 8. Открытые вопросы

Блокирующих вопросов нет: требование детально специфицировано, LLM-слой и доступ к CF подтверждены и
работают в проде, а граничные решения (целевой язык перевода, правило показа кнопок при
многосообщенческом результате) **явно делегированы архитектору** самим требованием. Оформление
документа не блокируется. Ниже — некритичные вопросы, уточняются при проектировании (`/adr`) или
приёмке.

- **ОВ-1 (архитектор, `/adr`).** Целевой язык перевода и способ его выбора: одна кнопка «Перевести»
  с авто-направлением (рус→en, иначе→ru) **либо** две кнопки EN/RU. По умолчанию допустимо
  авто-направление рус→en/иначе→ru. Делегировано требованием.
- **ОВ-2 (архитектор, `/adr`).** Где размещать логику действий (расширение `services/structure.py`
  vs новый сервис, например `services/actions.py`), где хранить промпты и подписи кнопок, как
  устроить callback-данные (`callback_data`) — детали реализации/архитектуры, не заказчика.
- **ОВ-3 (архитектор, `/adr`).** Точное поведение при сбое/таймауте CF по действию: формулировка
  служебного сообщения пользователю, политика ретраев (по умолчанию — единичная попытка с явным
  таймаутом, образец ADR-005). Заказчику достаточно «бот не падает, поведение понятное».
- **ОВ-4 (архитектор, `/adr`).** Поведение при повторном нажатии кнопки и на устаревших сообщениях:
  формировать ответ заново каждый раз vs дедуп; ответ на `callback.answer()` для снятия индикатора
  нажатия. По умолчанию — каждое нажатие формирует ответ заново, без падения.
- **ОВ-5 (архитектор, `/adr`).** Усечение слишком длинного исходного текста перед отправкой в CF
  (по образцу ADR-005, Решение 5: порог ~8000 символов) и его согласование с лимитами бесплатного
  tier и размером запроса. По умолчанию допустимо усечение по безопасному порогу.
- **ОВ-6 (архитектор, `/adr`).** Точное правило показа/скрытия кнопок при результате, отданном
  несколькими сообщениями или файлом (v0.7.0): не показывать вовсе vs показывать только под последней
  частью и т.п. По умолчанию — показывать только для одно-сообщенческого текстового результата
  (упрощение из требования).
- **ОВ-7 (проверяется при приёмке, не блокирует BA).** Финальное подтверждение работы действий
  (саммари/перевод) на проде ivahub на реальном тексте и корректного поведения при искусственном
  сбое CF (КП-7). Достижимость CF уже подтверждена работающими v0.5.0/v0.8.0 — риск низкий.

Противоречий действующим ADR нет. Релиз **переиспользует** ADR-005 (CF Workers AI: аккаунт/токен,
прямой HTTP через `httpx`, таймаут/фолбэк, паттерн промпта) и ADR-007 (`send_result` для отдачи
длинного ответа), **сохраняет** ADR-001 (профиль поставки «бот без HTTP-порта») и
ADR-002/ADR-003 (граница «хендлер ↔ сервис», неблокирующие операции). Профиль поставки ADR-001 — без
изменений.
