# BA-требования 013: Виральная петля — реферальные deep-link, шеринг бота и ненавязчивая подпись (v1.3.0)

- **Источник:** `requirements/req-textify-013.md`
- **Дата:** 2026-06-23
- **Статус:** черновик

## 1. Контекст и бизнес-цель

Textify — Telegram-бот, который **бесплатно** превращает голосовые/аудио и изображения в чистый
структурированный текст (см. `README.md`, `CLAUDE.md`). Релизы v0.1.0–v1.2.0 полностью в проде на
ivahub (1 vCPU / 1.9 GB RAM, long-polling, Docker, образ из GHCR; профиль поставки «бот без
HTTP-порта» — ADR-001). Продукт зафиксирован как 1.0 (контракт ADR-010); бесплатность под нагрузкой
застрахована дневным бюджетом Cloudflare с авто-деградацией (ADR-011) и глобальным семафором на
тяжёлые локальные операции (ADR-010). Релиз v1.2.0 (ADR-012) добавил режим групп как канал
органического роста и заложил фундамент под виральную петлю.

Сверхцель продукта — **максимум пользователей бесплатно**. Сейчас рост идёт пассивно (бот в
группах показывает своё имя участникам), но у пользователя **нет инструмента**, чтобы привести
других, и **нет видимости**, что бот распространяется. Релиз v1.3.0 (MINOR) превращает текущих
пользователей в активный канал привлечения новых — **нативными средствами Telegram и бесплатно для
всех**, тремя дополняющими друг друга механиками:

1. **Реферальные deep-link** — у каждого пользователя есть персональная ссылка-приглашение
   `https://t.me/YourTextifyBot?start=ref_<user_id>`; при переходе нового пользователя по ней
   фиксируется связь «кто кого привёл», а в `/start` пользователь видит свою ссылку и счётчик
   «приглашено: N» (лёгкая геймификация без денежных наград).
2. **Шеринг** — inline-кнопка «Поделиться ботом», ведущая на нативный экран Telegram
   `https://t.me/share/url?...` с персональной реф-ссылкой, чтобы пользователь в один тап
   переслал бота в любой чат.
3. **Ненавязчивая подпись (attribution)** — короткая строка `— @YourTextifyBot` под
   одно-сообщенческим результатом, чтобы пересланный/скопированный текст нёс имя бота (бесплатная
   реклама); управляется env `ATTRIBUTION_FOOTER` (дефолт on).

Вместе эти три механики образуют **самоподдерживающуюся петлю привлечения**: пользователь делится →
новый пользователь приходит по реф-ссылке → видит ценность и свою реф-ссылку → делится дальше; а
подпись обеспечивает фоновую экспозицию имени бота. Эффект виден владельцу через расширенную
команду `/stats` (реферальные метрики).

**Бесплатность по построению.** Все три механики — нативный Telegram (deep-link `?start=`, экран
`t.me/share/url`) + локальная SQLite (рядом со статистикой ADR-006). **Cloudflare/LLM не
используются** — затрат нет, на free-tier влияния нет.

Стейкхолдеры:
- **Владелец продукта.** Получает активный бесплатный канал роста (рефералы + шеринг + подпись) и
  видимость его эффекта через `/stats`, без денежных расходов и без рисков для бесплатности.
- **Конечный пользователь.** Получает персональную ссылку-приглашение, удобную кнопку «Поделиться
  ботом» и видимый счётчик приглашённых (лёгкая геймификация); поведение распознавания не меняется.
- **Приглашённый пользователь.** Приходит по реф-ссылке, получает обычный `/start` (полная
  функциональность бота сразу) — реферал прозрачен и ничего у него не требует.
- **Команда разработки (последующие агенты).** Получает первую таблицу связей между пользователями
  поверх существующего хранилища ADR-006 и опору под дальнейшие механики роста.

## 2. Область (scope)

### В рамках задачи

- **Реферальные deep-link (механика 1).**
  - У каждого пользователя есть персональная ссылка вида
    `https://t.me/<username_бота>?start=ref_<user_id>`. Username бота берётся из `Bot.get_me()`
    (уже используется в v1.2.0 для групп) — не хардкодится.
  - В `/start` бот **разбирает start-параметр**: если параметр имеет вид `ref_<id>`, пользователь
    **новый** (первое обращение к боту) и `id != user_id` самого нажавшего — фиксируется связь
    «кто кого привёл» (`referrer_id = id`, `referred_id = user_id`, время).
  - Связь хранится в **существующей SQLite** (рядом со статистикой ADR-006) в **новой таблице
    рефералов**. Учёт **идемпотентен**: один приглашённый учитывается **ровно один раз**;
    повторный `/start` (в т. ч. с другим/тем же реф-параметром) связь не меняет и не дублирует.
  - **Самоприглашение игнорируется** (`ref_<own_id>` не создаёт связи).
  - В `/start` пользователю показываются: **его персональная реф-ссылка** и **счётчик «приглашено:
    N»** (число уникальных пользователей, пришедших по его ссылке).

- **Шеринг (механика 2).**
  - inline-кнопка **«Поделиться ботом»**, ведущая на нативный экран Telegram
    `https://t.me/share/url?url=<реф-ссылка>&text=<короткий текст>` с **персональной реф-ссылкой
    нажавшего** (его `user_id` в `ref_`-коде), чтобы те, кто перейдёт по расшаренной ссылке,
    засчитались этому пользователю.
  - Кнопка показывается **минимум в `/start`**; размещение под результатом — опционально (за
    архитектором, ОВ-1), при условии, что это не ломает существующие кнопки действий
    «Кратко»/«Перевести» (ADR-009) и правило «markup только при одиночном сообщении» (ADR-007/009).

- **Ненавязчивая подпись / attribution (механика 3).**
  - К результату распознавания добавляется короткая строка-подпись `— @YourTextifyBot`.
  - **Только для одно-сообщенческого результата** (когда результат уходит одним сообщением).
    Подпись **не добавляется** при разбиении на серию сообщений и при переходе на файл `.txt`
    (контракт `send_result`, ADR-007) — чтобы **не ломать** сплиттер/файл из v0.7.0 и не влиять на
    подсчёт длины частей.
  - Управляется переменной окружения **`ATTRIBUTION_FOOTER`** (дефолт **on**); при `off` подпись
    не добавляется. Переменная добавляется как плейсхолдер в `.env.example` и
    `deploy/.env.example`.
  - Имя бота в подписи — из `Bot.get_me()` (не хардкодить).

- **Расширение `/stats` реферальными метриками.** Существующая админ-команда `/stats` (только для
  `ADMIN_USER_ID`, ADR-006) дополняется реферальными метриками: **всего рефералов** (число
  зафиксированных связей) и **топ приглашающих** (несколько пользователей с наибольшим числом
  приглашённых). Прочее поведение `/stats` (авторизация по `ADMIN_USER_ID`, ответ не-админу) не
  меняется.

- **Бесплатность и профиль поставки.** Всё нативно (Telegram deep-link/share + локальная SQLite);
  **CF/LLM не используются**. Профиль поставки ADR-001 сохраняется; новых портов/внешних
  зависимостей/направлений трафика нет; добавляется ровно одна опциональная env
  (`ATTRIBUTION_FOOTER`, с дефолтом) и одна новая таблица в **той же** SQLite, что уже монтируется
  в volume (ADR-006).

### Вне рамок (explicitly out of scope)

- **Денежные/платные награды за рефералы.** Бот бесплатный, наград нет — только геймификация
  счётчиком/лидербордом (исходное требование).
- **Inline-режим обработки медиа (inline mode)** — отдельная история, в этот релиз не входит.
- **Антифрод рефералов сверх правила «новый пользователь + не сам».** Дополнительные проверки
  (мультиаккаунты, накрутка) — позже при необходимости.
- **Многоуровневые/каскадные рефералы** (реферал реферала, «дерево» приглашений) — учитывается
  только прямая связь «привёл → приглашённый».
- **Изменение логики распознавания, структурирования, действий и базовой отдачи результата** по
  существу (ADR-004/005/007/008/009): подпись добавляется поверх уже сформированного результата в
  одно-сообщенческом режиме, пайплайн не переделывается.
- **Изменение профиля поставки** (ADR-001) в части long-polling/портов/внешних зависимостей.
- **Использование Cloudflare/LLM** для любой из механик — затрат не вводим.
- **Расширение `/stats` сверх двух новых метрик** (например, временные срезы рефералов, конверсия,
  графики) — в этот релиз не входит.

## 3. Действующие лица и сценарии использования

**Конечный пользователь (приглашающий).** Цель — привести знакомых в бота и видеть результат.

- **Сценарий A (получение своей реф-ссылки).** Пользователь отправляет `/start` → в ответе видит
  свою персональную ссылку `https://t.me/<bot>?start=ref_<его_id>` и счётчик «приглашено: N».
- **Сценарий B (шеринг в один тап).** Пользователь нажимает inline-кнопку «Поделиться ботом» →
  открывается нативный экран Telegram «Поделиться» с подставленной его реф-ссылкой и коротким
  текстом → выбирает чат/контакт и отправляет.
- **Сценарий C (рост счётчика).** Знакомый перешёл по реф-ссылке и впервые запустил бота → у
  приглашающего при следующем `/start` счётчик «приглашено» увеличивается на 1 (один приглашённый
  засчитывается один раз).

**Приглашённый пользователь.** Цель — начать пользоваться ботом.

- **Сценарий D (приход по реф-ссылке, новый пользователь).** Человек переходит по
  `?start=ref_<id>` и впервые запускает бота → бот фиксирует связь «id привёл его», отвечает
  обычным приветствием `/start` (с его собственной реф-ссылкой и счётчиком) — реферал прозрачен,
  ничего дополнительно не требуется.
- **Сценарий E (повторный /start у уже известного пользователя).** Пользователь, который уже
  обращался к боту, переходит по чьей-то реф-ссылке или повторно жмёт `/start` с `ref_`-параметром
  → связь **не** создаётся и не меняется (учёт идемпотентен; реферал — только для нового
  пользователя при первом обращении).
- **Сценарий F (самоприглашение).** Пользователь переходит по **своей** реф-ссылке
  (`ref_<own_id>`) → связь не создаётся (самоприглашение игнорируется).
- **Сценарий G (обычный /start без реф-параметра).** Пользователь запускает бота без deep-link
  или со start-параметром, не похожим на `ref_<id>` → бот работает как раньше; реферальная связь
  не создаётся; пользователь всё равно видит свою реф-ссылку и счётчик.

**Любой пользователь (получатель результата).** Цель — получить распознанный текст.

- **Сценарий H (подпись под коротким результатом).** Пользователь прислал медиа; результат
  помещается в одно сообщение → под результатом присутствует строка `— @YourTextifyBot` (если
  `ATTRIBUTION_FOOTER` включён). Пересланный/скопированный текст несёт имя бота.
- **Сценарий I (длинный результат — без подписи).** Результат длинный и уходит серией сообщений
  или файлом `.txt` → подпись **не добавляется**; разбиение/файл по ADR-007 работают как прежде.

**Владелец продукта / администратор.** Цель — видеть эффект виральной петли.

- **Сценарий J (реферальные метрики в /stats).** Администратор отправляет `/stats` → в сводке,
  помимо существующих метрик (ADR-006), видит «всего рефералов» и «топ приглашающих».

**Эксплуатант.** Цель — включить/выключить подпись без передеплоя кода.

- **Сценарий K (управление подписью через env).** Эксплуатант задаёт `ATTRIBUTION_FOOTER=off` в
  `.env` и перезапускает контейнер → подпись перестаёт добавляться; `on`/незаданное (дефолт) —
  подпись добавляется.

## 4. Функциональные требования

- **ФТ-1 (персональная реф-ссылка).** Для каждого пользователя бот формирует персональную ссылку
  вида `https://t.me/<username_бота>?start=ref_<user_id>`. Username бота определяется из
  `Bot.get_me()` (как в v1.2.0), **не хардкодится**.
- **ФТ-2 (разбор start-параметра в /start).** При обработке `/start` бот извлекает start-параметр
  (часть после `?start=` в deep-link). Если параметр соответствует формату `ref_<id>`, где `<id>`
  — целочисленный Telegram id, бот переходит к проверке условий фиксации реферала (ФТ-3).
- **ФТ-3 (фиксация реферальной связи).** Связь «кто кого привёл» (`referrer_id`, `referred_id`,
  время) фиксируется **тогда и только тогда**, когда одновременно: (а) start-параметр — валидный
  `ref_<id>`; (б) нажавший пользователь — **новый** (первое обращение к боту); (в) `id != user_id`
  нажавшего (не самоприглашение). При невыполнении любого условия связь не создаётся.
- **ФТ-4 (идемпотентность реферала).** Один приглашённый (`referred_id`) учитывается **ровно один
  раз**: после фиксации его связи повторные `/start` (с тем же или другим `ref_`-параметром) **не**
  создают новую связь и **не** переписывают существующую. Самоприглашение (`id == user_id`)
  игнорируется всегда.
- **ФТ-5 (хранение рефералов в существующей SQLite).** Реферальные связи хранятся в **новой
  таблице** в той же SQLite-БД, что и статистика (ADR-006), расположенной в примонтированном
  volume, и **переживают пересоздание контейнера** при передеплое. Доступ к БД — **неблокирующий**
  (вне event loop, через `asyncio.to_thread`, преемственность ADR-002/003/006); сбой записи
  реферала **не прерывает** обработку `/start` (тихая деградация, преемственность ADR-006).
- **ФТ-6 (счётчик «приглашено: N» в /start).** В ответе на `/start` бот показывает пользователю
  его персональную реф-ссылку (ФТ-1) и счётчик **«приглашено: N»** — число уникальных
  пользователей, пришедших по его ссылке (число связей с `referrer_id = user_id`).
- **ФТ-7 (кнопка «Поделиться ботом»).** Бот показывает inline-кнопку **«Поделиться ботом»**,
  ведущую на нативный экран Telegram `https://t.me/share/url?...` с **персональной реф-ссылкой
  нажавшего** (его `ref_<user_id>`) и коротким сопроводительным текстом. Кнопка показывается
  **минимум в ответе `/start`**; размещение под результатом распознавания — опционально (ОВ-1).
- **ФТ-8 (подпись под одно-сообщенческим результатом).** К результату распознавания добавляется
  строка-подпись `— @YourTextifyBot` (имя бота из `Bot.get_me()`) **только когда результат уходит
  одним сообщением**. При разбиении результата на серию сообщений или переходе на файл `.txt`
  (ADR-007) подпись **не добавляется**.
- **ФТ-9 (управление подписью через env).** Добавление подписи управляется переменной окружения
  **`ATTRIBUTION_FOOTER`**: дефолт **on** (подпись добавляется); значение «off» отключает подпись.
  Переменная читается в `config.py`, добавляется плейсхолдером в `.env.example` и
  `deploy/.env.example`; **не хардкодится**.
- **ФТ-10 (подпись не ломает отдачу).** Добавление подписи не должно ломать существующую логику
  `send_result` (ADR-007): сплиттер, переход на файл, плейсхолдер длины и существующие inline-кнопки
  «Кратко»/«Перевести» (ADR-009) продолжают работать корректно; в одно-сообщенческом режиме итоговый
  текст с подписью остаётся в пределах лимита одного сообщения (учёт длины подписи при проверке
  границы одиночного сообщения — за реализацией, ОВ-3).
- **ФТ-11 (реферальные метрики в /stats).** Команда `/stats` (для администратора по `ADMIN_USER_ID`,
  ADR-006) дополнительно выводит: **всего рефералов** (общее число зафиксированных связей) и **топ
  приглашающих** (несколько пользователей с наибольшим числом приглашённых; точный формат и размер
  топа — за реализацией, ОВ-4). Существующие метрики и авторизация `/stats` не меняются.
- **ФТ-12 (сохранение поведения /start и распознавания).** Помимо добавления реф-ссылки, счётчика
  и кнопки «Поделиться ботом», `/start` сохраняет прежнее приветствие/назначение; пайплайн
  распознавания, структурирования и базовая отдача результата (ADR-004/005/007/008/009) по существу
  не меняются — подпись добавляется поверх готового результата в одно-сообщенческом режиме.
- **ФТ-13 (приватность).** В таблице рефералов хранятся только Telegram id (`referrer_id`,
  `referred_id`) и время связи — без содержимого сообщений и иных персональных данных
  (преемственность приватности ADR-006).

## 5. Нефункциональные требования

- **НФТ-1 (бесплатность по построению).** Ни одна из трёх механик не использует Cloudflare/LLM и
  не вводит платных путей: deep-link и `t.me/share/url` — нативный Telegram; рефералы — локальная
  SQLite. Затрат и влияния на free-tier нет.
- **НФТ-2 (преемственность поставки).** Сохраняется профиль поставки «бот без HTTP-порта»
  (ADR-001): long-polling, без `ports:`/Traefik/`edge`, образ из GHCR, `restart: unless-stopped`,
  `extra_hosts`-пиннинг Telegram, volume данных (ADR-006). Релиз **не вводит** новых внешних
  зависимостей и новых направлений исходящего трафика; добавляется ровно одна опциональная env
  (`ATTRIBUTION_FOOTER`, с дефолтом) и одна новая таблица в существующей SQLite.
- **НФТ-3 (неблокирование / отзывчивость).** Операции с таблицей рефералов (запись связи в
  `/start`, чтение счётчика и метрик `/stats`) выполняются вне event loop (через
  `asyncio.to_thread`, преемственность ADR-002/003/006); бот остаётся отзывчивым.
- **НФТ-4 (устойчивость учёта).** Сбой записи/чтения реферальных данных **не должен ломать**
  основную функцию: `/start` отвечает пользователю, распознавание работает; при ошибке БД счётчик
  деградирует (например, «приглашено: 0» или скрывается) без падения бота (преемственность
  политики тихой деградации ADR-006).
- **НФТ-5 (персистентность).** Реферальные связи сохраняются между передеплоями за счёт размещения
  в той же SQLite в примонтированном volume (ADR-006). Потеря накопленных связей при штатном
  релизе недопустима.
- **НФТ-6 (идемпотентность и корректность счёта).** Счётчик «приглашено» и метрика «всего
  рефералов» считают **уникальные** связи; повторные `/start` не должны накручивать счётчики
  (следствие ФТ-4); самоприглашения не входят в счёт.
- **НФТ-7 (приватность / минимизация данных).** В таблице рефералов — только Telegram id и время
  (ФТ-13); содержимое сообщений и распознанный текст не сохраняются (преемственность НФТ-1
  ADR-006).
- **НФТ-8 (безопасность доступа к метрикам).** Реферальные метрики `/stats` доступны только
  администратору (`ADMIN_USER_ID`, ADR-006); не-админу статистика (включая реферальную) не
  раскрывается.
- **НФТ-9 (локализация интерфейса).** Все новые пользовательские строки (текст про реф-ссылку,
  «приглашено: N», подпись кнопки «Поделиться ботом», сопроводительный текст шеринга, реферальные
  строки `/stats`) — на **русском**, в стиле существующих сообщений. Сама подпись `— @YourTextifyBot`
  — латиница (имя бота). Реф-код `ref_<id>` и URL — служебные, не локализуются.
- **НФТ-10 (управление конфигурацией).** `ATTRIBUTION_FOOTER` задаётся только через
  `.env`/окружение с дефолтом on; в репозиторий коммитятся лишь плейсхолдеры в
  `.env.example`/`deploy/.env.example` (соответствует разделу «Секреты»/конфигурации `CLAUDE.md`).
- **НФТ-11 (наблюдаемость).** Факт фиксации реферала и (при наличии) сбои записи видны в
  `docker logs` (преемственность ADR-010/006); секреты в логи не пишутся.
- **НФТ-12 (сопровождаемость).** Работа с таблицей рефералов локализована в сервисе статистики
  (`services/stats.py`) или соседнем сервисе; разбор start-параметра, формирование реф-ссылки и
  кнопки «Поделиться» — на уровне хендлеров/клавиатур (граница «хендлер ↔ сервис» ADR-002/003);
  добавление подписи — в единой точке отдачи `services/reply.py` (ADR-007), без дублирования.

## 6. Ограничения и допущения

Ограничения:
- **ОГР-1 (Telegram deep-link / start-параметр).** Реферальный механизм опирается на штатный
  Telegram deep-link `?start=<param>`: параметр доставляется боту только при запуске через ссылку
  и подчиняется ограничениям Telegram на формат/длину start-параметра (только латиница, цифры,
  `_` и `-`, до 64 символов) — формат `ref_<id>` этому удовлетворяет.
- **ОГР-2 («новый пользователь» определяется собственным учётом).** Признак «новый пользователь»
  (первое обращение) опирается на собственное хранилище (ADR-006: `first_seen`/наличие записи).
  Пользователи, обращавшиеся к боту **до** внедрения учёта или **до** этого релиза, для целей
  рефералов могут считаться «уже известными» — ретроспективной фиксации рефералов нет.
- **ОГР-3 (подпись — только одно-сообщенческий режим).** Подпись принципиально добавляется лишь
  когда результат помещается в одно сообщение (ADR-007): для серии/файла источник целостности
  отдачи (v0.7.0) важнее, и подпись не добавляется. Это сознательное ограничение, согласованное с
  ограничением кнопок действий из ADR-009 (markup только при одиночном сообщении).
- **ОГР-4 (профиль поставки и границы).** Сохраняются профиль поставки (ADR-001), границы
  «хендлер ↔ сервис» и неблокирование (ADR-002/003); новых портов, внешних БД/Redis, новых
  направлений трафика нет; используется существующий volume и существующая SQLite (ADR-006).
- **ОГР-5 (без CF/LLM).** Ни одна механика не вызывает Cloudflare/LLM (исходное требование);
  затрат не вводим.
- **ОГР-6 (username бота).** Реф-ссылка и подпись используют username/имя бота из `Bot.get_me()`
  (как в v1.2.0); хардкодинг username запрещён.

Допущения:
- **ДОП-1.** Релизы v0.1.0–v1.2.0 в проде на ivahub; существующая SQLite в volume (ADR-006), учёт
  через middleware (ADR-006), `/start` и команды (`handlers/commands.py`), единая отдача
  `send_result` (ADR-007), inline-кнопки и `handlers/actions.py` (ADR-009), получение username из
  `Bot.get_me()` (v1.2.0) — работают; релиз **переиспользует** их.
- **ДОП-2.** Объёмы малы (единицы активных пользователей); новая таблица рефералов в той же SQLite
  адекватна нагрузке; выбор конкретной схемы таблицы, индексов и SQL-запросов для метрик — задача
  архитектора (ОВ-2, ОВ-4).
- **ДОП-3.** Граничные параметры (кнопка «Поделиться» под результатом или только в `/start`; точная
  схема таблицы рефералов; размер и формат «топ приглашающих»; учёт длины подписи при проверке
  границы одного сообщения; формулировки русских строк и текста шеринга) **делегированы
  архитектору/реализации** — BA это не блокирует (ОВ-1…ОВ-5).
- **ДОП-4.** Образ обновляется и деплоится тем же CI/CD на ivahub (ADR-001); релиз не требует
  новых обязательных секретов. Единственная новая env (`ATTRIBUTION_FOOTER`) — опциональна, с
  дефолтом on, добавляется в `.env.example`/`deploy/.env.example`, реальные значения не коммитятся.
- **ДОП-5.** Новая таблица рефералов создаётся в существующей БД при старте (по образцу
  инициализации схемы ADR-006); добавление таблицы не требует ручной миграции данных и не ломает
  существующую таблицу `user_stats`.
- **ДОП-6.** Релиз — **MINOR (v1.3.0)**: обратимо-совместимое добавление функциональности;
  публичный контракт 1.0.0 (ADR-010) не ломается.

## 7. Критерии приёмки

- **КП-1.** В ответе на `/start` пользователь видит свою персональную реф-ссылку
  `https://t.me/<bot>?start=ref_<его_id>` (username из `Bot.get_me()`, не захардкожен) и счётчик
  «приглашено: N». (ФТ-1, ФТ-6, ОГР-6)
- **КП-2.** Переход нового пользователя по `?start=ref_<id>` (где `id` — другой существующий
  пользователь) и первый запуск бота фиксирует связь «id привёл его» в таблице рефералов
  (`referrer_id`, `referred_id`, время). (ФТ-2, ФТ-3, ФТ-5)
- **КП-3.** Реферал идемпотентен: повторный `/start` приглашённого (с тем же/другим `ref_`) не
  создаёт новую связь и не переписывает существующую; один приглашённый учитывается один раз.
  (ФТ-4, НФТ-6)
- **КП-4.** Самоприглашение (`ref_<own_id>`) не создаёт связи. (ФТ-3, ФТ-4)
- **КП-5.** Реферал фиксируется только для **нового** пользователя: переход по реф-ссылке уже
  известного боту пользователя связь не создаёт. (ФТ-3)
- **КП-6.** Счётчик «приглашено: N» в `/start` равен числу уникальных пользователей, пришедших по
  ссылке этого пользователя; после прихода нового приглашённого счётчик увеличивается на 1. (ФТ-6,
  НФТ-6)
- **КП-7.** В `/start` присутствует inline-кнопка «Поделиться ботом», ведущая на
  `https://t.me/share/url?...` с персональной реф-ссылкой нажавшего; нажатие открывает нативный
  экран «Поделиться» Telegram. (ФТ-7)
- **КП-8.** Если кнопка «Поделиться ботом» размещена под результатом (опция ОВ-1), она не ломает
  существующие кнопки «Кратко»/«Перевести» (ADR-009) и правило «markup только при одиночном
  сообщении» (ADR-007/009). (ФТ-7, ФТ-10)
- **КП-9.** Под одно-сообщенческим результатом распознавания присутствует подпись
  `— @YourTextifyBot` (при `ATTRIBUTION_FOOTER` on); при разбиении на серию сообщений и при файле
  `.txt` подпись отсутствует. (ФТ-8, ОГР-3)
- **КП-10.** Подпись не ломает `send_result`: сплиттер, переход на файл и существующие кнопки
  работают как прежде; одно-сообщенческий результат с подписью укладывается в лимит одного
  сообщения. (ФТ-10)
- **КП-11.** `ATTRIBUTION_FOOTER=off` отключает подпись; значение on/незаданное (дефолт) —
  включает; переменная присутствует плейсхолдером в `.env.example` и `deploy/.env.example`, не
  захардкожена. (ФТ-9, НФТ-10)
- **КП-12.** `/stats`, вызванная администратором, дополнительно выводит «всего рефералов» и «топ
  приглашающих»; существующие метрики `/stats` сохранены. (ФТ-11)
- **КП-13.** Реферальные метрики не раскрываются не-администратору: `/stats` для не-админа ведёт
  себя как прежде (ADR-006), реферальная информация не показывается. (ФТ-11, НФТ-8)
- **КП-14.** В таблице рефералов хранятся только Telegram id и время связи; содержимое сообщений и
  распознанный текст нигде не сохраняются. (ФТ-13, НФТ-7)
- **КП-15.** После передеплоя (пересоздания контейнера) накопленные реферальные связи сохраняются
  (та же SQLite в volume), счётчики и метрики непрерывны. (ФТ-5, НФТ-5)
- **КП-16.** Операции с рефералами не блокируют event loop; сбой записи/чтения реферала не роняет
  бота и не прерывает `/start` и распознавание. (ФТ-5, НФТ-3, НФТ-4)
- **КП-17.** Ни одна механика не обращается к Cloudflare/LLM; профиль поставки сохранён
  (long-polling, без `ports:`/Traefik/`edge`, образ из GHCR, `restart: unless-stopped`, пиннинг
  Telegram, существующий volume); новых внешних зависимостей и направлений трафика нет. (НФТ-1,
  НФТ-2, ОГР-4, ОГР-5)
- **КП-18.** Новые пользовательские строки — на русском, в стиле существующих сообщений; имя бота в
  подписи/ссылке берётся из `Bot.get_me()`; факт фиксации реферала виден в `docker logs`, секреты
  не логируются. (НФТ-9, НФТ-11, ОГР-6)
- **КП-19.** Релиз обратимо-совместим (MINOR): публичный контракт 1.0.0 (ADR-010) не ломается;
  поведение распознавания/структурирования/действий не изменилось по существу. (ФТ-12, ДОП-6)

## 8. Открытые вопросы

Блокирующих вопросов нет. Все три механики и их условия (формат реф-ссылки, правило фиксации «новый
+ не сам», идемпотентность, подпись только в одно-сообщенческом режиме под управлением
`ATTRIBUTION_FOOTER`, реферальные метрики в `/stats`) заданы исходным требованием однозначно, а вся
инфраструктура (SQLite в volume и учёт ADR-006, `/start` и команды, `send_result` ADR-007,
inline-кнопки ADR-009, username из `Bot.get_me()`) уже существует и переиспользуется. Граничные
параметры явно делегированы архитектору/реализации. Оформление документа не блокируется. Ниже —
некритичные вопросы для `/adr` или приёмки.

- **ОВ-1 (архитектор, `/adr`).** Размещать ли кнопку «Поделиться ботом» только в `/start` или также
  под результатом распознавания. Под результатом она должна сосуществовать с кнопками
  «Кратко»/«Перевести» (ADR-009) и подчиняться правилу «markup только при одиночном сообщении»
  (ADR-007/009). По умолчанию — минимум в `/start`; под результатом — на усмотрение архитектора.
- **ОВ-2 (архитектор, `/adr`).** Конкретная схема таблицы рефералов (поля, ключи/уникальность по
  `referred_id` для идемпотентности, индексы, формат времени — UTC/ISO-8601 как в ADR-006), а также
  где разместить функции работы с рефералами (расширить `services/stats.py` или соседний сервис).
- **ОВ-3 (архитектор/реализация).** Как учитывать длину подписи при определении одно-сообщенческого
  режима: добавлять подпись до проверки границы `MAX_MESSAGE_LEN` (ADR-007) или после; гарантировать,
  что подпись не вытолкнет результат за лимит одного сообщения (иначе подпись не добавлять, оставив
  поведение ADR-007).
- **ОВ-4 (архитектор/реализация).** Формат и размер «топ приглашающих» в `/stats` (сколько позиций;
  как идентифицировать приглашающего — по `user_id` или с числом приглашённых; что показывать при
  отсутствии рефералов).
- **ОВ-5 (архитектор/реализация).** Точные русские формулировки новых строк (текст про реф-ссылку и
  счётчик в `/start`, подпись кнопки «Поделиться ботом», сопроводительный `text` для
  `t.me/share/url`, реферальные строки `/stats`) — в стиле существующих сообщений (НФТ-9).
- **ОВ-6 (архитектор, `/adr`).** Поведение реферала в **групповых** чатах (v1.2.0): deep-link
  `?start=ref_<id>` — это запуск в личке; в группе `/start` ведёт себя иначе. Базово реферал
  фиксируется при запуске бота в **личном** чате по deep-link; нужно подтвердить, что групповой
  `/start` (если есть) не создаёт некорректных связей.
- **ОВ-7 (проверяется при приёмке, не блокирует BA).** Подтверждение на проде ivahub: переход по
  реф-ссылке нового пользователя фиксирует связь и увеличивает счётчик; идемпотентность и
  игнор самоприглашения; подпись присутствует под коротким результатом и отсутствует у серии/файла;
  `ATTRIBUTION_FOOTER=off` отключает подпись; `/stats` показывает реферальные метрики админу;
  персистентность после передеплоя; отсутствие обращений к CF (КП-1…КП-18).

Противоречий действующим ADR нет. Релиз **сохраняет** ADR-001 (профиль поставки «бот без
HTTP-порта», расширяется лишь одной опциональной env и новой таблицей в существующей SQLite),
ADR-002/003 (границы и неблокирование), ADR-006 (учёт/SQLite в volume/`/stats`/`ADMIN_USER_ID` —
переиспользуются и расширяются реферальной таблицей и метриками), ADR-007 (`send_result` — подпись
добавляется в его одно-сообщенческом режиме, серия/файл не трогаются), ADR-009 (inline-кнопки —
кнопка «Поделиться» сосуществует с действиями, правило markup сохраняется), ADR-010 (публичный
контракт 1.0.0 — не ломается), ADR-012 (режим групп — учитывается в ОВ-6). CF/LLM (ADR-005/008/011)
данной задачей **не используются** — затрат нет. Публичный контракт 1.0.0 не ломается — релиз MINOR.
