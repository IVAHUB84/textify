# BA-требования 015: Прогрессивное раскрытие результата (суть → кратко → полностью), удаление «Перевести»

- **Источник:** `requirements/req-textify-015.md`
- **Дата:** 2026-06-23
- **Статус:** черновик

## 1. Контекст и бизнес-цель

Сейчас на любое голосовое/аудио/изображение бот сразу отдаёт **полный** структурированный
текст (результат `structure_text` — Markdown с заголовками и списками) и две inline-кнопки:
«Кратко» и «Перевести». На коротком голосовом или картинке это даёт «стену текста» в ответ
на простой запрос: пользователь видит много текста ещё до того, как понял, нужны ли ему
детали.

Бизнес-цель — снизить визуальный шум и дать пользователю **самому** выбирать глубину ответа:
сначала короткая суть, а детали — по запросу. Это улучшает воспринимаемое качество ответа,
особенно на мобильных, и убирает редко используемую функцию перевода, которая усложняет
интерфейс без явной ценности.

Стейкхолдеры: владелец продукта (заказчик задачи), конечные пользователи бота
[@YourTextifyBot](https://t.me/YourTextifyBot) в личных чатах и (потенциально) в группах.

## 2. Область (scope)

### В рамках задачи
- Заменить немедленную выдачу полного текста на **прогрессивное раскрытие** в три уровня
  детализации:
  - **Суть (превью)** — 1–2 предложения, показывается сразу, без действий пользователя.
  - **Кратко** — саммари по ключевым пунктам (текущее поведение кнопки «Кратко»), по кнопке.
  - **Полностью** — полный структурированный текст (текущий результат `structure_text`),
    по кнопке.
- Под превью-сутью разместить две inline-кнопки: **«Показать полностью»** и **«Кратко»**.
- Полностью убрать кнопку **«Перевести»** из выдачи (личка; при едином решении — и группы).
- Сохранить атрибуцию/подпись (`ATTRIBUTION_FOOTER`) в текущем виде.
- Сохранить прежнее поведение при нераспознанном медиа (нет речи / нет текста на картинке):
  служебное сообщение без превью и без кнопок.
- Переиспользовать существующую логику доставки длинного текста (`send_result`: разбиение
  на части / переход на файл при превышении лимита, ADR-007) для уровня «Полностью».

### Вне рамок (explicitly out of scope)
- Перевод в любом виде (кнопка или функция) — **убирается, не заменяется**. Удаление самой
  функции `translate` из `services/llm.py` (если она больше нигде не используется) — техническое
  решение в рамках реализации, отдельной продуктовой ценности не несёт.
- Пользовательские настройки уровня детализации по умолчанию — отложено.
- Кэширование результатов между разными медиа — отложено.
- «Красивое» форматирование Markdown через `parse_mode` — вне рамок (см. ADR-007).

## 3. Действующие лица и сценарии использования

**Пользователь в личном чате** отправляет голосовое/аудио/изображение. Его цель — быстро
понять содержание, не читая всё подряд.

- **Сценарий A — короткое медиа.** Пользователь шлёт голосовое на 10 секунд. Бот распознаёт,
  присылает суть в 1–2 предложениях и кнопки «Показать полностью» / «Кратко». Пользователю
  достаточно сути — он не нажимает ничего.
- **Сценарий B — нужны детали.** Пользователь хочет полный текст и нажимает «Показать
  полностью» — бот присылает полный структурированный текст (с разбиением/файлом при
  необходимости).
- **Сценарий C — нужно саммари.** Пользователь нажимает «Кратко» — бот присылает саммари по
  ключевым пунктам.
- **Сценарий D — медиа не распознано.** Речь не распознана / текста на картинке нет — бот
  присылает прежнее служебное сообщение, без превью и кнопок.
- **Сценарий E — группа** (v1.4.0). Пользователь нажимает «Распознать» под медиа в группе.
  Поведение выдачи определяется решением по охвату (см. п. 6 и п. 8). Минимум — кнопка
  «Перевести» в группах отсутствует.

## 4. Функциональные требования

- **ФТ-1.** При успешно распознанном медиа (личка) бот **первым** сообщением присылает
  «суть» — краткое изложение в 1–2 предложениях на **языке исходного текста** (язык не
  меняется).
- **ФТ-2.** Под сообщением с сутью отображаются ровно две inline-кнопки: «Показать полностью»
  и «Кратко».
- **ФТ-3.** Нажатие «Показать полностью» отдаёт пользователю **полный структурированный
  текст** (текущий результат `structure_text`).
- **ФТ-4.** Нажатие «Кратко» отдаёт **саммари по ключевым пунктам** (текущее поведение
  операции `summarize`), по объёму длиннее сути и короче полного текста.
- **ФТ-5.** Уровень «Полностью» доставляется через существующую логику `send_result`: при
  превышении лимита Telegram текст разбивается на части либо отдаётся файлом (ADR-007).
- **ФТ-6.** Кнопка «Перевести» отсутствует в выдаче бота (личка; при едином решении — и
  группы).
- **ФТ-7.** Если медиа не распознано (нет речи / нет текста на изображении), бот присылает
  прежнее служебное сообщение и **не** присылает превью и кнопки.
- **ФТ-8.** Атрибуция/подпись (`ATTRIBUTION_FOOTER`) сохраняется в текущем виде.
- **ФТ-9.** Три уровня детализации должны быть **явно различимы по объёму**: суть (1–2
  предложения) < кратко (ключевые пункты) < полностью (полный структурированный текст).
- **ФТ-10.** Поведение при невозможности сформировать ответ (LLM недоступен / ответ пуст /
  бюджет исчерпан) должно деградировать предсказуемо и не приводить к падению обработчика;
  пользователь получает понятное сообщение. Конкретная стратегия деградации для уровня «суть»
  зависит от решения архитектора по бюджету (см. п. 8).

## 5. Нефункциональные требования

- **НФТ-1 (производительность/латентность).** Прод — 1 vCPU / 2 GB, бот на long-polling.
  Время до первого ответа (суть) не должно ощутимо ухудшать UX по сравнению с текущей выдачей
  полного текста. Дополнительные LLM-вызовы не должны блокировать event loop (ADR-002/003).
- **НФТ-2 (бюджет LLM).** Прод использует Cloudflare Workers AI с дневным бюджетом
  (`CF_DAILY_BUDGET`, `services/budget.py`). Изменение не должно молча выводить бота за рамки
  бюджета; при исчерпании бюджета поведение должно деградировать осознанно (см. п. 8, ОВ-2).
- **НФТ-3 (локализация контента).** Суть, кратко и полностью формируются на языке исходного
  текста (русский/английский). Язык не меняется ни на одном уровне.
- **НФТ-4 (совместимость поставки).** Профиль поставки сохраняется: long-polling, без
  HTTP-портов/Traefik/edge (ADR-001). Новые env-переменные/зависимости/изменения Docker —
  только если их обоснует архитектор; по умолчанию не вводятся.
- **НФТ-5 (надёжность кнопок).** Обработчики кнопок должны корректно работать после удаления
  кнопки «Перевести» и при изменении источника текста (текст сообщения теперь — суть, а не
  полный текст). Недоступность исходного текста обрабатывается понятным сообщением, без
  падения.

## 6. Ограничения и допущения

**Ограничения:**
- **ОГР-1.** Прод: 1 vCPU / 2 GB, long-polling — без тяжёлого стейта и фоновых очередей
  (см. memory: deploy-server ivahub).
- **ОГР-2.** Лимит сообщения Telegram 4096 символов — для уровня «Полностью» обязательна
  логика `send_result` (ADR-007).
- **ОГР-3.** Дневной бюджет LLM (`CF_DAILY_BUDGET`) ограничивает число вызовов модели в сутки.
- **ОГР-4.** Cloudflare Workers AI — дефолтный провайдер; Groq гео-блокирует RU IP (см.
  memory: groq-geoblock). Любая логика, завязанная на LLM, должна учитывать дефолт CF и его
  бюджет.

**Допущения:**
- **ДОП-1.** «Суть» формируется отдельной LLM-операцией (новая операция генерации 1–2
  предложений), аналогично `structure_text`/`summarize`. Это **продуктовое** допущение из
  исходного требования; способ реализации — за архитектором.
- **ДОП-2.** Текущее поведение `summarize` (ключевые пункты) удовлетворяет уровню «Кратко» без
  изменения формулировок.
- **ДОП-3.** Уровень «Полностью» = текущий результат `structure_text` без изменения его
  содержания/формата.

## 7. Критерии приёмки

- **КП-1.** При успешно распознанном голосовом/аудио в личке первым приходит сообщение с
  сутью в 1–2 предложениях на языке оригинала. *(ФТ-1, НФТ-3)*
- **КП-2.** При успешно распознанном изображении в личке первым приходит сообщение с сутью в
  1–2 предложениях на языке оригинала. *(ФТ-1, НФТ-3)*
- **КП-3.** Под сообщением с сутью присутствуют ровно две кнопки: «Показать полностью» и
  «Кратко»; кнопки «Перевести» нет. *(ФТ-2, ФТ-6)*
- **КП-4.** Нажатие «Показать полностью» приводит к выдаче полного структурированного текста,
  совпадающего по содержанию с прежней немедленной выдачей. *(ФТ-3, ФТ-9)*
- **КП-5.** Нажатие «Кратко» приводит к выдаче саммари по ключевым пунктам (как текущая кнопка
  «Кратко»). *(ФТ-4, ФТ-9)*
- **КП-6.** Полный текст, превышающий лимит Telegram, доставляется частями или файлом без
  потери содержания. *(ФТ-5, ОГР-2)*
- **КП-7.** При нераспознанном медиа приходит прежнее служебное сообщение, без превью и
  кнопок. *(ФТ-7)*
- **КП-8.** Подпись/атрибуция присутствует там же, где и раньше. *(ФТ-8)*
- **КП-9.** Объёмы трёх уровней монотонно различимы: суть короче «кратко», «кратко» короче
  «полностью». *(ФТ-9)*
- **КП-10.** Кнопка «Перевести» отсутствует во всех затронутых сценариях выдачи; перевод
  недоступен пользователю. *(ФТ-6, scope)*
- **КП-11.** При исчерпанном дневном бюджете LLM бот не падает и выдаёт понятное сообщение
  (конкретный текст/уровень деградации фиксирует архитектор). *(ФТ-10, НФТ-2)*
- **КП-12.** Повторные/некорректные нажатия кнопок не приводят к падению обработчика;
  при недоступном исходном тексте показывается понятное сообщение. *(НФТ-5)*

## 8. Открытые вопросы

Блокирующих **продуктовых** вопросов нет — задачу можно проектировать. Ниже — вопросы,
делегированные **архитектору** на этапе ADR (технические) и один продуктовый вопрос с
зафиксированным дефолтом, не блокирующий переход.

- **ОВ-1 (архитектура).** Источник полного/исходного текста для кнопок. Сейчас
  callback-обработчики читают `callback.message.text`, но теперь текст сообщения — это суть.
  Где брать полный/исходный текст (пересборка из распознанного текста, предрасчёт, скрытый
  блок, лёгкий стейт/кэш) — с учётом ОГР-1 (long-polling, 1 vCPU/2 GB, без тяжёлого стейта).
  *Решает архитектор.*
- **ОВ-2 (архитектура/бюджет).** Дополнительный LLM-вызов на «суть» на каждое медиа сверх
  структурирования: влияние на `CF_DAILY_BUDGET` и стратегия деградации при исчерпании бюджета
  (что показывать вместо сути — полный текст? служебное сообщение? сразу кнопки без сути?).
  *Решает архитектор; продуктовое требование — деградация без падения и понятное сообщение
  (ФТ-10, КП-11).*
- **ОВ-3 (архитектура).** Идемпотентность кнопок: повторные тапы «Показать полностью» /
  «Кратко» — слать заново или нет. *Решает архитектор; с продуктовой стороны допустимы оба
  варианта при отсутствии падения (КП-12).*
- **ОВ-4 (продукт, дефолт зафиксирован).** Охват групп (v1.4.0, `handlers/group.py`).
  Применять ли прогрессивное раскрытие (суть + кнопки) и в группах, или оставить группам
  прежнюю выдачу. **Продуктовый дефолт BA:** в группах оставить прежнюю выдачу полного текста,
  но **обязательно убрать кнопку «Перевести»** ради единообразия (ФТ-6, КП-10). Полный перенос
  превью-сути в группы — отдельная итерация (экономит бюджет, упрощает релиз). Если владелец
  продукта хочет единый UX лички и групп сразу — это нужно подтвердить; до подтверждения
  действует дефолт. *Финальное решение по реализации охвата — за архитектором.*

Соответствия действующим ADR не нарушаются: уровень «Полностью» опирается на `send_result`
(ADR-007), профиль поставки — ADR-001, неблокирующие операции — ADR-002/003. Противоречий с
принятыми ADR не выявлено.
