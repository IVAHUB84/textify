# BA-требования 016: HTML-вывод, честное «Кратко», расширенное меню действий, персистентный кэш и устойчивость

- **Источник:** `requirements/req-textify-016.md`
- **Дата:** 2026-06-24
- **Статус:** черновик

> Документ оформлен **задним числом**: функциональность уже реализована и закоммичена прямыми
> коммитами после тега `v1.5.3` (`d322a39`, `9bf3a2d`, `b39b564`, `6feb45e`, `0a13011`, `fc42bc8`,
> `fbbc09d`). Цель — формализовать намерение и контракт, чтобы за этим документом архитектор

## 1. Контекст и бизнес-цель

После v1.5.0 (прогрессивное раскрытие «суть → кратко → полностью», ADR-015) накопился второй
раунд улучшений из ресёрча и из практики конкурирующих транскрипционных ботов. Бизнес-цель —
закрыть три разрыва единым MINOR-релизом:

- **Качество выдачи.** Структурированный текст шлётся без `parse_mode`, поэтому Markdown LLM
  (`####`, `**`) виден пользователю буквально как сырой текст. «Кратко» на слабой модели
  `llama-3.1-8b` раздувалось вместо сжатия — иногда длиннее полного текста. И то и другое снижает
  воспринимаемое качество продукта.
- **Паритет по возможностям.** Конкуренты дают тайм-коды, субтитры, перевод, извлечение задач и
  searchable PDF. У Textify этих действий нет (а «Перевести» был удалён в v1.5.0 — ADR-015).
- **Устойчивость.** Кнопки действий перестают работать после редеплоя и по истечении 1h TTL
  (in-memory кэш не переживает рестарт — ADR-015). Файлы крупнее лимита Telegram getFile и
  истёкший кэш роняют обработку молча, без понятного сообщения.

Стейкхолдеры: владелец продукта (заказчик задачи), конечные пользователи бота
[@YourTextifyBot](https://t.me/YourTextifyBot) в личных чатах и группах (v1.4.0/v1.5.0).

## 2. Область (scope)

### В рамках задачи

- **HTML-форматирование вывода.** Конвертировать Markdown от LLM в безопасное подмножество
  Telegram HTML (жирные заголовки, маркированные списки, экранирование `< > &`) и слать с
  `parse_mode=HTML`; при ошибке форматирования Telegram — деградировать в плоский текст.
  Промпт структурирования переучить на Telegram-формат (жирные заголовки, списки через дефис,
  экономная эмодзи) вместо `#`-заголовков Markdown.
- **Реальное сжатие «Кратко».** Промпт `summarize` переписать на действительно более короткую
  выжимку (2–5 тезисов, без деталей и повторов, никогда не длиннее источника), с few-shot на
  сжатие — по аналогии с «сутью».
- **Расширение меню действий** контекстными кнопками:
  - **Тайм-коды** — для длинного аудио построчные `[MM:SS]` на основе сегментов whisper.
  - **Субтитры** — стандартный `.srt` из сегментов whisper, отправляется документом.
  - **Перевести** — возврат функции перевода (RU↔EN, язык-источник определяет LLM).
    **Это реверс удаления из v1.5.0 (ADR-015).**
  - **Задачи** — извлечение action items / решений в чек-лист.
  - **PDF** — для фото/изображения-документа: searchable PDF (оригинал + невидимый текстовый
    слой Tesseract `rus+eng`), текст остаётся выделяемым и ищущимся.
- **Персистентный кэш результатов.** Опциональный SQLite-бэкенд кэша (write-through,
  load-on-start): кнопки действий продолжают работать после редеплоя и после истечения TTL.
  Без инициализации — чистый in-memory (тесты, неконфигурированные окружения).
- **Гард на превышение 20 МБ.** Хендлеры голоса/аудио/изображений отвечают понятным сообщением,
  когда файл превышает лимит Telegram getFile, вместо тихого падения на скачивании.
- **Убрать menu-кнопку команд.** Перестать регистрировать команды через `set_my_commands`, чтобы
  Telegram не показывал кнопку Menu у поля ввода; команды продолжают работать при ручном вводе.

### Вне рамок (explicitly out of scope)

- Новые внешние зависимости — не вводятся (PDF через уже установленный Tesseract `rus+eng`).
- Пользовательские настройки набора кнопок и языка перевода по умолчанию — отложено.
- Веб-дашборд / расширенная аналитика — вне рамок (как и прежде).
- Хранение байтов изображения в SQLite — вне рамок (только маленький in-memory-кэш).

## 3. Действующие лица и сценарии использования

**Пользователь в личном чате** отправляет голосовое/аудио/изображение, его цель — быстро получить
чистый текст и при желании выполнить дополнительное действие над ним.

- **Сценарий A — чистая выдача.** Пользователь получает структурированный текст с жирными
  заголовками и аккуратными списками (а не сырой Markdown с `####`/`**`).
- **Сценарий B — честное «Кратко».** Пользователь нажимает «Кратко» — получает выжимку короче
  источника (2–5 тезисов), а не «раздутый» пересказ.
- **Сценарий C — длинное аудио.** Под результатом аудио ≥ 60 с доступны «Тайм-коды» и «Субтитры»;
  по нажатию приходит список `[MM:SS]` или файл `.srt`.
- **Сценарий D — перевод/задачи.** Пользователь нажимает «Перевести» (получает перевод RU↔EN)
  или «Задачи» (получает чек-лист action items).
- **Сценарий E — фото-документ.** Под результатом фото доступна кнопка «PDF»; по нажатию приходит
  searchable PDF с выделяемым/ищущимся текстовым слоем.
- **Сценарий F — устойчивость кэша.** Пользователь нажимает кнопку через сутки или после
  редеплоя бота; при включённой персистентности действие работает, при недоступности данных —
  понятная деградация («Текст недоступен» / «устарело»), а не падение.
- **Сценарий G — крупный файл.** Пользователь шлёт файл > 20 МБ; бот отвечает понятным
  сообщением с подсказкой сжать/обрезать, а не молчит.
- **Пользователь в группе** (v1.4.0/v1.5.0) — поведение групп не должно ломаться; охват новых
  кнопок в группах определяет архитектор (см. ОВ-1).

## 4. Функциональные требования

- **ФТ-1.** Структурированный текст отправляется в Telegram HTML (`parse_mode=HTML`): заголовки —
  жирным, списки — маркированными, символы `< > &` экранированы; форматирование строится из
  Markdown-ответа LLM.
- **ФТ-2.** При ошибке Telegram на HTML-разметке (`TelegramBadRequest`) выдача деградирует в
  плоский текст без падения обработчика; пользователь всё равно получает содержание.
- **ФТ-3.** Промпт структурирования формирует Telegram-ориентированный формат (жирные заголовки,
  списки через дефис, экономная эмодзи), а не `#`-заголовки Markdown.
- **ФТ-4.** Результат «Кратко» (`summarize`) — выжимка из 2–5 тезисов, без деталей и повторов,
  **никогда не длиннее** исходного текста.
- **ФТ-5.** Для длинного аудио (порог — наличие сегментов транскрипции, ориентир ≥ 60 с) под
  результатом доступна кнопка «Тайм-коды»; по нажатию приходят построчные метки `[MM:SS]`,
  построенные из сегментов whisper.
- **ФТ-6.** Для длинного аудио доступна кнопка «Субтитры»; по нажатию приходит документ `.srt`,
  собранный из сегментов whisper по стандарту SRT.
- **ФТ-7.** Тайм-коды и субтитры берут сегменты из кэша по клику (а не из заранее сформированной
  строки); при отсутствии сегментов в кэше — понятная деградация («недоступны»), не падение.
- **ФТ-8.** Доступна кнопка «Перевести»: переводит распознанный текст RU↔EN, язык-источник
  определяет LLM. Функция перевода возвращается в продукт (реверс удаления v1.5.0 / ADR-015).
- **ФТ-9.** Доступна кнопка «Задачи»: извлекает action items / решения из текста в виде чек-листа.
- **ФТ-10.** Для фото и изображения-документа доступна кнопка «PDF»; по нажатию приходит документ
  searchable PDF (исходное изображение + невидимый текстовый слой Tesseract), текст в PDF можно
  выделять и искать. Байты исходного изображения берутся из in-memory-кэша; при их отсутствии —
  понятная деградация («Изображение недоступно / устарело»), не падение.
- **ФТ-11.** Перевод и задачи переиспользуют существующий кэш исходного текста и общий LLM-бюджет;
  при исчерпании дневного бюджета — понятное служебное сообщение, без падения (как «суть» в
  v1.5.0). При сбое/пустом ответе LLM — понятное служебное сообщение.
- **ФТ-12.** Кэш результатов поддерживает опциональную персистентность (SQLite, сквозная запись +
  загрузка на старте): при включении кнопки действий работают после редеплоя и после истечения
  in-memory TTL. Без инициализации кэш работает чисто в памяти (поведение v1.5.0 сохраняется).
- **ФТ-13.** Хендлеры голоса/аудио/изображений при файле, превышающем лимит Telegram getFile
  (20 МБ), отвечают единым понятным сообщением **до** попытки скачивания, вместо тихого падения.
- **ФТ-14.** Команды бота больше не регистрируются через `set_my_commands` — Telegram не
  показывает кнопку Menu у поля ввода. Команды (`/start`, `/help`) продолжают работать при ручном
  вводе; их текстовые описания/ответы сохранены.
- **ФТ-15.** Подсказки в fallback-превью (когда суть недоступна) указывают на кнопки ниже
  обобщённо, без перечисления конкретных названий действий.
- **ФТ-16.** Поведение групп (v1.4.0/v1.5.0) при новых изменениях не ломается; как минимум
  сохраняется прежняя выдача и работоспособность существующих кнопок.

## 5. Нефункциональные требования

- **НФТ-1 (память на 1 vCPU / 2 GB).** Прод — ivahub, 1 vCPU / 2 GB, long-polling. Кэш байтов
  изображений ограничен (порядка 50 элементов, только память, без SQLite); персистентный
  SQLite-кэш текста/сегментов не должен заметно раздувать RSS. Объёмы — под целевое железо.
- **НФТ-2 (неблокирование).** Новые LLM-вызовы (`translate`, `extract_tasks`) и сборка PDF не
  блокируют event loop (LLM — async httpx; локальная сборка PDF — вне основного цикла, как OCR).
  Тяжёлые операции сопровождаются индикатором прогресса (ADR-010).
- **НФТ-3 (бюджет LLM).** Все новые LLM-операции учитывают дневной бюджет CF
  (`CF_DAILY_BUDGET`) и осознанно деградируют при исчерпании, не выводя бота за рамки бюджета
  молча (ОГР по ADR-011, преемственно «сути» v1.5.0).
- **НФТ-4 (локализация контента).** Структурирование, «Кратко», тайм-коды, субтитры, задачи
  формируются на языке исходного текста (русский/английский). Перевод — единственная операция,
  меняющая язык (RU↔EN, направление выбирает LLM по тексту).
- **НФТ-5 (совместимость поставки).** Профиль поставки сохраняется: long-polling, без
  HTTP-портов/Traefik/edge (ADR-001). Новые внешние зависимости не вводятся. Персистентность
  кэша — локальный SQLite-файл рядом с существующим стором учёта (v0.6.0), без сетевых стора/БД.
- **НФТ-6 (надёжность кнопок).** При недоступности данных в кэше (TTL/вытеснение/рестарт без
  персистентности) обработчики кнопок дают понятное сообщение, не падают; повторные/некорректные
  нажатия не приводят к падению.
- **НФТ-7 (формат вывода).** HTML-форматирование — изменение **представления** немедленной
  выдачи, совместимое: входные каналы, команды и логика доставки длинного текста (`send_result`,
  ADR-007) не меняются; при невозможности применить HTML гарантирована деградация в плоский текст
  (ФТ-2).

## 6. Ограничения и допущения

**Ограничения:**

- **ОГР-1.** Прод ivahub: 1 vCPU / 2 GB, long-polling — без тяжёлого стейта и фоновых очередей
  (memory: deploy-server ivahub).
- **ОГР-2.** Лимит Telegram getFile для ботов — 20 МБ; файлы крупнее не скачать (ФТ-13).
- **ОГР-3.** Лимит сообщения Telegram 4096 символов — длинные ответы доставляются через
  `send_result` (части/файл, ADR-007).
- **ОГР-4.** Дневной бюджет LLM (`CF_DAILY_BUDGET`) ограничивает число вызовов модели; CF Workers
  AI — дефолтный провайдер (Groq гео-блокирует RU IP — memory: groq-geoblock).
- **ОГР-5.** Слабая модель `llama-3.1-8b` требует few-shot для надёжного сжатия/перевода/задач.
- **ОГР-6.** Telegram HTML поддерживает только ограниченное подмножество тегов — форматтер обязан
  оставаться в безопасном подмножестве и экранировать спецсимволы.
- **ОГР-7.** Лимит `callback_data` 64 байта — короткие коды кнопок (сохраняется подход v1.5.0).

**Допущения:**

- **ДОП-1.** Whisper отдаёт сегменты с таймингами — основа для тайм-кодов и субтитров; сегменты
  кладутся в кэш при распознавании и читаются по клику.
- **ДОП-2.** Searchable PDF собирается средствами уже установленного Tesseract (`rus+eng`) без
  новых зависимостей; исходные байты изображения доступны из in-memory-кэша на момент клика.
- **ДОП-3.** Персистентный SQLite-кэш соседствует со стором учёта (v0.6.0): имя файла, схема
  таблицы, TTL-чистка и поведение при отсутствии конфигурации — детали реализации/архитектуры.
- **ДОП-4.** Перевод (`translate`) восстанавливается как функция сервиса; способ реализации
  совпадает с прежним подходом (LLM определяет направление по тексту).

## 7. Критерии приёмки

- **КП-1.** Структурированный текст приходит с жирными заголовками и маркированными списками
  (HTML), а не как сырой Markdown (`####`, `**`); спецсимволы `< > &` не ломают сообщение.
  *(ФТ-1, ФТ-3)*
- **КП-2.** При некорректной HTML-разметке сообщение всё равно доставляется в виде плоского
  текста, обработчик не падает. *(ФТ-2, НФТ-7)*
- **КП-3.** Ответ «Кратко» короче исходного текста и состоит из 2–5 тезисов без повторов.
  *(ФТ-4)*
- **КП-4.** Для длинного аудио под результатом есть кнопка «Тайм-коды»; по нажатию приходят
  построчные метки `[MM:SS]`, соответствующие сегментам. *(ФТ-5, ФТ-7)*
- **КП-5.** Для длинного аудио есть кнопка «Субтитры»; по нажатию приходит валидный документ
  `.srt`. *(ФТ-6, ФТ-7)*
- **КП-6.** Кнопка «Перевести» присутствует и переводит распознанный текст RU↔EN. *(ФТ-8)*
- **КП-7.** Кнопка «Задачи» присутствует и формирует чек-лист action items из текста. *(ФТ-9)*
- **КП-8.** Для фото/изображения-документа есть кнопка «PDF»; по нажатию приходит searchable PDF
  с выделяемым и ищущимся текстом. *(ФТ-10)*
- **КП-9.** При исчерпании дневного бюджета LLM любая LLM-операция (кратко/перевод/задачи) даёт
  понятное служебное сообщение и не роняет бота; при сбое LLM — также понятное сообщение.
  *(ФТ-11, НФТ-3)*
- **КП-10.** При включённой персистентности кнопки действий работают после редеплоя и после
  истечения in-memory TTL; без инициализации кэш работает чисто в памяти. *(ФТ-12, НФТ-5)*
- **КП-11.** При недоступности данных в кэше (тайм-коды/субтитры/PDF/текст) приходит понятное
  сообщение о недоступности, без падения обработчика. *(ФТ-7, ФТ-10, НФТ-6)*
- **КП-12.** Файл > 20 МБ (голос/аудио/изображение) приводит к единому понятному сообщению до
  скачивания, а не к тихому отсутствию ответа. *(ФТ-13, ОГР-2)*
- **КП-13.** Кнопка Menu у поля ввода в Telegram не отображается; команды `/start` и `/help`
  по-прежнему работают при ручном вводе. *(ФТ-14)*
- **КП-14.** Подсказка fallback-превью указывает на кнопки ниже обобщённо, без перечисления
  конкретных названий действий. *(ФТ-15)*
- **КП-15.** Поведение групп (v1.4.0/v1.5.0) не сломано: прежняя выдача и существующие кнопки
  работают. *(ФТ-16)*

## 8. Открытые вопросы

Блокирующих **продуктовых** вопросов нет — функциональность реализована, документ
ретроспективный. Ниже — вопросы и наблюдения, делегированные **архитектору** на этап ADR.

- **ОВ-1 (продукт/архитектура — наблюдение по факту).** В текущей реализации кнопка «Перевести»
  присутствует **и в личке, и в групповой** клавиатуре (`actions_keyboard`), то есть реверс
  ADR-015 шире, чем «вернуть кнопку в личке». Исходное требование описывает перевод как кнопку
  меню действий без явного ограничения охватом групп. Архитектору: зафиксировать охват «Перевести»
  (и прочих новых кнопок) в группах явно в follow-up ADR; при необходимости — подтвердить у
  владельца продукта. До подтверждения действует фактическое поведение (кнопка доступна в обоих
  контекстах).
- **ОВ-2 (контракт вывода, ADR-010).** HTML-форматирование меняет формат немедленной выдачи —
  раздел публичного контракта 1.0.0 (ADR-010, надстройка ADR-007 «plain-text без `parse_mode`»).
  Зафиксировать как **совместимое** изменение представления (MINOR), с гарантией деградации в
  плоский текст (ФТ-2). *Решает архитектор.*
- **ОВ-3 (реверс ADR-015).** Follow-up ADR должен **явно** пометить решение ADR-015 об удалении
  перевода как replaced/superseded, а не молча ему противоречить. *Решает архитектор.*
- **ОВ-4 (персистентность кэша).** SQLite-бэкенд `result_cache` соседствует со стором учёта
  (v0.6.0): зафиксировать имя файла/таблицы, схему, TTL-чистку и поведение при отсутствии
  конфигурации. *Решает архитектор.*
- **ОВ-5 (память на 1 vCPU / 2 GB).** Оценить RSS-влияние in-memory-кэша изображений (порог ~50)
  и SQLite-кэша текста/сегментов на целевом железе; зафиксировать лимиты. *Решает архитектор.*
- **ОВ-6 (единая точка проверки 20 МБ).** Единое сообщение и точка проверки (до скачивания) для
  трёх хендеров (голос/аудио/изображение). *Решает архитектор (реализация уже даёт общий хелпер
  `handlers/limits.py`).*

Противоречий с принятыми ADR, кроме сознательного реверса ADR-015 (ОВ-3) и эволюции формата
вывода относительно ADR-007/010 (ОВ-2), не выявлено.
