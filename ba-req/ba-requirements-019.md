# BA-требования 019: «Скан»-качество видимой страницы в PDF из фото

- **Источник:** `requirements/req-textify-019.md`
- **Дата:** 2026-06-25
- **Статус:** черновик

## 1. Контекст и бизнес-цель

Textify — бесплатный Telegram-бот, превращающий голос/аудио и изображения в чистый
структурированный текст (см. `README.md`, `CLAUDE.md`). С релиза v1.6.0 (BA-016, ADR-016) под
результатом распознавания фото/изображения-документа доступна кнопка **«📄 PDF»**, которая отдаёт
**searchable PDF**: исходное изображение + невидимый текстовый слой от Tesseract, благодаря которому
текст в PDF можно выделять и искать.

**Проблема.** В видимую страницу PDF вшивается **сырое фото как есть** — перекошенное, с тенями,
цветным фоном стола, перспективой (`services/ocr.py::_run_ocr_pdf` передаёт оригинал в
`pytesseract.image_to_pdf_or_hocr`). При этом очищенное изображение в проекте **уже строится**
(`services/preprocess.py::preprocess_image`: EXIF-поворот, ресайз, бинаризация, deskew), но
используется **только** для извлечения текста OCR и в видимую страницу PDF не попадает. Визуальный
результат кнопки практически не отличается от пересланного снимка, что обесценивает фичу: главная
ценность «PDF из фото» — отдать **аккуратный документ**, а не фото на столе.

**Цель релиза.** Рендерить видимую страницу PDF из **очищенного** изображения (выравнивание
перекоса, подъём контраста / убирание теней, ресайз к разумному размеру), добавить режимы вывода
(цвет / ч-б) и JPEG-сжатие страницы, при **обязательной деградации к оригиналу** при любом сбое
обработки. Очистка уже вычисляется ради OCR — отдать её пользователю в видимом слое почти «бесплатно»
по архитектуре.

**Стейкхолдеры:**
- **Владелец продукта** (заказчик; администратор бота) — цель: повысить воспринимаемую ценность
  фичи «PDF из фото», результат должен выглядеть как из сканер-приложения.
- **Конечный пользователь бота**, отправляющий фото документа и нажимающий «📄 PDF», — получает
  аккуратную, читаемую страницу вместо сырого снимка.
- **Команда разработки** (последующие агенты) — переиспользует существующий пайплайн очистки
  (`preprocess.py`) для рендера, не вводя новой тяжёлой CV-нагрузки.

**Это первый из двух релизов** улучшения PDF (вариант A+D из требования). Многостраничный PDF из
альбома (вариант B) — следующий слой, отдельное требование (см. раздел 2 «Вне рамок»).

## 2. Область (scope)

### В рамках задачи

- **Рендер видимой страницы PDF из очищенного изображения.** К изображению применяется пайплайн
  очистки **до** вшивания в PDF: EXIF-поворот → (опц.) ресайз к разумному размеру/DPI →
  выравнивание перекоса (deskew) → подъём контраста / убирание теней. Текстовый слой PDF —
  по-прежнему от OCR (Tesseract).
- **Согласованность текстового слоя с видимой страницей.** Координаты невидимого текстового слоя
  должны совпадать с видимой (повёрнутой/очищенной) страницей, чтобы выделение текста в PDF
  попадало в слова. Способ обеспечения согласованности — за архитектором (ОВ-4).
- **Режимы вывода страницы:**
  - **«Документ (цвет/серый)»** — выровненная, контрастная, но цветная или серая страница.
    **Разумный дефолт.**
  - **«Скан (ч/б)»** — бинаризованная страница (классический скан-вид, компактнее).
  - Конкретный UX выбора режима (одна кнопка с дефолтом + альтернатива, две кнопки, или выбор
    после нажатия) и итоговый дефолт — за архитектором (ОВ-1).
- **JPEG-сжатие видимой страницы** (вариант D): страница вшивается как JPEG с ресайзом, чтобы
  итоговый PDF был **меньше**, чем сегодня.
- **Обязательная деградация к оригиналу.** Если очистка/обработка падает или даёт некорректный
  результат — вшивается оригинальное изображение (текущее поведение v1.6.0). Результат не должен
  быть **хуже** сегодняшнего ни на одном входе.
- **«Не трогать ровное».** Если изображение уже ровное/контрастное — не ухудшать его лишней
  обработкой (критерий «достаточно ровное» — за архитектором, ОВ-3).
- **Соблюдение ресурсных ограничений** (1 vCPU / 2 GB): обработка для рендера идёт под уже
  существующим `HEAVY_LOCAL_SEMAPHORE` (как `recognize_pdf` сейчас); параллельная тяжёлая
  CV-нагрузка не плодится.
- **Переиспользование/рефакторинг `services/preprocess.py`** под две задачи: «получить очищенное
  изображение для рендера» (видимый слой, цвет/серый/ч-б) и «получить массив для OCR-текста»
  (как сейчас). Выделение общих шагов — за архитектором.
- **Обновление тестов** `tests/test_ocr.py`, `tests/test_actions.py` под новое поведение и режимы.

### Вне рамок (explicitly out of scope)

- **Многостраничный PDF из альбома** (вариант B) — следующее отдельное требование.
- **Авто-кроп по границам документа / коррекция перспективы** (вариант C) — позже, опционально;
  тяжело для 1 vCPU и рискованно на пёстром фоне.
- **Изменение логики извлечения текста OCR** — текстовый слой и качество распознавания не
  трогаем, меняем только видимый слой PDF (и согласование координат слоя с ним).
- **Метаданные/осмысленное имя файла PDF** (title, имя) — необязательная косметика; на усмотрение
  архитектора в рамках scope, не основная цель и не критерий приёмки.
- **Новые артефакты поставки** — профиль ivahub (бот без HTTP-порта, long-polling, GHCR) не
  меняется. Новые env-переменные, скорее всего, не требуются (параметры обработки — константы,
  если архитектор не решит иначе).

## 3. Действующие лица и сценарии использования

**Пользователь, отправивший фото документа.** Цель — получить аккуратный PDF-документ.
- **Сценарий A (перекошенное фото).** Шлёт фото документа со столa (перекос, тени, цветной фон),
  получает результат распознавания, жмёт «📄 PDF» в дефолтном режиме → приходит PDF с выровненной,
  контрастной, читаемой страницей и сохранённым выделяемым текстовым слоем.
- **Сценарий B (выбор «Скан ч/б»).** Хочет компактный классический скан → выбирает режим «Скан
  (ч/б)» → приходит PDF с бинаризованной страницей меньшего размера.
- **Сценарий C (уже ровный документ).** Шлёт ровное контрастное изображение → обработка не
  ухудшает его; PDF выглядит не хуже исходника.
- **Сценарий D (сбой обработки).** Очистка/рендер падает на конкретном входе → бот вшивает
  оригинал (как в v1.6.0) и всё равно отдаёт рабочий searchable PDF, не показывая ошибку.

**Владелец продукта / эксплуатация.** Цель — поставить улучшение без изменения профиля деплоя.
- **Сценарий E.** Релиз выкатывается на ivahub без новых артефактов поставки и (как правило) без
  новых env-переменных; поведение остальных кнопок не меняется.

## 4. Функциональные требования

- **ФТ-1.** Видимая страница PDF рендерится из **очищенного** изображения: к нему перед вшиванием
  применяется пайплайн очистки (EXIF-поворот → опц. ресайз к разумному размеру/DPI → выравнивание
  перекоса → подъём контраста / убирание теней).
- **ФТ-2.** Текстовый слой PDF остаётся **searchable** (невидимый слой Tesseract), и его
  координаты согласованы с видимой (очищенной/повёрнутой) страницей: выделение текста в PDF
  попадает в соответствующие слова. Качество распознавания текста не ухудшается относительно
  v1.6.0.
- **ФТ-3.** Поддерживаются как минимум два режима видимой страницы: **«Документ (цвет/серый)»** —
  выровненная контрастная цветная/серая страница (дефолт) и **«Скан (ч/б)»** — бинаризованная
  страница. Способ выбора режима в UI и итоговый дефолт определяет архитектор (ОВ-1).
- **ФТ-4.** Видимая страница вшивается с JPEG-сжатием и ресайзом так, что итоговый PDF **не
  больше**, а в типичном случае **меньше** сегодняшнего для того же входа.
- **ФТ-5.** При любом сбое или некорректном результате обработки видимой страницы бот **деградирует
  к оригиналу** (вшивает исходное изображение, как в v1.6.0) и всё равно отдаёт рабочий searchable
  PDF; пользователь не видит ошибки из-за обработки.
- **ФТ-6.** Результат не хуже текущего ни на одном входе: на любом входе пользователь получает
  searchable PDF, визуально не уступающий результату v1.6.0 (улучшение либо паритет, не регресс).
- **ФТ-7.** Если изображение уже достаточно ровное/контрастное, лишняя обработка не ухудшает его
  (страница не «перекашивается» и не «выжигается»). Критерий «достаточно ровное/контрастное» — за
  архитектором (ОВ-3).
- **ФТ-8.** Подъём контраста / убирание теней не «выжигает» бледный/мелкий текст: читаемость
  мелкого шрифта на выходе не хуже, чем на входе. Конкретный метод (autocontrast / CLAHE /
  adaptive threshold для ч/б) — за архитектором (ОВ-2).
- **ФТ-9.** Тяжёлая обработка для рендера выполняется под существующим `HEAVY_LOCAL_SEMAPHORE` и
  не порождает дополнительной параллельной CV-нагрузки сверх текущей модели `recognize_pdf`.
- **ФТ-10.** Поведение остальных кнопок-действий (Кратко/Тайм-коды/Субтитры/Задачи и др.,
  v1.5.0/v1.6.0) и логика дневного лимита (PDF и действия лимит **не** расходуют, BA-017 ФТ-2) не
  изменяются.

## 5. Нефункциональные требования

- **НФТ-1 (производительность на 1 vCPU / 2 GB).** Дополнительная CV-обработка для рендера не
  должна заметно ухудшать отзывчивость бота: выполняется вне event loop (как `recognize_pdf`,
  через `asyncio.to_thread`) и сериализуется существующим `HEAVY_LOCAL_SEMAPHORE` (ОГР-1).
- **НФТ-2 (надёжность / деградация).** Сбой обработки видимой страницы не ломает фичу: гарантирован
  откат к оригиналу (ФТ-5) и логирование события; пользователь всегда получает рабочий PDF.
- **НФТ-3 (размер результата).** Итоговый PDF за счёт JPEG-сжатия и ресайза не больше текущего и в
  типичном случае меньше (ФТ-4), что снижает трафик и время отправки документа в Telegram.
- **НФТ-4 (совместимость / контракт).** Публичный контракт кнопки «📄 PDF» не ломается: по-прежнему
  приходит searchable PDF с выделяемым текстом; релиз трактуется как **MINOR** (улучшение
  существующей функции, деградация гарантирует обратную совместимость поведения). Версию
  подтверждает архитектор (ориентир v1.9.0).
- **НФТ-5 (совместимость поставки).** Профиль поставки сохраняется: бот без HTTP-порта,
  long-polling, образ из GHCR, деплой на ivahub (ADR-001). Новых артефактов поставки не требуется;
  новые env-переменные не вводятся, если архитектор не решит иначе (тогда — плейсхолдеры в
  `.env.example` и `deploy/.env.example`).
- **НФТ-6 (локализация интерфейса).** Подписи кнопок режимов и сообщения — на русском, в стиле
  существующего меню действий (BA-016).
- **НФТ-7 (без новых внешних зависимостей по умолчанию).** Обработка опирается на уже
  установленные OpenCV/Pillow/Tesseract; введение новой зависимости — только с явного решения
  архитектора (правило «Ask first» из глобальных инструкций).

## 6. Ограничения и допущения

**Ограничения:**

- **ОГР-1.** Прод ivahub: 1 vCPU / 2 GB; CPU — узкое место. Тяжёлая локальная обработка идёт под
  `HEAVY_LOCAL_SEMAPHORE` и вне event loop (преемственно ADR-002/003/006 и текущему
  `recognize_pdf`).
- **ОГР-2.** Текущий `preprocess_image` сразу переводит изображение в оттенки серого и
  бинаризует — для цветного/серого режима страницы потребуется отдельная ветка обработки, не
  ломающая ветку для OCR-текста (детали — за архитектором).
- **ОГР-3.** Лимит `callback_data` 64 байта — если режимы вводятся отдельными кнопками, коды
  должны быть короткими (преемственно подходу проекта, BA-016).
- **ОГР-4.** Реальный `.env` не коммитится; при появлении параметров-настроек коммитятся только
  плейсхолдеры (раздел «Секреты» `CLAUDE.md`).

**Допущения:**

- **ДОП-1.** Шаги очистки, уже вычисляемые ради OCR (EXIF-поворот, ресайз, deskew), пригодны для
  переиспользования в рендере видимой страницы; цена дополнительной обработки для рендера остаётся
  в рамках текущего бюджета CPU под `HEAVY_LOCAL_SEMAPHORE`.
- **ДОП-2.** Рендер видимой страницы из уже очищенного/повёрнутого изображения целиком (когда
  Tesseract сам кладёт текстовый слой по этому изображению) — допустимый и более простой путь
  согласования слоя с картинкой, чем синхронизация двух разных преобразований (ФТ-2; финальное
  решение — за архитектором, ОВ-4).
- **ДОП-3.** Параметры обработки (целевой DPI/сторона, качество JPEG, метод контраста) можно
  задать константами в коде; вынос в env не обязателен (НФТ-5).
- **ДОП-4.** Текущее поведение v1.6.0 (вшивание оригинала + слой Tesseract) считается приемлемым
  «дном» качества — целью деградации (ФТ-5, ФТ-6).

## 7. Критерии приёмки

- **КП-1.** Для перекошенного/затенённого фото документа кнопка «📄 PDF» в дефолтном режиме
  возвращает PDF с **выровненной и контрастной** видимой страницей (визуально лучше исходного
  снимка). *(ФТ-1, ФТ-3)*
- **КП-2.** В возвращённом PDF сохраняется выделяемый/ищущийся текстовый слой, и выделение текста
  попадает в соответствующие слова на видимой странице. *(ФТ-2)*
- **КП-3.** Доступны режимы «Документ (цвет/серый)» (дефолт) и «Скан (ч/б)»; в режиме «Скан (ч/б)»
  страница бинаризована. *(ФТ-3)*
- **КП-4.** Для одного и того же входа размер итогового PDF не больше, чем в v1.6.0, и в типичном
  случае меньше. *(ФТ-4, НФТ-3)*
- **КП-5.** При искусственно вызванном сбое обработки видимой страницы бот возвращает searchable
  PDF с **оригинальным** изображением (поведение v1.6.0) и не показывает ошибку из-за обработки;
  факт деградации логируется. *(ФТ-5, НФТ-2)*
- **КП-6.** Ни на одном тестовом входе результат не хуже v1.6.0: всегда приходит рабочий
  searchable PDF (улучшение либо паритет). *(ФТ-6)*
- **КП-7.** Для уже ровного/контрастного изображения видимая страница не ухудшается (нет ложного
  перекоса, нет «выжигания» текста). *(ФТ-7, ФТ-8)*
- **КП-8.** Обработка рендера выполняется под `HEAVY_LOCAL_SEMAPHORE` и вне event loop; новой
  параллельной тяжёлой CV-нагрузки сверх текущей модели нет. *(ФТ-9, НФТ-1)*
- **КП-9.** Поведение прочих кнопок-действий и учёт дневного лимита (BA-017) не изменяются: PDF и
  действия по-прежнему не расходуют лимит. *(ФТ-10)*
- **КП-10.** Профиль поставки не меняется (бот без HTTP-порта, long-polling, GHCR, ivahub); новых
  артефактов поставки нет; при появлении настроек они вынесены плейсхолдерами в `.env.example` и
  `deploy/.env.example`. *(НФТ-5)*
- **КП-11.** Подписи кнопок режимов и сообщения — на русском, в стиле текущего меню действий.
  *(НФТ-6)*

## 8. Открытые вопросы

Блокирующих **продуктовых** вопросов нет: требование детально специфицировано (рендер из
очищенного изображения, два режима цвет/чб, JPEG-сжатие, обязательная деградация к оригиналу,
ресурсные рамки), а технические детали явно делегированы архитектору. Ниже — некритичные вопросы
на этап ADR; они не блокируют переход к проектированию.

- **ОВ-1 (архитектор, `/adr`) — UX режимов и дефолт.** Одна кнопка с дефолтным режимом + вторая
  для альтернативы, две кнопки, или выбор режима после нажатия «PDF»? Что является дефолтом —
  «Документ (цвет/серый)» или «Скан (ч/б)»? Требование называет дефолтом цветной выровненный
  вариант.
- **ОВ-2 (архитектор, `/adr`) — метод подъёма контраста / убирания теней.** autocontrast, CLAHE,
  adaptive threshold для ч/б — так, чтобы не «выжигать» бледный/мелкий текст (ФТ-8).
- **ОВ-3 (архитектор, `/adr`) — критерий «не трогать ровное».** Порог угла/контраста, при котором
  обработка пропускается, чтобы не ухудшать уже ровное изображение (ФТ-7).
- **ОВ-4 (архитектор, `/adr`) — согласование текстового слоя с видимой страницей.** Рекомендуемый
  подход — рендерить PDF из уже очищенного/повёрнутого изображения целиком (Tesseract кладёт слой
  по нему). Подтвердить выбор и убедиться, что координаты слоя совпадают со страницей (ФТ-2,
  ДОП-2).
- **ОВ-5 (архитектор, `/adr`) — DPI/размер и качество JPEG.** Целевой DPI (ориентир A4 @ ~200 dpi),
  максимальная сторона при ресайзе и качество JPEG — баланс размер/читаемость мелкого шрифта (ФТ-4).
- **ОВ-6 (архитектор, `/adr`) — рефакторинг `preprocess.py`.** Выделить «получить очищенное
  изображение для рендера» (цвет/серый/ч-б) отдельно от «получить массив для OCR-текста», не
  ломая текущую ветку OCR (ОГР-2).
- **ОВ-7 (архитектор, `/adr`) — версия и (опц.) метаданные PDF.** Подтвердить MINOR/версию
  (ориентир v1.9.0) и решить, добавлять ли осмысленное имя файла/title PDF в рамках scope как
  необязательную косметику.

Противоречий действующим ADR не выявлено: релиз **улучшает существующую функцию** searchable PDF
(BA-016 / ADR-016), сохраняет её публичный контракт за счёт обязательной деградации к текущему
поведению и не меняет профиль поставки «бот без HTTP-порта» (ADR-001).
