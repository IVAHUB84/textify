# BA-требования 020: Анонс новых версий пользователям (changelog-рассылка)

- **Источник:** `requirements/req-textify-020.md`
- **Дата:** 2026-06-25
- **Статус:** черновик

## 1. Контекст и бизнес-цель

Textify — бесплатный Telegram-бот для транскрипции голоса/аудио и OCR изображений
(см. `README.md`, `CLAUDE.md`), работает на VPS ivahub (1 vCPU / 2 GB, long-polling;
memory: deploy-server ivahub). Продукт активно развивается (кнопки-действия v1.5.0/v1.6.0,
дневной лимит и subscription-gate v1.7.0, реферальная награда v1.8.0), но **пользователи
об этом не узнают**: о новой версии знает только владелец продукта, новые возможности
остаются незамеченными, неактивные пользователи не возвращаются.

Требование вводит **changelog-рассылку**: при выходе новой версии бот сам присылает
пользователям короткий user-friendly «что нового». Это решает три задачи:

- **Retention / возврат пользователей.** Напоминание о боте и демонстрация развития
  возвращает неактивных пользователей.
- **Прозрачность.** Пользователь видит, что продукт живёт и улучшается.
- **Поддержка цели канала (v1.7.0).** Анонс — естественный момент мягко напомнить про
  канал/приглашения.

Все ключевые продуктовые решения **уже приняты заказчиком** и зафиксированы в требовании:
триггер — авто-детект новой версии при старте бота с обязательным **подтверждением админа**
перед фактической рассылкой; контент — **курируемый** короткий текст на каждую версию
(реестр в коде/файле), а не технические `release-notes`; **отписка** от анонсов поддерживается.

Стейкхолдеры:
- **Владелец продукта** (заказчик; администратор бота, `ADMIN_USER_ID`) — цель: retention и
  прозрачность; контролирует факт и момент рассылки.
- **Конечный пользователь бота** в личном чате — получает короткое уведомление о новой версии;
  может отписаться.
- **Команда разработки** (последующие агенты) — получает новый сервис анонсов/рассылки,
  реестр текстов и состояние поверх существующего стора `stats.db` (v0.6.0).

## 2. Область (scope)

### В рамках задачи

- **Единый источник версии в коде.** Вводится единая константа текущей версии приложения
  (например, `__version__`). Сейчас версия есть только в git-тегах и `release-notes`, в коде —
  нет.
- **Курируемый реестр текстов анонсов** вида `{версия: короткий текст «что нового»}` — на
  русском, user-friendly, без технических КП/ADR. Хранится в коде или отдельном data-файле.
  Технические `release-notes/notes-vX.Y.Z.md` для рассылки **не используются**.
- **Авто-детект ожидающего анонса при старте.** При запуске бот сравнивает текущую версию с
  **последней анонсированной** (persistent). Если текущая новее и для неё есть текст в реестре —
  фиксируется «ожидающий анонс».
- **Подтверждение админом перед рассылкой (gate).** Бот **не рассылает автоматически**: уведомляет
  админа (`ADMIN_USER_ID`) — присылает превью текста анонса и inline-кнопки «Разослать» и
  «Пропустить». Рассылка стартует только по «Разослать». По «Пропустить» версия помечается
  анонсированной без рассылки. Если админ не отреагировал — «ожидающий анонс» предлагается снова
  при следующем старте, без дублирующих записей.
- **Фоновая рассылка** по всем `user_id` из `user_stats`, минус отписавшиеся, минус заведомо
  заблокировавшие бота. Рассылка — фоновая задача, **не блокирует** обработку входящих сообщений.
- **Троттлинг под лимиты Telegram** с паузами и обработкой `RetryAfter`/`FloodWait` (backoff);
  конкретные значения — за архитектором.
- **Устойчивость доставки.** `TelegramForbiddenError` (пользователь заблокировал бота / удалил
  чат) — пропустить, по возможности пометить, не падать; прочие ошибки — залогировать и
  продолжить.
- **Идемпотентность.** После запуска рассылки версия помечается анонсированной (persistent);
  рестарт/редеплой контейнера **не приводит** к повторной рассылке той же версии.
- **Отчёт админу** по завершении: отправлено N, пропущено/заблокировано M, ошибок K.
- **Отписка пользователя.** Под анонсом — inline-кнопка «🔕 Не показывать анонсы» (и/или
  команда). Нажатие помечает пользователя отписавшимся (persistent), будущие анонсы ему не
  приходят. Возможна команда повторной подписки (вторично, за архитектором).
- **Хранение состояния** (последняя анонсированная версия, список отписавшихся, опц. пометка
  заблокировавших) — поверх существующего `stats.db`; новый отдельный стор по умолчанию не
  заводится.
- **Переиспользование `ADMIN_USER_ID`** (уже есть в `config.py`). Возможный env-флаг включения
  анонсов — на усмотрение архитектора.

### Вне рамок (explicitly out of scope)

- **Рассылка в группы** — только личные чаты пользователей из `user_stats`.
- **Произвольные маркетинговые рассылки / сегментация аудитории** — только changelog по версиям.
- **Локализация анонсов** под язык пользователя — тексты только на русском, как и остальной UX.
- **Автогенерация текста анонса LLM** — тексты курируются вручную.
- **Веб-панель управления рассылками, A/B-тесты, аналитика открытий.**
- **Изменение текущей обработки медиа, команд, лимитов/gate (v1.7.0) и рефералов (v1.8.0)** —
  кроме добавления кнопки/команды отписки от анонсов.
- **Новые артефакты поставки** — профиль ivahub/long-polling/GHCR не меняется; добавляются, при
  необходимости, только env-флаг и новые поля в `stats.db`.

## 3. Действующие лица и сценарии использования

**Администратор / владелец продукта.** Цель — контролировать факт и момент рассылки.
- **Сценарий A.** После редеплоя новой версии при старте бот присылает админу превью анонса с
  кнопками «Разослать»/«Пропустить».
- **Сценарий B.** Админ жмёт «Разослать» → запускается фоновая рассылка; по завершении админ
  получает отчёт (отправлено/пропущено/ошибок).
- **Сценарий C.** Админ жмёт «Пропустить» → версия помечается анонсированной без рассылки,
  превью больше не всплывает.
- **Сценарий D.** Админ не отреагировал → при следующем старте бот снова предлагает тот же
  ожидающий анонс, без дублей в БД.

**Конечный пользователь бота.** Цель — узнавать о новинках, либо отказаться от уведомлений.
- **Сценарий E.** Получает короткий анонс «что нового» с кнопкой «🔕 Не показывать анонсы».
- **Сценарий F.** Нажимает «Не показывать анонсы» → помечается отписавшимся; будущие анонсы не
  приходят.
- **Сценарий G.** Пользователь, заблокировавший бота, при рассылке пропускается — рассылка не
  падает.

**Команда эксплуатации.** Цель — корректный выход новой версии.
- **Сценарий H.** Релиз без текста анонса в реестре (например, технический патч) → рассылка
  тихо не запускается, событие логируется; админу превью не приходит.

## 4. Функциональные требования

- **ФТ-1.** В коде вводится единый источник текущей версии приложения (например, константа
  `__version__`), используемый механизмом анонсов как «текущая версия».
- **ФТ-2.** Существует курируемый реестр `{версия: текст анонса}` — короткие user-friendly
  тексты на русском, отдельные от технических `release-notes`. Технические `release-notes` для
  рассылки не используются.
- **ФТ-3.** При запуске бот сравнивает текущую версию с **последней анонсированной** (persistent).
  «Ожидающий анонс» фиксируется, только если текущая версия **новее** последней анонсированной
  **и** для неё есть текст в реестре. Откат на более старую версию анонс не триггерит. Правило
  сравнения версий (semver / точная `vX.Y.Z`) — за архитектором (ОВ-2).
- **ФТ-4.** Если для текущей версии текста анонса в реестре **нет**, рассылка не запускается тихо,
  с логом; превью админу не отправляется.
- **ФТ-5.** Бот не рассылает автоматически: при наличии ожидающего анонса он отправляет
  **админу** (`ADMIN_USER_ID`) превью текста и inline-кнопки «Разослать» и «Пропустить».
- **ФТ-6.** По кнопке «Разослать» запускается фоновая рассылка анонса. По кнопке «Пропустить»
  версия помечается анонсированной **без** рассылки, и превью этой версии больше не предлагается.
- **ФТ-7.** Если админ не отреагировал на превью, ожидающий анонс сохраняется и предлагается
  снова при следующем старте, без создания дублирующих записей в хранилище.
- **ФТ-8.** Адресаты рассылки — все `user_id` из `user_stats`, **за исключением** отписавшихся от
  анонсов и (по возможности) заведомо заблокировавших бота.
- **ФТ-9.** Рассылка выполняется как **фоновая задача** и не блокирует обработку входящих
  сообщений пользователей.
- **ФТ-10.** Рассылка соблюдает **троттлинг** под лимиты Telegram (паузы между отправками) и
  корректно обрабатывает `RetryAfter`/`FloodWait` с backoff. Конкретные значения скорости и
  стратегия backoff — за архитектором (ОВ-1).
- **ФТ-11.** При ошибке доставки `TelegramForbiddenError` (пользователь заблокировал бота /
  удалил чат) получатель пропускается (по возможности помечается), рассылка не прерывается.
  Прочие ошибки доставки логируются, рассылка продолжается.
- **ФТ-12.** После запуска рассылки версия помечается анонсированной (persistent). Рестарт /
  редеплой контейнера **не** приводит к повторной рассылке той же версии (идемпотентность).
  Поведение при прерывании рассылки в середине (частичная доставка) — за архитектором (ОВ-3).
- **ФТ-13.** По завершении рассылки бот отправляет админу краткий отчёт: отправлено N,
  пропущено/заблокировано M, ошибок K.
- **ФТ-14.** Под каждым анонсом пользователю доступна inline-кнопка «🔕 Не показывать анонсы»
  (и/или команда). Нажатие помечает пользователя отписавшимся (persistent); будущие анонсы ему
  не отправляются.
- **ФТ-15.** Все пользовательские и админские сообщения механизма (превью, анонс, кнопки,
  отчёт) — на русском.
- **ФТ-16.** Текущая обработка медиа, существующие команды, лимиты/gate (v1.7.0) и рефералы
  (v1.8.0) не затрагиваются, кроме добавления кнопки/команды отписки от анонсов.

## 5. Нефункциональные требования

- **НФТ-1 (производительность / отзывчивость на 1 vCPU / 2 GB).** Рассылка — фоновая задача с
  троттлингом; она не должна блокировать event loop aiogram и не должна заметно ухудшать
  отзывчивость бота при обработке входящего медиа (whisper — узкое место, memory: deploy-server
  ivahub). Доступ к SQLite — вне event loop (преемственно ADR-002/003/006).
- **НФТ-2 (надёжность / устойчивость рассылки).** Ошибки доставки отдельным получателям
  (`TelegramForbiddenError`, прочие) не должны прерывать рассылку и не должны ронять бота
  (ФТ-11). Сбой механизма анонсов не должен ломать основную функцию бота.
- **НФТ-3 (идемпотентность / персистентность).** Последняя анонсированная версия и список
  отписавшихся хранятся персистентно (на базе `stats.db` в примонтированном volume) и переживают
  рестарт/редеплой; повторная рассылка той же версии исключается (ФТ-12).
- **НФТ-4 (соблюдение лимитов Telegram).** Скорость рассылки удерживается в рамках лимитов
  Telegram Bot API на массовую отправку, с корректной реакцией на `RetryAfter`/`FloodWait`
  (ФТ-10), чтобы избежать троттлинга/временных блокировок бота.
- **НФТ-5 (приватность / минимизация данных).** Механизм использует только уже хранимые
  `user_id` и служебные метки (отписка, анонсированная версия); содержимое сообщений и
  распознанный текст не сохраняются (преемственно v0.6.0).
- **НФТ-6 (управление конфигурацией / секреты).** Возможный флаг включения анонсов задаётся через
  окружение; в репозиторий коммитятся только плейсхолдеры (`.env.example`, `deploy/.env.example`).
  `ADMIN_USER_ID` переиспользуется из существующего конфига.
- **НФТ-7 (совместимость поставки).** Профиль поставки сохраняется: long-polling, без
  HTTP-порта / Traefik / edge (ADR-001), образ из GHCR, `restart: unless-stopped`, деплой на
  ivahub, переиспользование volume `stats.db`. Новых артефактов поставки релиз не вводит.
- **НФТ-8 (локализация интерфейса).** Все сообщения механизма — на русском, согласно соглашениям
  проекта.
- **НФТ-9 (MINOR-совместимость).** Релиз обратимо-совместим: при отсутствии текста анонса для
  версии и без подтверждения админа поведение бота не меняется, — что согласуется с трактовкой
  релиза (ориентир v1.10.0) как MINOR.

## 6. Ограничения и допущения

**Ограничения:**

- **ОГР-1.** Прод ivahub: 1 vCPU / 2 GB, long-polling — без тяжёлых очередей/воркеров; фоновая
  рассылка должна быть лёгкой и троттлящейся, не конкурируя за CPU с whisper (memory:
  deploy-server ivahub).
- **ОГР-2.** Telegram Bot API ограничивает скорость массовой отправки (порядка десятков
  сообщений/сек) и может возвращать `RetryAfter`/`FloodWait`; точные параметры троттлинга
  уточняет архитектор (ОВ-1).
- **ОГР-3.** Telegram не позволяет «разослать всем» одним вызовом — отправка идёт поштучно по
  `user_id`; единственный способ узнать о блокировке бота пользователем — ошибка при отправке
  (`TelegramForbiddenError`).
- **ОГР-4.** Запись/чтение SQLite — блокирующая операция, выполняется вне event loop
  (преемственно ADR-002/003/006).
- **ОГР-5.** Лимит `callback_data` 64 байта — короткие коды для inline-кнопок «Разослать» /
  «Пропустить» / «Не показывать анонсы» (преемственно подходу проекта).
- **ОГР-6.** Реальный `.env` не коммитится; коммитятся только плейсхолдеры (НФТ-6).

**Допущения:**

- **ДОП-1.** Таблица `user_stats` (`services/stats.py`, v0.6.0) содержит всех пользователей бота
  и пригодна как источник адресатов рассылки; существующая SQLite-инфраструктура пригодна для
  хранения отписок и последней анонсированной версии — новый отдельный стор не нужен.
- **ДОП-2.** `ADMIN_USER_ID` (`config.py`) задан в проде и однозначно определяет администратора,
  которому приходит превью и отчёт.
- **ДОП-3.** Базовый дефолт идемпотентности — пометка версии анонсированной **после старта**
  рассылки; частичная потеря при прерывании рассылки в середине допустима (уточняется в ОВ-3).
- **ДОП-4.** Отсутствие текста анонса в реестре трактуется как явное «не анонсировать эту
  версию» (например, мелкие патчи) — отдельный флаг подавления анонса не обязателен.
- **ДОП-5.** Курируемые тексты анонсов короткие и не содержат персональных данных, поэтому
  одинаковый текст пригоден для рассылки всем адресатам без персонализации.

## 7. Критерии приёмки

- **КП-1.** В коде есть единый источник текущей версии; механизм анонсов берёт «текущую версию»
  из него. *(ФТ-1)*
- **КП-2.** Существует курируемый реестр текстов анонсов по версиям; для рассылки используется
  текст из реестра, а не `release-notes/notes-vX.Y.Z.md`. *(ФТ-2)*
- **КП-3.** При старте, когда текущая версия новее последней анонсированной и для неё есть текст
  в реестре, фиксируется ожидающий анонс; откат на более старую версию анонс не триггерит.
  *(ФТ-3)*
- **КП-4.** Если для текущей версии текста анонса нет, рассылка не запускается, событие
  логируется, превью админу не отправляется. *(ФТ-4)*
- **КП-5.** При наличии ожидающего анонса админ (`ADMIN_USER_ID`) получает превью текста и
  inline-кнопки «Разослать» и «Пропустить»; автоматической рассылки без нажатия не происходит.
  *(ФТ-5)*
- **КП-6.** По «Разослать» запускается фоновая рассылка; по «Пропустить» версия помечается
  анонсированной без рассылки и её превью больше не предлагается. *(ФТ-6)*
- **КП-7.** Если админ не отреагировал, при следующем старте предлагается тот же ожидающий
  анонс, без дублирующих записей в хранилище. *(ФТ-7)*
- **КП-8.** Рассылка идёт по `user_id` из `user_stats`, исключая отписавшихся и (по возможности)
  заблокировавших бота. *(ФТ-8)*
- **КП-9.** Во время рассылки бот продолжает обрабатывать входящие сообщения (рассылка не
  блокирует основной поток). *(ФТ-9, НФТ-1)*
- **КП-10.** Рассылка соблюдает троттлинг и корректно реагирует на `RetryAfter`/`FloodWait`
  (backoff), не превышая лимиты Telegram. *(ФТ-10, НФТ-4, ОГР-2)*
- **КП-11.** При `TelegramForbiddenError` получатель пропускается (по возможности помечается),
  рассылка не прерывается; прочие ошибки логируются, рассылка продолжается; бот не падает.
  *(ФТ-11, НФТ-2)*
- **КП-12.** После запуска рассылки версия помечена анонсированной; рестарт/редеплой контейнера
  не приводит к повторной рассылке той же версии. *(ФТ-12, НФТ-3)*
- **КП-13.** По завершении рассылки админ получает отчёт: отправлено N, пропущено/заблокировано
  M, ошибок K. *(ФТ-13)*
- **КП-14.** Под анонсом доступна кнопка «🔕 Не показывать анонсы» (и/или команда); после
  отписки будущие анонсы пользователю не приходят, отписка переживает рестарт. *(ФТ-14, НФТ-3)*
- **КП-15.** Все сообщения механизма (превью, анонс, кнопки, отчёт) — на русском. *(ФТ-15,
  НФТ-8)*
- **КП-16.** Существующая обработка медиа, команды, лимиты/gate (v1.7.0) и рефералы (v1.8.0) не
  ломаются; добавляется только кнопка/команда отписки от анонсов. *(ФТ-16, НФТ-9)*

## 8. Открытые вопросы

Блокирующих **продуктовых** вопросов нет: все ключевые решения (триггер — авто-детект + gate
админа, курируемый реестр текстов, поддержка отписки, только личные чаты) приняты заказчиком и
зафиксированы. Ниже — некритичные вопросы, делегированные **архитектору** на этап ADR; переход к
`/adr` они не блокируют.

- **ОВ-1 (архитектор, `/adr`) — троттлинг и backoff.** Конкретная скорость рассылки (порядка
  ~25–30 сообщений/сек), паузы и стратегия обработки `RetryAfter`/`FloodWait` под лимиты
  Telegram Bot API. *(ФТ-10, НФТ-4)*
- **ОВ-2 (архитектор, `/adr`) — формат и сравнение версий.** Ключ реестра — точная `vX.Y.Z`?
  Сравнение версий через semper-compare, чтобы анонс срабатывал только при росте версии и не
  триггерился при откате. *(ФТ-3)*
- **ОВ-3 (архитектор, `/adr`) — идемпотентность при гонках/прерывании.** Помечать версию
  анонсированной после **старта** рассылки (базовый дефолт, частичная потеря при рестарте в
  середине допустима) или вести учёт прогресса/докрутку для гарантированной полной доставки.
  *(ФТ-12)*
- **ОВ-4 (архитектор, `/adr`) — где объявить версию и показывать ли её пользователю.** Место
  объявления `__version__` и стоит ли заодно показывать версию в `/start`/`/stats`. *(ФТ-1)*
- **ОВ-5 (архитектор, `/adr`) — повторная подписка на анонсы.** Нужна ли команда повторной
  подписки и как её назвать (отписка обязательна, повторная подписка — вторична). *(ФТ-14)*
- **ОВ-6 (архитектор, `/adr`) — политика анонса патч-релизов.** Анонсировать ли мелкие патчи
  (например, v1.9.1) или только MINOR+. Базовый механизм уже даёт это через отсутствие текста в
  реестре (ДОП-4); нужен ли дополнительный явный флаг — за архитектором. *(ФТ-4, ДОП-4)*
- **ОВ-7 (архитектор, `/adr`) — хранение состояния и пометка заблокировавших.** Подтвердить
  размещение last-announced-версии и opt-out в `stats.db`; решить, вести ли persistent-пометку
  заблокировавших бота пользователей (для исключения из будущих рассылок) или достаточно
  пропускать их по `TelegramForbiddenError` в рантайме. *(ФТ-8, ФТ-11, НФТ-3)*
- **ОВ-8 (архитектор, `/adr`) — env-флаг включения анонсов.** Нужен ли отдельный флаг включения
  механизма (для dev/CI) и его дефолт. *(НФТ-6)*

Противоречий действующим ADR не выявлено: релиз **расширяет** существующий персистентный слой
(`stats.db`, v0.6.0), переиспользует `ADMIN_USER_ID` (`config.py`) и сохраняет профиль поставки
«бот без HTTP-порта», long-polling, секреты через `.env` (ADR-001). Рассылка опирается на штатный
aiogram/Bot API без новых внешних зависимостей.
