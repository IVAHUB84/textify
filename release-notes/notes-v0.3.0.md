# Release Notes v0.3.0

- **Дата:** 2026-06-22
- **Тип релиза:** minor

## Кратко

Релиз добавляет транскрипцию голосовых сообщений и аудиофайлов в текст через faster-whisper (модель
`base`, CPU, `int8`). Пользователь присылает голосовое, аудиофайл или аудио-документ — бот возвращает
сырой транскрипт с автоопределением языка (ru/en). Функция работает автономно: веса модели вшиты в
Docker-образ на этапе сборки, в рантайме сетевого доступа к HuggingFace не требуется.

## Что нового

### Возможности

- Приём голосового сообщения (`message.voice`) с возвратом сырого транскрипта.
- Приём аудиофайла (`message.audio`) — обрабатывается наравне с голосовым.
- Приём аудио как файла-документа (`message.document` с MIME-типом `audio/*`) — обрабатывается
  наравне с голосовым/аудио.
- Автоопределение языка (русский и английский) — без явного указания, faster-whisper определяет сам.
- При пустом результате транскрипции (тишина, шум) бот отвечает сообщением «Речь в аудио не распознана.».

### Улучшения

- Новый сервисный слой `services/transcribe.py`: асинхронная обёртка над faster-whisper; синхронный
  CPU-bound вызов `WhisperModel.transcribe` вынесен в пул потоков через `asyncio.to_thread` — event
  loop остаётся отзывчивым. Модель грузится единожды (ленивый потокобезопасный singleton).
- Новый хендлер `handlers/audio.py` зарегистрирован до текстовой заглушки — аудио не перехватывается
  текстовым роутером.
- `Dockerfile` обновлён: в runtime-стейдж добавлены предзагрузка весов модели `base` на этапе
  сборки (`HF_HOME=/opt/models`), рантайм-переменные offline-режима (`HF_HUB_OFFLINE=1`,
  `TRANSFORMERS_OFFLINE=1`) и корректные права на каталог весов для non-root-пользователя.
  Системный `ffmpeg` через apt не добавляется — декодирование идёт через транзитивный пакет `av`.

### Исправления

Нет (новая функциональность, не исправление).

## Изменения публичного контракта

Обратимо-совместимое изменение (minor):

- Бот начинает обрабатывать голосовые сообщения, аудиофайлы и аудио-документы — возвращает сырой
  транскрипт. Ранее на голосовое/аудио приходил ответ текстовой заглушки.
- OCR на изображения, команды `/start`/`/help`, текстовая заглушка на обычный текст и конфигурация
  (`BOT_TOKEN`) — без изменений.
- **Новых пользовательских переменных окружения нет.** HF-переменные (`HF_HOME`, `HF_HUB_OFFLINE`,
  `TRANSFORMERS_OFFLINE`) заданы непосредственно в `Dockerfile` и не требуют изменений
  `deploy/docker-compose.yml`, `deploy/.env.example` или `.github/workflows/deploy.yml`.
- Миграционный путь не требуется.

## Известные ограничения

- При первом запросе к транскрипции модель загружается в память (~150–200 MB) — разовая задержка.
- При нескольких одновременных длинных аудио возможна конкуренция за 1 vCPU / 1.9 GB RAM
  (риск OOM); жёсткое ограничение параллелизма и очередь — v0.5.0.
- При очень длинном транскрипте, превышающем лимит Telegram на размер сообщения, поведение не
  специфицировано; разбивка — v0.5.0 (ОВ-1).
- Качество транскрипции определяется возможностями модели `base` и качеством присланного аудио;
  гарантии точности не задаются.
- Поддерживаются только русский и английский; выбор языка пользователем — вне рамок.
- Рост размера Docker-образа: веса модели `base` + транзитивные пакеты faster-whisper / CTranslate2
  / PyAV / onnxruntime. На ivahub (~22 GB свободно) приемлемо; время CI build/push и pull на сервере
  увеличивается.

## Заметки по деплою

- При первом деплое v0.3.0 на ivahub требуется `docker pull` нового образа — он заметно крупнее
  предыдущего из-за вшитых весов модели.
- Никаких новых секретов и переменных в `deploy/.env` добавлять не нужно.
- `deploy/docker-compose.yml`, `deploy/.env.example`, `.github/workflows/deploy.yml` — без изменений.
