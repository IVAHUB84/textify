# Release Notes v0.4.0

- **Дата:** 2026-06-22
- **Тип релиза:** minor

## Кратко

Релиз повышает точность распознавания текста на изображениях за счёт конвейера предобработки
на OpenCV и более точных настроек Tesseract. Пользовательский контракт не меняется: те же входы,
тот же сырой текст на выходе — просто точнее. Дополнительно добавлена ненавязчивая подсказка
слать изображение как файл (документ), чтобы Telegram не пересжимал его.

## Что нового

### Возможности

- Новый модуль `services/preprocess.py` — конвейер предобработки изображений из семи шагов:
  1. Исправление ориентации по EXIF-тегам (фото с телефона часто физически повёрнуты).
  2. Перевод в оттенки серого (grayscale).
  3. Кэп размера: если большая сторона > 2500 px — пропорциональный даунскейл до 2500 px.
  4. Апскейл мелких изображений: если большая сторона < 1000 px — масштабирование до ~1500 px.
  5. Лёгкий деноизинг — `cv2.medianBlur` с малым ядром (нетяжёлый, оглядка на 1 vCPU).
  6. Адаптивная бинаризация — `cv2.adaptiveThreshold` (Gaussian, THRESH_BINARY), умеренные параметры.
  7. Устранение наклона (deskew) — оценка угла через `cv2.minAreaRect`, поворот `cv2.warpAffine`
     на белом фоне; применяется только в диапазоне ±15°, иначе пропускается.

### Улучшения

- `services/ocr.py` обновлён: изображение проходит через `preprocess_image()` перед вызовом
  Tesseract; настройки движка сменены на `--oem 1` (LSTM) и `--psm 3`; предобработка и OCR
  выполняются в одном вызове `asyncio.to_thread` (event loop не блокируется).
- Реализован фолбэк: при любом исключении на этапе предобработки Tesseract получает исходное
  изображение (grayscale), ошибка логируется — бот не падает.
- UX-подсказка в `/help` и в сообщении о нераспознанном тексте: рекомендация присылать изображение
  как файл (📎 → Документ), а не обычным фото, чтобы избежать пересжатия JPEG со стороны Telegram.

### Исправления

Нет (улучшение существующей функциональности, не исправление дефекта).

## Изменения публичного контракта

Изменений публичного контракта нет. Пользовательский контракт OCR неизменен:
- те же входы (`message.photo` и изображение-документ);
- тот же сырой распознанный текст без постобработки;
- сообщение о нераспознанном тексте сохранено (дополнено UX-подсказкой);
- сигнатура `services/ocr.recognize_text(image_bytes: bytes) -> str` не изменилась.

Новых переменных окружения нет. `deploy/docker-compose.yml`, `deploy/.env.example`,
`.github/workflows/deploy.yml` — без изменений. Миграция не требуется.

## Заметки по деплою

- `requirements.txt` пополнился `opencv-python-headless>=4.8.0` и `numpy>=1.24.0`;
  в `Dockerfile` добавлены системные библиотеки `libglib2.0-0` и `libgomp1` и
  проверочный шаг `RUN python -c "import cv2"` — `docker build` упадёт, если зависимость
  не удовлетворена.
- `libGL`/`libgl1` не добавляется: `opencv-python-headless` не требует GUI-библиотек.
- Новый образ заметно крупнее из-за OpenCV/numpy; на ivahub (~22 GB свободно) приемлемо.
- Никаких новых секретов в `deploy/.env` добавлять не нужно.

## Известные ограничения

- Предобработка выполняется на CPU и занимает дополнительное время (с кэпом 2500 px и лёгким
  деноизингом — в пределах нескольких секунд на 1 vCPU).
- Жёстких лимитов размера входного файла нет — это зона будущего UX-релиза (v0.6.0).
- Качество OCR по-прежнему зависит от Tesseract и качества исходного изображения; абсолютные
  гарантии точности не задаются.
- Структурирование текста через Groq API остаётся отложенным: Groq API недоступен с российских IP
  (гео-блокировка по RU IP; недоступно с ivahub). Запланировано на v0.5.0 — при решении проблемы
  с доступностью.
- При очень длинном распознанном тексте, превышающем лимит Telegram, поведение не специфицировано;
  разбивка — v0.6.0.
