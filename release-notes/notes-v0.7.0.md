# Release Notes v0.7.0

- **Дата:** 2026-06-23
- **Тип релиза:** minor

## Кратко

Релиз устраняет латентный дефект надёжности: длинный текстовый результат (OCR, транскрипт, структурированный ответ) теперь гарантированно доходит до пользователя целиком — через серию сообщений с разбивкой по естественным границам или через вложение `.txt` при очень длинном тексте. Поведение коротких ответов и служебных сообщений не изменилось.

## Что нового

### Возможности

- **Единый модуль отдачи результата `services/reply.py`.** Вводится переиспользуемая корутина `send_result(message, text)` и чистая функция-сплиттер `split_text(text, limit)`. Все пути вывода (OCR-фото, OCR-документ, голос, аудио, аудио-документ) используют `send_result` вместо прямого `message.answer(result)`.

- **Три режима отдачи по длине результата:**
  - Короткий (≤ 4000 символов) — одно сообщение (поведение не изменилось).
  - Длинный (> 4000, ≤ 5 частей по 4000) — серия сообщений с паузой 0.4 с между частями, разбитых по абзацам/строкам/предложениям без разрыва слов.
  - Очень длинный (> 5 частей) — один файл `result.txt` из памяти с подписью «Результат целиком во вложении.»; серия сообщений в этом режиме не отправляется.

- **Иерархический жадный сплиттер.** Границы частей ищутся по убыванию приоритета: двойной перевод строки (абзац) → одиночный перевод строки → конец предложения → пробел → жёсткий срез (крайний случай — аномально длинная строка без границ). Конкатенация частей восстанавливает исходный текст без потерь и дублирования.

### Улучшения

- `handlers/image.py` и `handlers/audio.py` переведены на `send_result`; служебные сообщения `NO_TEXT_MESSAGE` и `NO_SPEECH_MESSAGE` остаются на прямом `message.answer`.
- `roadmap.md` обновлён: v0.7.0 закрывает надёжную отдачу длинного результата; прочие UX-пункты (прогресс-индикатор, понятные сообщения об ошибках, лимиты входного файла) перенесены в последующие итерации темы «UX и устойчивость».

### Исправления

- Устранён латентный дефект «message is too long» при отправке длинного результата через Bot API Telegram (ошибка возникала при превышении лимита 4096 символов).

## Изменения публичного контракта

Изменение совместимое — контракт не ломается:
- Короткий результат и служебные сообщения ведут себя как прежде.
- Длинный результат теперь приходит несколькими сообщениями или файлом `.txt` вместо ошибки отправки.
- Новых переменных окружения, портов и зависимостей нет. Профиль поставки ADR-001 (long-polling, без `ports:`/Traefik/`edge`, образ из GHCR, `restart: unless-stopped`) сохранён без изменений.

Константы, вынесенные в `services/reply.py`:
- `TELEGRAM_LIMIT = 4096` — жёсткий лимит Telegram (справочная константа).
- `MAX_MESSAGE_LEN = 4000` — безопасный лимит части (запас на UTF-16-расхождение).
- `MAX_PARTS = 5` — порог перехода на файл по числу частей.
- `PART_DELAY_SEC = 0.4` — пауза между частями серии.

## Известные ограничения

- Markdown отдаётся plain-text (без `parse_mode`) — как в предыдущих версиях. Включение `parse_mode` с border-safe-разбивкой сущностей отложено (ADR-007, ОВ-1).
- Жёсткий срез как крайний случай может разорвать аномально длинное «слово» без естественных границ (неизбежно при отсутствии пробелов/переводов строк внутри лимита).
- Ретраи при сбое отправки части серии не реализованы (вне scope BA); сбой логируется, бот не падает.
