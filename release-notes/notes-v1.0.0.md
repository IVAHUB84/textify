# Release Notes v1.0.0

- **Дата:** 2026-06-23
- **Тип релиза:** major

## Кратко

Textify 1.0.0 — релиз продакшн-готовности без новых пользовательских фич. Закрыты эксплуатационные пробелы: прод становится наблюдаемым через `docker logs`, пользователь видит индикатор прогресса во время обработки, слабый сервер защищён от OOM/перегрузки глобальным семафором. Публичный контракт бота явно зафиксирован как semver-baseline.

## Что нового

### Возможности

- Нет новых пользовательских возможностей. Релиз — стабилизация и эксплуатационная обвязка.

### Улучшения

- **Наблюдаемость (logging).** В точке входа `bot.py` настраивается `logging.basicConfig` с уровнем из `LOG_LEVEL` (дефолт `INFO`) и форматом `время уровень имя_логгера сообщение`. Некорректное или пустое значение `LOG_LEVEL` нормализуется в `INFO` без падения старта. При старте бота выводится INFO-сообщение, гарантируя непустой `docker logs`.
- **Небуферизованный вывод (`PYTHONUNBUFFERED=1`).** Добавлен `ENV PYTHONUNBUFFERED=1` в `Dockerfile` — логи появляются в `docker logs` в реальном времени, без задержки Python-буфера.
- **Индикатор прогресса (`ChatActionSender`).** Во время обработки в Telegram-чате отображается chat action:
  - `upload_document` — при распознавании фото и изображений-документов (`handlers/image.py`);
  - `typing` — при транскрипции аудио всех форматов (`handlers/audio.py`);
  - `typing` — при выполнении действий «Кратко»/«Перевести» (`handlers/actions.py`).
  Реализация неблокирующая (`ChatActionSender` крутит фоновую задачу); индикатор гасится по завершении, в том числе при ошибке.
- **Защита от перегрузки (глобальный семафор).** В `services/__init__.py` объявлен `HEAVY_LOCAL_SEMAPHORE = asyncio.Semaphore(1)`. Семафор захватывается в `services/ocr.recognize_text` и `services/transcribe._transcribe_local` вокруг вызовов `asyncio.to_thread`. CF-вызовы (ASR, LLM) под семафор не попадают. Фолбэк CF→local захватывает семафор ровно один раз — дедлок исключён. Одновременно выполняется не более одной тяжёлой локальной операции на всём процессе.
- **Рерайт `README.md`.** Документ полностью переписан под фактическое состояние: возможности, все 10 переменных окружения с назначением/обязательностью/дефолтами, схема деплоя (ivahub, GHCR, volume), особенности (пиннинг Telegram-DC, офлайн-модель Whisper, гео-ограничения Groq), раздел эксплуатации (логи, данные статистики, переключение провайдеров), раздел «Публичный контракт 1.0.0».

### Исправления

- Нет багфиксов. Все изменения — обвязка и документация.

## Изменения публичного контракта

Релиз является MAJOR не по факту ломающего изменения, а по факту **фиксации публичного контракта** как semver-baseline. Сам контракт не ломается — он впервые явно объявляется стабильным.

**Зафиксированный контракт 1.0.0** (полный текст — в `adrs/adr-010.md` и `README.md`):

- Команды: `/start`, `/help`, `/stats`.
- Входные каналы: фото, изображение-документ (`image/*`), voice, audio, аудио-документ (`audio/*`).
- Формат ответа: plain text без `parse_mode`; отдача через `send_result`; фолбэки при недоступности LLM/ASR.
- Inline-действия: «Кратко» (`act:sum`), «Перевести» (`act:tr`).

Начиная с v1.0.0: ломающее изменение контракта = MAJOR, обратимо-совместимое добавление = MINOR, багфикс = PATCH.

**Новая переменная окружения** `LOG_LEVEL` (опциональная, дефолт `INFO`) — обратимо-совместимое добавление, контракт не ломает. Миграционный путь: не требуется.

## Известные ограничения

- Лимит семафора = 1 сериализует все тяжёлые локальные операции. При апгрейде сервера значение меняется в одном месте (`HEAVY_LOCAL_CONCURRENCY` в `services/__init__.py`).
- Per-user мягкий лимит «один тяжёлый запрос на пользователя» не введён (отложен после 1.0.0); при параллельных запросах от одного пользователя каждый ждёт под индикатором прогресса.
- Чанкинг крупного аудио для CF (> 8 МБ переключается на локальный путь) — техдолг из v0.8.0.
- mypy-stubs для `pytesseract` и `faster_whisper` отсутствуют — часть mypy-предупреждений ожидается.

## Примечания к деплою

Новых обязательных серверных шагов нет. Новых обязательных секретов нет.

- **`LOG_LEVEL`** опциональна (дефолт `INFO`); при необходимости детализации добавить в `deploy/.env` (пример — в `deploy/.env.example`).
- **Верификация наблюдаемости после деплоя:** `docker logs <контейнер> -f` — лог не пуст при штатной работе (минимум — старт бота), обновляется в реальном времени.
- **Проверка под нагрузкой:** отправить несколько параллельных тяжёлых запросов (фото, аудио) — контейнер не падает, не уходит в OOM, индикатор виден в Telegram, бот остаётся отзывчивым.
