# Release Notes v1.3.0

- **Дата:** 2026-06-23
- **Тип релиза:** minor

## Кратко

Textify v1.3.0 запускает виральную петлю роста нативными бесплатными средствами Telegram: реферальные deep-link с фиксацией связей и счётчиком приглашённых, inline-кнопка «Поделиться ботом», ненавязчивая подпись под одно-сообщенческим результатом. Все три механики работают без Cloudflare/LLM, не вводят новых зависимостей и сохраняют профиль поставки ADR-001 без изменений. Контракт v1.0.0 не нарушен — релиз MINOR.

## Что нового

### Возможности

- **Реферальные deep-link.** Каждый пользователь получает персональную ссылку `https://t.me/<bot>?start=ref_<user_id>` прямо в ответе `/start`. Переход нового пользователя по такой ссылке атомарно фиксирует связь «кто кого привёл» в новой таблице `referrals` существующей SQLite. Фиксация идемпотентна (`INSERT OR IGNORE` + PRIMARY KEY на `referred_id`): один приглашённый учитывается ровно один раз; повторный `/start`, самоприглашение и обращение уже известного пользователя связь не создают.

- **Счётчик «Приглашено: N» в `/start`.** В ответе `/start` дополнительно выводится число уникальных пользователей, пришедших по реф-ссылке этого пользователя. При сбое БД счётчик деградирует до 0 — `/start` не падает.

- **Inline-кнопка «Поделиться ботом».** К ответу `/start` прикрепляется кнопка, открывающая нативный экран Telegram «Поделиться» (`t.me/share/url`) с персональной реф-ссылкой нажавшего и коротким сопроводительным текстом. Нажатие в один тап пересылает бота в любой чат.

- **Ненавязчивая подпись attribution.** Под одно-сообщенческим результатом распознавания добавляется строка `— @<bot_username>`. Подпись добавляется только когда результат укладывается в одно сообщение и при этом подпись не выталкивает его за лимит `MAX_MESSAGE_LEN`. При серии сообщений и отправке файлом подпись не добавляется (контракт ADR-007 сохранён). Управляется env `ATTRIBUTION_FOOTER` (дефолт on).

- **Реферальные метрики в `/stats`.** Команда `/stats` для администратора теперь дополнительно выводит «Всего рефералов» и «Топ приглашающих» (топ-5 по числу приглашённых, идентификация по user_id). Авторизация по `ADMIN_USER_ID` и отказ не-админу не изменились — реферальные данные не-админу не раскрываются.

### Улучшения

- **Общий держатель username бота (`services/bot_identity.py`).** Username получается один раз на старте через `Bot.get_me()` и хранится в единой точке. Модули `/start`, подпись attribution и групповой обработчик читают из одного источника — без хардкода и без лишних round-trip к API.

- **Атомарное определение «нового пользователя».** `record_message` теперь возвращает `bool` (вставка vs обновление UPSERT); `StatsMiddleware` пробрасывает признак в хендлер через `data["is_new_user"]`. Это единственный корректный способ определить «новизну», поскольку middleware отрабатывает до хендлера — прямая проверка в хендлере всегда давала бы «известен». При сбое учёта middleware деградирует до `False` (тихая деградация, ADR-006).

### Исправления

- Нет багфиксов. Все изменения — новая функциональность и расширение существующих контрактов.

## Изменения публичного контракта

Все изменения аддитивны, обратимо-совместимы. Контракт v1.0.0 (ADR-010) сохранён:

- `/start` — сохраняется, получает аддитивный вывод: реф-ссылка, счётчик «Приглашено: N», кнопка «Поделиться ботом».
- `/help` — не изменён.
- `/stats` — сохраняется, получает аддитивный блок реферальных метрик (только для админа).
- Формат ответа на медиа — одно-сообщенческий результат аддитивно получает подпись attribution (управляется env, по умолчанию on); серия и файл не изменились.
- **Внутренние сигнатуры (не публичный контракт):** `record_message(...) -> bool` (было `None`) — обратимо-совместимое расширение; `send_result` — поведение расширено, сигнатура не изменена.
- **Схема БД:** добавляется таблица `referrals(referred_id PK, referrer_id, created_at)` и индекс `idx_referrals_referrer` в существующей SQLite. `user_stats` не изменена. Ручной миграции не требуется — таблица создаётся при старте.

## Деплой-заметки

- Новых обязательных секретов нет. Единственная новая env — `ATTRIBUTION_FOOTER` (опциональна, дефолт on); добавлена плейсхолдером в `.env.example` и `deploy/.env.example`. Реальное значение не коммитится. Добавить `ATTRIBUTION_FOOTER=false` в серверный `.env` при необходимости отключить подпись.
- Миграция БД не требуется: таблица `referrals` создаётся при старте в том же volume, что и `user_stats` (права на каталог настроены с v0.6.0, UID 10001).
- Профиль поставки ADR-001 не изменён: long-polling, без `ports:`/Traefik/`edge`, образ из GHCR, `restart: unless-stopped`, пиннинг Telegram, существующий volume.
- Реф-ссылки и кнопка «Поделиться» работают сразу после деплоя — нативный Telegram, без серверной настройки.

## Известные ограничения

- Пользователи, обращавшиеся к боту до этого релиза, для целей рефералов считаются «известными» — ретроспективной фиксации нет (ОГР-2, осознанно).
- Подпись attribution не добавляется, если текст результата близок к границе `MAX_MESSAGE_LEN` и добавление подписи вытолкнуло бы за лимит — в этом случае текст уходит как есть без подписи.
- Кнопка «Поделиться ботом» доступна только из `/start`; под результатом распознавания кнопки нет (отложено).
- Реферальные метрики в `/stats` идентифицируют приглашающих по Telegram user_id без имён.

## E2E / Playwright

Textify — Telegram-бот без web UI. E2E-тесты на Playwright неприменимы. Release gate в `.github/workflows/` для e2e не создаётся.
