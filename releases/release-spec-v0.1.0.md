# Спецификация релиза v0.1.0

- **Дата планирования:** 2026-06-22
- **Тип релиза:** minor (первый релиз продукта; репозиторий без тегов, базовая версия 0.x)
- **Цели:** доставить минимальный работающий и деплоящийся каркас Telegram-бота Textify
  (long-polling, команды `/start` и `/help`, ответ-заглушка на текст) и проверить канал поставки
  на сервер ivahub до появления логики распознавания.

## Источники
- BA-требования: `ba-req/ba-requirements-001.md`
- ADR: `adrs/adr-001.md`

## Объём релиза

Минимальность обязательна (см. `CLAUDE.md`, принцип «Минимальные изменения»): только каркас.
OCR, аудио и Groq — вне рамок (релизы v0.2.0–v0.4.0).

### Задача 1: Конфигурация из окружения
- **Что:** загрузка `BOT_TOKEN` из переменной окружения (с подхватом `.env` через `python-dotenv`)
  и валидация: при отсутствующем/пустом токене бот не стартует «молча», а завершается с понятной
  диагностикой.
- **Где (файлы/модули):** `config.py` (или модуль конфигурации в корне проекта; имя — на усмотрение
  разработчика, но единая точка чтения окружения).
- **Контракт:** читается только переменная `BOT_TOKEN`. Никаких ключей распознавания
  (`GROQ_API_KEY` и т. п.) в этом релизе не читать. Токен нигде не хардкодится.
- **Критерии готовности:** есть единая функция/объект конфигурации; пустой/незаданный `BOT_TOKEN`
  приводит к завершению процесса с сообщением об ошибке конфигурации (ненулевой код выхода).
- **Связь с ADR:** ADR-001, «Решение», п. 4.

### Задача 2: Точка входа и запуск long-polling
- **Что:** `bot.py` создаёт `Bot`/`Dispatcher` aiogram, регистрирует роутеры из `handlers/` и
  запускает long-polling. Входящий HTTP-порт не используется.
- **Где (файлы/модули):** `bot.py`.
- **Контракт:** запуск командой `python bot.py`; режим — long-polling; никаких `ports`/webhook.
- **Критерии готовности:** `python bot.py` с валидным `.env` поднимает бота и он отвечает на `/start`.
- **Связь с ADR:** ADR-001, «Решение», пп. 1 и 4.

### Задача 3: Хендлеры команд и текста
- **Что:**
  - `/start` — приветствие с описанием назначения бота и явным указанием, что функции
    распознавания (аудио, изображения, структурирование) появятся в следующих релизах;
  - `/help` — короткая справка;
  - обычное текстовое сообщение (не команда) — ответ-заглушка (эхо присланного текста либо
    подсказка), ответ выдаётся всегда.
- **Где (файлы/модули):** `handlers/` (роутеры aiogram; разбивка на файлы — на усмотрение
  разработчика, например `handlers/commands.py` и `handlers/text.py`), с экспортом роутеров для
  регистрации в `bot.py`.
- **Контракт:** тексты бота — на русском языке (НФТ-5). Медиа-вложения (фото, голос, аудио,
  документы) профильной логикой не обрабатываются.
- **Критерии готовности:** все три сценария отвечают; роутеры подключены к диспетчеру в `bot.py`.
- **Связь с ADR:** ADR-001, «Влияние на код».

### Задача 4: Каркас каталогов и зависимости
- **Что:** каталог `services/` как заготовка под будущую логику распознавания (пустой модуль или
  `__init__.py`/заглушка, без бизнес-логики); `requirements.txt` с минимальным набором.
- **Где (файлы/модули):** `services/`, `requirements.txt`. (`handlers/` создаётся в Задаче 3.)
- **Контракт:** `requirements.txt` содержит как минимум `aiogram` и `python-dotenv`. Зависимости
  распознавания (faster-whisper, pytesseract, Groq SDK, ffmpeg-обёртки) НЕ включать.
- **Критерии готовности:** каталоги `handlers/` и `services/` существуют и являются пакетами;
  `pip install -r requirements.txt` ставит только каркасные зависимости.
- **Связь с ADR:** ADR-001, «Влияние на код».

### Задача 5: Артефакты поставки (профиль «бот без HTTP-порта»)
- **Что:** артефакты деплой-контракта с учётом осознанного отклонения, зафиксированного в
  ADR-001. Применимые правила `deploy-guide.md` соблюдаются; неприменимые (Traefik, `edge`,
  `loadbalancer.server.port`, `ports:`, health за прокси, общие БД/Redis) — НЕ добавляются.
  - `Dockerfile` — multi-stage, по возможности non-root, базовый образ python-slim; запуск
    `python bot.py`; БЕЗ `EXPOSE`/портов;
  - `deploy/docker-compose.yml` — production-манифест: `image:` из GHCR (НЕ `build:`),
    `restart: unless-stopped`, `env_file: .env`; БЕЗ `ports:`, БЕЗ traefik-лейблов, БЕЗ сети
    `edge`, БЕЗ своих `postgres`/`redis`;
  - `deploy/.env.example` — плейсхолдер `BOT_TOKEN` (и только то, что реально читает приложение);
    реальный `.env` не коммитится;
  - корневой `.env.example` — плейсхолдер `BOT_TOKEN` для локального запуска (ФТ-7);
  - `.github/workflows/deploy.yml` — активировать push-триггер (раскомментировать `push: branches:
    [main]`), нацелив деплой на сервер **ivahub**, а не на `app-01`.
- **Где (файлы/модули):** `Dockerfile`, `deploy/docker-compose.yml`, `deploy/.env.example`,
  `.env.example`, `.github/workflows/deploy.yml`.
- **Контракт:** см. `deploy-guide.md` (применимые железные правила) и ADR-001 (раздел «Осознанное
  отклонение»). Адресат деплоя и путь образа — ivahub: значения берутся из секретов/переменных
  CI; реальные хост/ключи в репозиторий не коммитятся. Если используются org-секреты `semsys-ams`,
  завязанные на `app-01`, — деплой ДОЛЖЕН быть перенацелен на ivahub (через секреты репозитория
  либо явные значения хоста/образа), иначе релиз уйдёт не на тот сервер.
- **Критерии готовности:** прод-манифест валиден и соответствует профилю отклонения; в репозитории
  нет реального `.env`; push-триггер активен; деплой нацелен на ivahub.
- **Связь с ADR:** ADR-001, «Осознанное отклонение от deploy-guide.md» и «Решение», п. 3.

## Изменения публичного контракта

Первый релиз — предшествующего публичного контракта нет, ломать нечего. Вводимый контракт:

- пользовательский интерфейс бота: команды `/start`, `/help`, ответ-заглушка на текст;
- конфигурация: переменная окружения `BOT_TOKEN` (обязательна);
- поставка: образ из GHCR + `deploy/docker-compose.yml` (профиль «бот без HTTP-порта»),
  деплой на ivahub.

Миграционный путь не требуется.

## Тесты

Минимальный набор, обязательный в этом релизе:

- **Т-1. Конфигурация — валидный токен:** при заданном `BOT_TOKEN` конфигурация загружается и
  отдаёт непустой токен.
- **Т-2. Конфигурация — отсутствует/пустой токен:** при незаданном/пустом `BOT_TOKEN` загрузка
  конфигурации завершается ошибкой (исключение/ненулевой выход), а не молча. (КП-4)
- **Т-3. Регистрация хендлеров:** роутеры из `handlers/` фактически зарегистрированы в диспетчере
  (проверка, что обработчики `/start`, `/help` и текста подключены).
- **Т-4. Поведение хендлеров (по возможности, через фикстуры aiogram):** `/start` отвечает
  приветствием с упоминанием будущих функций; `/help` — справкой; текст — ненулевым ответом-заглушкой.

Тесты — `pytest` (см. `CLAUDE.md`). Сетевые вызовы к Telegram не выполняются — использовать
заглушки/фикстуры.

## Риски и митигации

- **Риск:** деплой уйдёт на `app-01` вместо ivahub из-за org-секретов шаблона. →
  **Митигация:** Задача 5 обязывает перенацелить деплой на ivahub; мейнтейнер проверяет адресат в
  чек-листе перед `/deploy`.
- **Риск:** разработчик «починит» манифест под Traefik/`edge`, не зная об отклонении. →
  **Митигация:** отклонение явно зафиксировано в ADR-001 и продублировано как контракт Задачи 5.
- **Риск:** в образ/`requirements.txt` просочатся тяжёлые зависимости распознавания. →
  **Митигация:** КП-9 и контракт Задачи 4 запрещают их; арх-контроль и ревью проверяют состав.
- **Риск:** утечка `BOT_TOKEN`. → **Митигация:** токен только в `.env` (покрыт `.gitignore`),
  в репозитории — `.env.example`; в коде не хардкодится.
- **Риск (будущий, фиксируется заранее):** ресурс ivahub 1 vCPU / 1.9 GB RAM. Для каркаса нагрузки
  нет; выбор тяжёлых моделей (whisper) и их ресурсный бюджет — забота релизов v0.2.0–v0.4.0.
