# Спецификация релиза v0.3.0

- **Дата планирования:** 2026-06-22
- **Тип релиза:** minor (обратимо-совместимое добавление функции транскрипции аудио; публичный
  контракт v0.1.0/v0.2.0 не ломается).
- **Цели:** добавить вторую функцию распознавания — транскрипцию голосовых сообщений и аудиофайлов
  в текст через faster-whisper (ru/en, автоопределение). Пользователь присылает голосовое/аудио —
  бот возвращает **сырой** транскрипт. Функция работает на ivahub **автономно** (веса модели в
  образе, без скачивания в рантайме).

## Источники
- BA-требования: `ba-req/ba-requirements-003.md`
- ADR: `adrs/adr-003.md` (база — `adrs/adr-001.md`, `adrs/adr-002.md`)

## Объём релиза

Минимальность обязательна (`CLAUDE.md`, «Минимальные изменения»). В рамках — только транскрипция
аудио. **Вне рамок** (НЕ реализовывать): структурирование через Groq и любые Groq-зависимости /
`GROQ_API_KEY` (v0.4.0); лимиты длительности/размера аудио, индикация прогресса («распознаю...»),
очередь/ограничение параллелизма, разбивка длинного транскрипта, расширенная обработка ошибок и
фолбэки (v0.5.0); выбор пользователем языка; модели whisper крупнее `base`; хранение аудио/
транскриптов, аналитика, БД. Из обработки ошибок — только базовое сообщение о нераспознанной речи.

### Задача 1: Сервис транскрипции (`services/transcribe.py`)
- **Что:** асинхронная обёртка над faster-whisper, транскрибирующая аудио в текст с
  автоопределением языка (ru/en). Синхронный CPU-bound вызов `WhisperModel.transcribe` выполняется
  вне event loop через `asyncio.to_thread` (ADR-003, Решение 1). Модель — `base`,
  `compute_type="int8"`, `device="cpu"` (ADR-003, Решение 2), грузится **один раз** ленивым
  потокобезопасным singleton (ADR-003, Решение 5), не на каждый запрос. Возвращает строку сырого
  транскрипта (для пустого результата — пустая/«пробельная» строка; решение об ответе принимает
  хендлер).
- **Где (файлы/модули):** `services/transcribe.py`.
- **Контракт:**
  - публичная асинхронная функция, принимающая аудио как байты (или путь/буфер на усмотрение
    разработчика, но **не** объект aiogram) и возвращающая `str`;
  - жёстко зафиксировано: `WhisperModel("base", device="cpu", compute_type="int8")`; язык
    **не** передаётся явно — faster-whisper определяет ru/en автоматически;
  - сегменты, которые отдаёт `transcribe`, склеиваются в единый текст; результат `.strip()`-уется
    на стороне сервиса или хендлера согласованно (фиксируется в реализации);
  - экземпляр `WhisperModel` создаётся единожды (ленивый singleton с потокобезопасной
    инициализацией, т.к. вызовы идут из разных потоков `to_thread`) и переиспользуется;
  - модуль **не** импортирует aiogram и не обращается к Telegram — только faster-whisper;
  - модуль **не** инициирует сетевое скачивание весов: рассчитывает на локальный кэш `HF_HOME`,
    предзагруженный в образе (Задача 5); offline обеспечивается env `HF_HUB_OFFLINE=1`.
- **Критерии готовности:** функция асинхронная; на коротком аудио с речью возвращает непустой текст;
  блокирующий `transcribe` вынесен в поток (не исполняется напрямую в корутине); модель не
  создаётся повторно на каждый вызов; при аудио без распознаваемой речи возвращает пустой/
  пробельный результат без исключения.
- **Связь с ADR:** ADR-003, Решения 1, 2, 5.

### Задача 2: Хендлер аудио (`handlers/audio.py`)
- **Что:** роутер aiogram, принимающий:
  - голосовое сообщение (`message.voice`) — ФТ-1;
  - аудиофайл (`message.audio`) — ФТ-2;
  - аудио как файл-документ (`message.document` с audio-MIME) — ФТ-3;
  скачивающий файл штатным клиентом aiogram (`bot.download`) и вызывающий
  `services/transcribe.py`. При непустом результате отправляет пользователю **сырой** транскрипт;
  при пустом — понятное русскоязычное сообщение, что речь не распознана (ФТ-7).
- **Где (файлы/модули):** `handlers/audio.py`; экспорт роутера через `handlers/__init__.py`.
- **Контракт:**
  - три точки приёма: `F.voice`, `F.audio`, `F.document` с `mime_type`, начинающимся с `audio/`;
    не-audio документы этим роутером **не** перехватываются (чтобы не конфликтовать с image-
    роутером по документам — image ловит `image/`, audio ловит `audio/`);
  - скачивание — только штатным механизмом aiogram, без кастомных URL/хостов (ADR-003, Решение 8,
    риск Р-2);
  - ответ при непустом результате — ровно сырой транскрипт без форматирования/структурирования
    (ФТ-6, КП-5);
  - тексты системных сообщений — на русском (НФТ-3);
  - хендлер **не** импортирует faster-whisper напрямую — только `services/transcribe.py`.
- **Критерии готовности:** все три типа вложений (voice / audio / audio-документ) доходят до сервиса
  и дают ответ; пустой результат → сообщение о нераспознанной речи; не-audio документы хендлером не
  перехватываются.
- **Связь с ADR:** ADR-003, Решения 6, 8.

### Задача 3: Регистрация audio-роутера и порядок обработки
- **Что:** зарегистрировать audio-роутер в `bot.py` так, чтобы голосовое/аудио/audio-документ
  обрабатывалось audio-хендлером, а не текстовой заглушкой `handlers/text.py` (её фильтр
  `~Command("start","help")` перехватывает любое не-командное сообщение).
- **Где (файлы/модули):** `bot.py`, `handlers/__init__.py`.
- **Контракт:** audio-роутер включается в диспетчер **до** текстового роутера (`text_router`).
  Текущий порядок — `commands → image → text`; новый порядок —
  `commands → image → audio → text` (audio до text). Обработка изображений, обычного текста и
  команд `/start`, `/help` сохраняется без изменений поведения.
- **Критерии готовности:** голосовое/аудио/audio-документ обрабатывается audio-хендлером (не
  заглушкой); изображения по-прежнему обрабатываются OCR; обычный текст по-прежнему получает
  заглушку; команды работают как раньше.
- **Связь с ADR:** ADR-003, Решение 7.

### Задача 4: Зависимости (`requirements.txt`)
- **Что:** добавить `faster-whisper`. Транзитивно подтянутся `ctranslate2`, `av`, `tokenizers`,
  `onnxruntime`, `huggingface-hub` — их **не** нужно фиксировать явно (pip разрешит), если этого
  достаточно для импорта и запуска.
- **Где (файлы/модули):** `requirements.txt`.
- **Контракт:** добавляется только `faster-whisper` (плюс автоматические транзитивные). НЕ
  добавлять Groq SDK и любые зависимости структурирования (КП-11). Системный `ffmpeg` через pip
  не ставится — декодирование идёт через транзитивный `av` (ADR-003, Решение 4).
- **Критерии готовности:** `pip install -r requirements.txt` ставит faster-whisper и его
  транзитивные зависимости; импорт `from faster_whisper import WhisperModel` работает; Groq-
  зависимости отсутствуют.
- **Связь с ADR:** ADR-003, Решения 3, 4, «Влияние на код».

### Задача 5: Вшивание весов модели и env в Docker-образ (`Dockerfile`)
- **Что:** в **runtime**-стейдж `Dockerfile` (по аналогии с Tesseract из ADR-002) добавить:
  1. переменную окружения `HF_HOME=/opt/models` (фиксированный кэш-каталог модели);
  2. шаг **предзагрузки** весов модели `base` на этапе сборки — запустить Python, который
     инстанцирует `WhisperModel("base", device="cpu", compute_type="int8")`, что скачает веса в
     `HF_HOME`. Шаг выполняется на CI-раннере (где сеть рабочая), **до** `USER appuser`;
  3. рантайм-переменные offline-режима: `HF_HUB_OFFLINE=1` и `TRANSFORMERS_OFFLINE=1`, чтобы в
     рантайме faster-whisper не пытался ходить в сеть за моделью;
  4. права на `HF_HOME`, обеспечивающие чтение каталога пользователем `appuser` (предзагрузка идёт
     root-ом до переключения пользователя; каталог должен остаться читаемым для non-root).
  Системный `ffmpeg` через `apt` **не** ставится (ADR-003, Решение 4). Уже установленные пакеты
  Tesseract (`tesseract-ocr*` из v0.2.0) — **сохранить** без изменений.
- **Где (файлы/модули):** `Dockerfile`.
- **Контракт:**
  - предзагрузка весов и env `HF_HOME` находятся в **runtime**-стейдже (в builder бесполезны —
    туда не копируется кэш модели; см. ADR-003, Решение 3, механизм варианта B);
  - в собранном образе при `HF_HUB_OFFLINE=1` и **без сети** инстанцирование
    `WhisperModel("base", ...)` находит веса в `HF_HOME` и не идёт в сеть;
  - сохраняются multi-stage, non-root (`USER appuser`), запуск `python bot.py`, отсутствие
    `EXPOSE`/портов; профиль поставки ADR-001 не меняется;
  - размер образа удерживается: чистка кэшей в тех же слоях (`pip --no-cache-dir` уже есть; для apt
    при наличии установок — `rm -rf /var/lib/apt/lists/*`); лишние временные файлы предзагрузки не
    оставлять (ADR-003, Решение 2, «Отрицательные последствия» — рост образа);
  - **новых пользовательских env-переменных нет**: HF-переменные задаются в `Dockerfile`, а не в
    `.env` → `deploy/docker-compose.yml`, `deploy/.env.example`, `.env.example`,
    `.github/workflows/deploy.yml` править **не нужно** (зафиксировать в чек-листе как «без
    изменений»).
- **Критерии готовности:** в собранном образе присутствуют веса `base` в `HF_HOME`; транскрипция
  работает в контейнере без скачивания модели из интернета; offline-режим включён
  (`HF_HUB_OFFLINE=1`); образ остаётся non-root; кэши очищены; системный `ffmpeg` через apt не
  добавлен.
- **Связь с ADR:** ADR-003, Решения 2, 3, 4; деплой-контракт — ADR-001 (профиль «бот без
  HTTP-порта»), `deploy-guide.md`.

## Изменения публичного контракта

Обратимо-совместимое добавление (minor):
- бот начинает обрабатывать голосовые (`voice`), аудио (`audio`) и audio-документы → возврат сырого
  транскрипта; ранее на голосовое/аудио приходила текстовая заглушка;
- OCR на изображения, команды `/start`/`/help`, текстовая заглушка на обычный текст и конфигурация
  (`BOT_TOKEN`) — без изменений;
- поставка (профиль «бот без HTTP-порта», деплой на ivahub) — без изменений; новых пользовательских
  env-переменных нет. Миграционный путь не требуется. На сервере при первом деплое потребуется
  pull нового, более крупного образа (см. риск Р-3).

## Тесты

Минимальный обязательный набор (`pytest`; сетевые вызовы к Telegram и HuggingFace не выполняются —
мок/фикстуры). Тесты, требующие тяжёлой модели/`av` в окружении, при их отсутствии помечаются
`skip` с явной причиной (не удаляются) — по образцу `TESSERACT_SKIP` в `tests/test_ocr.py`.

- **Т-1. Транскрипция короткого аудио (сервис).** Либо сгенерировать/использовать фикстурное
  короткое аудио и убедиться, что `services/transcribe.py` возвращает непустую строку; либо —
  если модель/аудио недоступны в среде — **замокать** faster-whisper на уровне
  `WhisperModel.transcribe` (или фабрики модели), проверив, что обёртка корректно склеивает
  сегменты в текст и возвращает `str`. (КП-1, КП-4, ФТ-5, ФТ-6)
- **Т-2. Пустой результат.** Когда faster-whisper возвращает отсутствие сегментов (мок пустого
  результата или аудио-тишина), сервис возвращает пустой/пробельный `str` без исключения. (КП-6,
  ФТ-7)
- **Т-3. Регистрация audio-роутера и порядок.** `audio_router` фактически зарегистрирован в
  диспетчере; в нём есть хендлеры для voice/audio/audio-документа (минимум — для голосового и
  аудио); порядок относительно `text_router` обеспечивает, что аудио не перехватывается заглушкой
  (`audio_router` индексируется раньше `text_router`). По образцу `test_handlers.py`. (КП-7,
  ADR-003, Решение 7)
- **Т-4. Код не ходит в сеть за моделью (offline).** Проверяемо одним из способов:
  - тест, что сервис не падает/работает при `HF_HUB_OFFLINE=1` (модель берётся из локального кэша),
    при наличии предзагруженного `HF_HOME`; **или**
  - тест-проверка, что инстанцирование модели использует фиксированный кэш-путь и не инициирует
    сетевую загрузку (через мок/проверку, что при offline-режиме и отсутствии сети не возникает
    попытки download); **или**
  - проверка наличия непустого каталога модели внутри `HF_HOME` в собранном образе.
  Выбор способа — за реализацией; цель — зафиксировать автономность в рантайме (КП-8, КП-9,
  НФТ-2). При отсутствии модели в среде CI/локали — `skip` с явной причиной.

> Существующие тесты (`test_config.py`, `test_handlers.py`, `test_ocr.py`) должны продолжать
> проходить. `test_handlers.py` обновляется: добавляется регистрация `audio_router` в фикстуру `dp`
> и проверки Т-3.

## Риски и митигации

- **Р-1. OOM на 1.9 GB RAM.** `base`+`int8` помещается впритык; несколько одновременных длинных
  аудио (каждое держит буферы декодирования + конкурирует за модель) могут исчерпать память. →
  **Митигация:** модель грузится один раз (singleton, Задача 1), не плодит копии; `asyncio.to_thread`
  сохраняет отзывчивость; жёсткое ограничение параллелизма/очередь отнесены к v0.5.0. Известный риск,
  приемлем для текущих объёмов.
- **Р-2. Скачивание файла завязано на пиннинг `api.telegram.org`** (`extra_hosts` в compose). →
  **Митигация:** скачивать только штатным клиентом aiogram через `api.telegram.org` (без кастомных
  хостов) — тот же существующий пиннинг; при сбое подобрать рабочий DC в `extra_hosts`. Отдельных
  действий в этом релизе не требует.
- **Р-3. Рост размера образа** (веса модели `base` + `ctranslate2`/`av`/`onnxruntime`). Влияет на
  время CI build/push и pull на ivahub. → **Митигация:** multi-stage и чистка кэшей (Задача 5);
  системный `ffmpeg` не ставится; на ivahub свободно ~22 GB — приемлемо.
- **Р-4. Доступность весов модели в рантайме (КРИТИЧНО).** Если веса не вшиты или offline-режим не
  включён, faster-whisper попытается скачать модель с HuggingFace — на ivahub (битая маршрутизация)
  это упадёт/зависнет, функция неработоспособна. → **Митигация:** Задача 5 предзагружает веса в
  образ на этапе сборки и включает `HF_HUB_OFFLINE=1`/`HF_HOME`; Т-4 и арх-контроль проверяют
  автономность. **Блокирующий критерий приёмки** (КП-8/КП-9).
- **Р-5. PyAV не декодирует какой-то формат Telegram-аудио.** faster-whisper читает через `av`;
  для opus/.oga и типовых форматов кодеки в сборке `av` присутствуют. → **Митигация:** решение
  «без системного ffmpeg» зафиксировано (ADR-003, Решение 4); если на практике встретится
  недекодируемый формат — добавление системного `ffmpeg` в runtime-стейдж будет отдельной правкой.
- **Р-6. Просачивание Groq-зависимостей.** → **Митигация:** КП-11 и контракт Задачи 4 запрещают;
  арх-контроль и ревью проверяют состав `requirements.txt`/образа.
- **Р-7. Аудио перехватывает текстовая заглушка** (фильтр `~Command` в `handlers/text.py`). →
  **Митигация:** Задача 3 фиксирует порядок роутеров (audio до text); Т-3 проверяет.
