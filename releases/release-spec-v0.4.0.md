# Спецификация релиза v0.4.0

- **Дата планирования:** 2026-06-22
- **Тип релиза:** minor (улучшение существующей функции OCR; публичный контракт OCR не ломается —
  на вход то же фото/изображение-документ, на выходе тот же сырой текст, просто точнее).
- **Цели:** повысить точность распознавания текста на изображениях за счёт предобработки картинки
  (OpenCV) перед Tesseract и более удачных настроек движка, не меняя пользовательский контракт;
  добавить ненавязчивую UX-подсказку слать изображение файлом; переназначить слот roadmap v0.4.0.

## Источники
- BA-требования: `ba-req/ba-requirements-004.md`
- ADR: `adrs/adr-004.md` (база — `adrs/adr-002.md` (OCR, неблокирующий сервис), `adrs/adr-001.md`
  (профиль поставки))

## Объём релиза

Минимальность обязательна (`CLAUDE.md`, принцип «Минимальные изменения»). В рамках — только
улучшение OCR + UX-подсказка + roadmap. **Вне рамок** (НЕ реализовывать): структурирование через
Groq (`services/structure.py`, `GROQ_API_KEY`, Groq SDK — отложено из-за гео-блока RU IP), изменения
аудио-канала (faster-whisper), лимиты размера входного файла / индикация прогресса / разбивка
длинного текста / расширенная обработка ошибок приёма (прежний UX-слот), изменение способа приёма и
скачивания изображений, выбор языка пользователем, PDF/многостраничные файлы, хранение/аналитика/БД,
тяжёлый деноизинг (`fastNlMeansDenoising`), формальные метрики качества «до/после».

### Задача 1: Модуль предобработки (`services/preprocess.py`)
- **Что:** новый модуль с конвейером предобработки изображения на OpenCV/PIL/numpy. Публичная
  функция принимает изображение (байты или объект изображения — на усмотрение разработчика, но не
  путь к файлу и не объект aiogram) и возвращает подготовленное изображение в виде, пригодном для
  подачи в `pytesseract` (numpy-массив или `PIL.Image`). Порядок операций строго по ADR-004,
  Решение 2:
  1. EXIF-ориентация — `PIL.ImageOps.exif_transpose`.
  2. Grayscale.
  3. Кэп размера: если большая сторона > 2500 px — пропорциональный даунскейл до 2500 px.
  4. Апскейл мелких: если большая сторона < 1000 px — пропорциональный апскейл до ~1500 px, но не
     выше кэпа 2500 px.
  5. Лёгкий деноизинг — `cv2.medianBlur` малым ядром (тяжёлый деноизинг запрещён).
  6. Адаптивная бинаризация — `cv2.adaptiveThreshold` (Gaussian, `THRESH_BINARY`), умеренные
     `blockSize`/`C`.
  7. Deskew — оценка угла по `cv2.minAreaRect` облака текстовых пикселей, поворот `cv2.warpAffine`
     на белом фоне; применяется только при угле в диапазоне ±15°, иначе пропускается.
- **Где (файлы/модули):** `services/preprocess.py` (новый).
- **Контракт:** модуль импортирует только `cv2`/`numpy`/`PIL` — **не** импортирует `pytesseract`,
  Tesseract и aiogram. Функция детерминированно возвращает подготовленное изображение и не выполняет
  OCR. Числовые параметры (`blockSize`, `C`, ядро `medianBlur`, пороги 1000/1500/2500 px, диапазон
  угла) — на усмотрение разработчика в рамках методов/порядка/кэпов, заданных ADR-004.
- **Критерии готовности:** конвейер выполняет все семь шагов в указанном порядке; возвращает
  корректное изображение для разных входов (цветное/серое, мелкое/крупное, с EXIF-ориентацией);
  не падает на пустом/однотонном/очень маленьком изображении; большая сторона результата не
  превышает 2500 px; тяжёлый `fastNlMeansDenoising` не используется.
- **Связь с ADR:** ADR-004, Решения 1, 2, 3, 4, 5 (кэп).

### Задача 2: Интеграция предобработки и настроек Tesseract в OCR (`services/ocr.py`)
- **Что:** `recognize_text` перед вызовом Tesseract прогоняет изображение через
  `services/preprocess.py`, затем вызывает `pytesseract.image_to_string` с `lang="rus+eng"` и
  `config="--oem 1 --psm 3"`. Предобработка + OCR выполняются в одном вызове `asyncio.to_thread`
  (одна выгрузка в поток на запрос). Реализован фолбэк: при любом исключении предобработки в
  Tesseract подаётся исходное изображение (или его grayscale), ошибка логируется.
- **Где (файлы/модули):** `services/ocr.py`.
- **Контракт:** публичная сигнатура `recognize_text` снаружи **не меняется** (байты изображения →
  `str`); модуль по-прежнему не импортирует aiogram. Жёстко: `lang="rus+eng"`, `--oem 1`, `--psm 3`.
- **Критерии готовности:** контракт `recognize_text` сохранён (та же сигнатура и тип возврата);
  блокирующая работа (предобработка + Tesseract) не исполняется напрямую в корутине (вынесена в
  поток); используются `--oem 1 --psm 3` и `rus+eng`; при искусственно сломанной предобработке
  функция возвращает результат OCR по исходному изображению, а не исключение.
- **Связь с ADR:** ADR-004, Решения 3, 4, 6.

### Задача 3: UX-подсказка слать изображение файлом
- **Что:** ненавязчивая рекомендация присылать изображение как файл (документ), а не «фото», чтобы
  Telegram не пересжимал его в JPEG. Размещается в двух местах: строка в `HELP_TEXT`
  (`handlers/commands.py`) и дополнение `NO_TEXT_MESSAGE` (`handlers/image.py`). Логика приёма
  изображений не меняется.
- **Где (файлы/модули):** `handlers/commands.py` (`HELP_TEXT`), `handlers/image.py`
  (`NO_TEXT_MESSAGE`).
- **Контракт:** только текст сообщений; обработчики, фильтры и маршрутизация без изменений.
- **Критерии готовности:** `/help` содержит подсказку слать изображение файлом; сообщение о
  нераспознанном тексте содержит ту же подсказку; формулировки на русском, ненавязчивые; приём
  фото/документов работает как прежде.
- **Связь с ADR:** ADR-004, Решение 8.

### Задача 4: Зависимости (`requirements.txt`)

- **Где (файлы/модули):** `requirements.txt`.
- **Контракт:** Groq SDK и любые зависимости структурирования **не добавляются**; зависимости

- **Критерии готовности:** в `requirements.txt` присутствуют `opencv-python-headless` и `numpy`;
  обычный `opencv-python` отсутствует; Groq-зависимостей нет.
- **Связь с ADR:** ADR-004, Решение 7.

### Задача 5: Системные библиотеки для cv2 и проверка образа (`Dockerfile`)
- **Что:** в runtime-стейдже `Dockerfile` (до `USER appuser`, в существующем или отдельном
  `apt-get`-слое с чисткой кэша) добавить системные библиотеки для headless-сборки OpenCV:
  `libglib2.0-0` и `libgomp1`. `libGL`/`libgl1` для headless **не добавлять**. Добавить проверочный
  шаг сборки `RUN python -c "import cv2"` (после `COPY --from=builder /install /usr/local`, чтобы
  cv2 уже был установлен).
- **Где (файлы/модули):** `Dockerfile` (runtime-стейдж).
- **Контракт:** multi-stage и non-root (ADR-001) сохраняются; apt-кэш чистится
  (`rm -rf /var/lib/apt/lists/*`); установка системных пакетов и проверка `import cv2` выполняются
  до переключения на `USER appuser`. Существующая установка Tesseract и предзагрузка весов whisper —
  без изменений.
- **Критерии готовности:** `docker build` проходит; шаг `python -c "import cv2"` успешен внутри
  образа (КП-7); apt-кэш не остаётся в слое; non-root-запуск сохранён; `libgl1` не установлен.
- **Связь с ADR:** ADR-004, Решение 7.

## Изменения публичного контракта

Изменений публичного контракта нет. Пользовательский контракт OCR неизменен: те же входы
(`message.photo` и изображение-документ), тот же сырой распознанный текст без постобработки, то же
сообщение при нераспознанном тексте (дополнено UX-подсказкой). Внутренняя сигнатура
`services/ocr.recognize_text` сохраняется. Новых env-переменных нет — `deploy/.env.example`,
`.env.example`, `deploy/docker-compose.yml`, workflow без изменений. Миграция не требуется: обновляется
образ тем же CI/CD-пайплайном.

## Тесты

Новые тесты в `tests/` (минимум). OCR-тесты, требующие бинаря `tesseract`, делают `skip` с причиной,
если бинаря нет в среде прогона — по образцу `tests/test_ocr.py` (v0.2.0, `TESSERACT_SKIP`).
Тесты предобработки требуют только `cv2`/`numpy`/`PIL` и используют `pytest.importorskip("cv2")` для
skip с причиной, если cv2 недоступен (не требуют tesseract).

- **Предобработка не падает на разных входах:** конвейер возвращает корректное изображение для
  цветного, серого, очень маленького, однотонного/пустого входа и изображения с EXIF-ориентацией;
  большая сторона результата ≤ 2500 px.
- **Бинаризация:** результат предобработки бинарный/почти бинарный (текст отделён от фона) на
  сгенерированном изображении с текстом на неравномерном фоне.
- **Deskew на сгенерированном наклонном тексте:** изображение с заведомо повёрнутым текстом после
  предобработки имеет меньший наклон (проверяемо: оценённый остаточный угол ближе к 0, либо OCR
  узнаёт текст лучше — при наличии tesseract, иначе геометрическая проверка угла без OCR).
- **Фолбэк при ошибке предобработки:** при искусственно вызванном сбое предобработки (например,
  патч/monkeypatch функции предобработки на исключение) `recognize_text` не пробрасывает исключение,
  а возвращает результат OCR по исходному изображению (с tesseract — текст; без tesseract — тест
  под `TESSERACT_SKIP` либо проверка вызова фолбэк-ветки без реального OCR).
- **Контракт OCR сохранён:** `services/ocr.recognize_text` существует, асинхронна, принимает байты,
  возвращает `str`; модуль не импортирует aiogram; используются `--oem 1 --psm 3` и `lang="rus+eng"`
  (проверяемо мок-патчем `pytesseract.image_to_string` без реального бинаря).
- **Регистрация/маршрутизация изображений не сломана:** существующие тесты приёма фото/документов
  (`tests/test_handlers.py`) проходят; UX-подсказка присутствует в `HELP_TEXT` и `NO_TEXT_MESSAGE`.

## Риски и митигации

- **Риск:** скорость на 1 vCPU — предобработка + OCR на крупном фото затягивают ответ. →
  **Митигация:** кэп размера 2500 px по большей стороне, лёгкий `medianBlur` вместо
  `fastNlMeansDenoising`, deskew только в безопасном диапазоне угла; всё в `asyncio.to_thread` —
  event loop остаётся отзывчивым.
- **Риск:** системные deps для cv2 в slim-образе — `import cv2` падает без нужных библиотек. →
  **Митигация:** `opencv-python-headless` + `libglib2.0-0` + `libgomp1`; обязательный проверочный
  шаг `python -c "import cv2"` в `Dockerfile` (блокирует сборку при отсутствии deps); `libGL` не
  требуется для headless.
- **Риск:** регресс качества на «чистых» сканах из-за агрессивной адаптивной бинаризации. →
  **Митигация:** умеренные параметры бинаризации, лёгкий деноизинг до бинаризации, deskew с
  предохранителем по углу; фолбэк на исходное изображение при сбое; проверка отсутствия регресса на
  хорошем вводе при приёмке (НФТ-1/ДОП-2).
- **Риск:** рост размера образа из-за OpenCV/numpy. → **Митигация:** headless-сборка (без GUI),
  чистка apt-кэша; для ivahub (~22 GB свободно) приемлемо.
