# Спецификация релиза v0.5.0

- **Дата планирования:** 2026-06-22
- **Тип релиза:** minor (добавление функциональности; публичный контракт каналов не ломается — на
  вход те же фото/аудио, на выходе тот же распознанный текст, оформленный при доступности LLM либо
  сырой при фолбэке; новые env-переменные опциональны).
- **Цели:** довести продукт до исходного ценностного обещания — на вход голос/аудио или фото, на
  выходе структурированный Markdown; провайдер по умолчанию — бесплатный Cloudflare Workers AI
  (работает с RU-IP ivahub без прокси), Groq — переключаемая опция на будущее; при любом сбое LLM —
  тихий фолбэк на сырой текст; обновить roadmap.

## Объём релиза

Минимальность обязательна (`CLAUDE.md`, принцип «Минимальные изменения»). **Вне рамок** (НЕ
реализовывать): возврат к Groq как дефолту / relay-прокси под Groq; разбивка длинного текста на части
и склейка; пользовательские лимиты размера входа; индикация прогресса (typing/статус); понятные
пользовательские сообщения о таймаутах/ошибках (сейчас только тихий фолбэк); выбор модели/формата
пользователем в рантайме; кэширование/дедуп запросов; изменение логики OCR/транскрипции и способа
приёма/скачивания файлов; изменение профиля поставки (ADR-001); вендорские SDK (Cloudflare/Groq).

### Задача 1: Сервис структурирования с абстракцией провайдера (`services/structure.py`)
- **Что:** новый модуль-оркестратор. Публичная асинхронная функция (ориентир:
  `async def structure_text(raw_text: str) -> str`) принимает сырой распознанный текст и возвращает
  структурированный Markdown. Внутри:
  1. усекает входной текст до безопасного порога символов (константа модуля, ориентир ~8000) перед
     формированием промпта;
  2. строит сообщения с системным промптом структурирования (см. Контракт) и пользовательским
     контентом = усечённый сырой текст;
  3. выбирает реализацию провайдера по `LLM_PROVIDER` (дефолт/незадано → Cloudflare; `groq` →
     Groq-ветка) — через **явную абстракцию провайдера** (общий внутренний контракт + реализации);
  4. выполняет HTTP-вызов нативно асинхронно через `httpx.AsyncClient` с общим таймаутом 30 секунд,
     **единичная попытка, без ретраев**;
  5. при успехе извлекает структурированный текст из ответа (Cloudflare — `result.response`; Groq —
     `choices[0].message.content`) и возвращает его, если он непустой;
  6. при **любом** сбое (сеть/DNS, 4xx/5xx, таймаут, исчерпание лимита, пустой/некорректный ответ,
     отсутствие ключей провайдера) перехватывает исключение, логирует и возвращает **полный
     исходный** сырой текст (не усечённый) — фолбэк.
- **Где (файлы/модули):** `services/structure.py` (новый). Реализации провайдеров — в этом же модуле
  либо подмодуле (на усмотрение), но абстракция провайдера должна быть явной и тестируемой.
- **Контракт:**
  - модуль **не импортирует** aiogram/Telegram (граница ADR-002); знает только про HTTP и провайдеров;
  - Cloudflare-эндпоинт:
    `POST https://api.cloudflare.com/client/v4/accounts/{CF_ACCOUNT_ID}/ai/run/{CF_MODEL}`,
    заголовок `Authorization: Bearer {CF_API_TOKEN}`, тело
    `{"messages":[{"role":"system","content":<промпт>},{"role":"user","content":<сырой текст>}]}`;
    результат — `result.response`;
  - дефолтная модель Cloudflare — `@cf/meta/llama-3.1-8b-instruct`, переопределяема env `CF_MODEL`;
  - Groq-ветка — OpenAI-совместимый `chat/completions` с `GROQ_API_KEY` (код-путь существует и
    выбирается `LLM_PROVIDER=groq`; боевая работа не требуется и не проверяется на проде);
  - системный промпт (rus+eng): привести текст к читаемой структуре Markdown (заголовки/списки/
    ключевые пункты); **сохранять смысл, не выдумывать факты, не добавлять отсутствующее в
    исходнике**; не менять язык контента; вернуть только оформленный текст без преамбул;
  - неблокирование — через `await` нативного async-клиента; `asyncio.to_thread` **не** используется.
- **Критерии готовности:** функция существует, асинхронна, принимает `str`, возвращает `str`;
  при успешном моке HTTP-клиента возвращает структурированный текст из `result.response`; при
  моке, бросающем исключение/возвращающем ошибку/пустой ответ, возвращает **исходный** сырой текст
  без исключения; `LLM_PROVIDER` переключает ветку без правки кода; модуль не импортирует aiogram;
  используется `httpx.AsyncClient` с явным таймаутом; вход усекается до порога перед отправкой, но
  фолбэк отдаёт полный текст.
- **Связь с ADR:** ADR-005, Решения 1, 2, 3, 4, 5, 6, 7.

### Задача 2: Подключение структурирования к каналу изображений (`handlers/image.py`)
- **Что:** в обоих хендлерах (`handle_photo`, `handle_image_document`) после получения сырого
  результата OCR: если текст непустой (`text.strip()`) — прогнать через
  `services.structure.structure_text(...)` и отправить пользователю результат структурирования;
  если пустой — прежнее поведение (`NO_TEXT_MESSAGE`), структурирование **не вызывается** (ФТ-9).
- **Где (файлы/модули):** `handlers/image.py`.
- **Контракт:** хендлер не импортирует провайдера/`httpx` напрямую — только `structure_text` из
  сервиса; маршрутизация, фильтры, скачивание файла — без изменений; на пустом OCR LLM не вызывается.
- **Критерии готовности:** на непустом OCR пользователю уходит результат `structure_text`; на пустом —
  `NO_TEXT_MESSAGE` и `structure_text` не вызван; импорт провайдера/`httpx` в `handlers/image.py`
  отсутствует.
- **Связь с ADR:** ADR-005, Решение 8.

### Задача 3: Подключение структурирования к каналу аудио (`handlers/audio.py`)
- **Что:** в трёх хендлерах (`handle_voice`, `handle_audio`, `handle_audio_document`) после получения
  транскрипта: если непустой — прогнать через `structure_text(...)` и отправить результат; если
  пустой — прежнее поведение (`NO_SPEECH_MESSAGE`), структурирование не вызывается.
- **Где (файлы/модули):** `handlers/audio.py`.
- **Контракт:** аналогично Задаче 2 — хендлер знает только про `structure_text`, не про провайдера/
  `httpx`; приём/скачивание без изменений; на пустом транскрипте LLM не вызывается. Чтобы избежать
  дублирования общей логики в трёх хендлерах, допустима внутренняя хелпер-функция в модуле.
- **Критерии готовности:** на непустом транскрипте уходит результат `structure_text`; на пустом —
  `NO_SPEECH_MESSAGE` без вызова `structure_text`; импорт провайдера/`httpx` отсутствует.
- **Связь с ADR:** ADR-005, Решение 8.

### Задача 4: Конфигурация провайдера (`config.py`)
- **Что:** читать из окружения/`.env`: `LLM_PROVIDER` (дефолт `cloudflare`), `CF_ACCOUNT_ID`,
  `CF_API_TOKEN`, `CF_MODEL` (дефолт `@cf/meta/llama-3.1-8b-instruct`), `GROQ_API_KEY`, опционально
  `GROQ_MODEL`. В отличие от `BOT_TOKEN`, **отсутствие** ключей провайдера при старте **не валит**
  процесс — бот стартует, структурирование деградирует в фолбэк. Способ доступа (расширить словарь
  `config` или читать `os.environ` в сервисе) — на усмотрение, но значения берутся из окружения,
  не хардкодятся.
- **Где (файлы/модули):** `config.py` (и/или `services/structure.py`, если сервис читает env сам).
- **Контракт:** валидация `BOT_TOKEN` сохранена; ключи провайдера читаются без жёсткой валидации;
  дефолт `LLM_PROVIDER=cloudflare` и дефолтная модель применяются при отсутствии переменных.
- **Критерии готовности:** при незаданных `CF_*` бот импортируется/стартует без RuntimeError; при
  незаданном `LLM_PROVIDER` используется Cloudflare; дефолтная модель применяется при отсутствии
  `CF_MODEL`.
- **Связь с ADR:** ADR-005, Решение 9.

## Изменения публичного контракта

Пользовательский контракт каналов **не ломается**: те же входы (фото/изображение-документ; voice/
audio/audio-документ), но теперь ответ — структурированный Markdown при доступности LLM либо сырой
распознанный текст при фолбэке. Сообщения при нераспознанном тексте/речи (`NO_TEXT_MESSAGE`/
`NO_SPEECH_MESSAGE`) сохраняются. Внутренние сигнатуры `services/ocr.recognize_text` и
`services/transcribe.transcribe` не меняются. Новые env-переменные (`LLM_PROVIDER`, `CF_ACCOUNT_ID`,
`CF_API_TOKEN`, `CF_MODEL`, `GROQ_API_KEY`) **опциональны** — их отсутствие не ломает запуск (даёт
фолбэк). Миграция: обновить образ тем же CI/CD-пайплайном и добавить переменные провайдера в `.env`
на ivahub (Сценарий E BA).

## Тесты

Новые тесты в `tests/` (минимум), все — **без реальной сети**: HTTP-клиент мокается (по образцу
мок-патчей в `tests/test_transcribe.py`/`tests/test_handlers.py`). `httpx` доступен в среде прогона
(добавлен в зависимости); если по какой-то причине нет — `pytest.importorskip("httpx")`.

- **Успешное структурирование (Cloudflare):** при мок-ответе с `result.response = "<markdown>"`
  `structure_text` возвращает этот markdown; запрос ушёл на cloudflare-эндпоинт с
  `Authorization: Bearer` и телом с системным+пользовательским сообщениями.
- **Фолбэк при ошибке/таймауте:** при моке HTTP-клиента, бросающем исключение (сеть/таймаут) или
  возвращающем 4xx/5xx/пустой `result.response`, `structure_text` возвращает **исходный сырой
  текст**, а не исключение.
- **Фолбэк при отсутствии ключей провайдера:** при незаданных `CF_ACCOUNT_ID`/`CF_API_TOKEN`
  `structure_text` возвращает сырой текст без падения (LLM не вызывается либо вызов корректно уходит
  в фолбэк).
- **Переключение провайдера:** при `LLM_PROVIDER=groq` выбирается Groq-ветка (другой URL/форма
  запроса), при дефолте/незаданном — Cloudflare; проверяемо мок-патчем без сети.
- **Усечение длинного входа:** на входе длиннее порога в запрос провайдера уходит усечённый текст
  (≤ порога), но при фолбэке возвращается **полный** исходный текст.
- **Канал изображений:** при непустом OCR `handle_photo`/`handle_image_document` вызывают
  `structure_text` и отправляют его результат; при пустом OCR — `NO_TEXT_MESSAGE`, `structure_text`
  не вызван (мок `recognize_text` и `structure_text`).
- **Канал аудио:** при непустом транскрипте `handle_voice`/`handle_audio`/`handle_audio_document`
  вызывают `structure_text` и отправляют его результат; при пустом — `NO_SPEECH_MESSAGE`,
  `structure_text` не вызван (мок `transcribe` и `structure_text`).
- **Граница модуля:** `services/structure.py` не импортирует aiogram; `handlers/image.py` и
  `handlers/audio.py` не импортируют `httpx`/провайдера напрямую (только `structure_text`).
- **Неблокирование:** `structure_text` асинхронна и использует `httpx.AsyncClient` (а не
  `requests`/`asyncio.to_thread`) — проверяемо инспекцией/мок-патчем `httpx.AsyncClient`.
- **Существующие тесты не сломаны:** `tests/test_handlers.py`, `tests/test_transcribe.py`,
  `tests/test_ocr*.py` проходят (маршрутизация и сигнатуры сервисов сохранены).

## Риски и митигации

- **Риск:** Cloudflare-эндпоинт недостижим с ivahub (опровержение ДОП-1). → **Митигация:** фолбэк на
  сырой текст (бот не падает); реальная достижимость проверяется приёмкой деплоем (ОВ-1/КП-13); при
  неподтверждении — выбор дефолтного провайдера пересматривается отдельным ADR.
- **Риск:** исчерпание суточного лимита бесплатного tier → запросы начинают отказывать. →
  **Митигация:** единичная попытка без ретраев (не множит расход), усечение длинного входа (один
  запрос на сообщение), любой отказ → фолбэк на сырой текст.
- **Риск:** повисшее соединение блокирует обработку. → **Митигация:** явный общий таймаут 30 с на
  HTTP-запрос; нативный async (`await`) — event loop остаётся отзывчивым, поток пула не занят.
- **Риск:** модель искажает смысл / дописывает факты. → **Митигация:** системный промпт явно
  запрещает выдумывание и смену языка; качественная проверка на rus/eng примерах при приёмке
  (НФТ-7/ОВ-5); абсолютная точность не гарантируется (ДОП-3).
- **Риск:** утечка секретов провайдера. → **Митигация:** только `.env` (покрыт `.gitignore`); в
  репозитории — лишь плейсхолдеры; ключи не хардкодятся.
- **Риск:** случайное добавление вендорского SDK раздувает образ/связность. → **Митигация:** только
  `httpx`, прямой HTTP; SDK запрещены спецификацией.
