существующих хендлеров OCR/audio/text/structure; внешние БД/Redis.

### Задача 1: Сервис учёта на SQLite (`services/stats.py`)
- **Что:** новый модуль персистентного учёта. Реализовать:
  - `init_db()` — создаёт каталог БД при необходимости, открывает соединение, включает
    `PRAGMA journal_mode=WAL` и `PRAGMA busy_timeout=5000`, создаёт таблицу `user_stats`
    (`CREATE TABLE IF NOT EXISTS`, схема — ADR-006, раздел «Схема хранилища»). Вызывается один раз
    при старте бота.
  - `async def record_message(user_id: int, msg_type: str) -> None` — через `asyncio.to_thread`
    выполняет атомарный UPSERT: для нового `user_id` создаёт строку с
    `first_seen=last_seen=now_utc` и инкрементом счётчика типа =1; для существующего —
    `last_seen=now_utc` и инкремент нужного счётчика. `first_seen` не перезаписывается. Допустимые
    `msg_type`: `photo`, `audio`, `text`, `command`, `other` (маппинг типа на колонку — внутри).
  - `async def get_stats() -> dict` — через `asyncio.to_thread` агрегирует и возвращает:
    `unique_users` (`COUNT(*)`), `total_messages` (сумма всех счётчиков), `photo`/`audio`/`text`/
    `command`/`other` (`SUM` по каждому), `first_seen` (`MIN(first_seen)`), `last_seen`
    (`MAX(last_seen)`); при пустой БД — нули и `first_seen`/`last_seen` = `None`.
  - Соединение-на-операцию (открыть/выполнить/коммит/закрыть внутри каждой функции); общего
    изменяемого соединения между потоками нет.
- **Где (файлы/модули):** новый `services/stats.py`.
- **Контракт:** время — UTC ISO-8601 (`datetime.now(timezone.utc).isoformat()`); путь к файлу БД
  берётся из `config` (Задача 5). Содержимое сообщений не хранится.
- **Критерии готовности:** новый пользователь создаётся; повторные обращения сохраняют `first_seen`
  и двигают `last_seen`; счётчики инкрементируются по типу; `get_stats` агрегирует корректно;
  операции выполняются вне event loop.
- **Связь с ADR:** ADR-006, Решения 3, 4; «Схема хранилища».

### Задача 2: Outer-middleware учёта (`middlewares/stats.py`)
- **Что:** новый слой. Класс-middleware aiogram (`BaseMiddleware`), регистрируемый как
  **outer-middleware** на `dp.message`. На каждое сообщение:
  1. определяет `user_id` из `message.from_user` (если отправителя нет — пропускает учёт, но
     управление передаёт дальше);
  2. классифицирует тип сообщения (см. Контракт);
  3. вызывает `await record_message(user_id, msg_type)` в `try/except` — при любом исключении
     логирует и продолжает (тихая деградация);
  4. **всегда** вызывает и возвращает `await handler(event, data)` — учёт не влияет на обработку.
- **Где (файлы/модули):** новые `middlewares/__init__.py`, `middlewares/stats.py`.
- **Контракт (классификация, приоритетный порядок — ADR-006 Решение 2A):**
  1. `message.text` начинается с `/` → `command`;
  2. `message.photo` → `photo`;
  3. `message.document` и `message.document.mime_type` начинается с `image/` → `photo`;
  4. `message.voice` или `message.audio`, либо `message.document` с `mime_type` на `audio/` →
     `audio`;
  5. иной непустой `message.text` (не команда) → `text`;
  6. всё прочее → `other`.
  Функция классификации должна быть выделена так, чтобы её можно было протестировать юнитом.
- **Критерии готовности:** middleware зарегистрирован как outer на messages; срабатывает один раз
  на сообщение до хендлеров; не меняет вывод хендлеров; сбой `record_message` не прерывает цепочку.
- **Связь с ADR:** ADR-006, Решения 1, 2, 7.

### Задача 3: Команда `/stats` для администратора (`handlers/commands.py`)
- **Что:** добавить хендлер `@router.message(Command("stats"))`. Логика:
  - если `config["ADMIN_USER_ID"]` пуст/не задан → ответ «Команда недоступна.»;
  - если `message.from_user.id != ADMIN_USER_ID` → ответ «Команда недоступна.»;
  - иначе вызвать `await get_stats()` и вернуть отформатированную сводку на русском: уникальные
    пользователи, всего сообщений, разбивка по типам (фото/изображения, аудио/голос, текст, команды,
    прочее), дата первого и дата последнего обращения (UTC; при пустой БД — понятная пометка
    «нет данных»).
- **Где (файлы/модули):** `handlers/commands.py` (новый хендлер; импорт `get_stats`, `config`).
- **Контракт:** не-админ и отсутствующий админ → одинаковый короткий отказ; формат вывода — русский
  Markdown/текст с явными метками типов и времён.
- **Критерии готовности:** админ видит сводку; не-админ получает отказ без раскрытия данных;
  `ADMIN_USER_ID` читается из конфигурации, не хардкодится.
- **Связь с ADR:** ADR-006, Решение 6.

### Задача 5: Конфигурация `ADMIN_USER_ID` и путь к БД (`config.py`)
- **Что:** в `load_config()` добавить:
  - `ADMIN_USER_ID` — читается из ENV; пустое/неустановленное → `None`; непустое валидируется как
    целое (нечисловое → `None` или понятная ошибка конфигурации, на усмотрение реализации, но без
    падения старта бота из-за необязательной переменной);
  - `STATS_DB_PATH` — путь к файлу БД из ENV с дефолтом внутри каталога данных (ориентир:
    `/app/data/stats.db`).
- **Где (файлы/модули):** `config.py`.
- **Контракт:** `ADMIN_USER_ID` — только из окружения/`.env`, не хардкодится; `BOT_TOKEN` остаётся
  обязательным; новые переменные не обязательны для старта (бот стартует без `ADMIN_USER_ID` —
  тогда `/stats` недоступна).
- **Критерии готовности:** `config["ADMIN_USER_ID"]` и `config["STATS_DB_PATH"]` доступны; старт без
  `ADMIN_USER_ID` не падает.
- **Связь с ADR:** ADR-006, Решение 6, 8.

## Тесты
Минимальный набор (на временной SQLite в `tmp_path`, без сети, без реального Telegram):
- Учёт нового пользователя: после `record_message` создаётся строка с `first_seen`/`last_seen` и
  счётчиком типа =1.
- Инкремент счётчиков по типам: отдельные тесты на `photo`, `audio`, `text`, `command`, `other`.
- Идемпотентность `first_seen` / обновление `last_seen`: повторный `record_message` не меняет
  `first_seen`, обновляет `last_seen`, инкрементирует счётчик.
- Классификация типа в middleware: юнит на функцию классификации для всех веток (команда, фото,
  image-документ, voice/audio, audio-документ, обычный текст, прочее).
- Авторизация `/stats`: админ (`from_user.id == ADMIN_USER_ID`) получает сводку; не-админ — отказ;
  пустой `ADMIN_USER_ID` — отказ. (Через мок `Message`/`get_stats`.)
- Тихая деградация при сбое БД: при исключении в `record_message` middleware не пробрасывает
  ошибку и вызывает нижележащий handler (проверка, что цепочка продолжена).
- Формат вывода `/stats`: на заполненной временной БД `get_stats` возвращает корректные агрегаты
  (уникальные, всего, разбивка, first/last); проверка на пустой БД (нули, «нет данных»).

## Риски и митигации
- **Риск:** на сервере не настроены права на каталог данных → `appuser` не может писать в БД,
  персистентность не работает (и при отсутствии тихой деградации мог бы падать учёт). →
  **Митигация:** фиксированный UID/GID 10001 + обязательный шаг `chown` в чек-листе деплоя; тихая
  деградация учёта (Задача 2) защищает основную функцию даже при неверных правах.
- **Риск:** конкурентный доступ к SQLite (частые записи + чтение `/stats`) → блокировки/ошибки. →
  **Митигация:** WAL + `busy_timeout` + соединение-на-операцию (ADR-006 Решение 4); атомарный UPSERT.
- **Риск:** забыли volume в compose → данные теряются при передеплое. → **Митигация:** volume в
  Задаче 6 + пункт в чек-листе деплоя (КП-9); проверка мейнтейнером.
- **Риск:** утечка статистики не-админу при ошибке авторизации. → **Митигация:** единый отказ для
  не-админа и пустого `ADMIN_USER_ID`; тест авторизации.
