парсинг Markdown (ОВ-1 закрыт в пользу plain-text — ADR-007 Решение 1); индикатор прогресса

ADR-001) и артефактов поставки.

### Задача 1: Единый модуль отдачи результата (`services/reply.py`)
- **Что:** новый модуль. Реализовать:
  - **Константы** (именованные, ФТ-8): `TELEGRAM_LIMIT = 4096`; `MAX_MESSAGE_LEN = 4000`
    (безопасный лимit части); `MAX_PARTS = 5` (порог перехода на файл по числу частей);
    `PART_DELAY_SEC = 0.4` (пауза между частями, диапазон 0.3–0.5).
  - `def split_text(text: str, limit: int = MAX_MESSAGE_LEN) -> list[str]` — **чистая функция без
    сети и без Telegram-типов**. Иерархический жадный сплиттер по убыванию приоритета границ:
    двойной перевод строки (абзац `\n\n`) → одиночный перевод строки (`\n`) → конец предложения
    → жёсткий срез по `limit` (крайний случай для аномально длинного «слова»/строки без границ).
    Жадно набирать максимально длинные части ≤ `limit`, **не разрывая слова**. Гарантии:
    каждая часть `len(part) <= limit`; конкатенация частей восстанавливает исходный текст по
    содержимому **без потерь и без дублирования** (допустима нормализация ведущих/замыкающих
    пробелов на стыках); для текста `len <= limit` возвращается список из одного элемента.
  - `async def send_result(message: Message, text: str) -> None` — единая точка отдачи. Логика:
    1. `len(text) <= MAX_MESSAGE_LEN` → одно `await message.answer(text)`;
    2. иначе `parts = split_text(text)`; если `len(parts) <= MAX_PARTS` → отправлять части
       последовательно через `await message.answer(part)`, между **соседними** частями
       `await asyncio.sleep(PART_DELAY_SEC)` (после последней части паузы нет);
    3. если `len(parts) > MAX_PARTS` → отправить **один** файл `.txt` через
       `message.answer_document(BufferedInputFile(text.encode("utf-8"), filename="result.txt"))`
       с короткой русской подписью (например, `caption="Результат целиком во вложении."`);
       серия сообщений в этом режиме НЕ отправляется.
  - **Контракт входа:** `send_result` принимает **уже непустой** результат; решение
    «не распознано/речь не распознана» остаётся в хендлерах. На пустой/пробельной строке
    `send_result` деградирует тихо (без падения), штатно так не вызывается.
- **Где (файлы/модули):** новый `services/reply.py`.
- **Контракт:** plain-text, без `parse_mode` (ADR-007 Решение 1); файл формируется из памяти
  (`BytesIO`/`bytes`, utf-8), без записи на диск; имя файла осмысленное (`result.txt`).
- **Критерии готовности:** `split_text` — чистая функция, каждая часть ≤ `MAX_MESSAGE_LEN`, без
  потерь/дублей, режет по границам; `send_result` реализует три режима; константы вынесены;
  паузы — `asyncio.sleep`; новых зависимостей нет.
- **Связь с ADR:** ADR-007, Решения 2, 3, 4, 6, 7.

### Задача 2: Применение единой отдачи в OCR-хендлере (`handlers/image.py`)
- **Что:** заменить в обоих хендлерах (`handle_photo`, `handle_image_document`)
  `await message.answer(await structure_text(text))` на
  `await send_result(message, await structure_text(text))`. Импортировать `send_result` из
  `services.reply`. Ветка «не распознано» (`NO_TEXT_MESSAGE`) остаётся на прямом
  `await message.answer(NO_TEXT_MESSAGE)` — НЕ через `send_result`.
- **Где (файлы/модули):** `handlers/image.py`.
- **Контракт:** `structure_text(...)` вызывается ровно один раз на путь; результат передаётся в
  `send_result` один раз (без двойной отправки). Поведение OCR/структурирования не меняется.
- **Критерии готовности:** оба пути OCR используют `send_result`; служебное сообщение не разбивается;
  двойной отправки нет.
- **Связь с ADR:** ADR-007, «Влияние на код»; ФТ-1, ФТ-2, ФТ-10.

## Тесты
Минимальный набор. Тесты на разбивку — на **чистой функции** `split_text` (без сети, без Telegram):
- **Короткий → одно сообщение:** `len(text) <= MAX_MESSAGE_LEN` → `split_text` возвращает список из
  одного элемента, равного исходному тексту.
- **Граница ровно на лимите:** `len(text) == MAX_MESSAGE_LEN` → одна часть; `len == MAX_MESSAGE_LEN + 1`
  → две части, каждая ≤ `MAX_MESSAGE_LEN`.
- **Длинный → несколько частей:** каждая часть `len(part) <= MAX_MESSAGE_LEN`; конкатенация частей
  по содержимому равна исходному тексту (ничего не потеряно и не задвоено).
- **Резка по границам:** на тексте из абзацев (`\n\n`) границы частей попадают на границы абзацев/
  строк/предложений, а не в середину слова (ни одна часть не начинается/не заканчивается обрывком
  слова, кроме крайнего случая жёсткого среза).
- **Крайний случай (жёсткий срез):** «слово»/строка без естественных границ длиннее `MAX_MESSAGE_LEN`
  → режется жёстко по лимиту, каждая часть ≤ `MAX_MESSAGE_LEN`, без потерь.
- **Очень длинный → один `.txt`:** при числе частей `> MAX_PARTS` `send_result` вызывает отправку
  документа один раз (через мок `Message.answer_document`), а `Message.answer` не вызывается для
  частей. (Юнит через мок `Message`, без сети.)
- **Серия частей с паузой:** при `MAX_MESSAGE_LEN < len(text)` и числе частей `<= MAX_PARTS`
  `send_result` вызывает `message.answer` по числу частей и `asyncio.sleep` между ними. (Мок
  `Message` + мок/патч `asyncio.sleep`.)
- **Короткий путь не задет:** на коротком тексте `send_result` делает ровно один `message.answer` и
  НЕ вызывает `answer_document` и `asyncio.sleep`.
- **Регресс служебных сообщений:** проверить (на уровне хендлеров/мока), что ветки `NO_TEXT_MESSAGE`/
  `NO_SPEECH_MESSAGE` идут через прямой `message.answer` и не проходят через `send_result`.

## Риски и митигации
- **Риск:** серия сообщений упирается в rate limit Telegram. → **Митигация:** пауза
  `PART_DELAY_SEC` между частями + жёсткое ограничение серии `MAX_PARTS` (далее — один файл).
- **Риск:** некорректные границы разбивки — потеря/дублирование текста или разрыв слов. →
  **Митигация:** инвариант «конкатенация частей == исходный текст по содержимому» и «часть не рвёт
  слово» закреплены тестами на чистой функции; жёсткий срез — только крайний случай.
- **Риск:** превышение 4096 из-за UTF-16-расхождения (`len` в code points vs Telegram в code units)
  на тексте с эмодзи/символами вне BMP. → **Митигация:** запас `MAX_MESSAGE_LEN = 4000` от 4096.
- **Риск:** регресс короткого ответа или служебных сообщений (двойная отправка, разбивка заглушек).
  → **Митигация:** `send_result` принимает только непустой результат; служебные сообщения — прямой
  `message.answer`; тесты на короткий путь и на ветки заглушек; `structure_text` вызывается один раз.
- **Риск:** блокировка event loop паузами/формированием файла. → **Митигация:** паузы —
  `await asyncio.sleep` (не блокирует loop), число пауз ограничено `MAX_PARTS - 1`; формирование
  `BytesIO` — лёгкий CPU (ADR-007 Решение 7; ADR-002/003).
