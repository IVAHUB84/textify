# Спецификация релиза v0.9.0

- **Дата планирования:** 2026-06-23
- **Тип релиза:** minor (обратимо-совместимое добавление функциональности; контракт каналов и
  сигнатура `send_result` не ломаются)
- **Цели:** дать пользователю быстрые действия над уже полученным результатом — inline-кнопки
  «Кратко» (саммари 3–5 пунктов) и «Перевести» (перевод ru↔en) под текстовым результатом OCR/аудио;
  по нажатию — CF Workers AI LLM над текстом сообщения, ответ через `send_result`.

## Источники

- BA-требования: `ba-req/ba-requirements-009.md`
- ADR: `adrs/adr-009.md` (основной); переиспользуются `adrs/adr-005.md` (паттерн CF LLM),
  `adrs/adr-007.md` (`send_result`); сохраняются `adrs/adr-001.md` (профиль поставки),
  `adrs/adr-002.md`/`adrs/adr-003.md` (граница «хендлер ↔ сервис»), `adrs/adr-006.md` (учёт —
  callback'и не считаются).

## Объём релиза

### Задача 1: LLM-действия над текстом — `services/llm.py`
- **Что:** новый сервис с действиями над текстом через CF Workers AI: саммари и перевод.
  Переиспользует провайдерный слой из `services/structure.py` (выбор провайдера по `LLM_PROVIDER`,
  `_CloudflareProvider`/`_GroqProvider`, async POST через `httpx`), **не дублируя** его. Усекает вход
  по тому же порогу, что структурирование (`_MAX_INPUT_CHARS` = 8000 из `structure.py`). Единичная
  попытка, явный таймаут (`_TIMEOUT` = 30 с), без ретраев. Любой сбой → лог + возврат `None`.
- **Где (файлы/модули):** `services/llm.py` (новый); при необходимости — повышение видимости имён в
  `services/structure.py` (`_build_provider`/провайдеры/`_MAX_INPUT_CHARS`/`_TIMEOUT`) для
  переиспользования без копирования.
- **Контракт:**
  - `async def summarize(text: str) -> str | None` — саммари исходного текста в 3–5 пунктов на языке
    исходника; `None` при сбое/недоступности провайдера/пустом ответе.
  - `async def translate(text: str) -> str | None` — перевод на целевой язык (см. Задачу 2); `None`
    при сбое.
  - Эвристика направления перевода: функция определения наличия кириллицы (по диапазону Unicode,
    без сторонних библиотек) → кириллица → целевой английский, иначе → русский.
  - Промпты без выдумок: summarize — оформить смысл в 3–5 пунктов, не менять язык, не добавлять
    факты, вернуть только результат; translate — перевести, не добавляя содержание, вернуть только
    перевод. Текст подаётся как оформляемый материал, а не как обращение к модели (образец
    `structure.py`).
- **Критерии готовности:** при работающем провайдере `summarize`/`translate` возвращают непустой
  текст; при сбое httpx/некорректном ответе возвращают `None` без проброса исключения; вход усекается
  до 8000 символов перед отправкой; новый провайдер не вводится.
- **Связь с ADR:** ADR-009, Решения 2, 3, 4 (промпты), 5; переиспользование ADR-005.

### Задача 2: Перевод — эвристика направления
- **Что:** одна кнопка «Перевести» с авто-направлением. Направление определяется эвристикой по
  символам: наличие кириллицы → на английский; иначе → на русский. Без сторонних библиотек
  определения языка.
- **Где (файлы/модули):** `services/llm.py` (функция эвристики + использование в `translate`).
- **Контракт:** детерминированная функция «текст → целевой язык (`en`/`ru`)»; пустой/пробельный
  текст обрабатывается без падения.
- **Критерии готовности:** кириллический текст → целевой `en`; латинский текст → целевой `ru`;
  результат используется в промпте `translate`.
- **Связь с ADR:** ADR-009, Решение 3 (закрывает ОВ-1).

### Задача 3: Callback-хендлеры действий — `handlers/actions.py`
- **Что:** новый роутер с двумя `CallbackQuery`-хендлерами и фабрикой inline-клавиатуры.
- **Где (файлы/модули):** `handlers/actions.py` (новый); фабрика клавиатуры — здесь же (или
  `handlers/keyboards.py`, на усмотрение разработчика; подписи и `callback_data` в одной точке).
- **Контракт:**
  - `callback_data`: `"act:sum"` (саммари), `"act:tr"` (перевод) — компактно, в пределах 64 байт.
  - Фабрика клавиатуры возвращает `InlineKeyboardMarkup` с кнопками «Кратко» (`act:sum`) и
    «Перевести» (`act:tr`).
  - Хендлер: вызвать `callback.answer()` (снять «часики»; допустимо `callback.answer("Готовлю…")`);
    взять `callback.message.text`; если пуст/`None` → `callback.answer("Текст недоступен")` и выход
    (LLM не вызывается); иначе вызвать `summarize`/`translate`; при `None` → понятное русское
    служебное сообщение пользователю (через `callback.answer`/`message.answer`), без падения; при
    успехе → `send_result(callback.message, result)` **без** `reply_markup`.
- **Критерии готовности:** оба хендлера зарегистрированы на свои `callback_data`; нажатие снимает
  индикатор; пустой/`None` текст обрабатывается служебным сообщением без вызова LLM; сбой LLM →
  служебное сообщение, бот не падает; успешный результат уходит через `send_result`.
- **Связь с ADR:** ADR-009, Решения 5, 6, 10.

### Задача 4: Расширение `send_result` опциональным `reply_markup`
- **Что:** `send_result` принимает опциональный `reply_markup`, который прикрепляется **только** в
  одно-сообщенческом режиме (`len(text) <= MAX_MESSAGE_LEN`). В режимах «серия сообщений» и «файл
  `.txt`» markup игнорируется (кнопки не показываются).
- **Где (файлы/модули):** `services/reply.py`.
- **Контракт:**
  `async def send_result(message: Message, text: str, reply_markup: InlineKeyboardMarkup | None = None) -> None`.
  Markup передаётся в `message.answer` только в ветке одиночного сообщения; ветки `split_text`
  (серия) и `answer_document` (файл) markup не получают. Поведение при `reply_markup=None`
  идентично текущему (нулевой регресс).
- **Критерии готовности:** одиночное сообщение с переданным markup → кнопки видны; длинный текст
  (серия/файл) с переданным markup → кнопки не видны; вызов без `reply_markup` ведёт себя как сейчас.
- **Связь с ADR:** ADR-009, Решение 1 (закрывает ОВ-6); ADR-007.

### Задача 5: Подключение клавиатуры к каналам OCR и аудио
- **Что:** на непустом результате распознавания результат отдаётся с клавиатурой действий; на пустом
  результате — без изменений.
- **Где (файлы/модули):** `handlers/image.py` (фото + image-документ), `handlers/audio.py`
  (`_handle_audio_bytes`).
- **Контракт:** замена
  `await send_result(message, await structure_text(text))` на
  `await send_result(message, await structure_text(text), reply_markup=<клавиатура действий>)`.
  Пустой результат (`NO_TEXT_MESSAGE`/`NO_SPEECH_MESSAGE`) — прежний `message.answer(...)` без кнопок.
- **Критерии готовности:** под одно-сообщенческим текстовым результатом OCR и аудио видны кнопки;
  под пустым результатом кнопок нет; под длинным (серия/файл) кнопок нет (за счёт Задачи 4).
- **Связь с ADR:** ADR-009, Решение 7; ФТ-1, ФТ-13.

### Задача 6: Регистрация роутера в `bot.py` и экспорт
- **Что:** зарегистрировать роутер действий; экспортировать его из пакета хендлеров.
- **Где (файлы/модули):** `bot.py`, `handlers/__init__.py`.
- **Контракт:** `handlers/__init__.py` экспортирует `actions_router`; `bot.py` вызывает
  `dp.include_router(actions_router)`. `StatsMiddleware` остаётся **только** на
  `dp.message.outer_middleware` — на `dp.callback_query` middleware **не вешается** (callback'и не
  учитываются, ADR-006/Решение 4).
- **Критерии готовности:** роутер действий обрабатывает нажатия; статистика и схема `user_stats` не
  изменены.
- **Связь с ADR:** ADR-009, Решения 8, 9.

## Тесты

Минимальный набор, обязательный к появлению в релизе (моки `httpx`, без реальной сети и без
Telegram-сервера):

- **`services/llm.py` — summarize:** мок httpx-ответа CF → корректный парсинг и возврат текста
  саммари; усечение длинного входа до 8000 символов перед отправкой.
- **`services/llm.py` — translate:** мок httpx-ответа → возврат перевода; целевой язык в промпте
  соответствует эвристике.
- **`services/llm.py` — эвристика направления:** кириллический текст → целевой `en`; латинский →
  целевой `ru`; пустой/пробельный — без падения.
- **`services/llm.py` — сбой CF:** httpx бросает исключение / некорректный/пустой ответ →
  `summarize`/`translate` возвращают `None`, исключение не пробрасывается.
- **`handlers/actions.py` — диспетч действия:** callback с `act:sum` вызывает `summarize` и шлёт
  результат через `send_result`; `act:tr` вызывает `translate` (на моках сервисов и `send_result`).
- **`handlers/actions.py` — пустой текст:** `callback.message.text` пуст/`None` → служебный ответ,
  `summarize`/`translate` **не вызываются**, без падения.
- **`handlers/actions.py` — сбой LLM:** сервис вернул `None` → пользователю служебное сообщение, бот
  не падает; `callback.answer()` вызван.
- **`services/reply.py` — markup только при одиночном сообщении:** короткий текст + `reply_markup` →
  `answer` получает markup; длинный текст (серия) + `reply_markup` → части без markup; очень длинный
  (файл) + `reply_markup` → `answer_document` без markup.

## Риски и митигации

- **Риск:** `callback_data` превышает 64 байта → ошибка Telegram при формировании кнопки.
  → **Митигация:** фиксированные короткие коды `"act:sum"`/`"act:tr"`; тест/проверка длины при ревью.
- **Риск:** `callback.message.text` равен `None` (file-сообщение) или сообщение устарело/отредактировано.
  → **Митигация:** правило «markup только при одиночном сообщении» гарантирует наличие текста под
  кнопкой; хендлер всё равно проверяет `None`/пустоту и отвечает «Текст недоступен» без вызова LLM.
- **Риск:** устаревшее/повторное нажатие.
  → **Митигация:** каждое нажатие формирует ответ заново (без дедупа); `callback.answer()` всегда
  снимает индикатор; отсутствие текста → служебный ответ.
- **Риск:** лимиты free-tier CF при сериях нажатий / большой вход раздувает запрос.
  → **Митигация:** усечение входа до 8000 символов; единичная попытка с таймаутом; сбой/лимит →
  служебное сообщение (не падение). Объёмы малы (ДОП-2).
- **Риск:** эвристика направления ошибается на смешанном ru/en тексте.
  → **Митигация:** простое детерминированное правило (наличие кириллицы); приемлемо для ru/en (вне
  рамок прочих языков); при необходимости — две кнопки в будущем релизе.
- **Риск:** двойная отдача результата (структурирование уже в хендлере).
  → **Митигация:** `send_result` вызывается ровно один раз с результатом `structure_text`; ответ
  действия — отдельный поток без своих кнопок.
