# Спецификация релиза v1.0.0

- **Дата планирования:** 2026-06-23
- **Тип релиза:** major
- **Цели:** Перевести Textify в продакшн-готовность 1.0: наблюдаемые логи, индикатор прогресса для
  пользователя, защита слабого сервера от перегрузки/OOM, актуальная документация и зафиксированный
  публичный контракт как semver-baseline. Без новых пользовательских фич.

## Источники

- BA-требования: `ba-req/ba-requirements-010.md`
- ADR: `adrs/adr-010.md` (опирается на ADR-001, ADR-002/003, ADR-005, ADR-006, ADR-007, ADR-008, ADR-009)

## Объём релиза

Все изменения обратимо-совместимы. MAJOR обусловлен фиксацией публичного контракта (ADR-010, раздел
«Публичный контракт Textify 1.0.0»), а не ломающим изменением.

### Задача 1: Наблюдаемость — настройка логирования в точке входа

- **Что:** В начале `main()` (до `dp.start_polling`) настроить корневое логирование через
  `logging.basicConfig`. Уровень — из env `LOG_LEVEL` (дефолт `INFO`). Формат — с временем, уровнем
  и именем логгера (например `%(asctime)s %(levelname)s %(name)s %(message)s`). Некорректное/пустое
  значение `LOG_LEVEL` **не должно ронять старт** — откат на `INFO` (нормализация: привести к
  верхнему регистру, проверить, что это валидный уровень `logging`; иначе `INFO`). Допускается одна
  `INFO`-строка о старте бота, чтобы `docker logs` был непуст при штатной работе. Не плодить
  дубль-хендлеры (один `basicConfig`).
- **Где (файлы/модули):** `bot.py`. `LOG_LEVEL` читать напрямую из окружения в `bot.py` (не через
  `config.py` — см. ADR-010, Решение 1).
- **Контракт:** новая env `LOG_LEVEL` (опциональная, дефолт `INFO`). Секреты (`BOT_TOKEN`,
  `CF_API_TOKEN`) в логи не писать.
- **Критерии готовности:** при `LOG_LEVEL=DEBUG` корневой уровень — DEBUG; при отсутствии/пустом/
  мусорном значении — INFO, старт не падает; существующие `logger.*` пишут в stdout; ровно один
  root-handler.
- **Связь с ADR:** ADR-010, Решение 1; контракт env — раздел «Публичный контракт».

### Задача 2: Небуферизованный вывод в контейнере

- **Что:** Добавить `ENV PYTHONUNBUFFERED=1` в `Dockerfile`. Это **единственное** изменение
  `Dockerfile`; профиль поставки (ADR-001: multi-stage, non-root, без портов, GHCR) сохраняется.
- **Где (файлы/модули):** `Dockerfile`.
- **Контракт:** артефакт поставки не меняет своего профиля (см. `deploy-guide.md` — профиль «бот без
  HTTP-порта» из ADR-001).
- **Критерии готовности:** в `Dockerfile` присутствует `ENV PYTHONUNBUFFERED=1`; прочие директивы
  (HF_HOME, HF_HUB_OFFLINE, TRANSFORMERS_OFFLINE, non-root USER, CMD) не изменены.
- **Связь с ADR:** ADR-010, Решение 2.

### Задача 3: Индикатор прогресса (ChatActionSender)

- **Что:** Обернуть тяжёлую обработку в каждом релевантном хендлере в async-контекст
  `aiogram.utils.chat_action.ChatActionSender`. Action фиксирован per-хендлер:
  - `handlers/image.py` (`handle_photo`, `handle_image_document`): **`upload_document`**;
  - `handlers/audio.py` (`_handle_audio_bytes`, охватывает voice/audio/audio-document): **`typing`**;
  - `handlers/actions.py` (`_handle_action` для `act:sum`/`act:tr`): **`typing`** вокруг вызова LLM.
  Контекст охватывает распознавание/структурирование (или LLM-действие); отдача `send_result` может
  оставаться внутри контекста. Реализация неблокирующая; путь `send_result` (ADR-007) не меняется.
- **Где (файлы/модули):** `handlers/image.py`, `handlers/audio.py`, `handlers/actions.py`.
- **Контракт:** наблюдаемое поведение каналов (распознавание, структурирование, действия, отдача)
  неизменно — добавляется только индикатор.
- **Критерии готовности:** при обработке фото/изображения виден `upload_document`; при аудио — `typing`;
  при «Кратко»/«Перевести» — `typing`; индикатор гаснет по завершении/ошибке; существующая отправка
  результата и служебные сообщения работают как раньше.
- **Связь с ADR:** ADR-010, Решение 2 (фиксация «какой action где»).

### Задача 4: Защита от перегрузки — глобальный семафор на тяжёлые локальные операции

- **Что:** Объявить общий `asyncio.Semaphore` в `services/__init__.py` с лимитом **1** через
  именованную константу (например `HEAVY_LOCAL_CONCURRENCY = 1`, `HEAVY_LOCAL_SEMAPHORE =
  asyncio.Semaphore(HEAVY_LOCAL_CONCURRENCY)`). Захватывать семафор вокруг тяжёлых **локальных**
  операций:
  - `services/ocr.recognize_text`: `async with HEAVY_LOCAL_SEMAPHORE: await asyncio.to_thread(_run_ocr, ...)`;
  - `services/transcribe._transcribe_local`: захват вокруг `asyncio.to_thread(_transcribe_sync, ...)` —
    **единственная** точка захвата локального пути (покрывает и `ASR_PROVIDER=local`, и фолбэк CF→local).
  CF-вызовы (`transcribe._transcribe_cloudflare`, `structure.structure_text`, `llm.summarize`/
  `translate`) семафором **не** оборачивать. Семафор должен оставаться единственным экземпляром на
  процесс (объявлен на уровне модуля).
- **Где (файлы/модули):** `services/__init__.py` (новый код), `services/ocr.py`, `services/transcribe.py`.
- **Контракт:** результат успешной обработки не меняется; ограничивается только параллелизм/очерёдность.
  Дедлок/двойной захват исключены: CF-путь до вызова `_transcribe_local` семафор не держит.
- **Критерии готовности:** одновременно выполняется не более одной тяжёлой локальной операции;
  параллельные тяжёлые запросы не роняют процесс; фолбэк CF→local не зависает на семафоре; результат
  при наличии ресурсов идентичен дорелизному.
- **Связь с ADR:** ADR-010, Решение 3 (вариант A, лимит 1, правило отсутствия дедлока).

## Изменения публичного контракта

С 1.0.0 фиксируется публичный контракт (ADR-010, раздел «Публичный контракт Textify 1.0.0»):
команды (`/start`, `/help`, `/stats`), входные каналы (photo, document image/*, voice, audio,
document audio/*), формат ответа (plain text, отдача через `send_result`, фолбэки), inline-действия
(`act:sum`, `act:tr`). Дальнейшее ломающее изменение этого контракта = MAJOR.

Новая env-переменная `LOG_LEVEL` (опциональная, дефолт `INFO`) — обратимо-совместимое добавление, не
ломает контракт. Сигнатуры публичных сервисов (`recognize_text`, `transcribe`, `structure_text`,
`summarize`, `translate`, `send_result`) **не меняются**.

Миграционный путь: не требуется. Передеплой образа; `LOG_LEVEL` опционален.

## Тесты

Минимальный набор тестов, который обязан появиться/обновиться в этом релизе:

- **Логирование из LOG_LEVEL:** тест, что настройка логирования читает `LOG_LEVEL` и применяет
  уровень (например, через `monkeypatch.setenv` + вызов функции настройки/импорт `main`-пути и
  проверка `logging.getLogger().level` или мок `basicConfig`); проверить, что мусорное/пустое
  значение даёт `INFO` и не бросает исключение.
- **Семафор сериализует тяжёлые локальные операции:** тест на ограничение параллелизма — через
  мок/счётчик активных: подменить `_run_ocr`/`_transcribe_sync` на корутиноподобную задержку,
  запустить N параллельных вызовов `recognize_text`/`_transcribe_local`, проверить, что максимум
  одновременно активных = 1 (счётчик «вошёл/вышел»).
- **CF-путь не держит семафор:** тест, что при `ASR_PROVIDER=cloudflare` с успешным CF-ответом
  семафор не захватывается (или захват/освобождение не сериализует CF) — например, мок CF-ответа +
  проверка, что параллельные CF-вызовы не упираются в лимит 1; и что фолбэк CF→local при сбое CF
  корректно проходит через семафор без зависания.
- **ChatActionSender не ломает отправку результата:** тесты хендлеров на моках — при наличии текста
  результат уходит через `send_result` как и раньше; при пустом результате — служебное сообщение;
  оборачивание `ChatActionSender` (мокнутый) не меняет число/аргументы вызовов `send_result`/
  `message.answer`.
- **README содержит все env-переменные:** тест-проверка наличия ключевых строк в `README.md` (все 10
  имён переменных, три команды `/start`/`/help`/`/stats`) — по образцу `test_transcribe.py::
  test_hf_home_env_configured` (чтение файла + `assert ... in content`).
- **Dockerfile содержит PYTHONUNBUFFERED:** тест-проверка наличия `PYTHONUNBUFFERED=1` в `Dockerfile`
  (по образцу проверок Dockerfile в `test_transcribe.py`).

Существующие тесты (`test_actions.py`, `test_transcribe.py`, `test_ocr*.py`, `test_reply.py`,
`test_structure*.py`) должны остаться зелёными — обвязка их не ломает.

## Риски и митигации

- **Риск:** дедлок/двойной захват семафора на пути CF→local (CF-ветка держит семафор и ждёт его же в
  фолбэке). → **Митигация:** единственная точка захвата — `_transcribe_local`; CF-путь до неё семафор
  не держит (ADR-010, Решение 3); покрыто тестом «фолбэк CF→local не зависает».
- **Риск:** регресс отзывчивости — семафор/индикатор блокируют event loop. → **Митигация:** ожидание
  семафора асинхронно (`async with asyncio.Semaphore`), тяжёлая работа в `to_thread`,
  `ChatActionSender` неблокирующий; тест на параллелизм подтверждает, что бот не «встаёт».
- **Риск:** неверный уровень логов в проде (мусорный `LOG_LEVEL` роняет старт или DEBUG зашумляет). →
  **Митигация:** нормализация на `INFO` при некорректном значении, старт не падает; дефолт `INFO`;
  уровень управляется эксплуатантом без правки кода.
- **Риск:** лимит семафора = 1 чрезмерно снижает пропускную способность. → **Митигация:** дефолтный
  ASR — облачный CF (локальный whisper лишь фолбэк), OCR быстр; лимит — именованная константа, легко
  поднять при апгрейде сервера.
- **Риск:** `ChatActionSender` оставляет подвисший индикатор при исключении. → **Митигация:**
  контекстный менеджер гасит фоновую задачу на выходе, в т.ч. при исключении.

### Примечания к деплою

- Новых **обязательных** серверных шагов нет; новых обязательных секретов нет.
- `LOG_LEVEL` опционален (дефолт `INFO`); задаётся в `.env` при необходимости детализации.
- **Верификация наблюдаемости:** после деплоя `docker logs` становится непустым (минимум — старт
  бота) и обновляется в реальном времени — это и есть проверка наблюдаемости.
- **Лёгкая проверка под нагрузкой:** отправить несколько параллельных тяжёлых запросов (например,
  несколько изображений/аудио подряд) — контейнер не падает и не уходит в OOM; индикатор виден; бот
  остаётся отзывчивым.
