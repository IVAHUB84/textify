# Спецификация релиза v1.1.0

- **Дата планирования:** 2026-06-23
- **Тип релиза:** minor
- **Цели:** Гарантировать бесплатность для владельца при росте трафика — дневной бюджет обращений к
  Cloudflare с авто-деградацией на локальные бесплатные движки (без отказа пользователю). Усилить
  органический приток (дискаверабилити: ключевые слова метаданных + текст листинга для каталогов).

## Источники

- BA-требования: `ba-req/ba-requirements-011.md`
- ADR: `adrs/adr-011.md` (опирается на ADR-001, ADR-002/003, ADR-005, ADR-006, ADR-007, ADR-008,
  ADR-009, ADR-010)

## Объём релиза

### Задача 1: Модуль дневного бюджета CF (`services/budget.py`)

- **Что:** новый модуль учёта обращений к Cloudflare Workers AI за текущие UTC-сутки и проверки
  дневного бюджета. Паттерн доступа к SQLite — как в `services/stats.py` (соединение-на-операцию, WAL,
  `busy_timeout`, атомарный UPSERT, доступ через `asyncio.to_thread`). **Тот же файл БД**
  (`config["STATS_DB_PATH"]`, тот же volume — ADR-006), **отдельная таблица** `cf_usage`.
- **Где (файлы/модули):** новый `services/budget.py`.
- **Контракт (публичные функции/схема):**
  - Таблица:
    ```sql
    CREATE TABLE IF NOT EXISTS cf_usage (
        date  TEXT PRIMARY KEY,            -- UTC-дата 'YYYY-MM-DD'
        count INTEGER NOT NULL DEFAULT 0
    );
    ```
  - `def init_cf_usage_db() -> None` — создаёт таблицу/PRAGMA (по образцу `init_db()` в `stats.py`;
    использует тот же `STATS_DB_PATH`).
  - `async def cf_budget_allow() -> bool` — `True`, если счётчик за текущую UTC-дату `< CF_DAILY_BUDGET`.
    Ключ даты — `datetime.now(timezone.utc).date().isoformat()`. **fail-open:** при любом исключении
    чтения возвращает `True` и пишет `logger.warning` (учёт вспомогательный — ADR-011 Решение 7).
  - `async def cf_budget_consume() -> None` — атомарный UPSERT-инкремент счётчика за текущую UTC-дату
    (`INSERT ... ON CONFLICT(date) DO UPDATE SET count = count + 1`). При исключении —
    `logger.warning`, без проброса.
  - Чтение бюджета: целое из env `CF_DAILY_BUDGET` через `config` (см. Задачу 7), дефолт 300.
- **Критерии готовности:**
  - При `count < бюджета` → `cf_budget_allow()` возвращает `True`; при `count >= бюджета` → `False`.
  - `cf_budget_consume()` увеличивает счётчик ровно на 1 для текущей UTC-даты; конкурентные вызовы не
    теряются и не роняют процесс (атомарный UPSERT, WAL).
  - При смене UTC-даты чтение/инкремент идут по новому ключу — старый счётчик не влияет (сброс).
  - При недоступности БД `cf_budget_allow()` → `True` (fail-open) с логом; `cf_budget_consume()` не
    бросает исключение.
- **Связь с ADR:** ADR-011, Решения 1–4, 7; паттерн — ADR-006 Решение 4A.

### Задача 2: Гейт бюджета в CF ASR (`services/transcribe.py`)

- **Что:** перед фактическим обращением к CF Whisper проверять бюджет; при исчерпании — деградировать
  на локальный faster-whisper без отказа; при разрешении — инкрементировать счётчик вокруг CF-вызова.
- **Где:** `services/transcribe.py`, функция `transcribe()` — **на CF-ветке**, после проверок
  `ASR_PROVIDER`, размера файла (`_CF_MAX_AUDIO_BYTES`) и наличия `CF_ACCOUNT_ID`/`CF_API_TOKEN`,
  **перед** `_transcribe_cloudflare(...)`.
- **Контракт:**
  - Если `not await cf_budget_allow()` → `logger.warning(...)` о деградации ASR и
    `return await _transcribe_local(audio_bytes)`.
  - Иначе → `await cf_budget_consume()` и вызов CF (порядок «проверка → вызов → инкремент»; допускается
    инкремент непосредственно перед/при CF-вызове — overshoot принят, ADR-011 Решение 6).
  - Ручной режим `ASR_PROVIDER=local`, фолбэк по размеру и по отсутствию кредов идут в локальный путь
    **до** гейта — счётчик не растёт.
  - Существующий фолбэк CF→local при сбое CF (try/except) сохраняется; семафор ADR-010 захватывается
    ровно один раз в `_transcribe_local` (дедлока нет).
- **Критерии готовности:**
  - Бюджет не исчерпан → вызывается CF, счётчик +1 (поведение ASR как в 1.0.0).
  - Бюджет исчерпан → CF **не вызывается**, результат от локального whisper, счётчик не растёт, в
    логах — `warning` о деградации.
  - `ASR_PROVIDER=local` → CF не вызывается и `consume` не вызывается.
- **Связь с ADR:** ADR-011 Решение 5 (п.1), композиция «переключатель + бюджет»; ADR-008, ADR-010.

### Задача 3: Гейт бюджета в CF структурировании (`services/structure.py`)

- **Что:** перед обращением к CF LLM в `structure_text()` проверять бюджет; при исчерпании —
  существующий тихий фолбэк на сырой текст; при разрешении — инкремент вокруг CF-вызова. Гейт — только
  для Cloudflare-провайдера (для `LLM_PROVIDER=groq` не применяется).
- **Где:** `services/structure.py`, `structure_text()` — после `_build_provider()`.
- **Контракт:**
  - Ввести способ определить, что активный провайдер — Cloudflare (например,
    `isinstance(provider, _CloudflareProvider)` — внутренняя деталь модуля).
  - Если провайдер — Cloudflare и `not await cf_budget_allow()` → `logger.warning(...)` о деградации
    структурирования и `return raw_text` (существующий фолбэк ADR-005).
  - Иначе при CF-провайдере → `await cf_budget_consume()` вокруг `provider.complete(...)`.
  - Для не-CF провайдера (Groq) и для `provider is None` — поведение прежнее (бюджет CF не трогается).
- **Критерии готовности:**
  - Бюджет не исчерпан (CF) → CF вызывается, счётчик +1, структурирование как в 1.0.0.
  - Бюджет исчерпан (CF) → CF не вызывается, возвращается сырой текст, счётчик не растёт, в логах —
    `warning`.
  - `LLM_PROVIDER=groq` → бюджет CF не проверяется и не инкрементируется.
- **Связь с ADR:** ADR-011 Решение 5 (п.2); ADR-005.

### Задача 4: Гейт бюджета и sentinel в CF действиях (`services/llm.py`)

- **Что:** в `summarize()` и `translate()` перед обращением к CF LLM проверять бюджет; при исчерпании
  возвращать sentinel `BUDGET_EXCEEDED` (локальной замены нет); при разрешении — инкремент вокруг
  CF-вызова. Только для Cloudflare-провайдера.
- **Где:** `services/llm.py`, функции `summarize`, `translate`.
- **Контракт:**
  - Определить модульный sentinel: `BUDGET_EXCEEDED` (отдельный синглтон-объект; **не** `None` и не
    строка результата). Тип возврата `summarize`/`translate` расширяется: `str | None |
    <тип BUDGET_EXCEEDED>`.
  - Определить, что активный провайдер (`build_provider()`) — Cloudflare (переиспользовать механизм из
    Задачи 3).
  - Если провайдер — Cloudflare и `not await cf_budget_allow()` → `logger.warning(...)` и
    `return BUDGET_EXCEEDED`.
  - Иначе при CF-провайдере → `await cf_budget_consume()` вокруг `provider.complete(...)`.
  - `provider is None` → прежний `return None`; Groq-провайдер → бюджет CF не трогается.
- **Критерии готовности:**
  - Бюджет не исчерпан (CF) → результат summarize/translate, счётчик +1.
  - Бюджет исчерпан (CF) → возвращается `BUDGET_EXCEEDED`, CF не вызывается, счётчик не растёт, в логах
    — `warning`.
- **Связь с ADR:** ADR-011 Решения 5 (п.3), 8; ADR-009.

### Задача 5: Сообщение о лимите в действиях (`handlers/actions.py`)

- **Что:** обработать sentinel `BUDGET_EXCEEDED` в хендлере действий понятным русским сообщением;
  сохранить прежнее сообщение для `None`.
- **Где:** `handlers/actions.py`, `_handle_action`.
- **Контракт:**
  - Импортировать `BUDGET_EXCEEDED` из `services.llm`.
  - `result is BUDGET_EXCEEDED` → `await raw_msg.answer("Дневной бесплатный лимит исчерпан, попробуйте завтра.")`
    и `return` (точная формулировка — зафиксирована).
  - `result is None` → прежнее `await raw_msg.answer("Не удалось выполнить действие. Попробуйте позже.")`.
  - Иначе → `await send_result(raw_msg, result)`.
  - «Часики» callback снимаются как сейчас (`callback.answer("Готовлю…")` уже вызывается ранее).
- **Критерии готовности:**
  - При `BUDGET_EXCEEDED` пользователь видит ровно сообщение «Дневной бесплатный лимит исчерпан,
    попробуйте завтра.»; бот не падает; CF не вызывался.
  - При `None` — прежнее сообщение; при тексте — `send_result`.
- **Связь с ADR:** ADR-011 Решение 8; ADR-009.

### Задача 6: Усиление ключевых слов в метаданных (`handlers/commands.py`)

- **Что:** усилить ключевые слова в `BOT_DESCRIPTION` (≤ 512) и `BOT_SHORT_DESCRIPTION` (≤ 120) под
  поиск Telegram, сохранив стиль и смысл. Набор команд (`/start`, `/help`) — часть контракта 1.0.0
  (ADR-010) — **не менять**.
- **Где:** `handlers/commands.py`, константы `BOT_DESCRIPTION`, `BOT_SHORT_DESCRIPTION` (применяются в
  `setup_bot_profile`).
- **Контракт / ключевые слова:** распознавание речи, транскрипция, OCR, перевод, голос в текст, RU/EN
  (естественно вплетены, без спама). Лимиты: description ≤ 512 символов, short ≤ 120 символов.
- **Критерии готовности:**
  - `len(BOT_DESCRIPTION) <= 512` и `len(BOT_SHORT_DESCRIPTION) <= 120`.
  - В текстах присутствуют ключевые слова (транскрипция/распознавание речи, OCR, перевод, RU/EN).
  - `setup_bot_profile` по-прежнему вызывает `set_my_commands`/`set_my_description`/
    `set_my_short_description` без изменения набора команд.
- **Связь с ADR:** ADR-011 Решение 9; ADR-010 (контракт команд сохраняется).

### Задача 7: Конфиг `CF_DAILY_BUDGET` (`config.py`, `.env.example`, `deploy/.env.example`)

- **Что:** читать `CF_DAILY_BUDGET` (целое) с консервативным дефолтом 300; некорректное/неположительное
  значение нормализовать к дефолту без падения старта; добавить плейсхолдер в оба `.env.example`.
- **Где:** `config.py` (в `load_config`), `.env.example`, `deploy/.env.example`.
- **Контракт:**
  - `config["CF_DAILY_BUDGET"]` — `int`, дефолт `300`. Дефолт — именованная константа в `config.py`
    (например, `_DEFAULT_CF_DAILY_BUDGET = 300`).
  - Нечисловое/пустое значение → дефолт + `logger.warning` (по образцу `ADMIN_USER_ID`).
  - Значение `<= 0` → дефолт + `logger.warning` (бюджет 0 = вечная деградация — не цель; отключение CF
    — через `ASR_PROVIDER`/`LLM_PROVIDER`).
  - `.env.example` и `deploy/.env.example`: строка `CF_DAILY_BUDGET=300` (с пояснительным комментарием
    о назначении). Несекретная конфигурация.
- **Критерии готовности:**
  - При отсутствии env `config["CF_DAILY_BUDGET"] == 300`; старт не падает.
  - При `CF_DAILY_BUDGET=abc` или `CF_DAILY_BUDGET=-5` → 300 + warning, старт не падает.
  - Плейсхолдер присутствует в обоих `.env.example`.
- **Связь с ADR:** ADR-011 Решение 2; НФТ-10.

- **Критерии готовности:** после старта таблица `cf_usage` существует в файле `STATS_DB_PATH`;

- **Связь с ADR:** ADR-011 Решения 3, 4; ADR-006.

## Изменения публичного контракта

Публичный контракт 1.0.0 (ADR-010) **сохраняется** — релиз MINOR:

- Команды (`/start`, `/help`, `/stats`), входные каналы, формат ответа (`send_result`), inline-действия
  (`act:sum`/`act:tr`) — без изменений.
- **Новое поведение при исчерпании дневного бюджета** (обратимо-совместимое, без отказа): ASR →
  локальный whisper; структурирование → сырой текст; «Кратко»/«Перевести» → сообщение «Дневной
  бесплатный лимит исчерпан, попробуйте завтра.».
- **Новая опциональная env `CF_DAILY_BUDGET`** (дефолт 300) — несекретная; отсутствие не ломает старт.
- Тексты `BOT_DESCRIPTION`/`BOT_SHORT_DESCRIPTION` усилены ключевыми словами (не часть semver-контракта).
- Миграция БД: добавляется таблица `cf_usage` через `CREATE TABLE IF NOT EXISTS` в существующем файле
  (тот же volume) — обратно совместимо, ручных миграций не требуется.

## Тесты

Все тесты — на **моках**, без реальной сети/CF; дату мокать (`datetime`/функция получения UTC-даты),
БД — временный файл/`tmp_path`.

1. **Счётчик инкрементируется при CF-вызове и НЕ при локальном:**
   - `transcribe()` на CF-ветке (бюджет не исчерпан, креды заданы, размер ≤ порога) → `cf_budget_consume`
     вызван, CF вызван.
   - `ASR_PROVIDER=local` → `_transcribe_local`, `cf_budget_consume` НЕ вызван.
   - `structure_text()` с CF-провайдером, бюджет не исчерпан → `consume` вызван; с Groq-провайдером →
     `consume` НЕ вызван.
   - `summarize`/`translate` с CF, бюджет не исчерпан → `consume` вызван.
2. **Гейт блокирует при достижении бюджета → деградация по каждому из трёх путей:**
   - ASR: `cf_budget_allow()→False` → результат от локального whisper (мок `_transcribe_local`), CF не
     вызван, в логах warning.
   - Структурирование: `allow→False` → `structure_text` возвращает `raw_text`, CF не вызван, warning.
   - Действия: `allow→False` → `summarize`/`translate` возвращают `BUDGET_EXCEEDED`, CF не вызван,
     warning.
3. **Сброс по смене UTC-даты:** записать счётчик на дату D до бюджета; замокать дату на D+1 →
   `cf_budget_allow()` снова `True`, счётчик дня D+1 стартует с 0.
4. **fail-open при ошибке БД:** замокать сбой соединения/запроса → `cf_budget_allow()` возвращает
   `True` (логируется warning); `cf_budget_consume()` не бросает исключение.
5. **Сообщение о лимите в actions:** `_handle_action` при `result is BUDGET_EXCEEDED` → `raw_msg.answer`
   вызван ровно с текстом «Дневной бесплатный лимит исчерпан, попробуйте завтра.»; при `None` →
   прежнее сообщение; при тексте → `send_result`.
6. **Конкурентный инкремент:** несколько параллельных `cf_budget_consume()` (через `asyncio.gather`)
   не роняют процесс; счётчик увеличивается (точное равенство не требуется — overshoot допустим, но
   потери учёта быть не должно — проверить, что итог `>= числа неблокированных consume`).
7. **Конфиг:** отсутствие `CF_DAILY_BUDGET` → 300; нечисловое/неположительное → 300 (без падения).
8. **Лимиты метаданных:** `len(BOT_DESCRIPTION) <= 512`, `len(BOT_SHORT_DESCRIPTION) <= 120`.

## Риски и митигации

- **Риск:** overshoot счётчика при конкуренции (несколько корутин прошли `allow` до `consume`). →
  **Митигация:** принят осознанно (ADR-011 Решение 6); бюджет консервативен с запасом в десятки раз до
  free-tier; overshoot ограничен числом одновременных CF-запросов (на ivahub — единицы).
- **Риск:** неверно подобранный бюджет (слишком низкий → ранняя деградация; слишком высокий → выход за
  free-tier). → **Митигация:** консервативный дефолт 300 (именованная константа), переопределяемый env;
  проверка/подстройка на проде при заниженном `CF_DAILY_BUDGET` (ОВ-7).
- **Риск:** деградация качества при исчерпании (whisper `base` медленнее/слабее; структурирование
  пропадает). → **Митигация:** это и есть цель (бесплатность важнее качества при пике); факт
  логируется (`logger.warning`), виден в `docker logs` (ADR-010).
- **Риск:** длительный сбой учёта (fail-open) временно отключает страховку. → **Митигация:** сбой
  персистентного SQLite в volume — редкая аномалия, логируется; приоритет — работа бота (ADR-011
  Решение 7); БД та же, что для `/stats` (стабильна с v0.6.0).

## Границы (вне рамок)

- Режим групп (v1.2.0) и реферальная/виральная петля (v1.3.0) — не реализуются.
- Точный учёт нейронов, веса по типам вызовов, индикация остатка бюджета в `/stats` — не реализуются.
- Автоматическая подача листинга в каталоги — только текст; подача ручная.
