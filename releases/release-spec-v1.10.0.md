# Спецификация релиза v1.10.0

- **Дата планирования:** 2026-06-25
- **Тип релиза:** minor
- **Цели:** Ввести **changelog-рассылку анонсов новых версий**: при выходе новой версии бот, **с
  подтверждения админа**, рассылает пользователям короткий «что нового». Авто-детект при старте → превью
  админу с кнопками «Разослать»/«Пропустить» → фоновая троттлящаяся рассылка по личным чатам →
  отчёт админу. Поддержка отписки. Поведение не ломает обработку медиа, лимиты/gate (v1.7.0) и рефералы
  (v1.8.0).

## Источники
- BA-требования: `ba-req/ba-requirements-020.md`
- ADR: `adrs/adr-020.md` (закрывает ОВ-1…ОВ-8; стор поверх `stats.db` — паттерн ADR-006; профиль
  поставки ADR-001 сохраняется; стиль inline-кнопок/`callback_data` — преемственно ADR-017/018).

## Профиль поставки

Проект — **поставляемый сервис**: Telegram-бот на long-polling, **без HTTP-порта/Traefik/edge**
(профиль ADR-001), образ из GHCR, прод ivahub (1 vCPU / 2 GB). Контракт `deploy-guide.md` в части
артефактов **почти не задействован**: новых портов, volume, внешних зависимостей и направлений трафика
релиз не вводит (volume `stats.db` уже примонтирован с v0.6.0). **Единственное изменение артефактов** —
один новый env-флаг `ANNOUNCEMENTS_ENABLED` в `.env.example` и `deploy/.env.example`. `Dockerfile`,
`deploy/docker-compose.yml`, `.github/workflows/deploy.yml` — **без правок**. Миграция БД аддитивная и
выполняется в коде при старте (новая колонка с DEFAULT + новая таблица `CREATE IF NOT EXISTS`) — **без
ручных деплой-предусловий** сверх v0.6.0 (chown каталога данных уже сделан).

## Объём релиза

Не добавлять ничего сверх перечисленного. Все сообщения пользователю и админу — на **русском**,
дружелюбные, без давления. Идентификаторы/имена файлов кода — на английском. Учёт прогресса/докрутка
рассылки и persistent-пометка заблокировавших в этот релиз **не входят** (ADR-020 Решения 5A, 3A).

### Задача 1: Единый источник версии (`version.py`)
- **Что:** ввести единую константу текущей версии приложения и хелпер сравнения semver (ADR-020 Решение 1A).
- **Где:** `version.py` (новый, верхний уровень рядом с `bot.py`/`config.py`).
- **Контракт:**
  - `__version__: str = "1.10.0"` (формат `"X.Y.Z"`, **без** префикса `v`).
  - `parse_version(s: str) -> tuple[int, int, int]` — разбор `"X.Y.Z"` в тройку int; на нечисловом/
    неполном формате — поднять/вернуть значение, по которому сравнение трактует версию как невалидную
    (например `(-1, -1, -1)` или исключение, перехватываемое вызывающим). Решение детали — за
    разработчиком, но **сравнение должно быть числовым по тройкам, не лексикографическим по строкам**
    (`"1.10.0" > "1.9.1"`).
- **Критерии готовности:** `__version__ == "1.10.0"`; сравнение версий числовое (`1.10.0` новее
  `1.9.1`); невалидная строка не приводит к падению механизма анонсов. *(ФТ-1; КП-1)*
- **Связь с ADR:** ADR-020 Решение 1.

### Задача 2: Курируемый реестр анонсов (`announcements.py`)
- **Что:** реестр коротких user-friendly текстов «что нового» по версиям, отдельный от технических
  `release-notes` (ADR-020 Решение 2A).
- **Где:** `announcements.py` (новый, верхний уровень).
- **Контракт:**
  - `ANNOUNCEMENTS: dict[str, str]` с ключами `"X.Y.Z"` (как `__version__`).
  - Запись для `"1.10.0"` — короткий русский текст: что нового в этой версии (анонс самого механизма
    анонсов формулировать не нужно — это служебное; текст должен описывать пользу для пользователя.
    Допустим обзор недавних возможностей: тайм-коды/субтитры, PDF/скан, кнопки Кратко/Перевести/Задачи,
    повышение лимита за рефералов/подписку — кратко, без технических КП/ADR). Длина — несколько строк.
  - Только данные, без логики/импортов бизнес-кода.
- **Критерии готовности:** `ANNOUNCEMENTS["1.10.0"]` существует и непустой; ключи в формате `"X.Y.Z"`;
  для рассылки используется текст отсюда, **не** `release-notes/notes-*.md`. *(ФТ-2; КП-2)*
- **Связь с ADR:** ADR-020 Решения 2, 4.

### Задача 3: Стор анонсов и фоновая рассылка (`services/announce.py`)
- **Что:** persistent last-announced-версия поверх `stats.db` и фоновая троттлящаяся рассылка с backoff,
  обработкой ошибок и отчётом (ADR-020 Решения 3A, 5A, 6).
- **Где:** `services/announce.py` (новый сервис; без aiogram-роутинга, `Bot` передаётся аргументом).
- **Контракт:**
  - **Схема/init:** `init_announce_db()` — `CREATE TABLE IF NOT EXISTS announce_state(key TEXT PRIMARY
    KEY, value TEXT)` в `config["STATS_DB_PATH"]`; PRAGMA WAL + `busy_timeout` как в ADR-006.
  - `async get_last_announced_version() -> str | None` — значение строки `key='last_announced_version'`
    (через `asyncio.to_thread`); нет строки → `None`.
  - `async set_last_announced_version(version: str) -> None` — UPSERT строки `last_announced_version`
    (через `asyncio.to_thread`).
  - `async run_broadcast(bot: Bot, text: str) -> tuple[int, int, int]` — фоновая рассылка:
    - адресаты — `await stats.all_active_recipient_ids()` (исключает opt-out, Задача 4);
    - последовательная отправка `bot.send_message(uid, text, reply_markup=<кнопка отписки>)` с паузой
      `ANNOUNCE_SEND_INTERVAL` (константа `= 0.05`, ~20 msg/s) между отправками;
    - `TelegramRetryAfter` → `await asyncio.sleep(e.retry_after)` и **один повтор** тому же получателю;
      повторный `RetryAfter` → снова ждать `retry_after`;
    - `TelegramForbiddenError` → пропустить, инкремент `skipped`, **не прерывать**;
    - прочие исключения отправки → `logger.warning(..., exc_info=True)`, инкремент `errors`, продолжить;
    - вернуть `(sent, skipped, errors)`.
  - Кнопка отписки под анонсом («🔕 Не показывать анонсы», `callback_data="ann:off"`) строится здесь или
    импортируется из `handlers/announce.py` — **не дублировать** код клавиатуры (выбрать один источник).
  - Никаких прямых импортов aiogram-роутеров; только `Bot` и типы клавиатуры.
- **Критерии готовности:** last-announced читается/пишется и переживает рестарт (volume); `run_broadcast`
  шлёт по всем активным адресатам с троттлингом; `RetryAfter` обрабатывается через `sleep(retry_after)`+
  повтор; `TelegramForbiddenError` → пропуск без прерывания; прочие ошибки → лог + продолжение; возвращает
  тройку счётчиков. *(ФТ-8, ФТ-9, ФТ-10, ФТ-11, ФТ-12; КП-8, КП-9, КП-10, КП-11, КП-12)*
- **Связь с ADR:** ADR-020 Решения 3, 5, 6.

### Задача 4: opt-out и выборка адресатов (`services/stats.py`)
- **Что:** аддитивная миграция колонки отписки в `user_stats` и функции работы с ней + выборка активных
  адресатов (ADR-020 Решение 3A).
- **Где:** `services/stats.py` (дополнить; существующие `record_message`/`get_stats` не менять по смыслу).
- **Контракт:**
  - В `init_db()` после создания таблицы — **идемпотентная** миграция: проверить `PRAGMA table_info(user_stats)`;
    если нет колонки `announcements_optout` → `ALTER TABLE user_stats ADD COLUMN announcements_optout
    INTEGER NOT NULL DEFAULT 0`. На существующих строках значение по умолчанию `0` (не отписан).
  - `async all_active_recipient_ids() -> list[int]` — `SELECT user_id FROM user_stats WHERE
    announcements_optout = 0` (через `asyncio.to_thread`).
  - `async set_announcements_optout(user_id: int, optout: bool) -> None` — `UPDATE user_stats SET
    announcements_optout = ? WHERE user_id = ?` (через `asyncio.to_thread`). Если строки нет (теоретически)
    — допустимо ничего не делать (адресат и так не в рассылке) либо UPSERT минимальной строки; выбрать
    простейшее без усложнения.
  - `async is_announcements_optout(user_id: int) -> bool` — для тестов/команды подписки.
- **Критерии готовности:** колонка добавляется на старой БД без потери данных; `all_active_recipient_ids`
  возвращает всех, кроме отписавшихся; `set_announcements_optout` переживает рестарт. *(ФТ-8, ФТ-14;
  КП-8, КП-14)*
- **Связь с ADR:** ADR-020 Решение 3.

### Задача 5: Хук старта — детект и превью админу (`bot.py`)
- **Что:** при старте детектировать ожидающий анонс и при наличии — отправить превью админу с кнопками;
  инициализировать стор анонсов; зарегистрировать роутер (ADR-020 Решения 4, 5).
- **Где:** `bot.py` (`main`).
- **Контракт:**
  - Добавить `init_announce_db()` рядом с прочими `init_*`.
  - Зарегистрировать `announce_router` (`dp.include_router(announce_router)`).
  - После `setup_bot_profile` (в отдельном `try/except`, **не** блокируя `start_polling`) — детект:
    1. если `config["ANNOUNCEMENTS_ENABLED"]` ложно **или** `config["ADMIN_USER_ID"]` не задан → ничего;
    2. `cur = __version__`; `last = await get_last_announced_version()`;
    3. если `cur` **не новее** `last` (semver) → ничего;
    4. `text = ANNOUNCEMENTS.get(cur)`;
    5. если `text is None` → `await set_last_announced_version(cur)` + `logger.info("нет текста анонса
       для версии %s — рассылка не запускается", cur)` (тихо «проглотить», ADR-020 Решение 4);
    6. иначе → `bot.send_message(ADMIN_USER_ID, превью)` с клавиатурой «Разослать»/«Пропустить»;
       last-announced **здесь не трогать** (пометится при действии админа).
  - Любая ошибка детекта/превью логируется и не роняет старт бота.
- **Критерии готовности:** при новой версии с текстом админу приходит превью + кнопки; при версии без
  текста — превью не приходит, событие логируется, last-announced обновляется; при выключенном флаге/
  отсутствии админа — ничего; автоматической рассылки без нажатия не происходит; сбой не роняет бота.
  *(ФТ-3, ФТ-4, ФТ-5, ФТ-7; КП-3, КП-4, КП-5, КП-7)*
- **Связь с ADR:** ADR-020 Решения 4, 5, 10.

### Задача 6: Callback-роутер и команды отписки/подписки (`handlers/announce.py`)
- **Что:** обработчики «Разослать»/«Пропустить» (админ), «Не показывать анонсы» (пользователь), команды
  `/announces_off`, `/announces_on` (ADR-020 Решения 5, 8).
- **Где:** `handlers/announce.py` (новый), экспорт `announce_router` в `handlers/__init__.py`.
- **Контракт:**
  - `callback_data`-коды (≤ 64 байта, ОГР-5): `ann:send`, `ann:skip`, `ann:off`.
  - **`ann:send`** (только если `callback.from_user.id == ADMIN_USER_ID`, иначе `callback.answer`
    «Недоступно» без действия):
    1. **сначала** `await set_last_announced_version(__version__)` (идемпотентность до цикла, ADR-020
       Решение 5A);
    2. запустить фоновую рассылку как `asyncio.create_task(run_broadcast(bot, ANNOUNCEMENTS[__version__]))`
       (не блокировать обработку апдейтов, ФТ-9);
    3. по завершении задачи — отправить админу отчёт «Рассылка v{__version__} завершена: отправлено N,
       пропущено/заблокировано M, ошибок K» (можно через callback внутри корутины задачи);
    4. ответить на callback (`callback.answer("Рассылка запущена")`), убрать/деактивировать кнопки превью.
  - **`ann:skip`** (только админ): `await set_last_announced_version(__version__)` **без** рассылки;
    ответ «Версия помечена анонсированной без рассылки»; кнопки превью убрать. Повторно превью этой
    версии не предлагается (last-announced == cur).
  - **`ann:off`** (любой пользователь): `await set_announcements_optout(user_id, True)`; ответ «Больше
    не будем присылать анонсы. Вернуть — /announces_on».
  - **`/announces_off`**: то же, что `ann:off`, командой. **`/announces_on`**:
    `set_announcements_optout(user_id, False)`; ответ «Снова будем присылать анонсы новых версий».
  - Клавиатура **превью админу**: ряд — «Разослать» (`ann:send`), «Пропустить» (`ann:skip`).
  - Клавиатура **анонса пользователю**: «🔕 Не показывать анонсы» (`ann:off`) — единый источник с
    Задачей 3 (не дублировать).
  - Все тексты — русские, дружелюбные.
- **Критерии готовности:** «Разослать» помечает версию (до рассылки) и запускает фоновую рассылку, по
  завершении приходит отчёт; «Пропустить» помечает без рассылки и превью больше не всплывает; «Не
  показывать анонсы»/`/announces_off` отписывают (persistent), `/announces_on` возвращает; не-админ не
  может запустить рассылку. *(ФТ-5, ФТ-6, ФТ-13, ФТ-14; КП-5, КП-6, КП-12, КП-13, КП-14)*
- **Связь с ADR:** ADR-020 Решения 5, 6, 8.

## Изменения публичного контракта

- **Новое поведение:** при выходе новой версии (после подтверждения админа) бот рассылает короткий
  анонс пользователям в личных чатах. Под анонсом — кнопка отписки. Добавлены команды `/announces_off`,
  `/announces_on`. В `/stats` (только админ) — строка `Версия: X.Y.Z`.
- **Команды:** `/start`, `/help` — **без изменений** (версию в `/start` не добавляем). `/stats` —
  одна новая строка версии. Новые: `/announces_off`, `/announces_on`. Обработка медиа, точка списания
  лимита, gate/рефералы — **не меняются**.
- **Миграция БД:** аддитивная, **в коде при старте** — новая таблица `announce_state` (`CREATE IF NOT
  EXISTS`) и новая колонка `user_stats.announcements_optout` (`ALTER TABLE ADD COLUMN ... DEFAULT 0`,
  идемпотентно через `PRAGMA table_info`). Старые данные сохраняются; ручных шагов деплоя нет.
- **Новые env:** `ANNOUNCEMENTS_ENABLED` (дефолт `true`).
- **Новые внешние зависимости:** нет (рассылка на штатном aiogram Bot API).
- Релиз — **MINOR**.

## Риски и митигации

- **Риск:** рассылка блокирует event loop / конкурирует с whisper на 1 vCPU. → **Митигация:** фоновая
  `asyncio.Task` + троттлинг `0.05 с` между отправками (Задача 3, ADR-020 Решение 6); не блокирует
  обработку апдейтов (ФТ-9, КП-9).
- **Риск:** превышение лимитов Telegram → временная блокировка бота. → **Митигация:** консервативный
  интервал (~20 msg/s) + backoff на `RetryAfter` (ждать `retry_after` и повтор) (КП-10).
- **Риск:** рестарт во время рассылки → повторная массовая рассылка после редеплоя. → **Митигация:**
  пометка версии анонсированной **до** цикла отправки (ADR-020 Решение 5A, КП-12); частичная доставка
  при рестарте в середине допустима (осознанный дефолт).
- **Риск:** заблокировавший бота получатель роняет рассылку. → **Митигация:** пропуск
  `TelegramForbiddenError`, рассылка продолжается; прочие ошибки — лог + продолжение (КП-11).
- **Риск:** лексикографическое сравнение версий ошибочно сочтёт `1.10.0` старше `1.9.1`. → **Митигация:**
  числовое сравнение semver-троек (Задача 1, КП-3); тест `test_version.py`.
- **Риск:** сбой механизма анонсов роняет бота или ломает медиа/лимиты. → **Митигация:** изоляция —
  стартовый хук и рассылка в `try/except`, отдельные модули; обработка медиа/лимиты/gate/рефералы не
  затрагиваются (ФТ-16, КП-16, НФТ-2).
- **Риск:** миграция колонки повредит существующую `stats.db`. → **Митигация:** идемпотентный `ALTER
  TABLE ADD COLUMN ... DEFAULT 0` через проверку `PRAGMA table_info`; тест миграции на старой схеме.
