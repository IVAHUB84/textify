# Спецификация релиза v1.11.0

- **Дата планирования:** 2026-06-25
- **Тип релиза:** minor
- **Цели:** Ввести **новый входной канал — транскрипцию речи из видео** (не OCR кадров): бот принимает
  `video` / `video_note` / документ `video/*`, извлекает аудиодорожку через уже подключённый PyAV
  (`av`, **без** системного ffmpeg) и подаёт её в существующий whisper-пайплайн; результат — как у
  аудио (прогрессивное раскрытие + те же кнопки). Переиспользуются гард 20 МБ, дневной лимит/gate
  (v1.7.0), рефералы (v1.8.0) и `result_cache`. Видео без аудио/без речи → дружелюбное сообщение без
  падения.

## Источники
- BA-требования: `ba-req/ba-requirements-021.md`
- ADR: `adrs/adr-021.md` (закрывает ОВ-1…ОВ-6: Решение 1A — единое извлечение аудио PyAV → неизменный
  `transcribe_with_timestamps`; Решение 2A — общий путь для трёх форм видео, `video_note` по типу, не
  mime; Решение 3A — **группы отложены, первый релиз только личка**; Решение 4A — только гард 20 МБ;
  Решение 5 — чанкинг вне рамок; Решение 6A — видео без речи списывает единицу, точка списания v1.7.0
  не меняется).
- Переиспользуемые ADR: ADR-001 (профиль поставки), ADR-008 (пайплайн транскрипции), ADR-010 (семафор,
  публичный контракт), ADR-015 (прогрессив), ADR-016 (кнопки + `result_cache`), ADR-017 (лимит/gate),
  ADR-018 (рефералы).

## Профиль поставки

Проект — поставляемый сервис, но Telegram-бот на **long-polling, без HTTP-порта/Traefik/edge**
(профиль ADR-001), образ из GHCR, прод ivahub (1 vCPU / 2 GB). **Контракт `deploy-guide.md` в части
порт/traefik/edge НЕ применяется** (бот не слушает HTTP). Релиз **не вводит** новых внешних
зависимостей (PyAV уже транзитивно есть через faster-whisper), **системного ffmpeg НЕ добавляет**,
новых портов/volume/env/артефактов поставки нет. `Dockerfile`, `deploy/docker-compose.yml`,
`deploy/.env.example`, `.github/workflows/deploy.yml` — **без правок**.

## Объём релиза

Не добавлять ничего сверх перечисленного (принцип «минимальные изменения», CLAUDE.md). Не дублировать
логику аудио-канала — переиспользовать `handlers/audio.process_audio`. Не трогать OCR/перевод/контракт
лимита по смыслу. Все пользовательские сообщения — на **русском**; идентификаторы/имена файлов кода —
на английском.

### Задача 1: Хелпер извлечения аудиодорожки из видео (PyAV)
- **Что:** новая функция, извлекающая аудиодорожку из байтов видео в стандартный аудио-payload через
  PyAV (`av`), без системного ffmpeg (ADR-021 Решение 1A).
- **Где:** `services/transcribe.py` (рядом с существующим декодом) **или** новый `services/media.py`
  — на усмотрение разработчика; контракт `transcribe_with_timestamps`/`transcribe` при этом **не
  меняется**.
- **Контракт:**
  - `async extract_audio(video_bytes: bytes) -> bytes | None`:
    - открыть `av.open(BytesIO(video_bytes))`, найти аудиопоток; **если аудиопотока нет → вернуть
      `None`** (не исключение);
    - перекодировать аудиопоток в стандартный аудио-контейнер (рекомендация: WAV PCM 16 kHz mono либо
      MP3 — конкретный кодек/контейнер на усмотрение, критерий: декодируется faster-whisper **и**
      принимается CF Whisper, размер минимизирован под 1 vCPU); вернуть **аудио**-байты;
    - **системный ffmpeg не использовать** — только кодеки, доступные в текущей сборке PyAV; если
      целевой энкодер недоступен — выбрать иной общедоступный аудио-кодек без доп. системных пакетов;
  - CPU-работа (демукс/энкод) выполняется **под `HEAVY_LOCAL_SEMAPHORE`** (`from services import
    HEAVY_LOCAL_SEMAPHORE`) через `asyncio.to_thread` — синхронный PyAV не должен блокировать event
    loop (ADR-010);
  - любая ошибка декода/демукса либо логируется и пробрасывается для обработки в хендлере, либо
    возвращается `None` — выбрать **один** контракт и придерживаться его в Задаче 2 (рекомендация:
    `None` для «нет аудио», исключение — для реальной ошибки декода, перехватываемое в хендлере).
- **Критерии готовности:** из видео с аудиодорожкой возвращаются ненулевые аудио-байты; из видео без
  аудиодорожки — `None`; синхронный PyAV вызывается в `to_thread` под семафором; новых системных
  зависимостей/ffmpeg нет. *(ФТ-2, ФТ-9, ФТ-10, НФТ-1, НФТ-3; КП-2, КП-7, КП-8, КП-10)*
- **Связь с ADR:** ADR-021 Решение 1, ADR-010 (семафор).

### Задача 2: Хендлер видео-канала (`handlers/video.py`)
- **Что:** новый роутер и обработчики приёма видео во всех трёх формах; единый путь обработки,
  переиспользующий `process_audio` (ADR-021 Решения 2A, 3A, 4A, 6A).
- **Где:** `handlers/video.py` (новый).
- **Контракт:**
  - `router = Router()`; `router.message.filter(F.chat.type == "private")` (как `audio`/`image` —
    **только личный чат**, группы вне этого релиза).
  - **Три тонких хендлера-входа** (по образцу `handlers/audio.py`):
    - `@router.message(F.video)` → attachment `message.video`;
    - `@router.message(F.video_note)` → attachment `message.video_note` (диспетчеризация **по типу**,
      mime не используется — у кружочков его нет);
    - `@router.message(F.document & F.document.mime_type & F.document.mime_type.startswith("video/"))`
      → attachment `message.document`.
  - Общая `process_video(message, bot, attachment)` (или эквивалент) — единый путь:
    1. `if is_oversized(attachment.file_size): answer(OVERSIZED_MESSAGE); return` (гард 20 МБ,
       `handlers/limits.py` — без изменений);
    2. `user_id = message.from_user.id if message.from_user else None`; `if user_id is None or not
       await enforce_limit(message, user_id, is_private=True): return` (лимит/gate/рефералы v1.7.0/
       v1.8.0 — **точка списания и контракт не меняются**; вызывается до обработки, как у аудио);
    3. скачать attachment в `BytesIO` через `bot.download(...)`;
    4. `audio = await extract_audio(buffer.getvalue())`;
    5. `if audio is None: await message.answer(NO_AUDIO_MESSAGE); return` (видео без аудиодорожки —
       дружелюбное русское сообщение);
    6. иначе `await process_audio(message, message, bot, audio, progressive=True)` — переиспользует
       прогрессив (v1.5.0), кнопки (v1.6.0), `result_cache`, тайм-коды и сообщение «речь не
       распознана» (`NO_SPEECH_MESSAGE` уже в `handlers/audio.py`);
    7. ошибки `extract_audio`/декода обернуть `try/except` так, чтобы бот не падал и пользователь
       получил понятное русское сообщение (например `DECODE_ERROR_MESSAGE`); обработка прочих
       сообщений не нарушается.
  - Константа `NO_AUDIO_MESSAGE` — русское дружелюбное сообщение по образцу `NO_TEXT_MESSAGE`/
    `NO_SPEECH_MESSAGE` (например «В этом видео нет звуковой дорожки — распознавать нечего.»).
  - **Не дублировать** логику выдачи/прогрессива/кнопок — только `process_audio`.
- **Критерии готовности:** `video`/`video_note`/документ `video/*` принимаются и инициируют
  транскрипцию; ответ идентичен аудио (превью-суть + кнопки Полностью/Кратко/Перевести/Задачи и,
  при длительности ≥ порога, Тайм-коды/Субтитры — через `process_audio`); видео > 20 МБ → `OVERSIZED_MESSAGE`;
  видео без аудиодорожки → `NO_AUDIO_MESSAGE`; нераспознанная речь → `NO_SPEECH_MESSAGE` (из
  `process_audio`); видео списывает единицу лимита как любое медиа; ошибка декода не роняет бота.
  *(ФТ-1, ФТ-4, ФТ-5, ФТ-6, ФТ-7, ФТ-8, ФТ-9, ФТ-10, ФТ-11, ФТ-12; КП-1, КП-3, КП-4, КП-5, КП-6, КП-7, КП-8)*
- **Связь с ADR:** ADR-021 Решения 2, 3, 4, 6; ADR-015/016/017/018 (переиспользование).

- **Команды, формат ответа, набор кнопок, OCR/аудио-каналы** — без изменений.

  - видео без аудиопотока → `None`;
  - вызов идёт через `to_thread`/под семафором (достаточно проверить, что функция async и не блокирует;
    глубокое тестирование семафора — как в существующих тестах transcribe, если есть).
- Регрессия: существующие тесты аудио/OCR/групп/лимита/рефералов/анонсов проходят без изменений

## Риски и митигации

- **Риск:** синхронный PyAV (демукс/энкод) блокирует event loop или конкурирует с whisper на 1 vCPU.
  → **Митигация:** извлечение в `asyncio.to_thread` **под `HEAVY_LOCAL_SEMAPHORE`** (Задача 1, ADR-010);
  одна перекодировка на запрос (ADR-021 Решение 1A).
- **Риск:** CF Whisper отвергнет видео-байты. → **Митигация:** в CF/local идёт **извлечённый
  аудио-payload**, а не видео; `transcribe_with_timestamps` не модифицируется (ADR-021 Решение 1A; КП-3).
- **Риск:** видео без аудиодорожки / нераспознаваемый формат роняет бота. → **Митигация:** `extract_audio`
  → `None` для «нет аудио» (→ `NO_AUDIO_MESSAGE`); `try/except` вокруг декода (→ понятное сообщение);
  бот не падает, прочие сообщения не нарушаются (ФТ-9, ФТ-10; КП-7, КП-8).
- **Риск:** `video_note` без mime не диспетчеризуется. → **Митигация:** фильтр `F.video_note` по типу,
  не по mime (ADR-021 Решение 2A; КП-1).
- **Риск:** дублирование логики аудио → расхождение поведения видео и аудио. → **Митигация:**
  переиспользование `process_audio` без копирования логики выдачи/прогрессива/кнопок/кэша (ADR-021
  Решение 4; КП-4).
- **Риск:** регрессия существующих каналов/групповой диспетчеризации. → **Митигация:** `handlers/group.py`
  **не трогаем** (видео в группах отложено); новый изолированный `video_router`; прогон существующих
  тестов (КП-9).
- **Риск:** скрытое добавление системного ffmpeg/новой зависимости. → **Митигация:** `requirements.txt`
  и `Dockerfile` **без правок**; только PyAV-кодеки; проверка в чек-листе (КП-2, КП-10).
- **Риск:** видео ≤ 20 МБ с длинной дорожкой перегрузит CPU. → **Митигация:** тот же риск, что у
  аудио ≤ 20 МБ (уже принят); сверхбольшое аудио деградирует на локальный whisper (ADR-008); отдельный
  лимит длительности не вводится (ADR-021 Решение 4A).
