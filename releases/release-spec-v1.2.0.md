# Спецификация релиза v1.2.0

- **Дата планирования:** 2026-06-23
- **Тип релиза:** minor
- **Цели:** Дать боту работать в группах/супергруппах по явному запросу (reply на медиа +
  команда `/textify` или упоминание `@bot`), без спама, переиспользуя существующий пайплайн.
  Открыть главный канал органического роста, не ломая поведение в личке (контракт 1.0.0) и не
  тратя CF-бюджет лички на групповой трафик.

## Источники

- BA-требования: `ba-req/ba-requirements-012.md`
- ADR: `adrs/adr-012.md` (опирается на ADR-001, ADR-002/003, ADR-004, ADR-005, ADR-006, ADR-007,
  ADR-008, ADR-009, ADR-010, ADR-011)

## Объём релиза

### Задача 1: Рефактор общей логики обработки медиа в переиспользуемые функции

- **Что:** вынести логику «скачать медиа → распознать → структурировать → отдать через `send_result`
  с кнопками / служебное сообщение» из приватных хендлеров в переиспользуемые функции, которые
  принимают **сообщение-носитель медиа** (откуда качать вложение) и **сообщение-адресат ответа**
  (куда отвечать). В личке носитель = адресат = само сообщение; в группе носитель = адресат =
  `reply_to_message` (ответ становится reply на исходное медиа — ФТ-6). Суть OCR/ASR/структурирования
  не меняется (ADR-004/005/008).
- **Где (файлы/модули):** `handlers/image.py`, `handlers/audio.py`.
- **Контракт (внутренние переиспользуемые функции):**
  - `handlers/image.py`:
    - функция обработки фото: качает `media_message.photo[-1]`, OCR (`recognize_text`), при наличии
      текста → `send_result(reply_target, await structure_text(text), reply_markup=actions_keyboard())`,
      иначе → `reply_target.answer(NO_TEXT_MESSAGE)`. `ChatActionSender(action=upload_document)`
      вокруг тяжёлой части (ADR-010 сохранить).
    - функция обработки изображения-документа: то же для `media_message.document`.
  - `handlers/audio.py`:
    - функция обработки аудио, принимающая помимо сообщений признак форсирования локального ASR
      (например `force_local: bool`): качает соответствующее вложение
      (`voice`/`audio`/`document`), `transcribe(audio_bytes, force_local=force_local)`, при наличии
      текста → `send_result(reply_target, …, reply_markup=actions_keyboard())`, иначе →
      `reply_target.answer(NO_SPEECH_MESSAGE)`. `ChatActionSender(action=typing)` сохранить.
  - Имена функций — на усмотрение разработчика; важно: приватные хендлеры и групповой хендлер
    вызывают **одни и те же** функции (нет дубля пайплайна).
- **Критерии готовности:** приватные хендлеры используют вынесенные функции; поведение в личке
  идентично v1.1.0; `send_result`/`actions_keyboard`/служебные сообщения не изменены.
- **Связь с ADR:** ADR-012, Решение 2 (рефактор общей логики), Решение 4 (`force_local` для ASR).

### Задача 2: Ограничение существующих хендлеров приватными чатами

- **Что:** сузить существующие медиа-хендлеры и catch-all text до `private`, чтобы в группах они
  НЕ срабатывали автоматически (НФТ-1, ОВ-3).
- **Где (файлы/модули):** `handlers/image.py`, `handlers/audio.py`, `handlers/text.py`.
- **Контракт (фильтры):**
  - `handlers/image.py`: `@router.message(F.photo & (F.chat.type == "private"))`,
    `@router.message(F.document & F.document.mime_type.startswith("image/") & (F.chat.type == "private"))`.
  - `handlers/audio.py`: `F.voice`, `F.audio`, `F.document & …audio/*` — каждый `& (F.chat.type == "private")`.
  - `handlers/text.py`: `@router.message(~Command("start", "help") & (F.chat.type == "private"))`.
  - Альтернативно допускается роутер-уровневый `router.message.filter(F.chat.type == "private")`
    (ADR-012 Решение 1) — но эффект обязан быть тем же: в группах эти хендлеры не матчатся.
- **Критерии готовности:** в чате типа `group`/`supergroup` ни один из этих хендлеров не
  срабатывает; в `private` все срабатывают как раньше.
- **Связь с ADR:** ADR-012, Решение 1.

### Задача 3: Новый групповой роутер (`handlers/group.py`)

- **Что:** новый роутер, обрабатывающий медиа в группах **только** по явному запросу: сообщение в
  `group`/`supergroup`, являющееся reply на сообщение с поддерживаемым медиа, и содержащее команду
  `/textify` ИЛИ упоминание бота.
- **Где (файлы/модули):** новый `handlers/group.py`; экспорт `group_router` в `handlers/__init__.py`.
- **Контракт:**
  - Фильтр чата: `F.chat.type.in_({"group", "supergroup"})`.
  - Триггер: `Command("textify")` (ловит `/textify` и `/textify@<bot>`) **ИЛИ** упоминание бота —
    кастомный фильтр, читающий `message.entities`/`caption_entities`: сущность `mention` с текстом
    `@<bot_username>` (регистронезависимо) или `text_mention` на самого бота. Username бота приходит
    из `bot.py` (Задача 5).
  - Условие reply: обрабатывается `message.reply_to_message`. Поддерживаемые типы медиа —
    те же, что в контракте 1.0.0: `voice`, `audio`, `photo`, документ `image/*`, документ `audio/*`.
  - Диспетчеризация: по типу медиа в `reply_to_message` вызвать переиспользуемую функцию из Задачи 1
    с `media_message = reply_to_message`, `reply_target = reply_to_message`. Для аудио передать
    `force_local` согласно `GROUP_ASR_LOCAL` (Задача 6).
  - Если `reply_to_message` отсутствует или не содержит поддерживаемого медиа — короткая русская
    подсказка (Решение 5): например «Ответьте этой командой на голосовое, аудио или фото с текстом.»
    (`message.answer(...)`). Подсказка отвечает только на явный триггер.
  - На нерелевантные сообщения (без триггера) роутер не реагирует (хендлеры не матчатся).
- **Критерии готовности:** reply+`/textify`/упоминание на поддерживаемое медиа → обработка
  `reply_to_message`, ответ reply на исходное медиа с кнопками; триггер без подходящего медиа →
  подсказка; прочий трафик в группе → молчание.
- **Связь с ADR:** ADR-012, Решения 1, 2, 4, 5, 7.

### Задача 4: Группа — ответ reply на исходное медиа, кнопки работают (проверка)

- **Что:** убедиться, что отдача в группе идёт через `send_result(reply_to_message, …)` (reply на
  исходное медиа — ФТ-6) и что inline-действия «Кратко»/«Перевести» работают в группе без правок
  `actions.py`.
- **Где (файлы/модули):** `handlers/actions.py` (проверка, что не завязан на тип чата — изменений
  не требует), `handlers/group.py` (использование `send_result`).
- **Контракт:** `actions_keyboard()` и callback-хендлеры `act:sum`/`act:tr` (ADR-009) неизменны;
  `callback.message` доступен в группе.
- **Критерии готовности:** под одно-сообщенческим результатом в группе видны кнопки; нажатие
  «Кратко»/«Перевести» в группе отрабатывает (тот же путь, что в личке).
- **Связь с ADR:** ADR-012, Решение 4; ADR-009; ADR-007.

### Задача 5: Регистрация роутеров и порядок в `bot.py`, username бота для фильтра упоминания

- **Что:** получить username бота на старте и прокинуть в фильтр упоминания; зарегистрировать
  `group_router` в фиксированном порядке.
- **Где (файлы/модули):** `bot.py`, `handlers/__init__.py`.
- **Контракт:**
  - Получить username бота через `Bot.get_me()` на старте (до/в `main()`), передать в фильтр
    упоминания группового роутера (через замыкание/DI/workflow_data — на усмотрение разработчика).
  - Порядок роутеров: `commands_router` → `actions_router` → `group_router` → `image_router` →
    `audio_router` → `text_router`.
- **Критерии готовности:** бот стартует; в группе `/textify`-reply и упоминание-reply ловятся
  групповым роутером; контрактные команды `/start`/`/help`/`/stats` работают в обоих типах чата.
- **Связь с ADR:** ADR-012, Решения 2, 9.

### Задача 6: Конфиг `GROUP_ASR_LOCAL` и форсирование локального ASR в группах

- **Что:** ввести опциональный булев `GROUP_ASR_LOCAL` (дефолт `true`); при `true` групповой ASR
  идёт принудительно локальным faster-whisper (без CF, без расхода CF-бюджета); при `false` —
  группы идут общим путём `transcribe()`. OCR в группах — всегда локальный (ничего менять не надо).
- **Где (файлы/модули):** `config.py`, `services/transcribe.py`, `handlers/group.py` (и аудио-функция
  из Задачи 1), `.env.example`, `deploy/.env.example`.
- **Контракт:**
  - `config.py`: читает `GROUP_ASR_LOCAL` (булев; `true/1/yes` → True, `false/0/no` → False;
    некорректное/пустое → дефолт `True`); кладёт в `config["GROUP_ASR_LOCAL"]`. Старт не падает.
  - `services/transcribe.py`: `transcribe(audio_bytes, force_local: bool = False)`. При `force_local`
    → сразу `_transcribe_local(audio_bytes)` (под семафором, без CF, без `cf_budget_*`). По дефолту
    (`force_local=False`) — текущее поведение (CF + бюджет + фолбэк) полностью сохраняется.
  - Групповой аудио-путь вызывает обработку с `force_local = config["GROUP_ASR_LOCAL"]`; личка
    всегда `force_local=False`.
  - `.env.example` / `deploy/.env.example`: добавить строку с дефолтом и комментарием, например:
    ```
    # В группах ASR по умолчанию идёт локальным faster-whisper (не тратит CF-бюджет лички).
    # false → группы используют общий путь ASR_PROVIDER + дневной бюджет CF.
    GROUP_ASR_LOCAL=true
    ```
- **Критерии готовности:** при `GROUP_ASR_LOCAL=true` групповой аудио-запрос не вызывает CF и не
  инкрементирует `cf_usage`; личка не затронута; при `GROUP_ASR_LOCAL=false` группы идут общим путём;
  бот стартует без переменной.
- **Связь с ADR:** ADR-012, Решение 4; ADR-008; ADR-010 (семафор); ADR-011 (бюджет).

### Задача 7: Учёт статистики в группах (без изменений схемы)

- **Что:** подтвердить, что групповые обращения учитываются существующим `StatsMiddleware` как
  обычные сообщения; схему `user_stats` и контракт `/stats` НЕ менять.
- **Где (файлы/модули):** `middlewares/stats.py` (без изменений), `services/stats.py` (без изменений).
- **Контракт:** `classify_message` и схема БД неизменны; групповое сообщение учитывается по
  содержимому (триггер `/textify` → `command`, текст с упоминанием → `text`).
- **Критерии готовности:** изменений в учёте нет; учёт не ломается от групповых чатов; сбой учёта не
  прерывает обработку.
- **Связь с ADR:** ADR-012, Решение 6; ADR-006.

## Изменения публичного контракта

- **Контракт 1.0.0 для лички (ADR-010) — НЕ меняется.** Входные каналы, формат ответа `send_result`,
  служебные сообщения, inline-действия, команды `/start`/`/help`/`/stats` в личке идентичны
  v1.0.0/v1.1.0.
- **Добавление (обратимо-совместимое, MINOR):**
  - бот реагирует в `group`/`supergroup` на новый триггер (reply на медиа + `/textify` или
    упоминание бота), отвечая reply на исходное медиа через тот же `send_result` с кнопками;
  - команда `/textify` распознаётся как триггер в группах; в меню (`BOT_COMMANDS`) **не показывается**;
  - новый опциональный env `GROUP_ASR_LOCAL` (дефолт `true`); реальный `.env` не коммитится.
- Миграции данных нет; схема БД неизменна. Новых внешних зависимостей и направлений трафика нет
  (профиль поставки ADR-001 сохранён).

## Тесты

Минимальный набор (на моках aiogram `Message`/`CallbackQuery`; тяжёлые сервисы OCR/ASR/LLM
мокируются):

- **Группа: авто-медиа без запроса НЕ обрабатывается** — сообщение с `photo`/`voice` в
  `group`/`supergroup` без reply/триггера → ни один хендлер не отвечает (бот молчит).
- **Группа: reply + `/textify` на голосовое** → обрабатывается `reply_to_message`, результат
  отправляется reply на исходное медиа (адресат = `reply_to_message`).
- **Группа: reply + упоминание `@bot`** на фото/аудио → обработка `reply_to_message` (OCR/ASR).
- **Группа: команда/упоминание БЕЗ reply (или reply на сообщение без поддерживаемого медиа)** →
  короткая русская подсказка, обработки нет.
- **Личка — регресс не допустить** — `photo`/`voice`/`audio`/документ-медиа в `private`
  обрабатываются автоматически без reply (поведение v1.1.0); catch-all text в личке отвечает
  заглушкой как раньше.
- **Кнопки в группе** — под результатом в группе присутствует `actions_keyboard`; callback
  `act:sum`/`act:tr` с `callback.message` из группы отрабатывает (мок).
- **Групповой ASR форсирует локальный путь при `GROUP_ASR_LOCAL=true`** — групповой аудио-запрос
  вызывает локальный путь, CF/`cf_budget_*` не вызываются; при `GROUP_ASR_LOCAL=false` — общий путь.
- **Фильтры чата корректны** — фильтр упоминания распознаёт `mention`/`text_mention` на бота и не
  срабатывает на чужие упоминания/произвольную подстроку; `Command("textify")` ловит
  `/textify@<bot>`.
- **`transcribe(force_local=True)`** идёт сразу в локальный путь без обращения к CF; дефолт
  (`force_local=False`) сохраняет текущее поведение (CF + бюджет + фолбэк).

## Риски и митигации

- **Риск:** privacy mode настроен не так → бот не получает команды/упоминания/reply в группе и не
  видит триггеры. → **Митигация:** ручной шаг чек-листа деплоя (проверить privacy mode в @BotFather:
  команды/упоминания/ответы получаем; полное отключение privacy НЕ делаем).
- **Риск:** неверные фильтры → спам в группе (срабатывание авто-обработки/catch-all). →
  **Митигация:** все приватные хендлеры сужены до `private`; групповой путь требует явного триггера;
  отдельный тест «авто-медиа в группе не обрабатывается».
- **Риск:** групповая нагрузка на 1 vCPU / 1.9 GB (всплеск голосовых). → **Митигация:** глобальный
  семафор (ADR-010, лимит 1) сериализует тяжёлые локальные операции; локальный ASR в группах
  (`GROUP_ASR_LOCAL`) + дневной бюджет CF (ADR-011) защищают бесплатность и сервер.
- **Риск:** хардкод username бота в фильтре упоминания → хрупкость. → **Митигация:** username берётся
  из `Bot.get_me()` на старте, не хардкодится.
- **Риск:** регресс в личке при рефакторе. → **Митигация:** переиспользуемые функции вызываются и
  приватными хендлерами; тесты регресса лички.
