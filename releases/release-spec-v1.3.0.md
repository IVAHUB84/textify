# Спецификация релиза v1.3.0

- **Дата планирования:** 2026-06-23
- **Тип релиза:** minor
- **Цели:** Запустить виральную петлю роста нативными бесплатными средствами Telegram: реферальные
  deep-link со счётчиком приглашённых, кнопку «Поделиться ботом» и ненавязчивую подпись под
  результатом. Без Cloudflare/LLM, без новых внешних зависимостей и портов. Завершает текущий план
  роста.

## Источники

- BA-требования: `ba-req/ba-requirements-013.md`
- ADR: `adrs/adr-013.md` (основной); опора: `adrs/adr-006.md` (SQLite/учёт/middleware/`/stats`),
  `adrs/adr-007.md` (`send_result`), `adrs/adr-009.md` (inline-кнопки, правило markup),
  `adrs/adr-010.md` (контракт 1.0.0), `adrs/adr-012.md` (username из `Bot.get_me()`, группы),
  `adrs/adr-001.md` (профиль поставки)

## Объём релиза

### Задача 1: Таблица рефералов и сервис `services/referrals.py`

- **Что:** новый сервис работы с реферальными связями поверх существующей SQLite (тот же
  `STATS_DB_PATH`, паттерн доступа ADR-006: короткое соединение + WAL + `busy_timeout` +
  `asyncio.to_thread`). Идемпотентность и защита от гонок — на уровне БД.
- **Где (файлы/модули):** `services/referrals.py` (новый); вызов инициализации схемы — `bot.py`.
- **Контракт:**
  - Схема:
    ```sql
    CREATE TABLE IF NOT EXISTS referrals (
        referred_id  INTEGER PRIMARY KEY,
        referrer_id  INTEGER NOT NULL,
        created_at   TEXT NOT NULL
    );
    CREATE INDEX IF NOT EXISTS idx_referrals_referrer ON referrals(referrer_id);
    ```
  - `created_at` — ISO-8601 UTC (`datetime.now(timezone.utc).isoformat()`), как в `user_stats`.
  - Запись: `INSERT OR IGNORE INTO referrals (referred_id, referrer_id, created_at) VALUES (?,?,?)`
    (идемпотентность по PK `referred_id`).
  - Публичные корутины:
    - `async def record_referral(referrer_id: int, referred_id: int) -> None`
    - `async def count_referrals(referrer_id: int) -> int` (число строк с этим `referrer_id`)
    - `async def total_referrals() -> int` (`COUNT(*)`)
    - `async def top_referrers(limit: int) -> list[tuple[int, int]]`
      (`referrer_id, count` по убыванию count, `LIMIT limit`)
    - `def init_referrals_db() -> None` (создание таблицы и индекса; PRAGMA WAL/busy_timeout)
  - Все операции чтения/записи — синхронная функция внутри `asyncio.to_thread`; своё короткоживущее
    соединение на операцию.
- **Критерии готовности:**
  - Таблица и индекс создаются в `STATS_DB_PATH` при `init_referrals_db()`.
  - `record_referral` идемпотентна: повторная запись того же `referred_id` не меняет существующую
    строку и не падает.
  - `count_referrals`/`total_referrals`/`top_referrers` возвращают корректные значения на тестовой БД.
- **Связь с ADR:** ADR-013, Решение 2.

### Задача 2: Признак «новый пользователь» через middleware

- **Что:** определить «первое обращение» атомарно в `record_message` (UPSERT уже знает, вставка это
  или обновление) и пробросить признак в хендлер `/start` через `data`. Без второго чтения БД.
- **Где (файлы/модули):** `services/stats.py` (`record_message`, `_record_message_sync`),
  `middlewares/stats.py` (`StatsMiddleware`).
- **Контракт:**
  - `_record_message_sync(...)` возвращает `bool` — была ли строка создана (первое обращение).
    Определение через результат UPSERT (например, сравнение состояния до/после в одной транзакции
    или `INSERT ... ON CONFLICT` с детекцией вставки; конкретный приём — за разработчиком, но
    **атомарно** в одной транзакции).
  - `async def record_message(user_id: int, msg_type: str) -> bool` — пробрасывает этот `bool`.
  - `StatsMiddleware`: результат `record_message` кладётся в `data["is_new_user"]`. При исключении
    учёта (тихая деградация ADR-006) — `data["is_new_user"] = False`.
- **Критерии готовности:**
  - Первое обращение пользователя → `record_message` возвращает `True`; повторное → `False`.
  - При исключении в `record_message` middleware не падает, кладёт `is_new_user = False`, управление
    передаётся дальше.
  - Существующая логика учёта (инкременты, `first_seen` не перезаписывается) не изменена.
- **Связь с ADR:** ADR-013, Решение 1.

### Задача 3: Общий держатель username бота

- **Что:** вынести кэш username бота в общую точку, доступную `/start`, подписи и группам; без
  лишних round-trip к `Bot.get_me()` и без хардкода.
- **Где (файлы/модули):** общий держатель (`config.py` или новый `services/bot_identity.py` — на
  усмотрение разработчика, но в одной точке); `bot.py` (инициализация на старте); `handlers/group.py`
  (переход на общий держатель вместо локальной глобали `_bot_username`).
- **Контракт:**
  - `set_bot_username(username: str) -> None`, `get_bot_username() -> str` (пустая строка, если не
    задан).
  - `bot.py` после `me = await bot.get_me()` вызывает `set_bot_username(me.username)` один раз.
  - `handlers/group.py` использует общий держатель (нормализация `lstrip("@").lower()` для сравнения
    упоминаний сохраняется — может остаться локальной деталью группового модуля).
- **Критерии готовности:**
  - Username получается один раз на старте; `/start`, подпись и группы читают из общего держателя.
  - Групповые упоминания (ADR-012) продолжают работать.
- **Связь с ADR:** ADR-013, Решение 3.

### Задача 4: Реф-логика, ссылка, счётчик и кнопка «Поделиться» в `/start`

- **Что:** в приватном `/start` разобрать `ref_`-параметр, идемпотентно зафиксировать реферал для
  нового пользователя (не самоприглашение), показать персональную реф-ссылку, счётчик «приглашено: N»
  и inline-кнопку «Поделиться ботом».
- **Где (файлы/модули):** `handlers/commands.py` (`cmd_start`; фабрика клавиатуры — здесь или
  `handlers/keyboards.py`).
- **Контракт:**
  - Сигнатура: `async def cmd_start(message: Message, is_new_user: bool) -> None`.
  - Разбор start-параметра: через `CommandObject.args` (фильтр `Command("start")` с
    `CommandObject`) или разбор `message.text` после `/start`. Валидный формат — `ref_<int>`.
  - Фиксация реферала **только при всех** условиях (ФТ-3):
    1. `message.chat.type == "private"`;
    2. `is_new_user is True`;
    3. start-параметр распарсен как `ref_<int>` → `referrer_id`;
    4. `referrer_id != message.from_user.id` (не самоприглашение).
    Тогда `await record_referral(referrer_id, message.from_user.id)` в `try/except` (сбой не
    прерывает `/start`); факт фиксации — `logger.info(...)` без секретов.
  - Персональная ссылка: `https://t.me/<get_bot_username()>?start=ref_<user_id>`.
  - Счётчик: `await count_referrals(user_id)`; при сбое БД — деградирует до 0, `/start` не падает.
  - Кнопка: `InlineKeyboardButton(text="Поделиться ботом", url="https://t.me/share/url?url=<url-enc
    реф-ссылка>&text=<url-enc краткий русский текст>")`; прикрепляется к `message.answer(...,
    reply_markup=...)` ответа `/start` (НЕ через `send_result`). url/text кодируются
    `urllib.parse.quote`.
  - Реф-ссылка, счётчик и кнопка показываются и при `/start` без реф-параметра, и при невалидном
    параметре (поведение `/start` иначе как раньше).
- **Критерии готовности:**
  - КП-1…КП-7 (BA): ссылка с username из `get_bot_username` (не хардкод), счётчик, фиксация для
    нового по чужой ссылке, идемпотентность, игнор самоприглашения, реферал только для нового,
    кнопка «Поделиться» с персональной ссылкой.
  - В группах реф-логика не выполняется (`chat.type != "private"`).
- **Связь с ADR:** ADR-013, Решения 1, 2, 3, 5, 7.

### Задача 5: Подпись attribution в `send_result`

- **Что:** централизованно добавлять подпись `\n\n— @<bot_username>` к результату **только** в
  одно-сообщенческом режиме, при `ATTRIBUTION_FOOTER` on, и только если она не выталкивает текст за
  лимит одного сообщения.
- **Где (файлы/модули):** `services/reply.py` (`send_result`); `config.py` (`ATTRIBUTION_FOOTER`).
- **Контракт:**
  - `ATTRIBUTION_FOOTER` в `config.py` — булев, дефолт `True`; парсинг как `GROUP_ASR_LOCAL`
    (`true/1/yes` → on, `false/0/no` → off, иное/пусто → дефолт on); некорректное значение не роняет
    старт.
  - В `send_result`, в ветке `len(text) <= MAX_MESSAGE_LEN`:
    - подпись `= "\n\n— @" + get_bot_username()` добавляется к тексту **до** `message.answer`
      тогда и только тогда, когда: `config["ATTRIBUTION_FOOTER"]` истинно **и**
      `get_bot_username()` непустой **и** `len(text) + len(подпись) <= MAX_MESSAGE_LEN`;
    - иначе текст отправляется без подписи (поведение ADR-007 не меняется).
  - В ветках split (серия) и file подпись **не добавляется** (текст там уже `> MAX_MESSAGE_LEN`).
  - `reply_markup` (кнопки ADR-009) прикрепляется как раньше (к одиночному сообщению).
- **Критерии готовности:**
  - КП-9: подпись присутствует под коротким результатом (on), отсутствует при серии и файле.
  - КП-10: подпись не ломает сплиттер/файл/кнопки; одиночное сообщение с подписью укладывается в
    лимит.
  - КП-11: `ATTRIBUTION_FOOTER=off` отключает подпись.
  - Подпись добавляется только в `send_result` (результат распознавания/действий), не в
    `/start`/`/help`/`/stats`/служебных сообщениях.
- **Связь с ADR:** ADR-013, Решения 3, 4.

### Задача 6: Реферальные метрики в `/stats`

- **Что:** добавить в админский `/stats` блок реферальных метрик.
- **Где (файлы/модули):** `handlers/commands.py` (`cmd_stats`).
- **Контракт:**
  - После существующих метрик (ADR-006) — блок:
    - `Всего рефералов: <total_referrals()>`

  - Идентификация приглашающего — по `user_id` (без имён). Простой текст, стиль существующего

  - Авторизация (`ADMIN_USER_ID`) и ответ не-админу не меняются (ADR-006); реферальные метрики
    не-админу не раскрываются.
- **Критерии готовности:** КП-12 (метрики для админа), КП-13 (не раскрываются не-админу).
- **Связь с ADR:** ADR-013, Решение 6.

## Изменения публичного контракта

- **Команды:** `/start`, `/help`, `/stats` сохраняются (контракт 1.0.0, ADR-010). `/start` и `/stats`
  получают **аддитивный** вывод (реф-ссылка/счётчик/кнопка; реферальные метрики) — обратимо-совместимо.
- **Формат ответа:** под одно-сообщенческим результатом появляется аддитивная подпись (управляется
  env, по умолчанию on); серия/файл/кнопки (ADR-007/009) не меняются.
- **Внутренние сигнатуры (не публичный контракт):** `record_message(...) -> bool` (было `None`);
  `send_result` — поведение расширено (подпись), сигнатура не меняется. Изменения обратимо-совместимы.
- **Схема БД:** добавляется новая таблица `referrals` в существующей SQLite; `user_stats` не меняется.
  Ручной миграции не требуется (создаётся при старте). MINOR.

## Тесты

Все тесты — на моках aiogram-объектов; БД — во временном sqlite (override `STATS_DB_PATH` на tmp);
дату и username — мокать. Минимальный обязательный набор:

- **referrals — запись для нового приглашённого:** `record_referral` создаёт строку
  `(referred_id, referrer_id, created_at)`.
- **referrals — самоприглашение:** при `referrer_id == referred_id` `/start` не вызывает
  `record_referral` (проверка на уровне хендлера/логики).
- **referrals — идемпотентность:** повторный `record_referral` с тем же `referred_id` (тот же или
  другой `referrer_id`) не меняет существующую строку и не падает (`INSERT OR IGNORE`).
- **count_referrals:** возвращает число уникальных приглашённых для `referrer_id`; растёт на 1 после
  нового приглашённого.
- **total_referrals / top_referrers:** корректные значения и порядок (по убыванию count, `LIMIT`);
  пустая БД → 0 / пустой список.
- **is_new_user:** `record_message` возвращает `True` на первое обращение, `False` на повторное;
  `StatsMiddleware` кладёт корректный `data["is_new_user"]`; при исключении учёта → `False`.
- **/start фиксирует реферал только при new + private + valid ref + not-self:** комбинаторная
  проверка условий (новый/известный, private/group, валидный/невалидный ref, self/not-self).
- **/start без deep-link:** работает как раньше; реф-ссылка и счётчик показываются; реферал не
  создаётся.
- **/start счётчик и кнопка:** ответ содержит `?start=ref_<user_id>` с замоканным username и
  «приглашено: N»; присутствует URL-кнопка «Поделиться ботом» с `t.me/share/url` и персональной
  ссылкой.
- **подпись — только одиночное сообщение:** короткий текст → подпись присутствует; текст
  `> MAX_MESSAGE_LEN` (серия и файл) → подпись отсутствует.
- **подпись не толкает за лимит:** текст, у которого `len(text) <= MAX_MESSAGE_LEN`, но
  `len(text) + len(подпись) > MAX_MESSAGE_LEN` → подпись НЕ добавляется, текст уходит одним
  сообщением как есть.
- **ATTRIBUTION_FOOTER=off:** подпись не добавляется даже под коротким результатом.
- **подпись не ломает кнопки:** одиночное сообщение с `reply_markup` (ADR-009) уходит с кнопками и с
  подписью.
- **/stats реферальные метрики:** для админа вывод содержит «всего рефералов» и «топ приглашающих»;
  для не-админа — прежний отказ, реферальные данные не раскрыты.

## Риски и митигации

- **Риск:** «новый пользователь» определяется относительно outer-middleware-учёта (middleware пишет
  `first_seen` ДО хендлера) — наивная проверка наличия в `user_stats` всегда вернёт «известен», и ни
  один реферал не зафиксируется. → **Митигация:** признак вычисляется атомарно в `record_message`
  (вставка vs обновление) и пробрасывается через `data["is_new_user"]` (ADR-013, Решение 1);
  специальный тест на `is_new_user` (True/False/при сбое).
- **Риск:** гонка одновременного первого обращения нового пользователя → двойная запись реферала. →
  **Митигация:** `referred_id` PRIMARY KEY + `INSERT OR IGNORE` (идемпотентность и защита от гонок на
  уровне БД); атомарность UPSERT в `record_message`.
- **Риск:** подпись выталкивает текст за лимит → нежелательный split/попадание подписи в границы
  разбивки (ломает ADR-007). → **Митигация:** подпись добавляется только если
  `len(text) + len(подпись) <= MAX_MESSAGE_LEN` и только в одно-сообщенческой ветке; иначе не
  добавляется (ADR-013, Решение 4); специальные тесты на границу.
- **Риск:** username не получен (`get_me` упал/пуст) → ссылка/подпись битые. → **Митигация:** при
  пустом username подпись не добавляется; реф-ссылка формируется из держателя (пустой username —
  ссылка деградирует, но `/start` не падает); username получается тем же проверенным путём, что в
  v1.2.0.
- **Риск:** сбой БД рефералов прерывает `/start`. → **Митигация:** фиксация реферала и чтение
  счётчика — в `try/except`, тихая деградация (ADR-006); `/start` всегда отвечает.

## Деплой-заметки

- Новых обязательных секретов нет. Единственная новая env — `ATTRIBUTION_FOOTER` (опциональна, дефолт
