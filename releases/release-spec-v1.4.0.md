# Спецификация релиза v1.4.0

- **Дата планирования:** 2026-06-23
- **Тип релиза:** minor
- **Цели:** Сделать группы проактивными — бот сам замечает медиа и предлагает inline-кнопку
  «Распознать»; распознавание по тапу. Убрать трение (не нужно знать команду), усилить органический
  рост, не сломав контракт 1.0.0 лички и не перегрузив сервер.

## Объём релиза

### Задача 1: Проактивный оффер-кнопка на медиа в группе

- **Что:** В чате `group`/`supergroup` на новое сообщение с поддерживаемым медиа (`voice`, `audio`,
  `photo`, `document` с `mime_type` `image/*`, `document` с `mime_type` `audio/*`) бот отвечает
  **reply на это медиа-сообщение** одним сообщением с компактной inline-кнопкой «Распознать».
  Распознанный текст в момент оффера **не отправляется**. На обычный текст и прочий трафик в группе
  бот **не реагирует**.
- **Где (файлы/модули):** `handlers/group.py` — новый media-offer хендлер.
- **Контракт:**
  - Фильтр: срабатывает только в `group`/`supergroup` (роутер уже отфильтрован
    `F.chat.type.in_({"group","supergroup"})`) и только на поддерживаемое медиа (переиспользовать
    `_has_supported_media`, проверяя само сообщение). Хендлер должен **не** перехватывать
    сообщения-триггеры v1.2.0 (которые могут также не содержать медиа) — медиа-оффер реагирует на
    наличие медиа в самом сообщении, триггер-хендлеры — на команду/упоминание; пересечения по
    содержимому нет.
  - Подпись кнопки по типу медиа: для аудио/голоса (включая `document` `audio/*`) —
    **«Распознать голос»**; для изображений (`photo` и `document` `image/*`) — **«Распознать текст»**.
  - `callback_data` кнопки — единый `"grec"` (≤ 64 байт, без содержимого медиа).
  - Оффер отправляется как reply на медиа-сообщение (`message.answer(..., reply_markup=...)` в группе
    отвечает в тот же чат; для reply на конкретное сообщение использовать reply на исходное медиа,
    чтобы при тапе `callback.message.reply_to_message` указывал на медиа — это обязательное условие
    Задачи 3).
  - Один оффер на одно медиа: хендлер висит на апдейте нового `message` (не `edited_message`) —
    одно входящее медиа = один оффер. Персистентный дедуп не вводится.
- **Критерии готовности:**
  - На voice/audio/document-audio в группе бот шлёт reply-сообщение с кнопкой «Распознать голос».
  - На photo/document-image в группе бот шлёт reply-сообщение с кнопкой «Распознать текст».
  - На обычное текстовое сообщение в группе бот молчит (никакого ответа).
  - В момент оффера распознанный текст не отправляется.
- **Связь с ADR:** ADR-014, Решение 2, 3.

### Задача 2: Callback-хендлер «grec» — распознавание по тапу

- **Что:** По нажатию кнопки «Распознать» бот берёт исходное медиа из
  `callback.message.reply_to_message`, проверяет доступность, прогоняет через общий диспетчер
  обработки и постит результат reply на исходное медиа с кнопками «Кратко»/«Перевести».
- **Где (файлы/модули):** `handlers/group.py` — callback-хендлер на `F.data == "grec"` (разместить в
  `group_router`; альтернативно в `handlers/actions.py` — но логика группо-специфична и переиспользует
  `_dispatch_media`/`_has_supported_media` из `group.py`, поэтому предпочтительно `group.py`).
- **Контракт:**
  - `callback.answer()` вызывается **всегда** (снять «часики») — преемственность ADR-009.
  - Извлечь `reply = callback.message.reply_to_message`. Проверить: `isinstance(reply, Message)` (не
    `None`, не `InaccessibleMessage`) **и** `_has_supported_media(reply)`. Если проверка не пройдена —
    короткое русское служебное сообщение через `callback.answer("…", show_alert=True)`, пайплайн **не**
    вызывается, бот **не** падает.
  - При валидном медиа — `callback.answer("Распознаю…")` и вызов `_dispatch_media(reply, bot)`, где
    `bot = callback.bot`. `_dispatch_media` уже: определяет тип медиа, скачивает, вызывает
    `process_audio(reply, reply, bot, audio_bytes, force_local=config["GROUP_ASR_LOCAL"])` для
    аудио/голоса/документ-аудио либо `process_photo(reply, reply, bot)` / `process_image_document(reply,
    reply, bot)` для изображений; результат идёт через `send_result` reply на исходное медиа с
    `actions_keyboard()` (кнопки «Кратко»/«Перевести»), индикатор прогресса (`ChatActionSender`) — внутри
    общих функций (ADR-010).
- **Критерии готовности:**
  - Тап по кнопке на аудио → берётся `reply_to_message`, вызывается `process_audio` с
    `force_local=GROUP_ASR_LOCAL`, результат reply на медиа с кнопками «Кратко»/«Перевести».
  - Тап по кнопке на изображении → вызывается `process_photo`/`process_image_document`, результат reply
    на медиа с кнопками.
  - `reply_to_message` = `None`/`InaccessibleMessage`/без поддерживаемого медиа → служебное сообщение,
    пайплайн не вызван, без исключения.
  - `callback.answer()` вызывается во всех ветках.
- **Связь с ADR:** ADR-014, Решение 3, 4, 5, 6 (callback не считается в статистике).

### Задача 3: Сохранение прежнего триггера v1.2.0 и общего кода обработки

- **Что:** Прежний путь «reply на медиа + `/textify`/упоминание бота» сохраняется как дополнительный.
  Оба пути (триггер и оффер-тап) ведут в общий `_dispatch_media`.
- **Где (файлы/модули):** `handlers/group.py` — существующие `handle_group_textify_command`,
  `handle_group_mention`, `_handle_trigger`, `_dispatch_media`, `_has_supported_media`, `_is_bot_mention`
  **не удалять**, переиспользовать.
- **Контракт:** Триггер-хендлеры и media-offer хендлер не должны перехватывать сообщения друг друга
  (триггер — команда/упоминание; оффер — наличие медиа в сообщении). Дублирующего распознавания одного
  медиа не возникает: триггер и тап — разные пользовательские действия; оффер на медиа один.
- **Критерии готовности:**
  - Прежний триггер (reply на медиа + `/textify`/упоминание) продолжает распознавать (регресс
    `tests/test_v120_group.py` проходит).
  - На медиа-сообщение, которое одновременно не является reply-триггером, бот шлёт оффер, а не
    подсказку триггера.
- **Связь с ADR:** ADR-014, Решение 4.

### Задача 4: Личка и контракт 1.0.0 без изменений

- **Что:** Приватные медиа-хендлеры и `text_router` остаются private-only; авто-распознавание в личке
  без кнопок-офферов; контракт 1.0.0 сохранён.
- **Где (файлы/модули):** `handlers/image.py`, `handlers/audio.py`, `handlers/text.py` — **без
  изменений** (фильтр `F.chat.type == "private"` сохраняется).
- **Контракт:** В личке медиа распознаётся автоматически (как в 1.0.0); проактивные офферы в личке не
  появляются; на текст в группе `text_router` не реагирует (он private-only).
- **Критерии готовности:** Регресс-тесты лички (image/audio/text) проходят без изменений; в группе
  приватные хендлеры не срабатывают.
- **Связь с ADR:** ADR-014, Решение 9; ADR-010 (контракт 1.0.0).

## Изменения публичного контракта

- **Личка (контракт 1.0.0, ADR-010):** без изменений. Команды, входные каналы, формат ответа, inline-
  действия — прежние.
- **Группы:** добавляется проактивный оффер-кнопка и распознавание по тапу (новое поведение,
  обратимо-совместимое); прежний триггер v1.2.0 сохранён. Это расширение, не ломающее существующее.
- **Эксплуатация:** новое **обязательное** условие режима групп — Group Privacy OFF (ручной шаг
  @BotFather). Без него проактивные офферы не появляются (деградация без ошибок). Прежний v1.2.0-путь
  при privacy ON работал на reply+триггер — при privacy OFF он также работает.
- **Env / секреты / порты / зависимости:** без изменений (новой env нет — ADR-014, Решение 7).
- **Схема БД (`user_stats`):** без изменений.

## Тесты

Все тесты — на моках aiogram `Message`/`CallbackQuery` (по образцу `tests/test_v120_group.py`), без
сети и без реальных моделей.

Новый набор (например, `tests/test_v140_proactive_group.py`):
- **Оффер на аудио:** медиа-сообщение с `voice`/`audio`/`document audio/*` в группе → бот шлёт оффер
  reply на медиа с кнопкой, текст кнопки = «Распознать голос», `callback_data == "grec"`.
- **Оффер на изображение:** `photo`/`document image/*` в группе → оффер с кнопкой «Распознать текст».
- **Молчание на текст:** обычное текстовое сообщение в группе → media-offer хендлер не срабатывает
  (бот не отвечает); проверить, что `message.answer` не вызван media-offer хендлером.
- **Тап → аудио:** `CallbackQuery` с `data="grec"`, `message.reply_to_message` = мок аудио-сообщения →
  вызывается обработка аудио с `force_local` по `GROUP_ASR_LOCAL`; `callback.answer` вызван; результат
  идёт с `actions_keyboard()` (кнопки «Кратко»/«Перевести»). Допустимо мокать
  `_dispatch_media`/`process_audio` и проверять аргументы (в т. ч. `force_local`).
- **Тап → изображение:** аналогично для `process_photo`/`process_image_document`.
- **Тап при недоступном reply:** `reply_to_message` = `None` и `reply_to_message` = `InaccessibleMessage`
  (или объект без поддерживаемого медиа) → служебный ответ через `callback.answer(show_alert=True)`,
  пайплайн **не** вызван, исключения нет.
- **`callback.answer` всегда:** во всех ветках callback-хендлера `callback.answer` вызван.
- **force_local в группе:** при `GROUP_ASR_LOCAL=true` тап по аудио вызывает обработку с
  `force_local=True`; при `false` — `force_local=False`.
- **Регресс прежнего триггера:** `tests/test_v120_group.py` проходит без изменений (триггер +
  `_dispatch_media`).
- **Регресс лички:** существующие тесты image/audio/text лички проходят; приватные хендлеры не
  срабатывают в группе.

## Риски и митигации

- **Риск:** Group Privacy OFF → бот видит весь трафик группы (приватность участников). → **Митигация:**
  бот реагирует только на медиа (оффер) и на собственный callback; содержимое не сохраняется (ADR-006,
  НФТ-8); `callback_data` без содержимого медиа; компромисс осознанный и задокументирован (ADR-014).
- **Риск:** Оффер на каждое медиа в активной группе → умеренный шум. → **Митигация:** реакция только
  на поддерживаемое медиа, один оффер на одно медиа-сообщение; настройки на уровне группы и rate-limit
  — отдельным релизом при спросе.
- **Риск:** Нагрузка по тапам (несколько групп, всплеск медиа). → **Митигация:** распознавание только
  по тапу, под глобальным семафором (ADR-010, лимит 1); группы → локальный ASR (`GROUP_ASR_LOCAL`),
  CF-бюджет лички не расходуется; оффер — лёгкая операция, семафор не держит.
- **Риск:** Бот не настроен на privacy OFF на проде → офферы не появляются («режим не работает»). →
  **Митигация:** обязательный пункт чек-листа `/deploy` и раздел README (Задача 6); при privacy ON бот
  не падает, прежний триггер v1.2.0 продолжает работать.
- **Риск:** Дублирующее распознавание (триггер + тап на одно медиа). → **Митигация:** триггер и тап —
  разные осознанные действия пользователя; оффер на медиа один; дубля логики нет (общий
  `_dispatch_media`).
- **Риск:** `callback.message.reply_to_message` недоступно при тапе по старому офферу. → **Митигация:**
  явная проверка `Message` + поддерживаемое медиа, служебное сообщение, без падения (Задача 2).

## Границы (вне scope)

- Авто-распознавание медиа без тапа.
- Настройки на уровне группы (вкл/выкл оффера, белые/чёрные списки чатов).
- Rate-limit офферов сверх «один оффер на одно медиа», персистентный дедуп по `message_id`.
- Учёт callback «grec» в статистике.
- Новая env / новые секреты / платные пути.
- Изменение поведения лички и сути пайплайна (ADR-004/005/007/008/009).
- Изменение профиля поставки (ADR-001).
