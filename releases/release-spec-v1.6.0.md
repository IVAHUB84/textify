# Спецификация релиза v1.6.0

- **Дата планирования:** 2026-06-24
- **Тип релиза:** minor
- **Цели:** Читаемый HTML-вывод вместо сырого Markdown; честное «Кратко»; расширенное меню действий
  (возврат «Перевести», «Задачи», тайм-коды, субтитры, searchable PDF); персистентность кнопок после
  редеплоя/TTL; понятная реакция на файлы > 20 МБ; убрать кнопку Menu команд.

> **Релиз оформлен задним числом.** Вся функциональность v1.6.0 **уже реализована и закоммичена**
> прямыми коммитами после тега `v1.5.3` (`d322a39`, `9bf3a2d`, `b39b564`, `6feb45e`, `0a13011`,
> `fc42bc8`, `fbbc09d`); тесты добавлены. Спецификация фиксирует **то, что сделано**; раздел «Объём»
> описан как «уже сделано — осталась верификация и релиз». Критерии готовности проверяемы по коду.

## Профиль поставки

Проект — **поставляемый сервис**: Telegram-бот на long-polling, **без HTTP-порта/Traefik/edge**
(профиль ADR-001), образ из GHCR, прод ivahub (1 vCPU / 2 GB). Контракт `deploy-guide.md` (Dockerfile /
`deploy/docker-compose.yml` / `deploy/.env.example` / `.github/workflows/deploy.yml`) применим, но
**в части артефактов изменений нет**: новых env, портов, зависимостей и внешних направлений трафика
релиз не вводит (PDF — через уже установленный Tesseract `rus+eng`; персистентность — локальный
SQLite-файл `results.db` рядом со стором учёта). Проверка артефактов сводится к подтверждению «правок не
требуется».

## Объём релиза

Все задачи **реализованы**; ниже — что именно и критерии, по которым архитектор/тестировщик
верифицируют соответствие коду. Не добавлять ничего сверх перечисленного.

### Задача 1: HTML-форматирование вывода (уже сделано)
- **Что:** конвертировать Markdown LLM в безопасный Telegram-HTML и слать с `parse_mode=HTML`; при
  ошибке разметки — деградировать в плоский текст.
- **Где:** новый `services/telegram_format.py` (`to_telegram_html`); `services/reply.py`
  (`send_result`).
- **Контракт:** `to_telegram_html(text)` экранирует `& < >`, переводит `#`-заголовки и
  `**bold**`/`__bold__` в `<b>`, маркеры списка `-/*/+` в `•`; `send_result` в одно-сообщенческом
  режиме шлёт `to_telegram_html(text)` с `ParseMode.HTML`, при `TelegramBadRequest` — повтор тем же
  `text` без `parse_mode`. Доставка длинного текста (части/файл) — без изменений.
- **Критерии готовности:** заголовки → `<b>`; списки → `•`; `< > &` экранированы; битая разметка не
  роняет обработчик (плоский текст доходит).
- **Связь с ADR:** ADR-016, Решение 2; ОВ-2 BA.

### Задача 2: Telegram-формат в промпте структурирования (уже сделано)
- **Что:** промпт `structure_text` формирует жирные заголовки `**…**`, списки `- `, экономную эмодзи,
  без `#`-Markdown.
- **Где:** `services/structure.py` (`_SYSTEM_PROMPT`, few-shot).
- **Критерии готовности:** системный промпт предписывает `**bold**` и списки `- ` без `#`/таблиц/кода;
  few-shot демонстрирует целевой формат.
- **Связь с ADR:** ADR-016, Решение 2; ADR-005.

### Задача 3: Честное «Кратко» (уже сделано)
- **Что:** `summarize` даёт выжимку 2–5 тезисов, никогда не длиннее источника.
- **Где:** `services/llm.py` (`_SUMMARIZE_SYSTEM`, few-shot сжатия).
- **Контракт:** сигнатура `summarize(text) -> str | None | _BudgetExceededType` неизменна; тот же
  бюджет-гейт.
- **Критерии готовности:** промпт требует «заметно короче исходного», «2–5 пунктов с `- `», few-shot
  показывают сжатие (длинный вход → короткие пункты).
- **Связь с ADR:** ADR-016, Решение 3; ФТ-4/КП-3.

### Задача 4: Новые LLM-операции — перевод и задачи (уже сделано)
- **Что:** вернуть `translate` (RU↔EN, направление определяет LLM); добавить `extract_tasks` (action
  items в чек-лист) через общий хелпер.
- **Где:** `services/llm.py` (`_llm_transform`, `translate`, `extract_tasks`, промпты + few-shot).
- **Контракт:** обе — `… -> str | None | _BudgetExceededType`; тот же провайдерный слой и бюджет-гейт;
  усечение `MAX_INPUT_CHARS`; ловля исключений → `None`. `__all__` =
  `[BUDGET_EXCEEDED, summarize, summarize_gist, translate, extract_tasks]`.
- **Критерии готовности:** `translate`/`extract_tasks` возвращают строку при валидном ответе, `None`
  при сбое/пустом, `BUDGET_EXCEEDED` при исчерпанном бюджете; `_TRANSLATE_SYSTEM`/`_TASKS_SYSTEM`
  присутствуют; few-shot RU↔EN для перевода.
- **Связь с ADR:** ADR-016, Решения 1, 4; ФТ-8/9, КП-6/7. **Реверс ADR-015 Решение 6.**

### Задача 5: Тайм-коды и субтитры из сегментов whisper (уже сделано)
- **Что:** форматтеры построчных `[MM:SS]` и `.srt` из сегментов; кнопки для аудио ≥ 60 с; данные —
  из кэша по клику.
- **Где:** `services/transcribe.py` (`format_timestamps`, `format_srt`, `segments_duration`,
  `TIMESTAMP_MIN_SECONDS=60.0`); `handlers/actions.py` (callbacks `act:ts`/`act:srt`, `with_extras`);
  `handlers/audio.py` (кладёт сегменты в кэш при длине ≥ порога).
- **Контракт:** `format_timestamps(segments)` → `[MM:SS] текст` (часы для длинных); `format_srt` →
  нумерованные блоки `HH:MM:SS,mmm`, пропуск пустых, подстраховка нулевой длительности. Тайм-коды —
  текстом через `send_result`; субтитры — документом `subtitles.srt`. При отсутствии сегментов в кэше
  — `callback.answer("Тайм-коды/Субтитры недоступны", show_alert=True)`.
- **Критерии готовности:** для аудио ≥ 60 с в клавиатуре есть «⏱ Тайм-коды»/«Субтитры»; по клику
  приходят построчные метки / валидный `.srt`; пустой кэш сегментов → алерт, не падение; без LLM-вызова.
- **Связь с ADR:** ADR-016, Решение 5; ADR-008; ФТ-5/6/7, КП-4/5.

### Задача 6: Searchable PDF из фото (уже сделано)
- **Что:** PDF = исходное изображение + невидимый текстовый слой Tesseract; кнопка для изображений.
- **Где:** `services/ocr.py` (`recognize_pdf`); `handlers/actions.py` (callback `act:pdf`, `with_pdf`);
  `handlers/image.py` (кладёт байты изображения в кэш).
- **Контракт:** `recognize_pdf(image_bytes) -> bytes | None` через `image_to_pdf_or_hocr` (`rus+eng`)
  под `HEAVY_LOCAL_SEMAPHORE`/`to_thread`; байты берутся из `result_cache.get_image`; при отсутствии —
  `callback.answer("Изображение недоступно (устарело)", show_alert=True)`; при `None` от Tesseract —
  «Не удалось сформировать PDF». Документ — `document.pdf`.
- **Критерии готовности:** для изображений в клавиатуре есть «📄 PDF»; по клику приходит searchable PDF;
  пустой `_img_cache` → алерт; без новых зависимостей.
- **Связь с ADR:** ADR-016, Решение 6; ФТ-10, КП-8.

### Задача 7: Персистентный кэш результатов (уже сделано)
- **Что:** SQLite write-through + load-on-start для текста/сегментов; без `init` — чистый in-memory;
  байты изображения — только память.
- **Где:** `services/result_cache.py`; `bot.py` (`init_result_cache`).
- **Контракт:**
  - Составной ключ `(chat_id, message_id)`. Три стора: `_cache` (текст), `_seg_cache` (сегменты),
    `_img_cache` (байты). LRU + TTL `CACHE_TTL_SECONDS=3600`; размеры `CACHE_MAX_SIZE=500`,
    `IMG_CACHE_MAX_SIZE=50`.
  - `put`/`get`, `put_segments`/`get_segments`, `put_image`/`get_image`.
  - `init_result_cache(db_path)`: создаёт `results(chat_id, message_id, kind, payload, created_at,
    PK(chat_id, message_id, kind))`, `journal_mode=WAL`, `busy_timeout`; на старте чистит записи старше
    TTL и загружает свежие (kind `text`/`seg`) в память. Без вызова (`_DB_PATH is None`) — чисто
    in-memory.
  - `put`/`put_segments` делают сквозную запись (`_persist`, INSERT … ON CONFLICT DO UPDATE);
    `put_image` пишет **только в память** (байты не персистятся). Любой сбой SQLite ловится и логируется,
    не роняя бота.
  - `bot.py`: `init_result_cache(<dir(STATS_DB_PATH)>/results.db)`.
- **Критерии готовности:** put/get round-trip; неизвестный ключ → `None`; LRU-вытеснение; TTL-истечение
  → `None`; разные чаты с одинаковым `message_id` изолированы; персистентность переживает рестарт
  (`init` + новый процесс видит запись); без `init` запись на диск не идёт; сегменты независимы от
  текста.
- **Связь с ADR:** ADR-016, Решение 7; ADR-006; ФТ-12, КП-10/11, ОВ-4.

### Задача 8: Расширенная клавиатура и callback-хендлеры (уже сделано)
- **Что:** единая фабрика клавиатуры с флагами; хендлеры всех действий; чтение данных из кэша;
  деградации.
- **Где:** `handlers/actions.py`.
- **Контракт:**
  - `actions_keyboard(progressive=False, with_extras=False, with_pdf=False)`:
    - `progressive=True` (личка): ряды «Полностью»(`act:full`)/«Кратко»(`act:sum`),
      «Перевести»(`act:tr`)/«Задачи»(`act:task`);
    - `progressive=False` (группы/немедленная выдача): «Кратко»/«Перевести», «Задачи»;
    - `with_extras` → ряд «⏱ Тайм-коды»(`act:ts`)/«Субтитры»(`act:srt`); `with_pdf` → ряд
      «📄 PDF»(`act:pdf`). Все `callback_data` ≤ 64 байт.
  - `handle_full` (`act:full`): `answer("Готовлю…")`; текст из кэша; `None` →
    `answer("Текст недоступен", show_alert=True)`; иначе `structure_text` под `ChatActionSender` +
    `send_result` без markup.
  - `_run_llm_action` (для `act:sum`/`act:tr`/`act:task`): текст из кэша; `None` → «Текст недоступен»;
    `BUDGET_EXCEEDED` → «Дневной бесплатный лимит исчерпан, попробуйте завтра.»; `None`-результат →
    «Не удалось выполнить действие. Попробуйте позже.»; успех → `send_result` без markup.
  - `handle_timestamps`/`handle_srt`: сегменты из кэша; пустой → алерт.
  - `handle_pdf`: байты из кэша; пустой → алерт; `recognize_pdf`; `None` → служебное сообщение.
  - Источник данных — кэш по `(chat.id, message_id)`, не `callback.message.text`.
- **Критерии готовности:** `actions_keyboard(True)` → 4 кнопки (`act:full`,`act:sum`,`act:tr`,
  `act:task`); `actions_keyboard(False)` → 3 (`act:sum`,`act:tr`,`act:task`); `with_extras`/`with_pdf`
  добавляют свои ряды; `callback_data` ≤ 64 байт; каждый хендлер при пустом кэше даёт алерт/служебное
  сообщение без падения; LLM-действия деградируют по `BUDGET_EXCEEDED`/`None`.
- **Связь с ADR:** ADR-016, Решения 1, 5, 6; влияние на контракт 1.0.0.

### Задача 9: Прогрессивная выдача и наполнение кэша в хендлерах (уже сделано)
- **Что:** личка — суть + кнопки + кэш; группы — полный текст + кнопки + кэш; контекстные кнопки.
- **Где:** `handlers/audio.py`, `handlers/image.py`, `handlers/group.py`.
- **Контракт:**
  - `process_audio(..., progressive=False)`: при распознанном — `with_extras` если
    `segments_duration ≥ TIMESTAMP_MIN_SECONDS`; `progressive=True` (личка) → превью-суть
    (`summarize_gist`; деградация → служебное превью) + клавиатура + `result_cache.put` текста и (при
    наличии) сегментов по `message_id`; `progressive=False` (группы) → `structure_text` через
    `send_result` + кэш текста/сегментов по `message_id` результата. Личные хендлеры зовут
    `progressive=True`.
  - `process_photo`/`process_image_document(..., progressive=False)`: аналогично; всегда `with_pdf`;
    кладут байты изображения в `_img_cache`. Личные хендлеры — `progressive=True`.
  - `_dispatch_media` (группы) зовёт `process_*` с дефолтом `progressive=False`.
- **Критерии готовности:** личка — первым приходит суть/служебное превью с кнопками, текст в кэше;
  группа — полный текст с кнопками, текст в кэше; аудио ≥ 60 с → сегменты в кэше + кнопки экстра;
  изображения → байты в кэше + «PDF»; нераспознанное → прежнее служебное сообщение без превью/кнопок/
  кэша.
- **Связь с ADR:** ADR-016, Решения 5, 6, 7; ADR-015 (прогрессивное раскрытие сохраняется).

## Изменения публичного контракта

Контракт 1.0.0 (ADR-010), разделы «Inline-действия» и «Формат немедленной выдачи»:

- **Немедленная выдача** одно-сообщенческого результата — **Telegram-HTML** (жирные заголовки, `•`,
  экранирование `< > &`); при битой разметке — деградация в плоский текст. Длинный текст (части/файл) —
  без изменений.
- **Inline-действия (личка):** «Полностью»(`act:full`), «Кратко»(`act:sum`), «Перевести»(`act:tr`),
  «Задачи»(`act:task`); аудио ≥ 60 с — «⏱ Тайм-коды»(`act:ts`)/«Субтитры»(`act:srt`); изображения —
  «📄 PDF»(`act:pdf`).
- **Inline-действия (группы):** «Кратко»/«Перевести»/«Задачи» + те же контекстные по правилам.
- **«Перевести» возвращена** (реверс ADR-015).
- **Источник текста/сегментов/байт для действий:** кэш по `(chat_id, message_id)` (in-memory + SQLite
  для текста/сегментов); недоступность → служебное «… недоступно/устарело».
- **Кнопка Menu** команд не отображается.

Входные каналы, команды (`/start`/`/help`/`/stats`), логика доставки длинного текста `send_result`,
профиль поставки — **не меняются**. Миграция БД не требуется: `results.db` создаётся `init_result_cache`
на старте; без него — in-memory. Релиз — **MINOR**.

## Риски и митигации

- **Риск:** реверс ADR-015 (перевод) создаёт молчаливое противоречие в истории ADR. →
  **Митигация:** ADR-015 (статус + Решение 6) явно помечен как заменённый ADR-016 (выполнено).
- **Риск:** «Перевести»/«Задачи» доступны и в группах шире, чем подразумевал BA (ОВ-1). →
  **Митигация:** зафиксировано в ADR-016 как осознанный выбор + технический долг (подтверждение
  владельца); поведение групп v1.4.0/v1.5.0 не сломано (КП-15).
- **Риск:** HTML-разметка от слабой модели ломает сообщение. → **Митигация:** безопасное подмножество
  (`<b>`), самостоятельное экранирование, деградация в плоский текст при `TelegramBadRequest` (КП-2).
- **Риск:** write-through SQLite на каждую запись — синхронные операции в event loop. →
  **Митигация:** WAL + busy_timeout; короткие операции; сбой БД ловится и не роняет бота; объёмы лички/
  групп малы (ОГР-1).
- **Риск:** байты изображения раздувают RSS на 1 vCPU / 2 GB. → **Митигация:** `_img_cache` только
  память, лимит 50, без персистентности (НФТ-1, ОВ-5).
- **Риск:** исчерпание CF-бюджета ломает действия. → **Митигация:** все LLM-операции под бюджет-гейтом,
  `BUDGET_EXCEEDED` → служебное сообщение; тайм-коды/субтитры/PDF — локальные, без бюджета; ASR
  деградирует на локальный whisper (ADR-008/011).
