# Спецификация релиза v1.7.0

- **Дата планирования:** 2026-06-24
- **Тип релиза:** minor
- **Цели:** Ввести дневной лимит распознаваний на пользователя (базовый/подписчик, через env) для
  защиты CPU VPS ivahub от хеви-юзеров и мягкий subscription-gate на Telegram-канал ради роста
  канала под рекламную монетизацию. При незаданном `REQUIRED_CHANNEL` gate выключен — dev/CI и
  текущий прод не ломаются.

## Источники
- BA-требования: `ba-req/ba-requirements-017.md`
- ADR: `adrs/adr-017.md` (расширяет ADR-006 новой таблицей в `stats.db`; сохраняет профиль
  поставки ADR-001; соблюдает ADR-002/003; действия ADR-016 лимит не расходуют).

## Профиль поставки

Проект — **поставляемый сервис**: Telegram-бот на long-polling, **без HTTP-порта/Traefik/edge**
(профиль ADR-001), образ из GHCR, прод ivahub (1 vCPU / 2 GB). Контракт `deploy-guide.md` применим,
но **в части артефактов изменений почти нет**: новых портов, volume, внешних зависимостей и
направлений трафика релиз не вводит (таблица создаётся в уже смонтированном `stats.db`).
**Единственное изменение артефактов** — три новые env-переменные в `.env.example` и
`deploy/.env.example`. `Dockerfile`, `deploy/docker-compose.yml`, `.github/workflows/deploy.yml` —
без правок. **Деплой-предусловие:** бот должен быть **администратором канала** `REQUIRED_CHANNEL`
(иначе деградация по Задаче 3 / КП-11).

## Объём релиза

Не добавлять ничего сверх перечисленного. Все сообщения пользователю — на **русском**, дружелюбные,
без давления (ФТ-13). Идентификаторы/имена файлов кода — на английском.

### Задача 1: Слой учёта дневного лимита (`services/limits.py`)
- **Что:** новый сервис учёта дневного лимита поверх существующего `stats.db` (ADR-017 Решение 2A,
  схема). Паттерн ADR-006: соединение-на-операцию, WAL, `busy_timeout`, всё через `asyncio.to_thread`.
- **Где:** новый `services/limits.py`; инициализация из `bot.py`.
- **Контракт:**
  - Таблица `daily_usage(user_id INTEGER, day TEXT, count INTEGER, PRIMARY KEY(user_id, day))`.
  - `init_limits_db()` — создаёт таблицу (`CREATE TABLE IF NOT EXISTS`), `PRAGMA journal_mode=WAL`,
    `busy_timeout`. Путь — `config["STATS_DB_PATH"]` (тот же файл, что `user_stats`/`referrals`).
  - `_today_utc() -> str` — `datetime.now(timezone.utc).strftime("%Y-%m-%d")`.
  - `async record_recognition(user_id: int) -> None` — атомарный UPSERT за сегодня:
    `INSERT INTO daily_usage(user_id, day, count) VALUES(?, ?, 1) ON CONFLICT(user_id, day) DO UPDATE SET count = count + 1`.
  - `async usage_today(user_id: int) -> int` — `count` за сегодня; нет строки → `0`.
- **Критерии готовности:** первое распознавание создаёт строку с `count=1`; повторные за те же сутки
  инкрементируют; параллельные инкременты одного пользователя не теряются (атомарный UPSERT —
  КП-15); `usage_today` для нового пользователя/новых суток = 0; запись/чтение не блокируют event
  loop (через `to_thread`); схема не пересекается с `user_stats`/`referrals` (ОВ-8).
- **Связь с ADR:** ADR-017 Решение 2; ADR-006. *(ФТ-1, ФТ-4, ФТ-5, НФТ-1, НФТ-4; КП-1, КП-4, КП-5, КП-15)*

### Задача 2: Обёртка проверки подписки и кэш статуса (`services/subscription.py`)
- **Что:** обёртка над `bot.get_chat_member` с in-memory TTL-кэшем подтверждённой подписки и
  поддержкой выключенного режима (ADR-017 Решения 3A, 4).
- **Где:** новый `services/subscription.py`; чтение `REQUIRED_CHANNEL` — из `config`.
- **Контракт:**
  - `SUBSCRIPTION_TTL_SECONDS = 600`; in-memory `dict[int, float]` (user_id → expiry по
    `time.monotonic()`).
  - `is_gate_enabled() -> bool` — True, если `REQUIRED_CHANNEL` непустой.
  - `is_subscriber_cached(user_id) -> bool` — живая запись в кэше (не истёк TTL); иначе False.
    Истёкшие записи лениво удаляются.
  - `async check_subscription(bot, user_id) -> bool` — если gate выключен → False; иначе
    `bot.get_chat_member(REQUIRED_CHANNEL, user_id)` в `try/except`; статусы
    `member`/`administrator`/`creator` → True + запись в кэш на TTL; иначе False; любое исключение →
    `logger.warning(..., exc_info=True)` и False (деградация, бот не падает).
  - `channel_url() -> str | None` — ссылка на канал для кнопки «Открыть канал» (`@username` →
    `https://t.me/<username>`; числовой `chat_id` без публичного username — кнопку url не строить).
- **Критерии готовности:** при незаданном `REQUIRED_CHANNEL` `is_gate_enabled()` = False и
  `check_subscription` не вызывает API (КП-9); подтверждённая подписка кэшируется, повторная проверка
  в окне TTL не дёргает `getChatMember` (КП-10); по истечении TTL запись исчезает (отписавшийся снова
  неподтверждён — Сценарий F); сбой `getChatMember` → False + лог, без падения (КП-11).
- **Связь с ADR:** ADR-017 Решения 3, 4, 8. *(ФТ-7, ФТ-9, ФТ-10, ФТ-11, ФТ-12; КП-9, КП-10, КП-11)*

### Задача 3: Gate-логика и callback «Проверить» (`handlers/gate.py`)
- **Что:** проверка лимита перед распознаванием, формирование мягкого gate-сообщения с inline-кнопками
  и обработка кнопки «Проверить» (ADR-017 Решения 1A, 5A, 6).
- **Где:** новый `handlers/gate.py`; регистрация `gate_router` в `bot.py`.
- **Контракт:**
  - `async enforce_limit(message, user_id, is_private) -> bool` — возвращает True (разрешить
    обработку) или False (лимит исчерпан, сообщение уже отправлено вызывающей стороной не нужно):
    1. `subscriber = subscription.is_subscriber_cached(user_id)`;
    2. `limit = config["DAILY_LIMIT_SUBSCRIBED"] if subscriber else config["DAILY_LIMIT_FREE"]`;
    3. `used = await limits.usage_today(user_id)`;
    4. если `used >= limit`: при `is_private and subscription.is_gate_enabled() and not subscriber` →
       отправить **gate-сообщение** с клавиатурой («Открыть канал» url + «Проверить» callback);
       иначе (группа или подписчик или gate выключен) → отправить **нейтральное** «Дневной лимит
       распознаваний исчерпан, попробуйте завтра.»; вернуть **False**;
    5. иначе `await limits.record_recognition(user_id)` и вернуть **True**. Сбой записи учёта
       логируется и не препятствует обработке (НФТ-2).
  - Gate-клавиатура: ряд «Открыть канал» (url из `channel_url()`, если есть) + «Я подписался /
    Проверить» (`callback_data` короткий код, ≤ 64 байт, напр. `gate:chk`).
  - Callback `gate:chk`: `ok = await subscription.check_subscription(bot, user_id)`; при True →
    `callback.answer` + сообщение «Подписка подтверждена, лимит повышен — пришлите медиа ещё раз»
    (медиа повторно **не** обрабатывается, ADR-017 Решение 5A); при False → вежливое «Пока не вижу
    подписку, попробуйте после подписки».
  - Все тексты — русские, дружелюбные, без давления (ФТ-13).
- **Критерии готовности:** в пределах лимита `enforce_limit` инкрементирует и возвращает True;
  при исчерпании базового лимита неподтверждённым пользователем **в личке** приходит gate-сообщение с
  обеими кнопками, до исчерпания — не приходит (КП-6); «Проверить» с подпиской → подписчик + повышение
  (КП-7); «Проверить» без подписки → вежливый отказ, лимит не повышается (КП-8); в **группе** при
  исчерпании — нейтральное сообщение **без** кнопок подписки (КП-14); при выключенном gate
  gate-сообщение не строится (КП-9).
- **Связь с ADR:** ADR-017 Решения 1, 5, 6. *(ФТ-1, ФТ-6, ФТ-7, ФТ-8, ФТ-13, ФТ-15; КП-6, КП-7, КП-8, КП-12, КП-14)*

### Задача 4: Встраивание точки учёта в обработку медиа
- **Что:** вызвать `enforce_limit` в начале общих функций обработки медиа — до скачивания и
  распознавания (ADR-017 Решение 1A). Действия-кнопки (`handlers/actions.py`) не трогаются — они
  лимит не расходуют (ФТ-2).
- **Где:** `handlers/audio.py` (`process_audio`), `handlers/image.py` (`process_photo`,
  `process_image_document`).
- **Контракт:**
  - В начале каждой `process_*` определить инициатора (`media_message.from_user.id`) и приватность
    (`media_message.chat.type == "private"` — личка vs группа); вызвать
    `if not await enforce_limit(reply_target, user_id, is_private): return` **до** `bot.download` /
    распознавания. Существующий гард 20 МБ и остальная логика — без изменений.
  - В группах инициатор — `from_user` исходного медиа-сообщения (для callback «Распознать» — автор
    реплая); лимит учитывается по инициатору, gate-кнопки не показываются (через `is_private=False`).
- **Критерии готовности:** распознавание медиа (voice/audio/photo/image-документ) уменьшает лимит на
  единицу по `user_id` (КП-1); при исчерпании медиа **не скачивается и не распознаётся** (return до
  download); нажатие кнопок-действий над результатом лимит **не** меняет (КП-2); поведение групп
  v1.4.0/v1.5.0 не ломается (КП-14).
- **Связь с ADR:** ADR-017 Решения 1, 6. *(ФТ-1, ФТ-2, ФТ-15; КП-1, КП-2, КП-14)*

### Задача 5: Конфигурация новых env-переменных (`config.py`)
- **Что:** прочитать три новые переменные тем же паттерном, что `ADMIN_USER_ID`/`CF_DAILY_BUDGET`
  (strip, валидация, дефолт, `logger.warning` при некорректном).
- **Где:** `config.py` (`load_config`, словарь `config`).
- **Контракт:**
  - `REQUIRED_CHANNEL` — строка из `os.environ.get("REQUIRED_CHANNEL", "").strip()`; пусто → `""`
    (gate выключен).
  - `DAILY_LIMIT_FREE` — int, дефолт **3**; при нечисловом/≤ 0 → дефолт + `logger.warning`.
  - `DAILY_LIMIT_SUBSCRIBED` — int, дефолт **30**; та же валидация.
  - Ключи добавлены в возвращаемый `config`.
- **Критерии готовности:** значения берутся из окружения, не захардкожены; при отсутствии/некорректном
  значении применяется дефолт (КП-3); пустой `REQUIRED_CHANNEL` корректно трактуется как выключенный
  gate (КП-9).
- **Связь с ADR:** ADR-017 Решение 8. *(ФТ-3, ФТ-9, ФТ-10; КП-3, КП-9)*

### Задача 6: Показ остатка лимита (`handlers/commands.py`)
- **Что:** показать пользователю остаток дневного лимита в `/start`; добавить агрегат в `/stats`
  (ADR-017 Решение 7).
- **Где:** `handlers/commands.py` (`cmd_start`, `cmd_stats`).
- **Контракт:**
  - `/start` (только `private`, у пользователя есть `from_user`): добавить строку вида «Сегодня
    доступно N из M распознаваний», где `M = DAILY_LIMIT_SUBSCRIBED` если
    `is_subscriber_cached(user_id)` иначе `DAILY_LIMIT_FREE`, `N = max(0, M - usage_today(user_id))`.
    Сбой чтения учёта не должен ронять `/start` (try/except → строку не показывать).
  - `/stats` (админ): добавить в существующий вывод агрегат — число подтверждённых подписчиков в кэше
    и/или суммарные распознавания за сегодня (по месту, не ломая текущий формат).
- **Критерии готовности:** в `/start` в личке виден остаток лимита текущего уровня пользователя
  (КП-13); `/stats` дополнен агрегатом без слома существующих полей; сбой учёта не роняет команды.
- **Связь с ADR:** ADR-017 Решение 7. *(ФТ-14; КП-13)*

## Изменения публичного контракта

- **Новое поведение при заданном `REQUIRED_CHANNEL`:** распознавание медиа расходует дневной лимит
  (базовый `DAILY_LIMIT_FREE` / подписчика `DAILY_LIMIT_SUBSCRIBED`); при исчерпании базового лимита
  в личке — мягкое предложение подписаться (кнопки «Открыть канал» / «Проверить»); в группе и для
  подписчика — нейтральное сообщение о лимите. Кнопки-действия над результатом (ADR-016) лимит не
  расходуют.
- **При незаданном `REQUIRED_CHANNEL`:** проверок подписки и предложений подписаться нет; дневной
  лимит всё равно действует (базовый/подписчик; подписчиков без канала нет → базовый), но без gate.
  Поведение dev/CI/текущего прода по части распознавания сохраняется до создания канала и установки
  env.
- **Команды:** `/start` дополнен строкой остатка лимита; `/stats` — агрегатом. Входные каналаы,
  логика доставки результата, профиль поставки — **не меняются**.
- **Миграция БД:** не требуется — таблица `daily_usage` создаётся `init_limits_db()` на старте в
  существующем `stats.db`. Новый `getChatMember` требует **бот — администратор канала** (предусловие).
- Релиз — **MINOR**.

## Риски и митигации

- **Риск:** бот не админ канала / `getChatMember` падает на проде. → **Митигация:** деградация к
  базовому лимиту без gate, лог (Задача 2/3, КП-11); предусловие «админ канала» в чек-листе деплоя
  (Задача 9, ОВ-7).
- **Риск:** точка учёта в `process_*` случайно сработает на кнопки-действия. → **Митигация:**
  `handlers/actions.py` эти функции не вызывает; тест на «действие не инкрементирует» (КП-2).
- **Риск:** гонки при параллельных сообщениях одного пользователя. → **Митигация:** атомарный UPSERT
  под WAL + busy_timeout (Задача 1, КП-15).
- **Риск:** накопление строк прошлых суток в `daily_usage`. → **Митигация:** объёмы малы (ОГР-1);
  при необходимости ленивая чистка старых дней (опционально, тех. долг ADR-017).
- **Риск:** кэш подписки теряется при рестарте → пользователь временно на базовом лимите. →
  **Митигация:** повторный `getChatMember` по «Проверить» дёшев и редок; эфемерность осознана
  (ADR-017 Решение 3).
- **Риск:** включение gate ломает первое впечатление новых пользователей. → **Митигация:** gate
  мягкий и невидим до исчерпания базового лимита (НФТ-8, КП-6); базовый лимит выше типичного
  использования (ДОП-2).
