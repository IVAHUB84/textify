# Спецификация релиза v1.8.0

- **Дата планирования:** 2026-06-24
- **Тип релиза:** minor
- **Цели:** Ввести **реферальную награду** — повышение собственного дневного лимита распознаваний
  пользователя за приглашённых друзей (`эффективный = база + min(друзья·PER, CAP)`), как второй
  виральный рычаг роста поверх subscription-gate v1.7.0. Рычаги подписки и рефералов независимы и
  складываются. При нуле рефералов поведение идентично v1.7.0 (без регрессии).

## Источники
- BA-требования: `ba-req/ba-requirements-018.md`
- ADR: `adrs/adr-018.md` (надстройка над `enforce_limit` ADR-017; переиспользует реферальную
  механику v1.3.0 `services/referrals.py` и кэш-паттерн подписки v1.7.0 `services/subscription.py`;
  сохраняет профиль поставки ADR-001; доступ к `stats.db` — паттерн ADR-006).

## Профиль поставки

Проект — **поставляемый сервис**: Telegram-бот на long-polling, **без HTTP-порта/Traefik/edge**
(профиль ADR-001), образ из GHCR, прод ivahub (1 vCPU / 2 GB). Контракт `deploy-guide.md` применим,
но **в части артефактов изменений почти нет**: новых портов, volume, внешних зависимостей и
направлений трафика релиз не вводит (таблица `referrals` уже существует в `stats.db` с v1.3.0).
**Единственное изменение артефактов** — две новые env-переменные (`REFERRAL_BONUS_PER`,
`REFERRAL_BONUS_CAP`) в `.env.example` и `deploy/.env.example`. `Dockerfile`,
`deploy/docker-compose.yml`, `.github/workflows/deploy.yml` — **без правок**. Деплой-предусловий
сверх v1.7.0 нет.

## Объём релиза

Не добавлять ничего сверх перечисленного. Все сообщения пользователю — на **русском**, дружелюбные,
без давления. Идентификаторы/имена файлов кода — на английском. Уведомление пригласившему о новом
реферале (ФТ-12) в этот релиз **не входит** (ADR-018 Решение 7A).

### Задача 1: Кэш числа рефералов на горячем пути (`services/referrals.py`)
- **Что:** in-memory TTL-кэш числа рефералов с точечной инвалидацией, чтобы `enforce_limit` не
  выполнял `COUNT` на каждое распознавание (ADR-018 Решение 2A). Существующая механика рефералов
  v1.3.0 не меняется.
- **Где:** `services/referrals.py` (дополнить, не переписывать).
- **Контракт:**
  - `REFERRAL_COUNT_TTL_SECONDS = 600`; in-memory `dict[int, tuple[int, float]]`
    (user_id → (count, expiry по `time.monotonic()`)).
  - `async cached_referral_count(user_id: int) -> int` — живая запись в кэше (TTL не истёк) →
    вернуть закэшированный `count`, `COUNT` **не выполнять**; иначе `await count_referrals(user_id)`,
    положить в кэш на TTL, вернуть; любое исключение при чтении → `logger.warning(..., exc_info=True)`
    и вернуть `0` (деградация к лимиту без бонуса, НФТ-2).
  - В `record_referral` после успешной записи — **инвалидировать** запись пригласившего
    (`referrer_id`) из кэша (удалить ключ), чтобы рост числа рефералов отражался без ожидания TTL.
  - `count_referrals`, `total_referrals`, `top_referrers`, `record_referral` (его сигнатура/механика
    записи) — сохраняются; добавляется только кэш-обёртка и вызов инвалидации.
  - Для модуля нужен `logging` (если ещё не импортирован) — добавить `logger`.
- **Критерии готовности:** первое чтение для пользователя выполняет `COUNT` и кэширует; повторное в
  окне TTL `COUNT` **не выполняет** (КП-13); фиксация нового реферала через `record_referral`
  инвалидирует запись пригласившего → следующее `cached_referral_count` пересчитывает; по истечении
  TTL запись пересчитывается; сбой `COUNT` → возвращается `0` + лог, без падения (КП-12).
- **Связь с ADR:** ADR-018 Решения 2, 10. *(ФТ-8, НФТ-1, НФТ-2; КП-8, КП-12, КП-13)*

### Задача 2: Формула эффективного лимита (`services/limits.py`)
- **Что:** чистая функция расчёта эффективного дневного лимита из базы и числа рефералов
  (ADR-018 Решения 1A, 3). Без SQLite и без обращения к `referrals` внутри.
- **Где:** `services/limits.py` (дополнить; функции учёта `daily_usage` не трогать).
- **Контракт:**
  - `effective_daily_limit(base_limit: int, referral_count: int) -> int` =
    `base_limit + min(referral_count * config["REFERRAL_BONUS_PER"], config["REFERRAL_BONUS_CAP"])`.
  - При `referral_count <= 0` → `base_limit` (бонус 0, совместимость v1.7.0).
  - Функция синхронная, чистая (детерминированный результат от аргументов и config).
- **Критерии готовности:** при `referral_count = 0` → `base_limit` (КП-6); при `r` рефералах и
  `r·PER < CAP` → `base_limit + r·PER`; при `r·PER >= CAP` → `base_limit + CAP` (потолок, КП-3);
  результат не зависит от подписки (база передаётся снаружи) — складывается с любой базой (КП-2).
- **Связь с ADR:** ADR-018 Решения 1, 3, 4. *(ФТ-1, ФТ-2, ФТ-3, ФТ-6; КП-1, КП-2, КП-3, КП-6)*

### Задача 3: Применение бонуса в `enforce_limit` (`handlers/gate.py`)
- **Что:** между выбором базового уровня и сравнением остатка вставить расчёт реферального бонуса;
  допуск/отказ и остаток вести относительно **эффективного** лимита (ADR-018 Решение 4). Расширить
  gate-сообщение вторым путём (ADR-018 Решение 6).
- **Где:** `handlers/gate.py` (`enforce_limit`, `_gate_keyboard`, тексты сообщений).
- **Контракт:**
  - В `enforce_limit` после `subscriber`/`base` (как сейчас `limit`):
    1. `base = config["DAILY_LIMIT_SUBSCRIBED"] if subscriber else config["DAILY_LIMIT_FREE"]`;
    2. `count = await referrals.cached_referral_count(user_id)`;
    3. `limit = limits.effective_daily_limit(base, count)`;
    4. далее существующая логика `used >= limit` без изменений по структуре.
  - **Gate-сообщение (личка, неподписчик, gate включён)** — текст с **двумя путями**: подписка на
    канал И приглашение друзей (с упоминанием `+REFERRAL_BONUS_PER` за друга). Клавиатура:
    ряд 1 — «Открыть канал» (url из `channel_url()`, если есть) + «Пригласить друзей»
    (url на share-ссылку); ряд 2 — «Я подписался / Проверить» (callback `gate:chk`, как в v1.7.0).
    Url-кнопки `callback_data` не используют (ОГР-3 неактуален для url).
  - Share-ссылку для кнопки «Пригласить друзей» строить переиспользуя логику из
    `handlers/commands.py` (`_build_ref_link` + share-url) — вынести в общий хелпер или импортировать;
    **не дублировать** формулу URL.
  - **Нейтральное сообщение** (группа / подписчик / gate выключен) — без изменений: один путь не
    показывается, поведение v1.7.0.
  - callback `gate:chk` (`handle_gate_check`) — **без изменений**.
  - Все тексты — русские, дружелюбные, без давления.
- **Критерии готовности:** при `count > 0` допуск/отказ ведётся относительно `base + бонус`, а не
  бинарного уровня (КП-1, КП-7); неподписчик с рефералами получает бонус так же, как подписчик
  (КП-2); при исчерпании эффективного лимита в личке неподписчику приходит сообщение с **двумя
  путями** и url-кнопкой «Пригласить друзей» (КП-10); в группе/у подписчика/при выключенном gate —
  прежнее нейтральное сообщение без второго пути (совместимость v1.7.0); при нуле рефералов
  эффективный лимит = база, сообщения как в v1.7.0 (КП-6); сбой `cached_referral_count` → бонус 0,
  распознавание не падает (КП-12).
- **Связь с ADR:** ADR-018 Решения 4, 6, 10. *(ФТ-1, ФТ-2, ФТ-7, ФТ-10; КП-1, КП-2, КП-7, КП-10, КП-12)*

### Задача 4: Эффективный лимит в `/start` (`handlers/commands.py`)
- **Что:** строку остатка дневного лимита в `/start` считать от эффективного лимита; при наличии
  бонуса показать расшифровку слагаемых (ADR-018 Решение 5). Блок реф-ссылки и «Приглашено: N» —
  не дублировать, оставить как есть.
- **Где:** `handlers/commands.py` (`cmd_start`).
- **Контракт:**
  - `invited_count` уже читается в `cmd_start` (`count_referrals`) — переиспользовать его как число
    рефералов для расчёта (можно через `cached_referral_count`; допустимо текущее `count_referrals`,
    т.к. `/start` не горячий путь — выбрать согласованно, предпочтительно `cached_referral_count`).
  - `base = DAILY_LIMIT_SUBSCRIBED if is_subscriber_cached(user_id) else DAILY_LIMIT_FREE`;
    `effective = effective_daily_limit(base, invited_count)`;
    `used = usage_today(user_id)`; `remaining = max(0, effective - used)`; `bonus = effective - base`.
  - Строка остатка: «Сегодня доступно `<remaining>` из `<effective>` распознаваний.».
  - **Только при `bonus > 0`** — дополнительная строка-расшифровка вида
    «Лимит: `<base>` базовых + `<bonus>` за `<invited_count>` друзей.». При `bonus == 0` расшифровка
    не выводится (вывод визуально совпадает с v1.7.0).
  - Сбой чтения учёта/рефералов не должен ронять `/start` (существующий `try/except` сохранить;
    при ошибке — строку лимита/расшифровку не показывать).
- **Критерии готовности:** в `/start` в личке остаток считается от эффективного лимита (КП-9); при
  рефералах видна расшифровка база+бонус и число друзей (КП-9); при нуле рефералов расшифровки нет и
  вывод как в v1.7.0 (КП-6); реф-ссылка и «Приглашено: N» сохранены (не задвоены).
- **Связь с ADR:** ADR-018 Решение 5. *(ФТ-9; КП-6, КП-9)*

### Задача 5: Конфигурация новых env-переменных (`config.py`)
- **Что:** прочитать две новые переменные тем же паттерном, что `DAILY_LIMIT_FREE`/
  `DAILY_LIMIT_SUBSCRIBED` (ADR-018 Решение 9).
- **Где:** `config.py` (`load_config`, словарь `config`).
- **Контракт:**
  - `REFERRAL_BONUS_PER` — int, дефолт **3**; `os.environ.get(...).strip()`; при нечисловом/≤ 0 →
    дефолт + `logger.warning`.
  - `REFERRAL_BONUS_CAP` — int, дефолт **30**; та же валидация.
  - Оба ключа добавлены в возвращаемый `config`.
- **Критерии готовности:** значения берутся из окружения, не захардкожены; при отсутствии/некорректном
  значении применяется дефолт 3/30 (КП-5).
- **Связь с ADR:** ADR-018 Решение 9. *(ФТ-5; КП-5)*

## Изменения публичного контракта

- **Новое поведение:** дневной лимит пользователя теперь = `база (v1.7.0) + min(друзья·REFERRAL_BONUS_PER,
  REFERRAL_BONUS_CAP)`. Бонус начисляется независимо от подписки и складывается с уровнем подписчика.
  При исчерпании эффективного лимита в личке неподписчик видит **два пути** (подписка + пригласить
  друзей); `/start` показывает эффективный лимит и расшифровку слагаемых при наличии бонуса.
- **При нуле рефералов** (или при дефолтах) поведение **идентично v1.7.0** — регрессии нет.
- **Команды:** `/start` — строка лимита учитывает бонус (+опц. расшифровка). `/stats` —
  **не меняется** (реферальные метрики там уже есть с v1.3.0). Входные каналы, доставка результата,
  точка списания лимита в `process_*`, профиль поставки — **не меняются**.
- **Миграция БД:** **не требуется** — таблица `referrals` существует с v1.3.0, новой схемы нет.
- **Новые env:** `REFERRAL_BONUS_PER` (3), `REFERRAL_BONUS_CAP` (30).
- Релиз — **MINOR**.

## Риски и митигации

- **Риск:** второй SQLite-`COUNT` на горячем пути нагружает 1 vCPU. → **Митигация:** TTL-кэш 600 с с
  точечной инвалидацией (Задача 1, КП-13); `COUNT` по индексу `idx_referrals_referrer` дёшев; трафик
  лички низкий (ОГР-1).
- **Риск:** расхождение формулы лимита между gate и `/start`. → **Митигация:** единая
  `effective_daily_limit` переиспользуется обоими (Задача 2/3/4, ADR-018 Решение 1A).
- **Риск:** регрессия для пользователей без рефералов. → **Митигация:** `r=0 → база`; тесты на
  совместимость с v1.7.0 (КП-6).
- **Риск:** сбой чтения числа рефералов ломает распознавание. → **Митигация:** `cached_referral_count`
  при ошибке → `0`, бонус 0, распознавание продолжается + лог (Задача 1, КП-12).
- **Риск:** накрутка лимита самоприглашением/повторами. → **Митигация:** инварианты v1.3.0
  (`referred_id` PK, `referrer_id != user_id`, только `is_new_user`) + потолок `CAP` (ADR-018
  Решение 8); серьёзный анти-фрод вне рамок.
- **Риск:** кэш числа рефералов рассинхронизируется при многопроцессности. → **Митигация:** на ivahub
  процесс один; инвалидация в `record_referral` покрывает единственный источник роста.
