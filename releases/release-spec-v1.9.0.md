# Спецификация релиза v1.9.0

- **Дата планирования:** 2026-06-25
- **Тип релиза:** minor
- **Цели:** Видимую страницу в «PDF из фото» рендерить из **очищенного** изображения (выровненная,
  контрастная страница «как из сканера», а не сырое фото на столе), добавить режим «Скан (ч/б)»,
  вшивать страницу как JPEG (меньше итоговый PDF) и **обязательно** деградировать к оригиналу при
  любом сбое. Текстовый слой и качество распознавания не трогаем; результат не хуже v1.6.0 ни на
  одном входе.

## Источники
- BA-требования: `ba-req/ba-requirements-019.md`
- ADR: `adrs/adr-019.md` (надстройка над ADR-016 Решение 6 «Searchable PDF»; неблокирование под
  `HEAVY_LOCAL_SEMAPHORE` — ADR-002/003; профиль поставки ADR-001; PDF не расходует лимит — ADR-017).

## Профиль поставки

Проект — **поставляемый сервис**, но это **Telegram-бот на long-polling без HTTP-порта/Traefik/edge**
(профиль ADR-001), образ из GHCR, прод ivahub (1 vCPU / 2 GB). **Контракт артефактов поставки из
`deploy-guide.md` (один порт на `0.0.0.0`, traefik-лейблы, edge-сеть, отсутствие `ports:`) к этому
проекту НЕ применяется** — это не web-сервис на `app-01`. В этом релизе **нет новых артефактов
поставки, нет новых env-переменных, нет новых внешних зависимостей**: вся обработка — на уже
установленных OpenCV / Pillow / Tesseract. `Dockerfile`, `deploy/docker-compose.yml`,
`deploy/.env.example`, `.env.example`, `.github/workflows/deploy.yml` — **без изменений**.

## Объём релиза

Не добавлять ничего сверх перечисленного (принцип «минимальные изменения»). Меняется **только видимый
слой PDF** и UI его режимов; OCR-текст (`recognize_text`/`_run_ocr`), логика лимита, прочие кнопки —
**не трогать**. Все подписи/сообщения пользователю — на **русском**, в стиле меню действий BA-016.
Имена в коде — на английском.

### Задача 1: Рендер-ветка очистки страницы (`services/preprocess.py`)
- **Что:** функция, возвращающая **очищенную PIL-страницу, готовую к подаче в Tesseract**, для двух
  режимов; общие шаги (EXIF-поворот, ресайз, deskew-угол) переиспользуются; ветка OCR-текста
  (`preprocess_image`) не ломается (ADR-019 Решение 2A).
- **Где:** `services/preprocess.py`.
- **Контракт:**
  - `render_page(image_bytes: bytes, mode: str) -> PIL.Image.Image` — `mode ∈ {"doc", "scan"}`.
    Возвращает PIL-изображение готовой страницы (для `image_to_pdf_or_hocr`).
  - Общие приватные хелперы: `_load_oriented(image_bytes) -> PIL.Image` (open +
    `ImageOps.exif_transpose`); `_resize_to_target(...)` (ресайз к `_PAGE_MAX_SIDE`, апскейл мелких).
  - **Разделить `_deskew`** на `_skew_angle(binary: np.ndarray) -> float` (вычисление угла, с уже
    существующими отсечками `>45/<-45` и потолком `abs(angle) > 15° → 0`) и `_rotate(arr, angle)`
    (применение поворота к произвольному массиву, `BORDER_CONSTANT`, белый фон). **Пропускать**
    поворот при `abs(angle) < _MIN_DESKEW_ANGLE` (ADR-019 Решение 3).
  - Режим **`"doc"` (дефолт):** EXIF → ресайз → grayscale → **CLAHE** (`cv2.createCLAHE`,
    `clipLimit≈2.0`, `tileGridSize=(8,8)`) для снятия теней/контраста → вычислить угол по бинарной
    маске, повернуть **серую** страницу тем же углом → вернуть PIL (`mode "L"` или RGB).
  - Режим **`"scan"`:** EXIF → ресайз → `cv2.adaptiveThreshold` (как `preprocess_image`) →
    deskew бинарной страницы → вернуть PIL (бинарная).
  - Константы (в коде, **без env**): `_MIN_DESKEW_ANGLE ≈ 0.5`, `_PAGE_MAX_SIDE = 2000`, параметры
    CLAHE.
  - `preprocess_image(image_bytes) -> np.ndarray` (OCR-текст) — **возвращаемое значение и контракт не
    менять**; допускается переиспользовать вынесенные хелперы без изменения результата.
- **Критерии готовности:** `render_page(b, "doc")` отдаёт выровненную контрастную **не**
  бинаризованную страницу; `render_page(b, "scan")` — бинаризованную; на уже ровном изображении угол
  `< _MIN_DESKEW_ANGLE` → поворот не применяется (нет ложного перекоса); `preprocess_image` отдаёт тот
  же тип/форму массива, что и до релиза (ветка OCR не сломана).
- **Связь с ADR:** ADR-019 Решения 2, 3, 5. *(ФТ-1, ФТ-7, ФТ-8; КП-1, КП-7)*

### Задача 2: Рендер страницы в PDF + деградация к оригиналу (`services/ocr.py`)
- **Что:** строить видимую страницу PDF из очищенного изображения, с обязательным откатом к оригиналу
  при сбое; текстовый слой Tesseract кладётся по той же очищенной странице (согласован по построению)
  (ADR-019 Решения 1A, 7).
- **Где:** `services/ocr.py` (`_run_ocr_pdf`, `recognize_pdf`). `recognize_text`/`_run_ocr` —
  **не трогать**.
- **Контракт:**
  - `_run_ocr_pdf(image_bytes: bytes, mode: str = "doc") -> bytes | None`:
    1. `page = render_page(image_bytes, mode)`; `pdf = pytesseract.image_to_pdf_or_hocr(page,
       lang=_LANG, config=_CONFIG, extension="pdf")`; страница вшивается как **JPEG** (ресайз/качество
       из Задачи 1 / констант), `dpi=(200, 200)`; вернуть `bytes(pdf)`.
    2. При **любом** исключении/пустом результате шага 1 → `logger.warning`/`exception` (факт
       деградации логируется) → **fallback**: `image_to_pdf_or_hocr` по **оригиналу**
       (`Image.open(BytesIO(image_bytes))`) — поведение v1.6.0.
    3. Если и fallback падает → вернуть `None`.
  - `recognize_pdf(image_bytes: bytes, mode: str = "doc") -> bytes | None` — под
    `HEAVY_LOCAL_SEMAPHORE` через `asyncio.to_thread`, пробрасывает `mode`. Дефолт `mode="doc"`
    сохраняет совместимость вызовов без аргумента.
  - Один и тот же Tesseract-проход формирует видимый и текстовый слой из **одной** очищенной страницы
    (без ручного пересчёта bbox).
- **Критерии готовности:** для перекошенного фото `recognize_pdf(b, "doc")` отдаёт PDF с выровненной
  контрастной страницей и searchable-слоем (КП-1, КП-2); искусственный сбой `render_page` →
  возвращается PDF по **оригиналу** (как v1.6.0), исключение не пробрасывается, деградация
  залогирована (КП-5, КП-6); `recognize_pdf(b, "scan")` отдаёт PDF с бинаризованной страницей (КП-3);
  обработка идёт под `HEAVY_LOCAL_SEMAPHORE` вне event loop, новой параллельной CV-нагрузки нет
  (КП-8); `recognize_text` не изменён.
- **Связь с ADR:** ADR-019 Решения 1, 4, 7, 8. *(ФТ-1, ФТ-2, ФТ-4, ФТ-5, ФТ-6, ФТ-9; КП-1, КП-2,
  КП-4, КП-5, КП-6, КП-8)*

### Задача 3: Две кнопки режимов PDF (`handlers/actions.py`)
- **Что:** добавить вторую кнопку «📄 Скан (ч/б)» рядом с существующей «📄 PDF»; обе вызывают
  `recognize_pdf` с соответствующим `mode` (ADR-019 Решение 6A).
- **Где:** `handlers/actions.py` (`actions_keyboard`, callbacks/хендлеры PDF).
- **Контракт:**
  - Новый callback `_CB_PDF_BW = "act:pdfbw"` (короткий, в пределах 64 байт). Существующий
    `_CB_PDF = "act:pdf"` сохраняется и означает режим **`"doc"`** (дефолт «Документ»).
  - В `actions_keyboard` при `with_pdf` — ряд из **двух** кнопок: «📄 PDF» (`act:pdf`) и
    «📄 Скан (ч/б)» (`act:pdfbw`).
  - Логику обработки PDF вынести в общий хелпер `async _handle_pdf(callback, mode: str)` (чтение байт
    из кэша по `(chat_id, message_id)`, `recognize_pdf(image_bytes, mode)`, отправка документа /
    служебные сообщения — как в текущем `handle_pdf`). `handle_pdf` (`act:pdf`) → `_handle_pdf(cb,
    "doc")`; `handle_pdf_bw` (`act:pdfbw`) → `_handle_pdf(cb, "scan")`.
  - Имя файла остаётся `document.pdf` (имя/`title` — вне scope, ADR-019 Решение 9). Подписи и
    служебные сообщения — на русском, как сейчас.
- **Критерии готовности:** для изображений в клавиатуре две кнопки PDF; нажатие «📄 PDF» даёт
  PDF-документ (режим «Документ»), «📄 Скан (ч/б)» — бинаризованную страницу (КП-3); кэш-miss и сбой
  `recognize_pdf` → прежние служебные сообщения/alert без падения (поведение v1.6.0); прочие кнопки
  не изменены (КП-9).
- **Связь с ADR:** ADR-019 Решение 6. *(ФТ-3, ФТ-10; КП-3, КП-9, КП-11)*

## Риски и митигации

- **Риск:** CLAHE/deskew на 1 vCPU замедляет отзывчивость. → **Митигация:** вся обработка под
  существующим `HEAVY_LOCAL_SEMAPHORE` вне event loop (Задача 2, КП-8); масштаб ограничен ресайзом
  до `_PAGE_MAX_SIDE` (ОГР-1, НФТ-1).
- **Риск:** обработка «выжигает» бледный/мелкий текст или ложно перекашивает ровный документ. →
  **Митигация:** CLAHE (локальный, идемпотентный на контрастном входе) вместо глобального контраста;
  порог `_MIN_DESKEW_ANGLE` + потолок 15° (Задачи 1, ADR-019 Решения 3, 5; КП-7).
- **Риск:** сбой рендера ломает фичу. → **Митигация:** обязательная деградация к оригиналу + лог
  (Задача 2, КП-5/КП-6; ADR-019 Решение 7).
- **Риск:** текстовый слой не совпадает с видимой страницей при deskew. → **Митигация:** один проход
  Tesseract по очищенной странице — слой и картинка из одного массива (Задача 2, ADR-019 Решение 1A;
  КП-2).
- **Риск:** разделение `_deskew` на две функции вносит регрессию угла в ветке OCR. → **Митигация:**
  `preprocess_image` сохраняет контракт; тест на неизменность типа/формы массива (Задача 1, тесты).
- **Риск:** регрессия размера итогового PDF. → **Митигация:** JPEG quality 75 + ресайз до 2000 px →
  PDF не больше v1.6.0 (Задача 2, КП-4).
