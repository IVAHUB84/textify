# req-textify-017: Дневной лимит распознаваний + мягкий gate на подписку Telegram-канала

## Что нужно
Цель продукта — рост подписчиков Telegram-канала (монетизация через рекламу в канале) и защита
VPS от перегруза. Вводим **дневной лимит распознаваний на пользователя** и **мягкий
subscription-gate**: подписка на канал поднимает лимит.

### 1. Дневной лимит распознаваний
- У каждого пользователя есть лимит распознаваний в сутки (голос/аудио/фото/документ — любое
  распознавание считается за единицу).
- **Базовый лимит** (без подписки): по умолчанию **3/день**.
- **Лимит подписчика** (подписан на канал): по умолчанию **30/день**.
- Оба значения настраиваются через env (`DAILY_LIMIT_FREE`, `DAILY_LIMIT_SUBSCRIBED`).
- Счётчик сбрасывается в полночь (по времени сервера/UTC — уточнить у архитектора).
- Лимит — на распознавание (входящее медиа), не на кнопки-действия. Кнопки (Кратко, Перевести,
  PDF и т. п.) над уже распознанным результатом лимит **не** тратят.

### 2. Мягкий subscription-gate
- При исчерпании **базового** лимита пользователю, который ещё не подписан, показывается
  предложение: «Лимит на сегодня исчерпан. Подпишись на @канал — получишь 30 распознаваний в
  день» + inline-кнопки: «Открыть канал» (ссылка) и «Я подписался / Проверить».
- По кнопке «Проверить» бот вызывает `getChatMember(channel, user)`; если статус
  member/administrator/creator — пользователь помечается подписчиком, лимит поднимается до
  `DAILY_LIMIT_SUBSCRIBED`, текущее медиа можно сразу обработать (или попросить прислать заново —
  решает архитектор).
- Если не подписан — вежливое «пока не вижу подписку, попробуй ещё раз после подписки».
- Gate **мягкий**: пока базовый лимит не исчерпан, бот работает как обычно, ни о чём не просит.
  Первое впечатление новых пользователей не страдает.

### 3. Конфигурируемость и выключенный режим
- Канал задаётся через env `REQUIRED_CHANNEL` (например `@textify_news` или числовой chat_id).
- **Если `REQUIRED_CHANNEL` не задан** — gate выключен полностью: проверки подписки нет, всем
  действует базовый лимит (или, по решению архитектора, лимиты тоже можно отключить отдельным
  флагом для локальной разработки/тестов). Цель — не ломать dev/CI и текущий прод до создания
  канала.
- Бот должен быть администратором канала, иначе `getChatMember` недоступен — это
  деплой-предусловие (документировать), не задача кода.

## Поведение и детали
- Учёт лимита — по `user_id`, переживает рестарт (персистентно). Переиспользовать существующую
  SQLite-инфраструктуру учёта (`stats.db` / v0.6.0) — не заводить новый стор без нужды.
- Статус подписки можно кэшировать на короткое время (минуты), чтобы не дёргать `getChatMember`
  на каждое сообщение; но отписавшийся со временем должен снова упереться в базовый лимит.
- Группы: решить, применять ли лимит в группах (там распознавание идёт по кнопке «Распознать»,
  v1.4.0). Базовый дефолт — лимит считать по инициатору; gate в группах не навязывать (или
  отдельный флаг). Решение за архитектором.
- Сообщения о лимите/подписке — на русском, дружелюбные, без давления.

## Открытые вопросы для архитектора
- **Сброс счётчика:** скользящее окно 24 ч или календарные сутки (полночь UTC/локаль)? Простейшее —
  календарный день UTC.
- **getChatMember на канал:** требует бота-админа канала; что показывать, если канал недоступен /
  бот не админ (не падать — деградировать к базовому лимиту без gate, залогировать).
- **Кэш статуса подписки:** TTL и инвалидация при отписке.
- **Охват групп** (см. выше).
- **Гонки счётчика:** атомарный инкремент в SQLite (как `record_message` в v1.3.0).

## Зачем
- **Рост канала → монетизация рекламой** (цель владельца: 1000+ подписчиков канала как первая веха).
- **Защита 1 vCPU / 2 GB** от хеви-юзеров: базовый лимит сглаживает пики, не давая одному
  пользователю занять весь CPU whisper.
- Лимит выставлен выше типичного использования (3/день базово), поэтому обычный пользователь его
  почти не замечает; упираются хеви-юзеры — именно они конвертируются в подписчиков.
