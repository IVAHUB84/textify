# req-textify-019: «Скан»-качество страницы в PDF из фото

## Что нужно
Кнопка «📄 PDF» уже отдаёт searchable PDF (исходное изображение + невидимый текстовый слой от
Tesseract). Проблема: в PDF вшивается **сырое фото как есть** — перекошенное, с тенями, цветным
фоном стола, перспективой. При этом очищенное изображение у нас уже строится (`preprocess_image`:
EXIF-поворот, ресайз, бинаризация, deskew), но используется **только для извлечения текста OCR**, а
в видимую страницу PDF не попадает.

Нужно: видимую страницу PDF рендерить из **очищенного** изображения, чтобы результат выглядел как из
сканер-приложения, а не как фото на столе. Текстовый слой — по-прежнему от OCR.

Это **первый из двух релизов**. Многостраничный PDF из альбома — следующий слой, отдельное
требование (см. «Вне рамок»).

## Поведение и детали
- При генерации PDF применять к изображению пайплайн очистки перед вшиванием:
  EXIF-поворот → (опц.) ресайз к разумному DPI (~A4 @ 200dpi) → выравнивание перекоса (deskew,
  переиспользовать `_deskew` из `services/preprocess.py`) → подъём контраста / убирание теней.
- **Режимы вывода** (UX уточнит архитектор/BA):
  - «Документ (цвет)» — выровненная, контрастная, но цветная/серая страница. Разумный дефолт.
  - «Скан (ч/б)» — бинаризованная страница (как для OCR). Компактнее, классический скан-вид.
  - Вариант: одна кнопка с дефолтным режимом + вторая для альтернативного; либо выбор после нажатия.
- **Сжатие (D):** вшивать страницу как JPEG с ресайзом → меньше размер итогового PDF, чем сейчас.
- **Деградация — обязательна:** если очистка/обработка падает, вшивать оригинал (текущее поведение).
  Результат не должен быть **хуже** сегодняшнего ни на одном входе.
- Ограничение ресурсов: 1 vCPU / 2 ГБ. Обработка идёт под `HEAVY_LOCAL_SEMAPHORE` (как сейчас
  `recognize_pdf`); не плодить параллельную тяжёлую CV-нагрузку.

## Открытые вопросы для архитектора
- **UX режимов:** одна кнопка с дефолтом, две кнопки, или выбор после нажатия? Что дефолт —
  цветной выровненный или ч/б скан?
- **DPI/размер:** целевой DPI и максимальная сторона при ресайзе; качество JPEG (баланс
  размер/читаемость мелкого шрифта).
- **Подъём контраста / тени:** какой метод (autocontrast, CLAHE, adaptive threshold для ч/б);
  чтобы не «выжигать» бледный текст.
- **Совмещение с текстовым слоем:** при deskew страницы координаты текстового слоя должны
  совпадать с повёрнутым изображением (важно, чтобы выделение текста в PDF попадало в слова).
  Решить: рендерить PDF из уже очищенного/повёрнутого изображения целиком (тогда Tesseract сам
  кладёт слой по нему) — это проще, чем синхронизировать два разных преобразования.
- **Когда не трогать:** если изображение уже ровное/контрастное — не портить лишней обработкой.

## Совместимость / охват
- `services/ocr.py` — `recognize_pdf` / `_run_ocr_pdf` (основное место изменений).
- `services/preprocess.py` — переиспользовать deskew/ресайз; возможно выделить «получить очищенное
  изображение для рендера» отдельно от «получить массив для OCR-текста».
- `handlers/actions.py` — кнопки/режимы PDF (`_CB_PDF`, `actions_keyboard`, `handle_pdf`).
- Тесты: `tests/test_ocr.py`, `tests/test_actions.py` (моки `recognize_pdf`).

## Зачем
- Главная ценность фичи «PDF из фото» — отдать **аккуратный документ**, а не пересланное фото.
  Сейчас визуальный результат не отличается от исходного снимка, что обесценивает кнопку.
- Очистка уже вычисляется ради OCR — отдать её пользователю почти «бесплатно» по архитектуре.

## Дополнительный контекст
- Релиз — ориентировочно **v1.9.0** (MINOR: улучшение существующей функции без слома контракта;
  деградация гарантирует совместимость). Версию подтверждает архитектор.
- Профиль поставки прежний (бот без HTTP-порта, long-polling, ivahub, GHCR). Новых артефактов

  если архитектор не решит иначе).

## Вне рамок
- **Многостраничный PDF из альбома** — следующее отдельное требование (вариант B).
- **Авто-кроп по границам документа / коррекция перспективы** (вариант C) — позже, опционально;
  тяжело для 1 vCPU и рискованно на пёстром фоне.
- Изменение логики извлечения текста OCR — не трогаем, меняем только видимый слой PDF.
- Метаданные/имя файла PDF (title, осмысленное имя) — необязательная косметика, на усмотрение
  архитектора в рамках scope, не основная цель.
