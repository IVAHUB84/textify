# req-textify-020: Анонс новых версий пользователям (changelog-рассылка)

## Что нужно
Когда выходит новая версия бота, пользователи об этом не узнают. Нужно, чтобы бот **сам сообщал
пользователям о выходе новой версии** — присылал короткий «что нового» по релизу. Это рассылка
(broadcast) по всем, кто пользовался ботом, с защитой от случайной/повторной отправки и возможностью
отписаться.

## Поведение и детали

### 1. Источник версии и текста анонса
- Ввести единый источник текущей версии приложения в коде (например, константа `__version__`).
  Сейчас версии в коде нет — она только в git-тегах и release-notes.
- **Курируемый короткий текст анонса на каждую версию** (user-friendly, на русском, без технических
  КП/ADR). Хранится в коде или отдельном данных-файле как реестр `{версия: текст «что нового»}`.
  Технические `release-notes/notes-vX.Y.Z.md` для рассылки **не** использовать (длинные/технические).
- Если для текущей версии текста анонса в реестре нет — рассылка не запускается (тихо, с логом).

### 2. Триггер: авто-детект при старте + подтверждение админа
- При запуске бот сравнивает текущую `__version__` с **последней анонсированной версией**
  (persistent). Если текущая новее и для неё есть текст анонса — есть «ожидающий анонс».
- Бот **не рассылает автоматически**. Вместо этого уведомляет **админа** (`ADMIN_USER_ID`):
  присылает превью текста анонса + inline-кнопки «Разослать» и «Пропустить».
- По кнопке «Разослать» — запускается фоновая рассылка. По «Пропустить» — версия помечается
  анонсированной без рассылки (превью больше не всплывает).
- Если админ не отреагировал — «ожидающий анонс» сохраняется и предлагается снова при следующем
  старте (без дублей в БД).

### 3. Механика рассылки
- Адресаты: все `user_id` из `user_stats` (таблица уже хранит всех пользователей), **минус**
  отписавшиеся, минус заведомо заблокировавшие бота.
- Рассылка — **фоновая задача**, не блокирует обработку входящих сообщений.
- **Троттлинг** под лимиты Telegram (порядка ~25–30 сообщений/сек — точное значение уточнит
  архитектор), с паузами; обработка `RetryAfter`/`FloodWait` с backoff.
- Обработка ошибок доставки: `TelegramForbiddenError` (пользователь заблокировал бота / удалил чат) —
  пропустить, по возможности пометить, не падать. Прочие ошибки — залогировать и продолжить.
- **Идемпотентность:** после успешного запуска рассылки версия помечается анонсированной — рестарт
  контейнера не приводит к повторной рассылке той же версии.
- По завершении — краткий отчёт админу: отправлено N, пропущено/заблокировано M, ошибок K.

### 4. Отписка
- Под каждым анонсом — inline-кнопка «🔕 Не показывать анонсы» (и/или команда). Нажатие помечает
  пользователя отписавшимся (persistent), будущие анонсы ему не приходят.
- Желательна возможность повторной подписки (команда), но это вторично — решает архитектор.
- Текущая обработка медиа и команды отпиской **не** затрагиваются.

## Открытые вопросы для архитектора
- **Где хранить состояние:** последняя анонсированная версия, список отписавшихся, (опц.) пометка
  заблокировавших. Логично — `stats.db` (там уже `user_stats`); не плодить новый стор без нужды.
- **Точный троттлинг** и стратегия backoff на `RetryAfter` (Telegram Bot API лимиты на рассылку).
- **Идемпотентность при гонках:** что если рассылка прервалась на середине (рестарт во время
  broadcast) — версия уже помечена анонсированной → часть не получит. Допустимо ли, или нужен учёт
  прогресса/докрутка? Базовый дефолт — пометка после **старта** рассылки, частичная потеря допустима.
- **Формат версии в реестре:** ключ — точная `vX.Y.Z`? Анонсировать только при росте версии (semver
  compare), чтобы откат не триггерил.
- **Где объявить `__version__`** и стоит ли заодно показывать версию в `/start`/`/stats`.
- **Повторная подписка** — нужна ли команда, как назвать.
- **Патч-релизы:** анонсировать ли мелкие патчи (как v1.9.1) или только MINOR+? Возможен флаг «без
  анонса» для версии (просто отсутствие текста в реестре уже это даёт).

## Совместимость / охват
- Новый сервис рассылки/анонсов (`services/`), реестр текстов, состояние в `stats.db`.
- `bot.py` — хук на старте (детект ожидающего анонса, уведомление админа).
- Новый роутер/хендлеры: callback «Разослать»/«Пропустить» (админ), «Не показывать анонсы»
  (пользователь), возможная команда отписки/подписки.
- `services/stats.py` (или новый модуль) — чтение всех `user_id`, хранение opt-out / last-announced.
- `config.py` — переиспользовать `ADMIN_USER_ID` (уже есть). Возможный флаг включения анонсов.
- Тесты: детект версии, gate-подтверждение админа, троттлинг (мок), пропуск заблокировавших, opt-out,
  идемпотентность.

## Зачем
- **Retention/возврат пользователей:** напоминание о боте + демонстрация развития (новые кнопки,
  режимы PDF, лимиты) возвращает неактивных.
- **Прозрачность:** пользователь видит, что продукт живёт и улучшается.
- Связь с целью канала (v1.7.0): анонс — естественный момент мягко напомнить про канал/приглашения.

## Дополнительный контекст
- Релиз — ориентировочно **v1.10.0** (MINOR: новая функциональность, без слома контракта). Версию
  подтверждает архитектор.
- Профиль поставки прежний (Telegram-бот, long-polling, без HTTP-порта, ivahub, GHCR). Новых
  артефактов поставки не требуется; новых внешних зависимостей, скорее всего, тоже (рассылка — на
  aiogram/штатном Bot API).

## Вне рамок
- Рассылка в группы (только личные чаты пользователей `user_stats`).
- Произвольные маркетинговые рассылки/сегментация аудитории — только changelog по версиям.
- Локализация анонсов под язык пользователя (тексты — RU, как и остальной UX).
- Веб-панель управления рассылками, A/B, аналитика открытий.
- Автогенерация текста анонса LLM — тексты курируются вручную.
