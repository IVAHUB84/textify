# req-textify-021: Транскрипция речи из видео

## Что нужно
Сейчас бот распознаёт голос/аудио и изображения, но **видео игнорирует**. Нужно принимать видео и
извлекать из него **речь → текст** (транскрипция звуковой дорожки). Это новый входной канал поверх
существующего whisper-пайплайна, не OCR кадров.

## Поведение и детали
- Принимать видео во всех типичных формах Telegram:
  - `video` (обычный mp4-ролик),
  - `video_note` (кружочки),
  - документ с mime `video/*` (видео, присланное файлом).
- Из видео извлекать **аудиодорожку** и подавать в существующий пайплайн транскрипции
  (`services/transcribe.py`): Cloudflare Whisper (дефолт) с фолбэком на локальный faster-whisper.
  Извлечение аудио — через уже подключённый **PyAV** (`av`), системный ffmpeg не добавлять (тем же
  способом декодируются голосовые).
- Результат — как у аудио: транскрипт + прогрессивное раскрытие (суть → кратко → полностью) и те же
  кнопки-действия (Кратко, Перевести, Задачи, ⏱ Тайм-коды, Субтитры). Переиспользовать
  `handlers/audio.py`-логику и `services/reply.py`, не дублировать.
- Переиспользовать существующие механизмы: гард размера 20 МБ (`handlers/limits.py`), дневной лимит и
  gate (v1.7.0), реферальные бонусы (v1.8.0), `result_cache`.
- Если в видео **нет аудиодорожки** или речь не распознана — дружелюбное сообщение (как «текст не
  распознан» для картинок), без падения.
- Ограничение ресурсов: 1 vCPU / 2 ГБ. Длинные видео тяжелы — опираемся на те же лимиты (20 МБ —
  фактический потолок загрузки Telegram-ботом; при необходимости архитектор задаёт доп. кап
  длительности/размера).

## Открытые вопросы для архитектора
- **Способ извлечения аудио:** PyAV demux видео → в какой контейнер/кодек упаковать аудио для CF
  Whisper (CF ждёт аудио-payload, base64)? Можно ли для локального faster-whisper подавать байты
  видео напрямую в `av.open`, а для CF — извлекать/перекодировать аудио? Минимизировать число
  перекодировок (1 vCPU).
- **`video_note`:** у кружочков нет mime, всегда mp4 — отдельная ветка или общий путь.
- **Охват групп:** расширять ли распознавание видео в группах (v1.4.0 — кнопка «Распознать» по
  упоминанию); сейчас `_has_supported_media` покрывает аудио. Базовый дефолт — включить видео и в
  группах по тому же механизму, либо первый релиз только в личке (решает архитектор).
- **Кап размера/длительности:** оставить только гард 20 МБ или добавить лимит длительности.
- **Чанкинг длинных аудио** из видео — вне рамок или переиспользуется существующая логика
  транскрипции как есть.

## Совместимость / охват
- Новый хендлер видео (`handlers/video.py` или расширение `handlers/audio.py`): приём `video`,
  `video_note`, документ `video/*`.
- Извлечение аудио — новый хелпер (например в `services/transcribe.py` или отдельный
  `services/media.py`) на PyAV.
- Регистрация роутера в `bot.py`; `handlers/__init__.py`.
- Группы: `handlers/group.py` (`_has_supported_media`, диспетчеризация) — при включении видео в группах.
- Тесты: приём `video`/`video_note`/документ `video/*`; извлечение аудио (мок PyAV); транскрипция
  (мок); видео без аудио → сообщение; гард 20 МБ; лимит/gate не сломаны.

## Зачем
- Видео с речью (лекции, кружочки, пересланные ролики, голосовые-видео) — частый формат; сейчас бот
  на них молчит. Расширяет охват без новой инфраструктуры — переиспользует whisper-пайплайн.

## Дополнительный контекст
- Релиз — ориентировочно **v1.11.0** (MINOR: новый входной канал, без слома контракта). Версию
  подтверждает архитектор.
- Профиль поставки прежний (Telegram-бот, long-polling, без HTTP-порта, ivahub, GHCR). Новых внешних
  зависимостей быть не должно (PyAV уже есть); системный ffmpeg НЕ добавлять. Новых артефактов
  поставки не требуется.

## Вне рамок
- **OCR текста в кадре** (титры/надписи/слайды) — отдельная задача, не входит.
- Перевод/озвучка видео, субтитры поверх видеофайла, нарезка по сценам.
- Стриминг/онлайн-видео по ссылке (YouTube и т. п.) — только присланные в Telegram файлы.
- Поддержка форматов, которые PyAV не декодирует без доп. системных кодеков.
